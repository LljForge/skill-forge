# 后端接口边界维度表

本表是 `test-case-design` 的**召回轴**——**不是分类法，是点名清单**：拿着它逐条问「这条维度在本次任务里有没有对应用例」。

用例写完往哪归档，是四类分区（TC-F / TC-E / TC-ERR / TC-ST）的事。**两轴正交，别试图对齐**：一条维度会散落到多个类（D1 幂等的「重复提交」落 TC-E，「并发超限被拒」落 TC-ERR）。

> **成色（引用前必读）**：D1–D6 借自逆向场景的经验表，**正向未独立验证**；真实任务对照表明这 6 条**内容多属模型自带**——本表的价值在「**逐条点名强制交代 + 沉默落待澄清**」这个机制，不在维度本身的信息量。所以这里只留**点名锚**（名 + 一句），不铺教科书。

---

## §1 输入契约边界法

管「单个字段怎么取值」。**是方法、不是清单**：对每个输入字段做**等价类划分 + 边界值分析**——有效类取代表值；无效类**每种原因单列**（超范围 / 类型错 / 格式错 / 必填缺失是四类，不是一类）；每个有效区间**边界两侧各取一点**。适用面：数值范围、字符串长度、集合大小、日期区间、分页参数、金额精度。

---

## §2 领域维度清单（点名锚 · 逐条问「本次有没有对应用例」）

### D1 · 幂等 / 去重 / 并发
- **测什么**：重复提交同一请求的后果（幂等成功 / 去重不重复 / 报冲突——必须明确是哪一种）；并发的上限与拒绝策略。
- **两半边判据不同**：「重复 / 幂等」半边**默认点名**，事实集没提也按铁律 3 落「待澄清」；「并发上限 / 拒绝策略」半边算限流类、**受 §3 管**（有依据才生成，没依据不生成也不问）。
- **常落分区**：TC-E（重复提交）、TC-ERR（并发超限被拒，仅当有依据）

### D2 · 鉴权前置
- **测什么**：无凭据 / 凭据过期 / 凭据有效但无权限——**这三种是不同的**，别合并成「鉴权失败」；哪些路径豁免。是**黑盒可复现的端点前置契约**，不是安全审计。
- **常落分区**：TC-ERR

### D3 · 事务边界 / 回滚补偿
- **测什么**：一组写操作**部分失败**时，是整批回滚还是留脏数据；失败后有无补偿 / 重试自愈。
- **常落分区**：TC-ERR（部分失败）、TC-ST（失败后状态）

### D4 · 缓存读一致性
- **测什么**：写后立即读，能读到新值，还是存在「读旧值」窗口。
- **前提（判归宿）**：有架构线索表明无缓存层（如设计写明直连库读写）→ **N/A（形式 1，回指该线索）**；事实集沉默且无线索 → **待澄清**。
- **常落分区**：TC-E（写后立即读）

### D5 · 错误响应契约 + 成功信封
- **测什么**：出错返回什么状态码 / 业务错误码 / 错误体结构；成功报文的真实信封（字段名、分页缺省值）。调用方据此分支解析——**这是契约的另一半**。
- **常落分区**：TC-ERR（错误码）、TC-F（成功信封）、TC-E（分页边界）

### D6 · 异步最终一致
- **测什么**：**「接口返回成功 ≠ 下游已就绪」**。操作成功后向下游推送 / 发消息的，测下游状态何时可观察到变化。
- **常落分区**：TC-F（下游最终变化）、TC-ST（中间态）

## §3 不默认登记（元规则）

> **限流 / 防刷、多租户数据隔离、版本 / 向后兼容 —— 不默认登记。**

**很多项目根本没有这些。** 默认写进用例会臆测出不存在的契约（比如把单租户系统臆测成有租户过滤）。

**本 skill 比 module-spec-baseline 更危险**：它至少能去代码里探测有没有，本 skill 正向——**连代码都没有，更容易凭空造**。

**硬规则**：这类维度**只有在待测事实集里有明确依据时才生成用例，否则连问都不问**。

判别：事实集里有「限流阈值 100 QPS」→ 生成；事实集只字未提限流 → **不生成、也不问**（问了就是在诱导用户补一个本来不存在的需求）。
