# 测试用例设计

产出**「测什么」的规范**，与「怎么测」解耦。下游执行者（另一个 AI agent / skill）拿到后自行决定框架、请求构造、数据准备、断言方式。

| 项 | 取向 |
|----|------|
| 产物 | 测试用例规范 markdown |
| 抽象层次 | **执行无关**：可写被测契约，禁写测试实现（判据见「铁律 1」） |
| 领域维度表 | 当前只有**后端接口**一张；前端 E2E 维度表尚未建立（见下） |
| 边界召回 | 靠 [`references/backend-api.md`](references/backend-api.md) **点名扫**，不靠临场想 |
| 依赖 | **无** |

**不做**：写测试代码、选框架、mock 策略、覆盖率、bug-hunting。

## 调用方式

```
/test-case-design <本次要测的开发任务或业务领域>
```

**遇前端 E2E 任务**：纪律层（三铁律 + 出口三门）与输入契约边界法照常适用，但**没有 E2E 维度表**——如实告知「领域维度未覆盖，边界召回只能靠临场想」，不要拿后端那 6 条硬套。

**遇后端 + 前端混合任务**：不是二选一——后端半按维度表点名扫，前端半按上一条走纪律层；维度扫描表里把纯前端用例标注「无领域维度」，别拿后端 6 条硬套到前端半。

## 三条铁律

### 铁律 1 · 用例没有「步骤」

**用例只有 GIVEN / WHEN / THEN，没有「测试步骤」字段。**

- **GIVEN** = 状态事实（不是构造动作）
- **WHEN** = 意图（不是操作序列）
- **THEN** = 可观察结果（不是断言写法）

**为什么是铁律**：一个叫「步骤」的槽位天然只能填操作序列，填进去的一定是「怎么做」。这是结构性防御 —— 不靠自律，靠没有那个槽位。

**界线**（后端接口）：

| 可写（被测契约本身） | 禁写（测试实现） |
|---|---|
| 接口语义、HTTP 状态码、业务错误码、响应字段、状态名 | 框架名、请求怎么构造、数据怎么造、mock 谁、怎么断言 |

**裁决 —— HTTP 方法 + 路径可写**：`WHEN 调 POST /orders/{id}/cancel` 不算泄漏。过执行者替换测试：换成 Postman / RestAssured / 人手测，方法 + 路径依然成立——它是接口契约本身，不是执行者的选择。

**裁决 —— 非接口层的被测契约也可写**：design 点名要单测的纯函数（如某转换函数的列映射正确性）不算泄漏——它是「映射对不对」（测什么），不是「怎么测」。上面「可写」列举的是接口面的典型项、**不是全集**；**别因为它不在 HTTP 端点层、Postman 测不了就当泄漏删掉**。观察层（这个结果在哪儿看得到）≠ 执行手段（门 B 要拦的「怎么测」）。

**判据 —— 执行者替换测试**：
> 「把执行者换成 人手测 / RestAssured / Postman / Playwright / 手写单测，这条用例还成立吗？」
> 成立 → 执行无关；不成立 → 泄漏。
> **判据判的是「测什么 vs 怎么测」，不是「在哪一层测」**——换执行者后描述的仍是被测契约（测什么）就成立，别拿某个具体执行者跑不跑得动来判（否则纯函数缝隙会被误当泄漏，见上方「非接口层的被测契约也可写」裁决）。

最易漏的是 **GIVEN 的状态事实 vs 构造方式**：

- ✅ `GIVEN 一个处于开启状态的订单`（状态事实，执行者自行决定怎么造）
- ❌ `GIVEN 预置一条状态为开启的订单数据`（构造方式——已经替执行者决定了「预置数据」这个手段，只是换了个不触发禁用词的说法）

两句描述同一个前置，后者是**软泄漏**——词表拦的是**词**，不是**形状**：换一种措辞就绕开硬拦，只能靠逐条问执行者替换测试。

**WHEN 的意图 vs 操作序列**：

- ✅ `WHEN 对它连续发起两次相同的取消请求`（契约行为，意图明确）
- ❌ `WHEN 先登录拿 token，再带 token 调取消接口`（操作序列——把「怎么拿到调用资格」这个执行细节也写了进去）

**THEN 的可观察结果 vs 断言写法**：

- ✅ `THEN 订单最终处于已取消状态，且只产生一条取消记录`（可观察结果，不规定怎么验证）
- ❌ `THEN 断言响应体中 data.status 字段值为 "CANCELLED"`（断言写法——已经替执行者决定了从哪个字段、用什么方式验证）

### 铁律 2 · 无源即非法

**每条用例的「来源」字段必须回指待测事实集里的某一条。回指不上的用例不许存在。**

**为什么是铁律**：没有锚就会编造用例。**待测事实集就是锚。**

想测但事实集里没依据 → **不是偷偷补一条用例**，是落进「待澄清」回头问用户，补进事实集后再生成。

**源分两型——各自只能 ground 特定的东西**：

| 源 | 只能 ground | 不能 ground |
|---|---|---|
| **文档 / 对话**（需求、设计、验收、业务规则） | **范围**（测什么）与**预期**（应该怎样） | — |
| **代码**（可选，有则用） | **结构与坐标**：字段名、类型、端点路径、DDL 列 | **预期**——代码只说明「现在怎样」，不说明「应该怎样」 |

**判据 —— 改了它算不算功能变了**：
> 把这个值换成另一个，**算不算功能变了**？
> **不算** → **坐标**。代码可 ground，**读到即填死**，不必标记。（字段从 `records` 改叫 `list`、兜底时间换个格式——只是叫法，没人会当成 bug）
> **算** → **答案**。代码只能提供**现状**：照填，但**必须标 `⚠️ 歧义点`**（Step 5）。（零行导入从拒绝改成成功、越界从钳制改成报错——有人会说「原来那个是 bug」）

**未标注地**拿现状当预期＝把实现的偶然行为洗成产品契约，测试永远绿、永远发现不了问题。标了，它就还是个待审的现状，不是契约——**标记是这条铁律的全部分量所在**。

**代码是可选输入**：没有代码可读时（TDD / 需求评审 / 实现未开工），结构空位照常进「待澄清」，流程不变。

### 铁律 3 · 不猜

**事实集没说的，不编造一个契约出来。** 按 Step 5 四情形表落定：能从代码查到现状就照填并标 `⚠️ 歧义点`，查不到才落「待澄清」。

这是铁律 2 的泄压阀 —— 没有它，铁律 2 只会逼着自己偷偷编一条来源出来。**「不猜」管的是「别把猜的东西说成契约」，不是「不许写」。**

**但「不猜」≠「弃案」—— 分清场景与预期。** 一条用例有两处会沉默：**场景**（GIVEN / WHEN，测什么）和**预期**（THEN，结果如何）。

- **场景无据** → 不生成（尤其 §3 治下：限流 / 多租户 / 版本兼容、并发上限）。
- **场景有据、只预期沉默**（字段 / 操作 / 边界在事实集里，只是结果事实集没定义）→ **写案，别弃案**。预期怎么落，按 Step 5 四情形表：
  - 代码可读 → **按代码填 + 标 `⚠️ 歧义点`**（注明取自实现、非契约，且**不会发现此处缺陷**）
  - 代码不可读 / 查不到 → `⟨预期待定⟩` + 进「待澄清」

  两条都**不替源选定一个具体值当作契约**——填了也明写它只是现状。

**不列候选**：陈述「源未定义什么」+「现状是什么」，**不罗列可能的答案**。列候选是在猜答案空间——猜漏了反而把人框进你给的选项。

广度（场景被枚举）与纪律（现状不冒充契约、缺口不被藏起来）两头都要。裸写要么把预期编死（造错测试）、要么丢整条（丢覆盖），这条两头都躲开。

**特别地**：限流 / 多租户隔离 / 版本兼容这类维度，事实集无依据时**不生成、也不问**（见 [`references/backend-api.md`](references/backend-api.md) §3）—— 问了就是在诱导用户补一个本来不存在的需求。**注意**：这是「场景无据」的典型 —— 既不走「歧义点」、也不走「预期待定」，是**根本不生成用例**。

## 流程（7 步）

### Step 1 · 输入归一化 → 待测事实集

**待测事实集**是流程唯一的输入面，四类：**验收标准 / 业务规则 / 异常场景 / 接口契约**。任何来源都先归一成这四类——**按语义取，不认任何来源的固定章节号 / 路径 / 分节**。

来源不限，常见几种、一律平权：

| 来源 | 四类事实通常落在哪 |
|---|---|
| 对话 / PRD / 需求文档 | 正文里散列，缺的当场问 |
| 接口文档 / 设计稿 | 接口契约为主，其余当场补 |
| recon-driven-dev 这类五阶段产物 | 分散在它的 `requirements` / `design` markdown 里——**按标题语义定位，别绑它的章节号**（号随场景数漂移） |

**读产物即可，不调用任何上游 skill、不要求它跑过。** recon 产物只是「四类事实恰好落在它文件里」的一种常见来源，不是特权入口。

**接口契约只取「被测契约本身」**（接口语义 / 状态码 / 错误码 / 字段 / 状态名）；来源里若夹带测试实现（框架 / 断言写法 / mock / 数据构造），**不带入事实集**——门 B 出口再兜一层。

四类事实归一化后**落盘进产物「概览」**，逐条编号列出（如 `业务规则 §3.2`）。这不是可选存档：铁律 2 / 门 A 要求每条用例的 `来源` 回指事实集某条，没有落盘编号，回指的就是当场编的号、用完即弃，产物读者解析不了、复盘者核不了，门 A 会退化为 agent 自我声明。

**后续步骤只认这份事实集，不认原始输入。**

### Step 2 · 抽场景

按四类分区抽：

| 分区 | ID 前缀 | 抽什么 |
|---|---|---|
| 功能测试 | `TC-F` | 事实集里的正常流程 |
| 边界测试 | `TC-E` | 边界值、空值、上限 |
| 错误处理 | `TC-ERR` | 异常场景、失败模式 |
| 状态迁移 | `TC-ST` | 有状态时的合法/非法迁移 |

### Step 3 · 逐条对维度表点名扫

**读 [`references/backend-api.md`](references/backend-api.md)。**

- **§1 输入契约边界法**：对每个输入字段推导等价类 + 边界值（每个有效区间边界两侧各取一点；无效类每种原因单列）。
- **§2 领域维度清单**：**逐条点名** D1–D6，问「这条维度在本次任务里有没有对应用例」。

**这是点名，不是「想想还有什么边界」。** 拿着清单一条条过——这是维度表存在的唯一理由。

### Step 4 · 产用例

按下方「用例格式」写。每条必须有 `维度` 和 `来源` 两个字段。

### Step 5 · 空位查证

Step 4 产出的用例里会留下两种空位，**先标出来，再去查**：

- **结构空位**：字段叫什么、状态码是多少、表有哪些列。
- **决策空位**：源没定义「应该怎样」（成功还是拒绝、钳制还是报错）。

**带着这张空位清单去代码里查——只查清单上的，不做探索性阅读。**

**没有代码可读时**（TDD / 需求评审 / 实现未开工）本步不跳过：所有空位一律走下表最后一行（`⟨预期待定⟩` + 待澄清）。

**这是「拉」不是「推」**：先有空位、才去查。顺序反了就会**照着实现写测试**——代码怎么写、断言怎么写，连 bug 一起固化，测试全绿却一个问题都发现不了。**Step 1–4 必须在不看代码的前提下完成**，这个顺序是安全机制，不是流程偏好。

查到之后，**按四种情形落定，不逐条问用户**：

**先定「文档有没有定义」——这一步是回查，不是回忆。**

> **判据**：「文档有定义」＝ **事实集里存在一条正面陈述了这个点应然结果的条目**。
> 判之前**回到产物「概览」里那份落盘的事实集逐条扫**，别凭对原始文档的印象答。

这一步判错，后面所有判据都用不上：源里写了却被当成沉默，冲突就悄悄降格成歧义点，Step 6 形同虚设。而**事实集就是为此存在的**——Step 1 已经把它落盘编号，且明令「后续步骤只认这份事实集」。手里握着表还凭回忆答题，是这条流程最容易犯的错。

**「覆盖」的松紧**（防两头跑偏）：

| | 例 |
|---|---|
| ✅ 算定义 | 事实集写「上传非 Excel → 提示**格式不支持**」，而实现统一返回「导入失败：\<异常\>」→ **源定义了应然结果，这是冲突** |
| ❌ 不算定义 | 事实集只提到「支持 .xls/.xlsx」，没说别的格式该怎么响应 → 源确实沉默，按代码填 + 歧义点 |

**只提及概念 ≠ 定义了结果。** 把「提到过」也算成定义，冲突数会爆，Step 6 就退回成批量提问——那条路走过，代价是每条都要人裁决。

| 情形 | 处置 |
|---|---|
| **文档有定义**，代码一致或未查 | 按文档填。**文档优先** |
| **文档有定义，代码却不一样** | → **冲突**（见下），转 Step 6。别当成歧义点 |
| **文档沉默、代码有** | **按代码填**，并标 `⚠️ 歧义点`（注明取自当前实现、非契约）。`来源` **追加**一条 `代码 <path>:<line>`（追加，不是替换——门 A 仍要求回指得上事实集） |
| **两边都没有** | `⟨预期待定⟩` + 进「待澄清」（罕见降级：如实现在仓外依赖包内、查不到） |

**为什么决策空位也直接填**：卡住等裁决会让整份规范停摆，而缺口本身并不会因为多等几天变清楚。填上现状 + 标歧义点，测试能先跑起来，缺口也没被藏起来——**审查权交给人，但不打断人**。

**`⚠️ 歧义点` 必须明写它的代价**：该断言取自当前实现、不是产品契约，**它不会发现此处的缺陷**（实现改了它才会红，实现错了它不会红）。不写这句，歧义点就退化成一个没人当回事的标签。

**冲突是另一回事**：源说 A、实现做 B，源**并没有**沉默——这不是空位，是两个权威打架。

- **单列一条，排在「待澄清」最前**，两侧都注出处（源哪一条、代码哪一行）。
- 相关用例的 `THEN` 保持待定，**不许默认哪一侧是对的**：实现不天然是契约，文档也不天然是现行需求。

### Step 6 · 冲突上报（有冲突才走；无冲突跳过）

**只报冲突，别的一律不问。** 文档与代码不一致是唯一值得打断用户的事——它意味着要么实现跑偏了，要么需求改过没同步，两种都是真问题。

- 一次性列出全部冲突，两侧各注出处。
- **不列候选、不替他选**：陈述「源说什么」「实现做什么」，把裁决权交出去。
- 用户裁决了 → 写成正常 `THEN`，`来源` 记 `用户裁决`，该条移出待澄清。
- 用户暂不裁定 / 无法交互 → 保持 `⟨预期待定⟩` 与待澄清条目原样。

**源沉默的缺口不进本步**——它们已在 Step 5 填好并标了歧义点，由用户在审查产物时自行过目；要改由用户主动发起。

### Step 7 · 出口自检（三门，折进一趟）

见下方「出口三门」。**一趟过完三门**，不分三趟。

## 用例格式

```markdown
#### TC-E-003 · 重复提交取消请求
- **维度**：D1 幂等/去重
- **来源**：业务规则 §3.2 / 验收标准 #4
- **优先级**：高
- **GIVEN** 一个处于开启状态的订单
- **WHEN** 对它连续发起两次相同的取消请求
- **THEN** 订单最终处于已取消状态，且只产生一条取消记录
- **THEN** 第二次请求以幂等方式返回成功，不报冲突错误
```

**歧义点用例**（场景有据、源未定义预期、代码给出了现状 —— 常态）：

```markdown
#### TC-QRY-010 · 多筛选条件组合
- **维度**：§1 输入边界
- **来源**：业务规则 §1.2（可筛字段：批次 / 日期 / …）／代码 `ProjectQuery.java:88`
- **优先级**：中
- **GIVEN** 列表中存在可被多个筛选字段命中的数据
- **WHEN** 同时指定两个及以上筛选条件
- **THEN** 仅返回同时满足全部条件的记录（AND 取交）
  ⚠️ **歧义点 歧-3**：源未定义多条件组合语义，此预期取自当前实现
  （`ProjectQuery.java:88`），**非产品契约**——它不会发现此处的缺陷，
  实现改了才会红。须人工审查确认。
```

- `维度`：D1–D6，或 `§1 输入边界`（纯字段边界用例）
- `来源`：回指事实集条目，**必填**（铁律 2）。代码出处 `代码 <path>:<line>` 是**追加项**、不能单独充当来源；经用户裁决的记 `用户裁决`
- `优先级`：高 / 中 / 低。高 = 核心流程、数据完整性、鉴权路径
- `THEN` 可多条
- `⚠️ 歧义点 歧-N`：源未定义、按代码现状填的预期**必须带**，且必须写全三件事——**源未定义什么**、**现状取自哪一行**、**它不会发现此处缺陷**。只写「歧义点」三个字等于没写，人不会为一个空标签停下来
- `⚠️ 可观察性存疑`：`THEN` 在给定 `GIVEN` 下没有确定性观察点时**必须带**（门 B 可观察性核判出）。写清**为什么观察不到** + **改写成的可观察等价物**（改不出则写「转待澄清」并回指 `澄-N`）。别把不可观察的断言当正常 `THEN` 混进去——下游只能拿它写永远不红的假绿测试
- **编号分两套，别混**：歧义点用 `歧-1 / 歧-2`（对应「⚠️ 歧义点」表），待澄清用 `澄-1 / 澄-2`（对应「待澄清」区）。都写成 `#1` 会串
- `THEN` **可待定**，两种写法：

  **(a) 源与代码都查不到**（罕见降级）
  ```
  THEN ⟨预期待定 · 源未定义 <X> · 现状未查得（<原因>）
        · 不得据此写死断言；见待澄清 澄-N⟩
  ```

  **(b) 源与实现相抵触、尚未裁定**（冲突。源**定义了**、现状也**查到了**，(a) 那两个字段都填不上）
  ```
  THEN ⟨预期未定 · 冲突未裁定
        · 源：<源说什么>（<事实集编号>）
        · 实现：<实现做什么>（<path>:<line>）
        · 不得选定任一侧写死断言；见待澄清 澄-N⟩
  ```

  两种都必须带「不得…写死断言」——这是写给下游实施者的**禁令**。**只描述状态而不下达指令时，下游会自行把待定填成具体值**

## 出口三门

**一趟过完，不分三趟。**

### 门 A · 无源门

逐条查：每条用例的 `来源` 能回指事实集某条吗？

- 回指不上 → **删除**，或落「待澄清」回头补事实集后重生成。**不许留。**
- **门 A 查的是「场景」有没有源**——`来源` 里回指事实集的那部分必须成立。`代码 <path>:<line>` 是**补充**（结构/现状的出处），不能单独充当来源：只有代码出处、没有事实集回指 = 场景无据 → 删。
- **歧义点用例、`⟨预期待定⟩` 用例**：场景回指得上就**放行**（预期未定 ≠ 场景无源，别误删）。

**反向核 —— 每条歧义点都要过一遍**。门 A 正向核「场景有没有源」，还要反向核**「声称的沉默是真沉默吗」**：

> 逐条读「⚠️ 歧义点」表的**「源未定义什么」**一栏，拿那句话回事实集**搜一遍**。
> 搜到正面陈述了该点应然结果的条目 → **这不是歧义点，是漏报的冲突**：升格、转 Step 6、从歧义点表移出。

**为什么必须在出口再查一次**：Step 5 判「文档沉默」时，agent 正忙着处理一堆空位，漏看事实集某条是常态；而判漏的代价是**一条真冲突被永久静音**——它会伪装成一条标注良好的歧义点，读产物的人不会怀疑。正向核只能保证场景有源，拦不住这种降格。**这是唯一能把降格捞回来的地方。**

### 门 B · 执行泄漏 + 可观察性门

**作用域 —— 只查契约文本，不查 provenance。** 门 B 查的是 `GIVEN` / `WHEN` / `THEN` 这三行**契约本身**。`⚠️ 歧义点` 与冲突注记里的**出处与现状描述**（`<path>:<line>`、「现状：会插入一条新记录」）**不在检查范围内**——那是「这个预期从哪来」的溯源，不是「怎么测」的指令。

**误伤过一次，别再来**：一条合法的现状描述（如「现状：重复导入会插入一条新记录（`Service.java:162`）」）会**命中下面的硬拦正则**；一个文件路径看着也像实现细节。照删就会成批干掉歧义点用例——而歧义点是本 skill 最常见的产物形态。**跑 rg 前先把注记行排除掉。**

**硬拦（机械可查）**——用例的 `GIVEN` / `WHEN` / `THEN` 契约文本里出现即违规：

```bash
rg -ni "RestAssured|Postman|JUnit|Mockito|Playwright|assertEquals|JSON ?path|点击|输入框|滚动|跳转|mock 掉|插入一条.*记录|执行 SQL" <产物文件>
```

**加 `-i`**：大小写不敏感——不加会漏 `JSONPath`（大写 P）、`restassured`、`postman` 这类变体。

**软拦（逐条判断）**——对每条用例问**执行者替换测试**：「换成人手测 / RestAssured / Postman，还成立吗？」

只查 `GIVEN` / `WHEN` / `THEN`：是**状态事实 / 意图 / 可观察结果**，还是**构造方式 / 操作序列 / 断言写法**？（铁律 1 的三对 ✅/❌ 范例就是这道题）**注记行同样跳过**——判的是契约成不成立，不是溯源写没写代码路径。

**别拿观察层当泄漏删**：一条指向纯函数 / 内部单元的**被测契约**（design 点名的映射 / 转换正确性），换成手写单测就成立、是「测什么」——它不在 HTTP 端点层、Postman 测不了，**不构成泄漏**（铁律 1「非接口层的被测契约也可写」裁决）。误删过一次——把后端范围内的映射单测缝隙连场景一起弃了——别再来。

**软泄漏不含禁用词，硬拦一个字都拦不住** —— 别只跑 rg 就收工。

**可观察性核 —— 每条 `THEN` 都要过一遍**。门 B 上两级查「有没有把『怎么测』写进契约」（泄漏），这一级查一件同源的事：**「这条 `THEN` 描述的结果，在它的 `GIVEN` 下真观察得到吗」**。写了个观察不到的 `THEN`，下游要么写不出测试、要么只能写一条永远不红的脆弱断言——和泄漏一样，是这行契约本身不成立。

**机械锚（先扫）**——`THEN` 命中即复核：

- 在**无序集合**上断言**顺序 / 位置 / 先后**：数据源是数据库表（无显式 `ORDER BY`）、Set、Map 迭代序，却断言「保持原顺序」「第 N 条是」「排在前 / 后」。
- 断言依赖**时间戳 / 并发**的默认行为（如同批 `import_time` 相同 → 默认排序退化为不确定）。

**软判（逐条问）**——每条 `THEN` 问：**「存在一个确定性的观察点，能把这个结果看出来吗？」** 三类看不出：① 无序集合上的顺序断言；② 时间 / 并发的默认行为；③ 没有任何接口 / 查询能暴露的内部中间态。

命中 → 标 `⚠️ 可观察性存疑`，两条出路：

- **能改就改成可观察的等价物**（`其余记录保持原顺序` → `按主键 id 升序观察时，其余记录相对顺序不变`）：`THEN` 注明原断言不可观察、已改写，回指 `澄-N`（那个不可观察的性质业务到底要不要，得问）。
- **改写不出 → 落「待澄清」**、`THEN` 保持待定。它和「源与代码都查不到」同类，是产物里给人过目的缺口，**不主动打断用户**（不进 Step 6）。

**为什么必须有、又为什么折进门 B**：三门此前只查有没有源（门 A）、有没有泄漏（门 B 上两级）、维度覆没覆盖（门 C），**没有一门问过「这条 `THEN` 观察得到吗」**——不可观察的断言会一路绿灯进产物、伪装成正常预期，直到下游写测试时才崩，或被写成永远不红的假绿测试。折进门 B，是因为它和泄漏判的是同一样东西——`THEN` 这行契约本身成不成立：泄漏是「写多了（把怎么测塞进来）」，不可观察是「写了个测不到的」，同一行、同一趟查。

### 门 C · 维度覆盖门

`references/backend-api.md` §2 的 D1–D6 **逐条**在「维度扫描表」里有交代：

- **已覆盖**：列出 TC-ID
- **N/A**：**+ 具体理由**（仅限下方两种法定形式之一）
- **⚠️ 歧义点覆盖**：状态写 `◐ 已覆盖·预期待审 · <TC-ID>（歧义点 歧-N）`。**限「场景有据、只预期沉默」**——该维度在事实集里有场景依据，只是结果没定义、由代码补的现状。它已有可跑用例，别塌成「待澄清」
- **待澄清**：**该维度整个在事实集里没依据**（连场景都无据，铁律 3 → 不生成用例），或有场景但预期在文档与代码里都查不到。**不是 N/A**（见下）——「待澄清」区要有对应条目

> **别把这两栏搞反**：维度整个沉默 → **不生成用例**、走待澄清；有场景只缺预期 → 生成用例、走歧义点。拿代码现状去补一个**事实集根本没有场景依据**的维度，就是在凭空造场景（铁律 2 正要防这个）。
- **D1 两半边判据不同**（幂等半默认点名 / 并发半受 §3）：一行内两半边状态不一致时，**行状态取更严的一半**（幂等半待澄清 > 并发半 N/A → 整行记 `⚠ 待澄清`），理由列分注两半边归宿

**不许空着。理由不许写「无」/「不涉及」** —— 要写清为什么不涉及（例：`N/A · 本次接口只读，无写操作，无事务边界`）。

**N/A 理由必须可核，且判据唯一（走 N/A 还是走待澄清，不留自由裁量）**：

- **形式 1（所有维度都能用）**：回指事实集里的具体条目。例：`N/A · 业务规则 §2.1 明确单机部署，无缓存层`。
- **形式 2（仅限 §3 治下维度——限流/防刷、多租户隔离、版本兼容，以及 D1「并发上限」半边）**：明写「事实集未提及」。例：`N/A · 事实集未提及并发上限，按 §3 不生成`。这是铁律 3 特别条款给这几类维度的专属豁免，**不能挪用到其它维度**。
- **非 §3 维度，事实集沉默 → 不许套用形式 2**：这不是 N/A。按上面两栏分流——**维度整个无场景依据** → `⚠ 待澄清 · <说明>` + 待澄清区补条目（不生成用例）；**有场景、只缺预期且代码查得到现状** → `◐ 已覆盖·预期待审 · <TC-ID>（歧义点 歧-N）`。**两条都不是 N/A**。

**理由不许自由发挥**——门 A 逼用例回指事实集，N/A 理由不能例外，否则编起来零成本、无人对账。「事实集未提及」不是万能挡箭牌：只在 §3 治下才是合法 N/A 理由，用到别的维度上就是把该标出来的缺口偷偷冲成免检——直接侵蚀「歧义点」与「待澄清」这两个自检信号（见文末）。

**「⚠️ 歧义点」是给人过目的清单，「待澄清」是真的卡住了的。** 两者别混：歧义点已经有可跑的预期、只是没人签字；待澄清连预期都写不出来。

**自检信号**：

- **歧义点为空但用例齐全** → 多半是把现状当契约直接写死了、没标。正向任务几乎不可能不撞上源沉默，回头重查。
- **待澄清里出现「源沉默 + 代码有现状」的条目** → 归错档了，那是歧义点、不是待澄清。
- **冲突条目**：必须两侧都注出处；只写一侧的不算冲突，是没查清。
- **`THEN` 里出现「顺序 / 先后 / 保持原样」却没说按什么字段排** → 多半在无序集合上断言了顺序，过门 B 可观察性核：改成可观察等价物，或落待澄清。
