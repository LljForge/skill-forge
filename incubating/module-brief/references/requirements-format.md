# requirements.md 格式（业务视角）

主上下文（Step 2）写 `requirements.md` 的格式权威。这份是**给业务方/测试/业务新人看**的——读者画像决定一切。

## 读者与失败画像（开篇先校准）

读者是 **PM / 测试 / 刚接手业务的开发**。他们要的是"这个模块在业务上干什么、有哪些规矩、典型流程怎么走"。**一旦混进代码黑话（类名、`status=0`、SQL、分层术语），他们就读不下去——那是失败**。所以本文件**禁一切代码层标识**，全用业务自然语言。

## 骨架（4 节）

```markdown
# {{MODULE_CN_NAME}} — 需求说明

> 本文档以业务视角描述 {{MODULE_CN_NAME}} 的概念、规则与流程。由 module-brief 自动生成、供人快速建立认知。

## 1. 模块为谁解决什么
（目标用户、解决的业务问题、价值。1-2 段大白话）

## 2. 核心业务能力与规则
（模块提供哪些业务能力；关键业务规则与约束，自然语言陈述。规则可编号 BR-1/BR-2 便于引用，但不强制、不凑数——只收真·业务规则，模块替换测试不过的删）

## 3. 术语表
（业务术语 → 含义；「别名」只填中文/业务同义词，禁代码字段名/方法名）

| 术语 | 含义 | 别名 |
|------|------|------|

## 4. 主要业务场景与流程
（按业务领域聚合，每个场景：触发 + 主流程（编号自然语言步骤）+ 异常分支。**不按 Controller 方法逐条列**，按业务场景组织）
```

## 写作要点（借鉴，但保持简报粒度）

1. **按业务领域聚合场景，不按接口方法**：读者关心"客户怎么建档、怎么变更、怎么停用"，不关心"哪个 Controller 方法"。把相关接口归并成一个业务场景叙述。
2. **主流程用编号自然语言**：`1. 用户提交开户申请 → 2. 系统校验证件 → 3. ...`。**禁** "Controller→Service→DAO" 分层引用、禁 `status=0` 这类黑话、禁 Given/When/Then 三段式（那是 openspec specs 的格式，不是给人读的叙述）。
3. **业务语义不清时定点查证再写**：模块是薄 CRUD 脚手架时，业务语义往往在调用方/上游。可在 survey 识别到的调用方/上游代码里定点 Grep 补足语义（如某状态字段的业务含义、某字典的口径），但**查到的代码位置不写进文档**（保持业务抽象）；查不清就在术语表/场景里标「业务侧待确认」，别硬编。

## 抽象层次约束（写完自检）

| 允许 | 禁止 |
|------|------|
| 业务术语、规则编号、自然语言流程 | 类名、方法名、变量名、SQL、文件路径、行号、HTTP 路径、**代码里的裸字面量**（枚举值如 `ALL`/`Y`/`N`、配置码如租户码 `YQJT`、魔法值如占位日期 `1900-01-01`）、Given-When-Then 三段式 |

自检：通读一遍，凡出现驼峰类名/`.method(`/`.java`/SQL 关键字/`status=`/HTTP 路径 → 删改成业务说法。**裸字面量同样禁**——枚举值 `ALL`、配置码 `YQJT`、魔法日期 `1900-01-01` 之类即便不是类名/方法名也是代码黑话，改成业务说法（「全部」「默认归属集团主租户」「无效占位日期」），别把括号里的原值也带出来。
