# Agent 通用前置约定

> 本文档由各 Analysis Agent 在执行前内联读取的"通用前置约定"。集中维护以避免在 3+ agent 模板里重复同一段。

## 1. 计划模式处理

**背景**：若主上下文处于计划模式（plan mode），子 agent 会继承该限制导致 `Write` / `Edit` 工具不可用，从而无法把分析结果落盘。

**处理**：
- IF 检测到无法写入文件（`Write` / `Edit` 工具调用提示「计划模式」或不可用）：
  - 调用 `AskUserQuestion`：「当前处于计划模式，需要退出才能写入分析结果（<目标文件>）。是否同意退出？」
  - 用户同意 → 调用 `ExitPlanMode` → 继续写入
  - 用户拒绝 → 返回 `[错误] 用户拒绝退出计划模式，无法写入文件`

**Why**：分析 Agent 的产出必须落盘，否则下游 Agent 拿不到输入；计划模式作为主上下文的全局开关，不能由子 agent 静默忽略。

## 2. 上下文变量读取约定

各 Agent 的上下文变量（如 `MODULE_NAME`、`PROJECT_ROOT`、`SCRATCHPAD_DIR` 等）从启动 Agent 时传入的 prompt 「上下文变量」章节获取，代入模板中的 `{{...}}` 占位符。**不要从环境变量或 stack-profile 重读已传入的变量**——避免上下文不一致。

## 3. 反幻觉机制（写作约束）

所有分析 Agent 共用：

1. **每项发现附带代码证据**：`ClassName.methodName()` + 具体行为描述（或前端：`文件路径:行号` + 代码片段）
2. **不确定的发现用 `[待验证]` 标记**：说明需要检查什么（而非推测）
3. **未读取到的源码用 `[未读取: <标识>]` 标注**：不要凭直觉补完
4. **每条发现按统一格式书写**：
   - **类别**：<结构追踪 / 领域与状态建模 / 前端追踪>
   - **发现**：[一句话描述]
   - **证据**：[代码位置 + 具体行为]
   - **影响**：[对模块的影响]
   - **置信度**：[高/中/低]

**Why**：本 Skill 的产出物是工程级文档，会被多个工程师作为权威参考；幻觉一条都会损害信任。证据 + 不确定标记让产出物可追溯、可核查、可后续验证。

> **grounding 基础**：本 skill 最终产出 openspec specs（行为契约基线），每条需求/场景都 MUST 落到真实代码事实——这里的代码证据正是 grounding 的取证。编造、查无实据的行为标 `[待验证]` 或丢弃，绝不为"看起来完整"而虚构。
>
> **基座/框架继承方法不算虚构**：MyBatis-Plus `IService` 的 `save`/`list`/`page` 等、JPA `JpaRepository` 的 `findById`/`save` 等、项目自定义基座（如 `BaseTableServiceImpl`）暴露的方法，在子类源文件里查不到定义是正常的（继承自基座）——**不据此判该行为不存在**。CRUD 类端点的可观察行为照常落 spec。

## 4. 失败返回约定

任何阶段出错时，统一返回以 `[错误]` 开头的单行描述：

```
[错误] <阶段简述>：<具体错误原因>。
```

**Why**：编排层用关键字 `[错误]` 判定是否触发重试/降级；统一前缀让判定逻辑稳定，不需要在 4+ 处维护不同的失败短语。

## 5. 跨模块事实约定（横切约定）

跨模块事实（本模块调谁 / 写谁 / 跨模块状态消费）从**目标依赖自身的代码**现读现写。

**深浅自判（地板，非天花板）**：按引用形态判要追多深读目标代码。
- 浅依赖（PO 类型透传 / 枚举或常量引用 / `@Autowired` 类内零调用）—— 登记即可、不深挖
- 深依赖（service 实调、状态契约消费、业务逻辑参与）—— 追到目标方法读其代码，把跨模块事实写准

## 6. 代码扫描卫生

Glob / find / grep 扫源码时排除构建产物目录（`target/` `build/` `dist/` `out/` `node_modules/`）——含编译副本/打包产物，会让文件计数翻倍、或让分析读到过期旧编译事实（如 `target/classes` 里的旧 XML）。
