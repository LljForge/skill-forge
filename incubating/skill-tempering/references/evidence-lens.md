# 路线 B · 实证透镜：消化用户实证的优化细则

> 本文件是 `skill-tempering` **路线 B（实证透镜 · 实证优化）** 的执行细则。它**只编排流程、只交代证据接收口径**——判据（七核 `#1`–`#7`）、归型 / 派生 / 完整性 critic / 三分诊 / 盲区声明，全在 `principles.md` 与 `derivation-protocol.md`；改进清单 / 部分符合声明模板复用 `audit-lens.md`。本文件**只引锚、不复述、不重造模板**（守 `principles.md` #5：单一权威源）。
>
> **路线 B 是什么**：用户**已经在真实使用中跑完**这个 skill，把攒下的实证带进来，要据此把 skill 优化到本仓机制层标准。它与路线 A 共用同一底座、同一改进清单产物，**唯一的差别是 ④ 的证据来源**——A 从文本静态推定，B 吃用户带来的真实运行留痕。故本文件极短:只讲"实证怎么接、三分诊在实证语境怎么用"，其余一律走路线 A 已定的流程。

---

## 一、守薄红线（先读这条,否则本路线会复辟被砍的机床）

路线 B **只消费用户已有的实证,自己绝不产生实证**。下列一旦在本路线冒头,即是被砍掉的评测机床复辟、违 #5,当场打回:

- ❌ 让 skill-tempering **自跑 baseline 双跑**（with/without 对照）——产生实证是用户在真实使用、或另用 skill-creator 等评测工具里的事,不是本路线的事。
- ❌ 织**覆盖账本网格 / 收敛账本**、追求"每格碰一次"式主动穷尽——本路线是被动的:用户实证撞到什么就消化什么,没撞到就诚实标盲区。
- ❌ 把本路线写厚成一条自带评测循环的重流水线——它的独有内容只有两样:**证据接收口径**（§二）+ **三分诊实证语境**（§三）,其余全借路线 A。

> 与 skill-creator 的边界:skill-creator 负责**产生实证**（跑测试 / benchmark / 迭代）;路线 B 负责**消化实证做机制层优化**。用户完全可以用 skill-creator 跑出结果,再把结果当实证喂进路线 B。两者互补,不重叠——本路线不碰"怎么跑测试"。

---

## 二、证据接收口径（本路线的第一独有环节）

路线 B 的输入 = 一份现成 skill **＋ 一批用户已跑完的实证**。什么算"实证"——**真实使用中留下的运行留痕**,典型四类（非穷列）:

| 实证形态 | 例 | 喂给底座哪一步 |
|---|---|---|
| **卡点记录** | "用它时在 X 步卡住 / 指令含糊不知怎么办" | ① B 失败形态（skips-rule / condition-dependent…） |
| **失败案例** | "产物形状跑偏 / 漏了必含要素 / 崩在某前置缺失上" | ① B 失败形态 + ②种子库对应纪律 |
| **运行 trace** | 真实会话记录 / 工具调用序列 / 报错栈 | ① 归型嗅事实 + ③ critic 找命脉 |
| **使用反馈** | 用户主观评价"这里总让我多绕路 / 这段没用" | ③ critic + #5（空转条目） |

**接收纪律**:
- **先验"实证够格吗"**（#3 输入异常分支 + 输入族「源受控性」——用户实证是外部输入,不天然可信,消化前先过一道闸）:**来源**是真实运行留痕,还是道听途说 / 事后臆测?几条实证**自相矛盾**吗（互斥则存疑）?**样本够吗**（单条孤例难判是 skill 通病还是任务特例）?不够格的实证**降级当线索、不当定论**——据它标"疑似、待更多实证",退静态推定,**绝不凭一条可疑或孤立实证就判 skill 缺陷逼改**（呼应第 4 步三分诊:孤例、可疑留痕往往正落在"任务特例 / 模型失误"两类,而非"skill 缺陷"）。
- 实证是**真实运行留痕、不是审计者臆想**——凡用户没带、要靠推断补的部分,退回静态推定,按路线 A 标"未实证"进盲区声明,不冒充实证。
- 实证**有多少用多少**:够格实证充足则以它为主;**单薄时降级**——退化为"路线 A 静态体检为主 ＋ 消化那几条实证",不必硬凑成完整实证优化（分流与降级的总规则在 `SKILL.md` 分流闸,本文件不复述）。

---

## 三、执行流程（借路线 A 七步,只换 ④ 证据来源 + 三分诊语境）

**整体沿用 `audit-lens.md` 的七步静态体检流程**（读 skill → 归型 → 逐条判三态 → 护栏+三分诊 → 完整性 critic → 产改进清单 → 产部分符合声明）,本文件**不复述那七步**,只标出路线 B 的两处不同:

1. **第 2 步归型 · ① B 失败形态**:路线 A 从文本静态推断"最可能怎么坏";**路线 B 直接读用户实证里"实际怎么坏了"**（走 `derivation-protocol.md` ④ 证据来源二分的 B 路）。实证印证的失败形态是**真实留痕、非静态推定**,不必进盲区声明"待验证";实证里从没撞到的纪律,仍是静态推定,照路线 A 标盲区。

2. **第 4 步三分诊 · 实证语境**:走 `derivation-protocol.md`「三分诊归因闸」节的**路线 B 语境**——每条实证暴露的问题归一次因:**skill 缺陷** → 进改进清单;**任务特例** → 不报 / 降 advisory;**模型失误** → 退盲区声明"待确认"。三类各自的含义与判法（凭真实 trace 怎么分得开)在上游那节,本文件**只列名与出口、不复述**（守 #5）——凭真实 trace 能真分开这三类,正是三分诊本为动态证据而设、在路线 B 满血之处。

其余五步（第 1 / 3 / 5 / 6 / 7 步）与路线 A **完全一致**,直接照 `audit-lens.md` 走。**产物同构**:一份按强制力档分级的改进清单 ＋ 一份结构化部分符合声明,**复用 `audit-lens.md`「模板一 / 模板二」,不另造**。

---

## 四、⏸ 暂停点(同路线 A)

**产出改进清单（＋ 部分符合声明）后停下,交给用户。** 路线 B 同样**只诊断、列改进项,不自动去改被测 skill**——改与不改、怎么改由用户定。这是本路线唯一的硬暂停,与路线 A 一致（脊柱 `SKILL.md` ⏸ 节是这条的权威,本文件不复述）。

---

> 自洽性回望:路线 B 是"同一套判据喂真实证据"——它相对路线 A 的全部增量,就是把 ④ 的证据从"文本静态推定"换成"用户实证",让三分诊满血、让失败形态从推断变留痕。它若消化不了真实实证、或消化时偷偷自跑起 baseline,即是坏的。
