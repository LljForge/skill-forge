### 待测事实集：验收标准

- 验收标准 #1（requirements.md“绑定客户经理”“验收标准”；design.md“个人主数据统一保存”）：已实名客户绑定有效客户经理后，本地保存新工号和名称；POST /person/save 收到同一客户的 mdmcode 及 source、orgCode、deptCode、empCode；两端成功后页面与本地缓存才显示新归属。
- 验收标准 #2（requirements.md“验收标准”；design.md“绑定事务内的数据流”）：本地 customerId 为空时，绑定按主数据新增或查重语义取得 mdmcode，并按个人主键回写本地。
- 验收标准 #3（requirements.md“关键异常”“验收标准”；design.md“同步强失败”）：主数据返回失败、空响应、空数据或空 mdmcode 时，绑定失败，本地保持原归属，页面和缓存保持原值。
- 验收标准 #4（requirements.md“验收标准”；design.md“员工与外部个人统一处理”）：主数据员工与外部个人客户执行绑定时走同一保存逻辑，不存在员工跳过或禁止绑定分支。
- 验收标准 #5（requirements.md“验收标准”；design.md“个人主数据统一保存”）：首次实名认证新增个人继续通过统一主数据保存组件生成既有请求字段，并保存响应 mdmcode。
- 验收标准 #6（requirements.md“验收标准”；design.md“个人主数据统一保存”）：已完成个人再次提交、本人手机号命中旧个人且接受新客户经理时均同步主数据；主数据失败时不确认本地新归属。
- 验收标准 #7（requirements.md“验收标准”；design.md“小程序实时预检与引导”）：主包和分包签署页均在请求签署地址前查询当前登录人的服务端最新绑定状态。
- 验收标准 #8（requirements.md“验收标准”；design.md“小程序实时预检与引导”）：小程序门禁只消费 /user/getMe 返回的当前登录人归属；其他客户的相反绑定状态不能导致串人放行或拦截。
- 验收标准 #9（requirements.md“验收标准”；design.md“后端统一绑定门禁”）：个人合同、企业合同、普通 /contract/getSignUrl 和企微 /qw/esign/contract/getSignUrl 均受后端当前登录人绑定门禁保护。
- 验收标准 #10（requirements.md“合同签署”“验收标准”；design.md“小程序实时预检与引导”）：签署人未绑定时显示“取消 / 前往绑定”；取消后不继续签署，选择前往绑定后切换到“我的”页，签署地址请求不发出。
- 验收标准 #11（requirements.md“新增企业认证”“验收标准”；design.md“小程序实时预检与引导”）：企业认证列表发起新增时，未绑定客户不进入表单，并可切换到“我的”页。
- 验收标准 #12（requirements.md“验收标准”；design.md“后端统一绑定门禁”）：直接请求企业保存时，未绑定门禁先于代理人短信验证码、企业重复绑定检查和企业主数据创建。
- 验收标准 #13（requirements.md“验收标准”“非目标”；design.md“企业范围控制”）：已绑定客户的合同签署与新增企业认证保持原成功流程；企业创建继续携带当前登录人的客户经理归属，但不新增企业归属变更。

### 待测事实集：业务规则

- 业务规则 §1（requirements.md“业务规则”；design.md“ADR-1”）：个人主数据新增和更新继续使用 POST /person/save；customerId 非空时作为 mdmcode，空时不传 mdmcode并采用既有新增或查重语义。
- 业务规则 §2（requirements.md“业务规则”；design.md“个人主数据统一保存”）：主数据请求携带 source=GMZB-BS-ECS、客户经理经营单位编码、部门编码和工号；客户经理名称只保存本地，不作为主数据归属字段发送。
- 业务规则 §3（requirements.md“业务规则”；design.md“ADR-3”）：绑定成功以本地归属更新和主数据保存均成功为准；任何一步失败都不得向客户显示成功。
- 业务规则 §4（requirements.md“业务规则”；design.md“ADR-4”）：不区分主数据员工和外部个人客户；凡已实名并执行绑定的人员均走相同个人主数据保存流程。
- 业务规则 §5（requirements.md“业务规则”；design.md“新增/更新共用一个主数据保存组件”）：首次实名认证、已完成个人再次提交、手机号命中旧个人、签约兜底创建或查重及显式绑定共用 PersonMasterDataService。
- 业务规则 §6（design.md“绑定事务内的数据流”）：绑定先按 wxUserId 取得已实名个人并解析唯一有效完整的客户经理，再更新本地归属、保存主数据，必要时按个人主键更新 customerId；本地更新零行或返回失败均抛出异常，全部成功后才返回 BindBusinessManagerVO。
- 业务规则 §7（requirements.md“业务规则”；design.md“个人主数据统一保存”）：savePersonInfo 的再次提交和手机号命中旧个人两条路径先保存主数据，成功后才覆盖本地 mdmcode、客户经理工号和名称；主数据失败时不执行本地覆盖。
- 业务规则 §8（requirements.md“业务规则”；design.md“所有签署对象检查当前登录人”）：个人与企业合同统一校验当前登录微信用户的个人归属，不按合同 customerType 分支，也不校验企业自身归属。
- 业务规则 §9（requirements.md“业务规则”；design.md“后端统一绑定门禁”）：saveCompany 在业务开始阶段校验当前登录人绑定状态，且校验先于短信验证码、重复企业检查和企业主数据创建。
- 业务规则 §10（requirements.md“业务规则”；design.md“前端实时预检 + 后端兜底”）：后端门禁覆盖普通与企微入口，不能被前端缓存、页面路径、普通小程序入口或企微入口绕过。
- 业务规则 §11（requirements.md“业务规则”；design.md“小程序实时预检与引导”）：前端以 /user/getMe 的登录态当前个人为放行依据，不读取 userInfo 缓存，也不使用未按当前用户过滤的分页首条记录；查询失败不继续原动作。
- 业务规则 §12（design.md“小程序实时预检与引导”）：共享前端门禁在已绑定时只继续原动作；未绑定或无个人记录时显示“取消 / 前往绑定”，确认后切换到“我的”页且不继续原动作；查询失败时复用既有请求错误展示。
- 业务规则 §13（design.md“页面接入”）：主包签署页、分包签署页和企业认证列表分别在原签署地址请求或进入新增表单前接入共享门禁。
- 业务规则 §14（requirements.md“业务规则”“非目标”；design.md“企业范围控制”）：企业创建仍将当前登录人的客户经理归属发送给主数据；不更新已认证企业归属，也不增加企业归属字段。
- 业务规则 §15（requirements.md“非目标”“依赖与风险”；design.md“本地/远程事务边界”）：本次采用同步强失败，不引入分布式事务、消息队列、异步补偿或人工补偿；远程主数据成功但本地随后失败时，允许出现已声明的两端短暂不一致，且不自动补偿远端。

- 异常场景 #4（requirements.md“关键异常”）：客户经理手机号存在多个精确结果时绑定失败，本地和主数据均不得确认新归属。
- 异常场景 #5（requirements.md“关键异常”）：客户经理已离职时绑定失败，本地和主数据均不得确认新归属。
- 异常场景 #6（requirements.md“关键异常”）：客户经理缺少工号时绑定失败，本地和主数据均不得确认新归属。
- 异常场景 #7（requirements.md“关键异常”）：客户经理缺少经营单位编码时绑定失败，本地和主数据均不得确认新归属。
- 异常场景 #8（requirements.md“关键异常”）：客户经理缺少部门编码时绑定失败，本地和主数据均不得确认新归属。
- 异常场景 #9（requirements.md“关键异常”；design.md“绑定事务内的数据流”）：本地归属更新零行时绑定失败，不调用主数据，也不返回成功。
- 异常场景 #10（requirements.md“关键异常”；design.md“个人主数据统一保存”）：主数据返回空响应时保存失败并抛出业务异常。
- 异常场景 #11（requirements.md“关键异常”；design.md“个人主数据统一保存”）：主数据返回非成功结果时保存失败并抛出业务异常。
- 异常场景 #12（requirements.md“关键异常”；design.md“个人主数据统一保存”）：主数据成功信封中的数据为空时保存失败并抛出业务异常。
- 异常场景 #13（requirements.md“关键异常”；design.md“个人主数据统一保存”）：主数据响应 mdmcode 为空时保存失败并抛出业务异常。
- 异常场景 #14（design.md“绑定事务内的数据流”）：返回的新 mdmcode 需要回写而本地 customerId 更新零行或返回失败时，绑定抛出异常且不返回成功结果。
- 异常场景 #15（requirements.md“关键异常”；design.md“小程序实时预检与引导”）：查询当前绑定状态失败时，不继续签署，也不进入企业认证新增表单。
- 异常场景 #16（design.md“后端统一绑定门禁”）：后端门禁找不到当前个人或发现未完成实名认证时，保持“请先完成实名认证”语义。
- 异常场景 #17（requirements.md“关键异常”；design.md“后端统一绑定门禁”）：当前登录人未绑定时，后端拒绝签署或企业保存，并提示“请先前往我的页面绑定客户经理”。
- 异常场景 #18（requirements.md“关键异常”；design.md“个人主数据统一保存”）：savePersonInfo 的再次提交或手机号命中旧个人路径中，主数据失败时不覆盖本地新归属。
- 异常场景 #19（requirements.md“依赖与风险”；design.md“本地/远程事务边界”）：远程主数据已成功但本地写入或最终提交失败时，远端不能随本地事务回滚，且本次没有自动补偿。

### 待测事实集：接口契约

- 接口契约 #1（design.md“个人主数据统一保存”）：PersonMasterDataService.saveOrUpdate 接收个人与客户经理信息，映射姓名、证件类型、证件号、source、orgCode、deptCode、empCode及可选 mdmcode；成功返回最终 mdmcode，响应不合法时抛出 BusinessException。
- 接口契约 #2（requirements.md“业务规则”；design.md“ADR-1”）：主数据沿用 POST /person/save；携带 mdmcode 表示更新，不携带时采用新增或查重语义；有效成功响应必须含非空数据和非空 mdmcode。
- 接口契约 #3（design.md“绑定事务内的数据流”）：显式绑定按 wxUserId 处理当前个人，只有本地归属、主数据及必要的 customerId 回写均成功时才返回 BindBusinessManagerVO。
- 接口契约 #4（design.md“后端统一绑定门禁”）：requireBusinessManagerBound 按 wxUserId 取得当前个人；无个人或未实名时给出实名认证提示，未绑定时给出绑定提示，已绑定时返回当前个人供后续业务使用。
- 接口契约 #5（requirements.md“业务规则”；design.md“小程序实时预检与引导”）：/user/getMe 依据后端登录态强制定位当前个人，其 belongBusiManager 是小程序门禁的权威输入。
- 接口契约 #6（design.md“小程序实时预检与引导”）：ensureBusinessManagerBound 接受 api 与 onBound；已绑定时调用 onBound，未绑定或无个人记录时提供两个选择，查询失败时不调用 onBound。
- 接口契约 #7（requirements.md“验收标准”；design.md“后端统一绑定门禁”）：普通 /contract/getSignUrl 与企微 /qw/esign/contract/getSignUrl 共享签署服务门禁，覆盖个人和企业签署对象。
- 接口契约 #8（requirements.md“验收标准”；design.md“后端统一绑定门禁”）：企业保存接口在业务开始阶段执行同一后端门禁，并使用门禁返回的当前个人解析最新客户经理归属。
- 接口契约 #9（design.md“小程序实时预检与引导”）：api.getUserInfo 增加可选失败回调并在请求失败时转发；原有只提供成功回调的消费者保持兼容。

## 1. 功能测试 (TC-F)

#### TC-F-001 · 已有主数据编号的客户绑定成功
- **维度**：D5 错误响应契约 + 成功信封
- **来源**：验收标准 #1 / 业务规则 §1 / 业务规则 §2 / 业务规则 §3 / 业务规则 §6 / 接口契约 #1 / 接口契约 #2 / 接口契约 #3
- **优先级**：高
- **GIVEN** 一个已实名且具有 customerId 的当前登录人，以及唯一、在职且归属信息完整的客户经理
- **WHEN** 当前登录人提交绑定该客户经理
- **THEN** 本地归属保存该客户经理的工号和名称
- **THEN** POST /person/save 收到该 customerId 对应的 mdmcode、source=GMZB-BS-ECS、经营单位编码、部门编码和工号，且不把客户经理名称作为主数据归属字段
- **THEN** 主数据返回有效 mdmcode 后绑定成功，页面与本地缓存显示新归属

#### TC-F-002 · 主数据员工执行绑定
- **维度**：§1 输入边界
- **来源**：验收标准 #4 / 业务规则 §4
- **优先级**：高
- **GIVEN** 一个已实名的主数据员工和一个有效客户经理
- **WHEN** 该员工提交客户经理绑定
- **THEN** 绑定采用与外部个人客户相同的个人主数据保存流程
- **THEN** 员工身份不导致跳过同步或拒绝绑定

#### TC-F-003 · 外部个人客户执行绑定
- **维度**：§1 输入边界
- **来源**：验收标准 #4 / 业务规则 §4
- **优先级**：高
- **GIVEN** 一个已实名的外部个人客户和一个有效客户经理
- **WHEN** 该客户提交客户经理绑定
- **THEN** 绑定采用统一的个人主数据保存流程
- **THEN** 本地与主数据均成功后返回绑定成功

#### TC-F-004 · 首次实名认证新增个人统一保存
- **维度**：D5 错误响应契约 + 成功信封
- **来源**：验收标准 #5 / 业务规则 §5 / 接口契约 #1 / 接口契约 #2
- **优先级**：高
- **GIVEN** 一个首次完成实名认证且本地尚无主数据编号的个人
- **WHEN** 实名认证流程接受该个人
- **THEN** 个人信息通过 PersonMasterDataService 保存到 POST /person/save
- **THEN** 请求保留姓名、证件、来源和客户经理归属字段，响应中的有效 mdmcode 保存到本地

#### TC-F-005 · 已完成个人再次提交后同步新归属
- **维度**：D5 错误响应契约 + 成功信封
- **来源**：验收标准 #6 / 业务规则 §5 / 业务规则 §7
- **优先级**：高
- **GIVEN** 一个已完成实名认证的个人再次提交资料，且系统接受新的客户经理
- **WHEN** savePersonInfo 处理该次提交
- **THEN** 新归属先通过统一个人主数据组件保存
- **THEN** 主数据成功后，本地 customerId、客户经理工号和名称更新为成功结果

#### TC-F-006 · 手机号命中旧个人后同步新归属
- **维度**：D5 错误响应契约 + 成功信封
- **来源**：验收标准 #6 / 业务规则 §5 / 业务规则 §7
- **优先级**：高
- **GIVEN** 本人手机号命中一条既有个人记录，且系统接受新的客户经理
- **WHEN** savePersonInfo 处理该个人归属覆盖
- **THEN** 新归属先通过统一个人主数据组件保存
- **THEN** 主数据成功后，既有个人记录保存返回的 mdmcode、新工号和新名称

#### TC-F-007 · 签约兜底创建或查重统一保存
- **维度**：D5 错误响应契约 + 成功信封
- **来源**：业务规则 §5 / 接口契约 #1 / 接口契约 #2
- **优先级**：高
- **GIVEN** 签约流程发现当前个人需要进行主数据兜底创建或查重
- **WHEN** 签约流程解析或创建该个人
- **THEN** 个人信息通过 PersonMasterDataService 调用既有 POST /person/save
- **THEN** 有效 mdmcode 成为后续签约使用的个人主数据编号

#### TC-F-008 · 主包签署页已绑定时继续原签署
- **维度**：无领域维度（纯前端）
- **来源**：验收标准 #7 / 验收标准 #13 / 业务规则 §11 / 业务规则 §12 / 业务规则 §13 / 接口契约 #5 / 接口契约 #6
- **优先级**：高
- **GIVEN** 主包签署页已完成文档与平台校验，且 /user/getMe 返回当前登录人已绑定
- **WHEN** 当前登录人发起签署
- **THEN** 共享门禁继续原签署动作

#### TC-ST-001 · 绑定从旧归属迁移到新归属

## 待澄清

- 待澄清 #1：客户经理手机号的必填性、空值语义、允许格式、长度上下界及各无效类的具体错误响应未定义；因此不猜测空串、缺失、边界两侧的期望。
