## 概览

- **输入来源**：`docs/recon-driven-dev/_archived/2026-07-17-sync-customer-manager-ownership/`（`requirements.md` §2.4 / §3 / §4 + `design.md` 接口契约）
- **跨栈提示**：本任务同时含**后端**（`edoc-be`）与**小程序前端**（`edoc-weixin-fe`）两个被测面。后端用例受 `backend-api.md` 维度表点名召回；**小程序门禁属 E2E/前端行为，本 skill 无对应领域维度表**——前端用例只受三铁律 + 出口三门 + 输入边界法约束，边界召回靠临场想，如实标注（见「维度扫描表」脚注）。

### 待测事实集（本次用例的唯一来源，`来源` 字段回指此处编号）

**验收标准（req §4）**
- **AC-1**：已实名客户绑定有效客户经理后，本地保存新工号/名称，`POST /person/save` 收到相同客户的 `mdmcode` 及 `source/orgCode/deptCode/empCode`。
- **AC-2**：本地 `customerId` 为空时，绑定仍按新增/查重语义取得 `mdmcode` 并回写本地。
- **AC-3**：主数据返回失败、空数据或空 `mdmcode` 时，绑定接口返回失败，本地保持原客户经理，页面和缓存保持原值。
- **AC-4**：主数据员工与外部个人客户走同一保存逻辑，无员工跳过或禁止绑定分支。
- **AC-5**：个人实名认证新增客户仍通过同一主数据保存组件生成原字段并保存响应 `mdmcode`。
- **AC-6**：已实名个人再次提交客户经理、手机号命中旧个人并更新客户经理时也同步主数据；主数据失败时不确认新本地归属。
- **AC-7**：主包/分包签署页在请求签署地址前查询当前登录人最新绑定状态。
- **AC-8**：小程序门禁通过 `/user/getMe` 取登录人归属；即使库中另一客户绑定状态相反，也不串人放行/拦截。
- **AC-9**：个人合同、企业合同、普通 `/contract/getSignUrl`、企微 `/qw/esign/contract/getSignUrl` 均受后端门禁保护。
- **AC-10**：未绑定时看到「取消 / 前往绑定」，选择后切到「我的」，签署请求不发出。
- **AC-11**：企业认证「新增认证」未绑定不进入表单，可切到「我的」。
- **AC-12**：直接调企业保存接口时，未绑定校验发生在代理人短信验证码和企业重复绑定检查之前。
- **AC-13**：已绑定客户签署和新增企业认证保持现有成功流程，不增加企业归属变更。
- **AC-14**：后端定向 + 小程序行为测试全通过，原有测试无回归。*(过程/元验收，不落用例)*

**业务规则（req §3）**
- **BR-1**：个人主数据新增与更新继续用 `POST /person/save`，不切 Feign 路径。
- **BR-2**：本地 `customerId` 作 `mdmcode`；无 `customerId` 按新增/查重语义并回写 `mdmcode`。
- **BR-3**：主数据请求必携带 `source=GMZB-BS-ECS`、经营单位编码、部门编码、工号。
- **BR-4**：客户经理名称只存本地，不作主数据归属字段发送。
- **BR-5**：不区分主数据员工与外部个人客户，同一保存流程。
- **BR-6**：绑定成功以「本地成功且主数据成功」为准，任一失败不显示成功。
- **BR-7**：个人/企业合同统一校验当前登录人个人归属；是操作人门禁，非企业归属校验。
- **BR-8**：新增企业认证前端预检 + 后端保存开始阶段再校验。
- **BR-9**：后端校验不能被前端缓存/页面路径/普通入口/企微入口绕过。
- **BR-10**：前端绑定状态查服务端最新，不以本地缓存放行。
- **BR-11**：前端绑定状态经 `/user/getMe` 取当前个人，不用未绑定当前用户的分页首条。
- **BR-12**：`savePersonInfo` 两条覆盖路径与显式绑定用同一组件和失败语义。
- **BR-13**：企业创建仍按现有逻辑发送当前登录人客户经理归属。

**关键异常（req §2.4）**
- **EX-1**：客户未实名 → 「请先完成实名认证」，不允许绑定。
- **EX-2**：客户经理手机号无效/查不到/多个精确结果/已离职/缺工号或经营单位或部门 → 绑定失败，本地和主数据均不确认新归属。
- **EX-3**：本地归属更新失败 → 不调主数据，不返回成功。
- **EX-4**：主数据更新失败或未返回有效客户编号 → 整次失败，本地回滚原值，页面缓存不刷新。
- **EX-5**：查询当前绑定状态失败 → 不继续签署，也不进入企业新增表单。
- **EX-6**：当前登录人未绑定 → 后端直接调用被拒，提示「请先前往我的页面绑定客户经理」。

**接口契约（design.md）**
- **IF-1**：`PersonMasterDataService.saveOrUpdate(person, managerInfo) → String mdmcode`。组装 `MdmPersonDTO`（`name`/身份证类型/身份证号/`source`/`orgCode`/`deptCode`/`empCode`），`customerId` 非空映射 `mdmcode`、为空不填；调 `POST /person/save`；校验响应非空+成功+数据非空+`mdmcode` 非空；失败抛 `BusinessException`。
- **IF-2**：`UserPersonService.requireBusinessManagerBound(wxUserId) → UserPersonPO`。找不到/未实名 → 「请先完成实名认证」；`belongBusiManager` 空 → 抛「请先前往我的页面绑定客户经理」；已绑定返回当前个人。
- **IF-3**：`ensureBusinessManagerBound({api, onBound})`（前端）。调 `/user/getMe` 读 `belongBusiManager`（不读缓存、不用分页首条）；已绑定调 `onBound`；未绑定/无记录显 Modal（取消 / 前往绑定），确认则 `switchTab` 到 `individual-center` 且不调 `onBound`；查询失败复用错误展示且不调 `onBound`。
- **IF-4**：绑定事务数据流（design §1）。读本地个人（须已实名）→ 按手机号取工号/名称/经营单位/部门 → 先更新本地 `belong_busi_manager(+name)`，未命中任何记录立即失败且不调主数据 → 调 `saveOrUpdate` → 返回 `mdmcode` 与本地 `customerId` 不同则同事务按主键更新 `customer_id`，未命中或返回 false 抛异常 → 任一主数据异常上抛回滚本地 → 全成功才组装 `BindBusinessManagerVO`。
- **IF-5**：`FadadaPlatformService.getSignUrl` 在查法大大认证记录前调 `requireBusinessManagerBound`；普通与企微 Controller 共享该服务（一处兜底两入口）。
- **IF-6**：`UserBusinessServiceImpl.saveCompany` 在进入代理人短信验证码、企业重复绑定检查、企业主数据创建前调 `requireBusinessManagerBound`。

## 1. 功能测试 (TC-F)

#### TC-F-001 · 已实名客户绑定有效客户经理（双写成功）
- **维度**：D5 成功信封
- **来源**：AC-1 / BR-6 / IF-1 / IF-4
- **优先级**：高
- **GIVEN** 一个已实名、当前归属为客户经理 A 的个人客户，提交了一个唯一有效、归属信息完整的客户经理 B 的手机号
- **WHEN** 发起绑定
- **THEN** 本地归属更新为客户经理 B 的工号与名称
- **THEN** 主数据收到同一客户的保存请求，其中承载了 B 的归属编码
- **THEN** 绑定以成功信封返回，页面与本地缓存刷新为客户经理 B

#### TC-F-002 · 本地无客户编号时按新增/查重取得并回写
- **维度**：§1 输入边界（`customerId` 空态）
- **来源**：AC-2 / BR-2
- **优先级**：高
- **GIVEN** 一个已实名、本地 `customerId` 为空的个人客户，提交了有效客户经理手机号
- **WHEN** 发起绑定
- **THEN** 主数据请求不携带 `mdmcode`，由接口按新增/查重语义处理
- **THEN** 响应返回的 `mdmcode` 被回写到本地客户编号

#### TC-F-003 · 个人实名认证新增客户走统一保存组件
- **维度**：D5 成功信封
- **来源**：AC-5 / IF-1
- **优先级**：中
- **GIVEN** 一个首次提交实名认证、需新建个人记录的客户
- **WHEN** 完成实名认证个人创建
- **THEN** 主数据保存请求由统一组件生成，字段与显式绑定路径一致
- **THEN** 响应 `mdmcode` 被保存到该个人

#### TC-F-004 · 已实名再次提交 / 手机号命中旧个人也同步主数据
- **维度**：D5 成功信封
- **来源**：AC-6 / BR-12
- **优先级**：高
- **GIVEN** 一个已实名个人，通过「再次提交个人认证」或「本人手机号命中已有个人记录」两条路径之一携带了新的所属客户经理
- **WHEN** 系统接受新的所属客户经理
- **THEN** 执行与显式绑定相同的主数据同步，不只改本地即提前返回

#### TC-F-005 · 已绑定客户签署 / 新增企业认证保持原成功流程
- **维度**：D5 成功信封
- **来源**：AC-13 / BR-13
- **优先级**：中
- **GIVEN** 一个当前登录人已绑定客户经理的场景
- **WHEN** 发起合同签署或新增企业认证
- **THEN** 沿用原有成功流程放行，不产生任何企业归属变更请求

#### TC-F-006 · 主数据员工与外部个人走同一保存逻辑
- **维度**：D5 成功信封（分支合并）
- **来源**：AC-4 / BR-5
- **优先级**：中
- **GIVEN** 一个 `workCode` 非空（主数据员工身份）的已实名客户，提交了有效客户经理手机号
- **WHEN** 发起绑定
- **THEN** 走与外部个人客户完全相同的主数据保存路径，不存在员工跳过或被禁止绑定的分支

#### TC-F-007 · 客户经理名称只存本地不发主数据
- **维度**：D5 成功信封（字段裁剪）
- **来源**：BR-4
- **优先级**：中
- **GIVEN** 一次携带客户经理名称的有效绑定
- **WHEN** 组装主数据保存请求
- **THEN** 客户经理名称只落到本地归属，主数据归属字段中不含客户经理名称

#### TC-F-008 · 主数据请求承载来源与归属三字段
- **维度**：D5 成功信封（字段完整性）
- **来源**：AC-1 / BR-3 / IF-1
- **优先级**：高
- **GIVEN** 一次有效绑定，其客户经理具备完整的经营单位、部门、工号
- **WHEN** 组装主数据保存请求
- **THEN** 请求同时承载 `source=GMZB-BS-ECS`、经营单位编码、部门编码、工号四项

#### TC-F-009 · 已绑定登录人门禁放行
- **维度**：D2 鉴权前置（放行态）
- **来源**：IF-2 / AC-13
- **优先级**：中
- **GIVEN** 一个已实名且已绑定客户经理的当前登录人
- **WHEN** 触发绑定门禁校验
- **THEN** 返回当前个人，供调用方继续使用其客户经理归属

#### TC-F-010 · 前端页面仅在门禁通过后调用原动作
- **维度**：前端行为（无领域维度，见脚注）
- **来源**：AC-7 / AC-10 / AC-11 / BR-8
- **优先级**：高
- **GIVEN** 主包签署页、分包签署页、企业认证列表页三处入口，当前登录人已绑定
- **WHEN** 各自完成本地前置校验后触发门禁
- **THEN** 三处均在门禁通过后才发起原动作（签署地址请求 / 进入企业表单）

## 2. 边界测试 (TC-E)

#### TC-E-001 · 客户编号空态两分支的 mdmcode 传递
- **维度**：§1 输入边界（`customerId` 有效/空 等价类）
- **来源**：BR-2 / IF-1
- **优先级**：高
- **GIVEN** 两个已实名客户，一个本地 `customerId` 非空、一个为空，均提交有效客户经理手机号
- **WHEN** 分别发起绑定
- **THEN** 非空者主数据请求携带 `mdmcode=customerId`
- **THEN** 空者主数据请求不携带 `mdmcode`

#### TC-E-002 · 绑定成功后本地缓存刷新为新值
- **维度**：D4 缓存读一致性（写后读）
- **来源**：AC-1 / req §2.1.4
- **优先级**：中
- **GIVEN** 一个本地与主数据均成功的绑定
- **WHEN** 绑定完成后读取本地缓存
- **THEN** 读到的是新客户经理，不残留旧值

#### TC-E-003 · 前端绑定状态取服务端最新而非本地缓存
- **维度**：D4 缓存读一致性
- **来源**：BR-10 / IF-3
- **优先级**：高
- **GIVEN** 本地缓存中的绑定状态与服务端最新状态不一致
- **WHEN** 前端门禁判定是否放行
- **THEN** 以服务端最新状态为准，不以本地缓存作为放行依据

#### TC-E-004 · 客户经理手机号命中多个精确结果
- **维度**：§1 输入边界（多结果无效类）
- **来源**：EX-2
- **优先级**：中
- **GIVEN** 一个提交的手机号在客户经理侧命中多个精确结果
- **WHEN** 系统尝试确认唯一客户经理
- **THEN** 绑定失败，本地与主数据均不确认新归属

#### TC-E-005 · 双用户隔离：另一客户绑定状态相反不串人
- **维度**：用户隔离（§3 多租户半边，事实集有依据）
- **来源**：AC-8 / BR-11 / IF-3
- **优先级**：高
- **GIVEN** 当前登录人未绑定，而库中另一名客户已绑定（状态相反）
- **WHEN** 前端门禁按当前登录态判定
- **THEN** 只消费当前登录人的归属结果，不因另一客户的相反状态而串人放行或拦截

#### TC-E-006 · 主数据返回编号与本地不同时回写本地
- **维度**：§1 输入边界（`mdmcode` 一致/相异）
- **来源**：IF-4
- **优先级**：中
- **GIVEN** 一次绑定中主数据返回的 `mdmcode` 与本地当前客户编号不同
- **WHEN** 处理主数据响应
- **THEN** 本地客户编号在同一事务内更新为主数据返回的编号

## 3. 错误处理 (TC-ERR)

#### TC-ERR-001 · 未实名客户被拒
- **维度**：D2 鉴权前置（未实名前置态）
- **来源**：EX-1 / IF-2
- **优先级**：高
- **GIVEN** 一个未完成实名认证的客户
- **WHEN** 发起绑定或触发门禁
- **THEN** 以「请先完成实名认证」被拒，不允许绑定

#### TC-ERR-002 · 客户经理手机号无效/查不到
- **维度**：§1 输入边界（无效/缺失类）
- **来源**：EX-2
- **优先级**：高
- **GIVEN** 一个提交的客户经理手机号无效或查询不到
- **WHEN** 系统尝试确认客户经理
- **THEN** 绑定失败，本地与主数据均不确认新归属

#### TC-ERR-003 · 客户经理已离职或归属信息不完整
- **维度**：§1 输入边界（数据完整性无效类）
- **来源**：EX-2
- **优先级**：中
- **GIVEN** 一个命中的客户经理已离职，或缺工号/经营单位/部门中的任一项
- **WHEN** 系统尝试确认客户经理
- **THEN** 绑定失败，本地与主数据均不确认新归属

#### TC-ERR-004 · 本地归属更新未命中记录
- **维度**：D3 事务边界
- **来源**：EX-3 / IF-4
- **优先级**：高
- **GIVEN** 一次绑定中本地归属更新未命中任何记录
- **WHEN** 处理本地更新结果
- **THEN** 立即失败，不调用主数据，不返回成功

#### TC-ERR-005 · 主数据失败触发本地回滚
- **维度**：D3 事务边界 / 回滚
- **来源**：EX-4 / AC-3 / IF-1 / IF-4
- **优先级**：高
- **GIVEN** 一次绑定中本地归属已更新成功，随后主数据返回失败、空数据或空 `mdmcode`
- **WHEN** 处理主数据响应
- **THEN** 整次绑定失败，本地归属回滚为原客户经理
- **THEN** 页面与缓存保持原值，不显示成功

#### TC-ERR-006 · 客户编号回写未命中或失败
- **维度**：D3 事务边界
- **来源**：IF-4
- **优先级**：中
- **GIVEN** 一次绑定中主数据已成功，但本地客户编号回写未命中记录或返回 false
- **WHEN** 处理回写结果
- **THEN** 抛出异常触发事务回滚，不返回成功信封

#### TC-ERR-007 · 后端门禁拒绝未绑定登录人
- **维度**：D2 鉴权前置（未绑定前置态）
- **来源**：EX-6 / IF-2 / BR-9
- **优先级**：高
- **GIVEN** 一个已实名但未绑定客户经理的当前登录人
- **WHEN** 直接调用后端签署或企业保存
- **THEN** 被门禁拒绝，提示「请先前往我的页面绑定客户经理」，前端缓存/页面路径/入口变体均无法绕过

#### TC-ERR-008 · 企业保存门禁前置于短信与重复检查
- **维度**：D2 鉴权前置（校验时序）
- **来源**：AC-12 / IF-6 / BR-8
- **优先级**：高
- **GIVEN** 一个未绑定登录人直接调用企业保存接口
- **WHEN** 进入企业保存流程
- **THEN** 门禁校验发生在代理人短信验证码、企业重复绑定检查、企业主数据创建之前，且这三步均未被触达

#### TC-ERR-009 · 签署门禁前置于法大大记录查询（两入口共享）
- **维度**：D2 鉴权前置（多入口一致）
- **来源**：AC-9 / IF-5 / BR-7
- **优先级**：高
- **GIVEN** 一个未绑定登录人，分别经普通 `/contract/getSignUrl` 与企微 `/qw/esign/contract/getSignUrl` 发起个人/企业合同签署
- **WHEN** 进入签署地址获取流程
- **THEN** 两入口均在查询法大大认证记录前被门禁拒绝，行为一致

#### TC-ERR-010 · 前端查询绑定状态失败不放行
- **维度**：前端行为（无领域维度，见脚注）
- **来源**：EX-5 / IF-3
- **优先级**：高
- **GIVEN** 前端查询当前绑定状态时接口失败
- **WHEN** 门禁处理查询失败
- **THEN** 不继续签署、不进入企业新增表单，复用请求层错误展示，不调用后续原动作

#### TC-ERR-011 · 覆盖路径主数据失败不覆盖本地
- **维度**：D3 事务边界
- **来源**：AC-6 / BR-12
- **优先级**：高
- **GIVEN** `savePersonInfo` 的「再次提交」或「手机号命中旧个人」路径携带新客户经理，随后主数据保存失败
- **WHEN** 处理主数据失败
- **THEN** 不执行本地新归属覆盖，两端不出现「本地新、主数据旧」

## 4. 状态迁移 (TC-ST)

#### TC-ST-001 · 绑定成功状态链
- **维度**：D3 / 状态迁移
- **来源**：IF-4 / AC-1
- **优先级**：中
- **GIVEN** 一个当前归属客户经理 A 的已实名客户
- **WHEN** 本地写成功 → 主数据成功 → 编号回写成功
- **THEN** 状态终态迁移为归属客户经理 B，且缓存刷新为 B

#### TC-ST-002 · 绑定失败回滚状态
- **维度**：D3 回滚 / 状态迁移
- **来源**：EX-4 / BR-6
- **优先级**：高
- **GIVEN** 一个当前归属客户经理 A 的已实名客户，绑定过程中主数据失败
- **WHEN** 失败触发回滚
- **THEN** 状态回到归属客户经理 A，无任何中间态残留（不出现「本地 B、主数据 A」）

#### TC-ST-003 · 前端未绑定引导状态机
- **维度**：前端行为（无领域维度，见脚注）
- **来源**：AC-10 / AC-11 / IF-3
- **优先级**：高
- **GIVEN** 当前登录人未绑定，触发签署或新增企业认证的门禁
- **WHEN** 呈现「取消 / 前往绑定」两个动作
- **THEN** 选「前往绑定」→ 切到「我的」页面，且原动作不发生
- **THEN** 选「取消」→ 停留原页，原动作不发生

#### TC-ST-004 · 同步失败后征信授权仍读旧归属
- **维度**：D3 消费方一致性 / 状态迁移
- **来源**：req §5 非目标（征信授权）/ design 消费方护栏
- **优先级**：中
- **GIVEN** 一次绑定或个人认证的主数据同步失败并已回滚本地归属
- **WHEN** 征信授权流程读取 `belongBusiManager`
- **THEN** 读到的是回滚后的原归属值，行为不受本次改动破坏
