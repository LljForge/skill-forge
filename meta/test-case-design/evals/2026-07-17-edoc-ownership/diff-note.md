## 二、横向读（skill 加没加值）

- **Q1 执行解耦**：两模型都大幅去泄漏，**Codex 落差更大（18→0 vs 10→1）**。裸 Codex 泄漏倾向比裸 Claude 还强，套 skill 后零命中。越能干、越自然把「测什么/怎么测」揉一起的模型，砍字段 + 门 B 越值钱。
- **Q2 D3 召回**：四格全 ✓，**零差异**。两模型裸跑都自己读 design 抓到了 D3 回滚。
- **Q3 臆造**：**只有 Codex 行显差异**。裸 Codex 自造并发 8 条（§3 治下），codex-skill-on 按 §3 全 N/A。裸 Claude 没造。（注：裸 Codex 的并发用例有对冲——标「策略未定义→登记高风险缺陷」，属「有对冲的思辨」而非盲目臆造；§3 该不该一律压制它，是可辩的取舍，非纯对错。）
- **Q4**：两 on 格都有逐条来源 + 结构化待澄清；off 格都缺逐条源。但裸 Codex 会标假设(7)，裸 Claude 直接静默填平——off 侧差距在 Codex 上更小。

## 三、纵向读（跨模型复现 + 纠上一轮误判）

**稳健复现：**
- **Q1 执行解耦** = skill 真价值内核，两模型都完胜，唯一在两模型上都差异化胜出的维度。
- **Q2 维度表召回 = 教已知** —— 两模型裸跑都自抓 D3，招牌卖点在 2 个模型上都无差异化。证据硬。

**跨模型抓出的两处上一轮（只跑 Claude）误判：**

1. **§3「不默认登记」上一轮判「价值未证」——假阴性。** 那只是**没被 Claude 触发**。换到更爱发挥的 Codex，裸跑立刻臆造 8 条并发，§3 当场救场。§3 不是没价值，是「防某些模型臆造倾向」的保险。**单模型 n=1 会给假阴性——这正是跨模型对照的价值。**
2. **上一轮担心「skill 收窄覆盖」（Claude on=31 条 < off=91 条）——不复现。** Codex 上 skill-on（691 行）反比 skill-off（395 行）更大。「31<91」多半是手跑 skill-on 时的保守，不是 skill 压制发挥。跨模型控制把这个假信号也纠了。

## 四、分层结论（对「加规则会不会影响发挥/教已知」的修正版回答）

| skill 部件 | 判定 | 证据 |
|---|---|---|
| 砍字段 / 门 B（执行解耦） | **真内核，留** | 两模型完胜，硬 |
| §3 不默认登记 | **留**，防臆造保险，模型越能发挥越需要 | Codex 上证实，中 |
| 门 A 无源 + 待澄清结构 | **留**，收益随模型自省力递减 | 两模型 on 占优，中 |
| 维度表点名召回（D3 这类显然维度） | **教已知，可砍/降级** | 两模型裸跑都自抓，硬 |
| §1 边界值例子 / 四分区解释 | **教已知，可砍** | 裸跑都自带结构 |

**一句话**：「加规则影响发挥」这个顾虑，对**知识类规则**成立、对**纪律类规则**不成立；且换个更能发挥的模型，纪律类规则不是更多余、是更必要。

## 五、前端半挖掘（回答「要不要建前端 E2E 维度表」）

skill 只建了后端维度表，前端 E2E 维度层是欠账（SKILL.md 自述、CHANGELOG「后续」）。edoc 是后端+前端混合，四格的前端半全靠纪律层 + 临场想跑（无维度表）。用四格判「前端召回表值不值得建」。

**七类前端风险 × 四格（已核实是真用例，非顺嘴一提）：**

| 前端风险 | claude-on | claude-off | codex-on | codex-off |
|---|:-:|:-:|:-:|:-:|
| R1 不因缓存放行 | ✓ | ✓ | ✓ | ✓ |
| R2 不串人隔离 | ✓ | ✓ | ✓ | ✓ |
| R3 分包页不绕过 | ✓ | ✓ | ✓ | ✓ |
| R4 switchTab 目标 | ✓ | ✓ | ✓ | ✓ |
| R5 Modal 双分支 | ✓ | ✓ | ✓ | ✓ |
| R6 getUserInfo fail 兼容 | ✓ | ✓ | ✓ | ✓ |
| **R7 门禁在本地校验后、请求前** | **✗** | ✓ | **✗** | ✓ |

1. **6/7 四格全中（含两裸格）**：前端不存在「临场想会漏、需点名清单兜」的召回缺口。
2. **R2 裸 Codex 反超**：不止 GUARD-009/010「不用分页首条」，还自加 SEC-004/005（他人身份注入、`/user/getMe` 拒绝身份覆盖）——裸模型是超额覆盖，彻底证明教已知。
3. **R7 唯一差异且方向反了**：两裸格都写了「门禁在『已选文档/同平台』本地校验后、发签署请求前」（claude-off FE-PAGE-03、codex-off FEP-005/REG-003），**两 skill-on 格都漏**（claude-on 那句「顺序」是后端企业保存门禁前移，非前端页面级）。成因推断：Step 3 逼着逐条走**后端**维度表，注意力被拽向后端 + 整套点名/三门仪式压着，前端半反比裸跑更易漏一个顺序细节。**这是「加规则影响发挥」顾虑的第一个实锤——虽小但已核实、跨两模型一致。**

**合并结论**：维度表（占行数最多、血统最弱）在**后端显然维度 / 后端隐性 D3 / 前端全部**三处都没挣到差异化价值，前端半甚至隐约有反效果。skill 至今唯一差异化胜出的是**纪律层**（执行解耦/门B/无源/待澄清），领域无关、两半都管用。

**决定：不建前端 E2E 维度表**——建了就是教已知（同后端显然维度的坑），且缺口（R7）不在「没有前端表」而在「后端仪式挤占前端注意力」，加表补不到。

## 六、仍未触达（别越界外推）
- 任务维度 n=1：没踩悬案 1（D4 缓存沉默），没有前端为主的任务。
- 前端结论有个前提：本任务前端风险在 design.md 里全写明了，裸模型是「读到」不纯是「凭空想到」。换一个前端风险没写全的任务，裸模型可能真会漏、那时前端表才可能显值。所以「不建前端表」= 这份证据不支持建，非「永远别建」。
- 维度表在别的任务的普适性、§3 在别的任务的价值，都需换任务再验。
- 「砍显然维度、只留反直觉维度」这个改法本身还没实测——它是本轮结论的推论，不是已验证的动作。
