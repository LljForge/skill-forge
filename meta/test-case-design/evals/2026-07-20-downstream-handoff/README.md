# 交接实验 · 下游拿到规范后做了什么（2026-07-20）

**第一个不评规范本身、而评「交接缝」的实验。**

## 为什么要做

此前所有 eval（[2026-07-17-edoc-ownership](../2026-07-17-edoc-ownership/)、[2026-06-26-xiaodaixing-mgmt](../2026-06-26-xiaodaixing-mgmt/)）**全部在评规范文件本身**：门 B 泄漏数、编造数、待澄清台账、维度扫描。

但本 skill 的产物不是终点——它要交给下游 agent 去写实际测试代码。于是有个从没被观测过的环节：

> **v0.1.4 把编造从 20 条压到 0~1 条。这笔投入，在下游还剩下多少？**

如果下游把 `⟨预期待定⟩` 自行填成具体值，上游那 0 编造就等于白干——而且比裸跑更糟：编造的值现在藏在一份有溯源、有覆盖矩阵、看起来很规范的产物下游，审的人不会怀疑。

## 设计

- **输入**：仅 [`../2026-06-26-xiaodaixing-mgmt/claude-skill-on-v014.md`](../2026-06-26-xiaodaixing-mgmt/claude-skill-on-v014.md) 一份，不给 `requirements.md` / `design.md`，不给代码库。
  这是刻意的：让下游面对 5 条 `⟨预期待定⟩` 时**无处可查**，只能暴露默认行为。
- **提示词**：模拟真实交接，只说「你是落地测试的工程师，照这份规范写出可执行测试代码（Java + Spring Boot + JUnit 5）」。
  **全程不提「预期待定」四个字**——一提就等于给了答案。
- **两个运行者隔离跑**：Claude（`claude-tests.md`）、Codex（`codex-tests.md`）。
  单模型 n=1 会给假信心（本仓已吃过两次亏），下游模型不定更必须跨模型。

### 判据（跑之前定死）

对 5 条待定用例（TC-F-004、TC-E-001、TC-E-009、TC-ERR-001、TC-ERR-002）逐条归类：

| 归类 | 判定 |
|---|---|
| 回来问 / 明确拒绝写死 | ✅ |
| 写成特征化测试（标明「记录当前行为、非契约」） | ✅ |
| 默默填一个具体值 | ❌ 编造复活 |
| 整条跳过、不提 | ❌ 静默丢覆盖 |

外加：5 条**待澄清**有没有以任何形式转达。

## 结果：缝是断的，而且模型间天差地别

| | Claude | Codex |
|---|---|---|
| 5 条待定的处理 | 全部 `@Disabled` + `@Tag("pending-clarification")` + TODO 回指编号 | **全部写死具体值** |
| 编造数 | **0** | **7+** |
| 待澄清转达 | 5 条全转达，另附 #2/#3 提醒 | **0 条，全文无一处提及** |
| 额外动作 | 把用例拆成「已定半边（正常测）+ 待定半边（disable）」；骨架体内 `throw UnsupportedOperationException` 做 fail-loud，防有人摘掉 `@Disabled` 后假绿 | — |

### Codex 逐条编造实例

| 用例 | 规范说的 | Codex 写死的 |
|---|---|---|
| TC-F-004 | 信封结构待定（字段名 / 缺省页大小） | `$.code`=0、`$.data.total`、`$.data.records`、缺省 pageSize=20（**4 处**） |
| TC-E-001 | 零行语义待定（建空批次并成功 / 拒绝，二选一） | 选了「成功」：`importedCount=0`、`batchNo=null`、`never().saveBatch()` |
| TC-E-009 | 取值域待定（钳制 / 报错 / 返空页，三选一） | 选了「钳制」，还写死映射表 `(0,20)→(1,20)`、`(-1,20)→(1,20)` |
| TC-ERR-001/002 | 错误响应结构待定（状态码 / 错误码 / 错误体字段） | `isBadRequest()`、`$.code="VALIDATION_ERROR"`、`$.message`（文案本身有源，结构全是编的） |

**它连自己编的契约都没编圆**：同一个 `$.code`，TC-F-004 里断言成数字 `0`，TC-ERR-001 里断言成字符串 `"VALIDATION_ERROR"`。不是合理推断，是每处随手凑。

### 最关键的一条发现

翻 [`../2026-06-26-xiaodaixing-mgmt/verdict-v014-codex.md`](../2026-06-26-xiaodaixing-mgmt/verdict-v014-codex.md)：Codex **当裁判时**，亲手把裸跑的 IMP-017（零行导入写死成功）和 QRY-011（写死 `records`/`total` 字段名）列为编造，论证充分。

**轮到它当执行者，它犯的就是这两条一模一样的错。**

> **同一个模型，当裁判时知道规则，当执行者时不用。**
> 不是不懂——是「写出能跑的测试」这个任务压力把纪律顶掉了。
>
> **推论：纪律不能留给模型自觉。即便是一个已被证明懂这条规则的模型，在执行压力下也不会应用它。指令必须焊在产物里。**

## 对设计的结论

产物当前的写法 `THEN ⟨预期待定 · 源未定义 X；见待澄清 #N⟩` **只描述状态、不下达指令**。陌生 agent 读到它仍需自行决定怎么办，而「自行决定」的结果在模型间方差极大。

由于下游**不受控**（用户明确：下游 skill 未定、agent 可能是 Codex 也可能是别的），必须按 Codex 这一档设计：**闸焊在产物里，不指望读的人自觉。**

这与本仓 skill 自包含硬约束是同一个道理——**出厂产物面对的是你控制不了的世界。**

## 效力边界

- **n=1 任务**（xiaodaixing），且该任务已被用于 v0.1.4 改法设计，有污染。但本实验测的是**下游行为**，与上游改法的拟合无关，污染影响小。
- **没给代码库**。真实下游多半能读代码，可能从实现里读出「实际行为」从而合法地写特征化测试。本实验测的是**无处可查时的默认行为**，是下界不是均值。
- **两格提示词有一处差异**：Codex 那格加了「禁止编译/运行」一句（它首跑去编 Java 卡死、30 分钟零产出）。该约束与「面对待定会不会编造」无关，判断不影响可比性，但记此备查。
