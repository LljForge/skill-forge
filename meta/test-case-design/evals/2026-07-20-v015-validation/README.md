# v0.1.5 验证 · 两阶段（2026-07-20）

**第一次把「跑 skill」与「跑交接缝」串成一条链的验证。** 阶段 1 测规范本身，阶段 2 拿阶段 1 的产物测下游。

被测版本：v0.1.5 正本（代码作可选第二输入源、`⚠️ 歧义点` 取代大部分 `⟨预期待定⟩`、冲突单列上报）。此前该版本**零外部验证**——三轮静态复查各有发现且严重度未收敛，结论是「要提高置信度只能跑，不能再读」，本实验即那次跑。

## 判据（跑之前定死）

### 阶段 1 · 规范本身

| 观测点 | 通过 | 失败 |
|---|---|---|
| 冲突 | 抓到「只追加 vs 覆盖」（用户确认过的真冲突，留作探针） | 漏报 → Step 5/6 白写 |
| 现状标注 | 凡按代码填的，`来源` 追加 `代码 path:line` 且带 `⚠️ 歧义点` | 有裸填现状而不标 → 铁律 2 架空 |
| 门 B 误伤 | 含代码路径/现状描述的用例活着 | 被删 → 三次复查那个修没兜住 |
| 歧义点数量 | 与 v0.1.4 的 5 条待定同量级 | 爆到 20+ → 变成偷懒出口 |
| 提问数 | ≤ 冲突数（只报冲突，别的不问） | 又开始批量提问 → 回到中途版老路 |

### 阶段 2 · 交接缝

对每条 `⚠️ 歧义点` 与 `⟨预期待定⟩`：

| 归类 | 判定 |
|---|---|
| 保留「这是现状/未定，非契约」的语义（标记、注释、特征化测试、`@Disabled`） | ✅ |
| 默默当作产品契约写死断言 | ❌ 洗白 / 编造复活 |
| 整条跳过、不提 | ❌ 静默丢覆盖 |

外加：「待澄清」有没有以任何形式转达。

## 设计

- **任务**：小贷星管理（xiaodaixing-mgmt），沿用 [`../2026-06-26-xiaodaixing-mgmt/`](../2026-06-26-xiaodaixing-mgmt/) 的素材，便于跨版本对读。
- **阶段 1**：两格（Claude / Codex）各开干净会话，读 SKILL.md + `references/backend-api.md`，文档源为 requirements + design，**代码库可读**。产物落 `../2026-06-26-xiaodaixing-mgmt/{claude,codex}-skill-on-v015-final.md`。
- **阶段 2**：**只用 Claude 那一份产物**作输入（单一变量），分发给 Claude 与 Codex 两个下游执行者写 JUnit 测试。产物为本目录 `claude-tests-v015.md` / `codex-tests-v015.md`。
  提示词全程**不提「歧义点」「预期待定」**——一提就等于给了答案。不给代码库、不给需求文档。

## 阶段 1 结果：五条判据两格全过

| 观测点 | Claude | Codex |
|---|---|---|
| 抓到探针冲突 | ✅（原澄-1） | ✅（U1） |
| 现状必标注 | ✅ 11 条全带 `path:line` | ✅ 12 条全带 |
| 门 B 未误伤 | ✅ 自检明写「已排除注记行」 | ✅「仅检查 GIVEN/WHEN/THEN」 |
| 歧义点量 | 11 | 12（判据线为 20） |
| 提问数 ≤ 冲突数 | ✅ 2 问 / 2 冲突 | ✅ 4 问 / 4 冲突 |

三次复查改的两处（门 B 作用域声明、Step 5 四格路由）**各自在两个模型的出口自检里留下了痕迹**——不是没触发，是触发了并按新规则走。批量提问也真停了：从中途版的 10+ 问降到只问冲突。

歧义点 11/12 条虽高于 v0.1.4 的 5 条，但逐条核对均为真实的源沉默点（默认排序、匹配方式、组合语义、trim、千分位、事务回滚……），不是偷懒贴标签。**数量上升是把原本沉默的面显性化了，不是新增了含糊。**

### 阶段 1 暴露的真缺陷：冲突 / 歧义边界不收敛

同一份文档、同一份代码，**Codex 报 4 条冲突，Claude 报 2 条**。差额里至少一条是 Claude 判错：

- 异常场景 #2 白纸黑字写了「上传非 Excel → 提示**格式不支持**」
- 实现不单列该分支，统一返回「导入失败：\<异常\>」
- Codex 判冲突（U4）✅ / **Claude 判成「歧-10 · 源未定义后端的错误提示语」❌**

源明明定义了，被当成了沉默——这正是 Step 5 四格表最该防的一格。

**根因**：skill 给了「坐标 vs 答案」的机械判据（`把这个值换成另一个，算不算功能变了`），但**「文档到底有没有定义这件事」没有任何判据**，全靠模型读文档读得多细。四格表的行选择第一步就悬空。

旁证：Claude 自己嗅到了边界模糊——它把歧-3、歧-5 标成「**可能升格为冲突的候选**」并提醒优先看。知道模糊，但没有工具去判。

→ 已记 [BACKLOG A2](../../BACKLOG.md)。

另有一处分歧不算错，是判据的正常张力：`/page` → `/search` 端点改名，Claude 按「坐标」直接填死并注明，Codex 当冲突问了用户。倾向 Claude 对，但不硬气——端点路径变了，调用方是要改的。**判据在「端点路径」这一类上确实不锋利。**

## 阶段 2 结果：缝补上了

| | Claude | Codex | 对照：v0.1.4 产物喂 Codex |
|---|---|---|---|
| 歧义点语义转达 | 11/11，`歧-N` 编号回指 + `@Tag("impl-derived")` | **11/11**，风险栏点名 10 条用例：「预期来自规范记录的当前实现，不是已确认产品契约」 | — |
| 待澄清转达 | 3/3，`澄-N` 回指 | **5 项全转达**（含鉴权三态） | **0/5，全文无一处提及** |
| 无据处编造 | 0 | **0**——反推的信封 / 分页参数 / 字段名 / 列序全部列进「必须由团队核对的适配假设」 | **7+ 处写死** |
| 防假绿 | `@Disabled` + `Assumptions.assumeTrue` | `@Disabled` + `fail()` 兜底 | 无 |

上次 Codex 亲手写死的那两条（`$.code`、`records`/`total`），这次**照样写了，但公开挂在「假设 #2：规范只约定 `Result`，没给 JSON 结构」之下**。从偷偷编造变成公开记账——这正是「闸焊在产物里」要的效果。

**两格还都独立指出**歧-3 / 歧-5 与业务规则 R2「零丢失、不清洗」的语义张力，都建议产品负责人签字。上游标的「可能升格为冲突」，两个下游都接住了。

### 阶段 2 的副产物：下游反向审出了上游的问题

Claude 格指出规范里 TC-E-001「版本号递增」在冲突存档中无出处、TC-E-002「保持文件原有先后顺序」严格说不可测（表无序、同批 `import_time` 相同）；Codex 格独立提出了后者同一条。**两个下游独立指认同一条不可测断言**，这条比任何单侧判断都可信——`THEN` 的可观察性目前没有任何一道门在管。

# 追加 · A2 修复验证（v0.1.6，同日）

阶段 1 暴露的 A2（冲突/歧义边界不收敛）当轮就改了并复跑。产物：`claude-a2.md` / `codex-a2.md`。

## 改法（一条判据 + 一道闸）

根因不是「没读到文档」——**Claude 自己的事实集里就写着**「异常场景 #2：上传非 Excel → 提示格式不支持」，Step 5 判断时却没回去比对。**手里握着表在凭回忆答题。**

- **判据（Step 5）**：「文档有定义」＝ 事实集里存在一条**正面陈述了该点应然结果**的条目；判之前回查那份落盘的事实集，不凭印象。附正反例卡松紧——**只提及概念 ≠ 定义了结果**（防冲突数爆掉、退回批量提问老路）。
- **闸（门 A）**：出口**反向核**。逐条读歧义点表的「源未定义什么」，拿那句话回事实集搜；搜到正面陈述 → 是**漏报的冲突**，升格转 Step 6。

不往每条用例里加必填字段——那是沉积。出口机械反查一次就够。

## 判据（沿用上一轮跑前定死的三条）

| | 判什么 | 结果 |
|---|---|---|
| **必过** | 「格式不支持」判成冲突，不再是歧义点 | ✅ **两格都判成冲突**（Claude 澄-2 / Codex 澄-2） |
| **必不能坏** | Codex 上轮对的冲突一条不少 | ✅ 见下表 |
| **不能过头** | 冲突数不许爆到 10 | ✅ Claude 3 条 / Codex 4 条 |

### 「必不能坏」逐条对账

| 上轮 Codex 的冲突 | 本轮 Codex | 本轮 Claude |
|---|---|---|
| U1 只追加 vs 同键覆盖 | ✅ 澄-1 | ✅ 澄-1 |
| U4 格式不支持 vs 统一兜底 | ✅ 澄-2 | ✅ 澄-2 |
| U2 上半 · 文本 trim vs 不清洗 | ✅ 澄-3 | ❌ **未出现**（见下「诚实记录」） |
| U2 下半 · 空壳行跳过 vs 零丢失 | ✅ 澄-4 | ✅ 澄-3 |
| U3 `/page` → `/search` | 判为**坐标**、不计冲突 | 判为**坐标**、不计冲突 |

U3 这轮**两格收敛到坐标侧**（上轮 Codex 当冲突问了用户）。这是收敛不是丢失——上轮记录里就已倾向「Claude 对」。

## 收敛度：2 vs 4 → 3 vs 4，且集合高度重叠

改动前两模型的冲突集合差 2 条且有实质误判；改动后差 1 条，且**剩下那 1 条不是判据分歧**（见下）。四条实质冲突里有三条被两个模型独立、一致地认定。

## 闸确实开了火，且开在预料之外的地方

Claude 格出口三门记录：

> 客户与项目号均空的行被跳过，起初按「源未定义非数据行如何处理」记为歧义点；回查 R2「零丢失」与异常场景 #2「不丢行」是**正面陈述了应然结果**的条目 → **升格为冲突 澄-3**，已从歧义点表移出、转 Step 6。

值得注意：**它捞回的不是靶子那条**。靶子（格式不支持）在 Step 5 就被新判据直接判对了，反向核额外捞回了另一条降格。判据管前段、闸管漏网，两者各自生效，不是同一处的重复保险。

Codex 格也逐条留了反向核记录（歧-1..7 各自声明「没有正面应然陈述」），未捞出升格项——它这轮的 Step 5 判得更准。

## 诚实记录：Claude 这轮漏了 trim 这个点

Claude 格全文没有 trim / 首尾空白的任何条目——**既不是判成歧义点，也不是判成冲突，而是这个空位压根没进 Step 5 的清单**。

- 这**不是** A2 的失败：A2 治的是「空位已在手上、判错了行」；这是「空位没被列出来」，属 Step 5 空位清单的完备性问题。
- 上一轮 Claude 是有这条的（歧-5，还标了「可能升格为冲突」的候选）。**同模型同任务两轮之间的覆盖波动**，n=1，无法归因到本次改动。
- 未升为 BACKLOG 待办：单次波动、无第二个数据点，记在此处备查。若后续复现，再立项。

## 本轮的方法差异（与阶段 1 对读时注意）

两格均由 subagent 跑、**无法与用户交互**，Step 6 走「无法交互」降级出口，冲突全部保持 `⟨预期未定 · 冲突未裁定⟩`。阶段 1 那两份产物是用户当场裁决过的。这一差异**不影响 A2 判据**——判据看的是 Step 5 的行选择，发生在 Step 6 之前；反而剥掉了「用户裁决」这层混淆。

## 效力边界（重要）

- **Codex 的改善不全归功于设计。** 这版规范把上次「无处可查」的点用代码现状填掉了大半，它面对的真空本来就比上次小。能干净归因到设计的证据只有一条，但很硬：**它把 11 条歧义点的「这是现状不是契约」完整转达了**——这句话除了产物里那行注记，没有第二个来源。
- **Codex 零使用 `歧-N` / `澄-N` 编号**，转达全靠自然语言重述。语义没丢，但追溯性弱于 Claude：上游想反查「我标的哪条被下游怎么处理了」，在 Codex 这格做不到。
- **阶段 2 只喂了 Claude 那一份产物**。Codex 产物（12 歧义点 / 9 条 `⟨预期待定⟩`，风格更保守）未进下游，其交接表现未知。
- **n=1 任务**，且该任务参与过 v0.1.4 与 v0.1.5 的改法设计，有拟合污染。阶段 2 测的是下游行为，受污染较小；阶段 1 受污染较大。
- **没跑盲评 A/B**。本轮全部是对硬指标的直接核对，没有去问「v0.1.5 的产物比 v0.1.4 的更好用吗」。规范可用性的相对判断仍停在 v0.1.4 那两份 verdict。
