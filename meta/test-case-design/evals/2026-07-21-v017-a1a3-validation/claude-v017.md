### 待测事实集（本次用例的唯一来源，`来源` 字段回指此处编号）

> Step 1 归一化落盘。后续步骤只认这份事实集，不认原始输入。门 A / 门 A 反向核都回到这份表逐条核。

**验收标准**
- **AC-1**：上传合法 Excel（.xlsx），逐行解析为考勤记录并入库，返回成功导入的行数与被跳过的行。
- **AC-2**：导入完成后，考勤列表页展示本次导入的记录；**列表默认按 Excel 中的原始行顺序展示**，方便逐行核对。
- **AC-3**：支持按「工号」「打卡日期区间」「部门」分页查询，每页默认 20 条。
- **AC-4**：工时（下班时间 − 上班时间）由后端计算并存储，保留两位小数，不接受前端传入。

**业务规则**
- **BR-1**：工号必须是 8 位数字；非法工号所在行整行跳过，不阻断其余行导入，并在结果里报告跳过的行号。
- **BR-2**：同一工号 + 同一打卡日期视为同一条考勤；重复导入同一 (工号, 日期) 时以最后一次为准（覆盖旧值）。
- **BR-3**：上班时间晚于下班时间的行判为异常行，整行跳过并报告。
- **BR-4**：打卡日期必须落在查询所选月份内，跨月的行跳过并报告。

**异常场景**
- **EX-1**：上传非 .xlsx 文件 → 返回「仅支持 .xlsx 格式」，不入库。
- **EX-2**：Excel 无数据行（空表）→ 返回成功，导入行数为 0。
- **EX-3**：某行缺「打卡日期」必填列 → 该行跳过并报告，不阻断其余行。

**接口契约**
- **IF-1**：`POST /api/attendance/import`，multipart/form-data，字段 `file`。响应 `{ code, message, data: { imported:int, skipped:[{row:int, reason:string}] } }`；`code`：`0` 成功 / `4001` 文件格式错误 / `5000` 服务器内部错误。
- **IF-2**：`GET /api/attendance/page`，query `empNo`(可选) / `dateFrom` / `dateTo` / `dept`(可选) / `pageNo`(默认 1) / `pageSize`(默认 20)。响应 `{ code, message, data: { total:int, records:[AttendanceVO] } }`；`AttendanceVO`：`empNo, workDate, clockIn, clockOut, workHours, dept`。
- **IF-3**：数据模型 `t_attendance`（`emp_no VARCHAR(8) NOT NULL`、`work_date DATE NOT NULL`、`clock_in TIME`、`clock_out TIME`、`work_hours DECIMAL(5,2)`、`dept VARCHAR(64)`、`created_time DATETIME`、`UNIQUE KEY uk_emp_date (emp_no, work_date)`）。**表无「原始行号」列**；批量导入 `saveBatch`，同一批记录 `created_time` 相同。
- **IF-4**：`parseAttendanceRow(Row)` 是纯函数（5 列映射 + 工时计算 + 格式解析），**设计点名必须单测覆盖其映射正确性**，并对「工时计算」「非法日期格式」各补一个失败用例。

> **代码坐标已确认（属「坐标」，读到即填死、不标歧义）**：`parseAttendanceRow` 列序 = 第 0 列工号 / 第 1 列打卡日期(`yyyy-MM-dd`) / 第 2 列上班(`HH:mm`) / 第 3 列下班(`HH:mm`) / 第 4 列部门（`AttendanceServiceImpl.java:12-17`）；`validate` 校验项 = 工号 8 位 / 上班早于下班 / 必填 / 月份边界（`:29`）；`importExcel` 逐行 `rowIdx` 自 1 递增（`:25-30`）。

## 1. 功能测试 (TC-F)

#### TC-F-001 · 全合规行批量导入
- **维度**：§1 输入边界（有效类代表值）
- **来源**：AC-1 / IF-1
- **优先级**：高
- **GIVEN** 一个 .xlsx，全部数据行均合规（工号 8 位数字、上班早于下班、日期落在基准月份内、必填列齐全）
- **WHEN** 提交导入
- **THEN** 全部行入库成为考勤记录，返回体 `imported` 等于合规行数，`skipped` 为空，`code` 为 0

#### TC-F-002 · 导入后列表可检索到本次记录
- **维度**：D5 成功信封
- **来源**：AC-2 / IF-2
- **优先级**：高
- **GIVEN** 刚完成一批考勤导入
- **WHEN** 按对应条件查询考勤列表
- **THEN** 查询结果能检索到本次导入的全部记录（记录存在性，可观察）
- **THEN** ⟨预期未定 · 冲突未裁定 · 源：列表默认按 Excel 原始行顺序展示（AC-2） · 实现：查询未指定排序、表无原始行号列、同批 `created_time` 相同，返回顺序由存储引擎决定（`AttendanceServiceImpl.java:39`；IF-3） · 不得选定任一侧写死断言；见待澄清 澄-2⟩
  ⚠️ **可观察性存疑**：即便撇开冲突，「原始行顺序」在当前 schema 下也无确定观察点——表无原始行号列、同批 `created_time` 相同，无字段可承载或还原「原始行序」。改写不出可观察等价物 → 落待澄清 澄-2，不主动打断用户。

#### TC-F-003 · 按工号分页查询
- **维度**：D5 成功信封
- **来源**：AC-3 / IF-2
- **优先级**：高
- **GIVEN** 库中存在多个工号的考勤记录
- **WHEN** 指定某工号发起分页查询
- **THEN** 返回体 `data.records` 仅含该工号的记录，`data.total` 为该工号命中总数，每条为完整 `AttendanceVO`（`empNo/workDate/clockIn/clockOut/workHours/dept`）

#### TC-F-004 · 按打卡日期区间分页查询
- **维度**：§1 输入边界（日期区间）
- **来源**：AC-3 / IF-2
- **优先级**：高
- **GIVEN** 库中存在跨多个日期的考勤记录
- **WHEN** 指定 `dateFrom`–`dateTo` 区间发起查询
- **THEN** 返回体仅含打卡日期落在该闭/半开区间内的记录（区间端点归属见 澄-9）

#### TC-F-005 · 按部门分页查询
- **维度**：D5 成功信封
- **来源**：AC-3 / IF-2
- **优先级**：中
- **GIVEN** 库中存在多个部门的考勤记录
- **WHEN** 指定某部门发起查询
- **THEN** 返回体仅含该部门的记录

#### TC-F-006 · 工时后端计算并存储、不接受前端传入
- **维度**：§1 输入边界（数值精度）
- **来源**：AC-4 / IF-4
- **优先级**：高
- **GIVEN** 一行合规考勤，含上班与下班时间
- **WHEN** 导入该行
- **THEN** 入库记录的 `workHours` 等于「下班 − 上班」并保留两位小数（舍入模式见 澄-3）
- **THEN** 前端若随行携带工时值，不被采纳，最终以后端计算值为准

#### TC-F-007 · parseAttendanceRow 列映射正确性（纯函数单测缝隙）
- **维度**：§1 输入边界（映射正确性）
- **来源**：IF-4（设计点名必须单测覆盖）
- **优先级**：高
- **GIVEN** 一行含 5 个单元格（工号 / 打卡日期 `yyyy-MM-dd` / 上班 `HH:mm` / 下班 `HH:mm` / 部门）的原始输入
- **WHEN** 对该行求值产出 `AttendanceRecord`
- **THEN** 产出实体各字段与源单元格一一对应：工号取第 0 列、打卡日期取第 1 列并解析为日期、上班取第 2 列、下班取第 3 列、部门取第 4 列、工时为「下班 − 上班」两位小数
  （本条是 IF-4 点名的纯函数映射正确性，属「测什么」；换成手写单测即成立，不因其非 HTTP 端点层而算泄漏。）

#### TC-F-008 · 导入成功响应信封结构
- **维度**：D5 成功信封
- **来源**：IF-1 / AC-1
- **优先级**：中
- **GIVEN** 一次含合规行与被跳过行的导入
- **WHEN** 导入返回
- **THEN** 返回体结构为 `{ code, message, data:{ imported, skipped } }`，其中 `skipped` 每项含 `row` 与 `reason` 两个字段，`imported` 与 `skipped` 数量之和覆盖本次全部数据行

#### TC-F-009 · 查询成功信封 + 默认分页参数
- **维度**：D5 成功信封
- **来源**：AC-3 / IF-2
- **优先级**：中
- **GIVEN** 库中存在超过 20 条命中记录
- **WHEN** 查询时不传 `pageNo` 与 `pageSize`
- **THEN** 按默认第 1 页、每页 20 条返回；返回体结构为 `{ code, message, data:{ total, records } }`，`records` 长度不超过 20

## 2. 边界测试 (TC-E)

#### TC-E-001 · 工号恰 8 位数字（有效边界）
- **维度**：§1 输入边界（有效区间边界）
- **来源**：BR-1 / IF-3（`emp_no VARCHAR(8)`）
- **优先级**：高
- **GIVEN** 一行工号恰为 8 位纯数字
- **WHEN** 导入该行
- **THEN** 该行通过工号校验并入库

#### TC-E-002 · 工号长度越界（7 位 / 9 位）
- **维度**：§1 输入边界（无效类·长度）
- **来源**：BR-1
- **优先级**：高
- **GIVEN** 一行工号为 7 位数字，另一行工号为 9 位数字
- **WHEN** 导入
- **THEN** 两行均判为非法工号、整行跳过，各自进入 `skipped` 并报告其行号，不阻断其余合规行

#### TC-E-003 · 工号含非数字 / 为空（无效类·各原因单列）
- **维度**：§1 输入边界（无效类·类型/缺失）
- **来源**：BR-1
- **优先级**：高
- **GIVEN** 一行工号含字母或符号（如 `1234abcd`），另一行工号为空
- **WHEN** 导入
- **THEN** 两行均判为非法工号、整行跳过并报告，不阻断其余行
  （`reason` 具体文案未被源定义、`validate` 返回串不在片段——见 澄-8；本条只断言「进入 skipped 且指明工号原因」，不写死文案）

#### TC-E-004 · 跨次重复导入同一 (工号, 日期)
- **维度**：D1 幂等/去重
- **来源**：BR-2 / IF-3（`uk_emp_date`）
- **优先级**：高
- **GIVEN** 库中已存在 (工号 A, 日期 D) 的一条考勤
- **WHEN** 再次导入含 (工号 A, 日期 D) 但字段值不同的一行
- **THEN** ⟨预期未定 · 冲突未裁定 · 源：以最后一次为准、覆盖旧值（BR-2） · 实现：`saveBatch(ok)` 为标准批量插入、片段内未见任何针对 `uk_emp_date` 的 upsert/覆盖机制，冲突默认抛异常而非覆盖（`AttendanceServiceImpl.java:33`） · 不得选定任一侧写死断言；见待澄清 澄-1⟩

#### TC-E-005 · 同一批 Excel 内出现重复 (工号, 日期)
- **维度**：D1 幂等/去重
- **来源**：BR-2 / IF-3（`uk_emp_date`）
- **优先级**：高
- **GIVEN** 同一个 Excel 内有两行同为 (工号 A, 日期 D)、其余字段不同
- **WHEN** 导入该文件
- **THEN** ⟨预期未定 · 冲突未裁定 · 源：视为同一条、以最后一次为准（BR-2） · 实现：两行均进入 `ok` 后同批 `saveBatch`，`uk_emp_date` 冲突无 upsert 处置（`AttendanceServiceImpl.java:31-33`） · 不得选定任一侧写死断言；见待澄清 澄-1⟩

#### TC-E-006 · 上班时间等于下班时间（关系边界）
- **维度**：§1 输入边界（区间边界·相等点）
- **来源**：BR-3
- **优先级**：中
- **GIVEN** 一行上班时间与下班时间完全相等
- **WHEN** 导入该行
- **THEN** 该行按现状被判为异常行、整行跳过并报告
  ⚠️ **歧义点 歧-1**：BR-3 只定义「上班**晚于**下班」跳过，未定义「相等」如何处置。此预期取自 `validate` 校验「上班早于下班」的现状（`AttendanceServiceImpl.java:29`，相等即非「早于」→ 落入跳过）；确切比较运算符（`<` 还是 `<=`）不在片段。**非产品契约——实现改了才会红，实现错了不会红。**须人工审查工时为 0 的同点打卡应否入库。

#### TC-E-007 · 打卡日期落在月份边界
- **维度**：§1 输入边界（日期区间两侧各取一点）
- **来源**：BR-4
- **优先级**：中
- **GIVEN** 基准月份为某月，四行分别为：该月第 1 天、该月最后 1 天、上月最后 1 天、次月第 1 天
- **WHEN** 导入
- **THEN** 落在基准月份内的两行（月初、月末）通过月份校验；跨月的两行整行跳过并报告
  （「基准月份」在导入接口无对应入参、来源不明——见 澄-4；本条断言「月内保留 / 跨月跳过」的规则本身，基准月份一旦明确即可参数化实例化）

#### TC-E-008 · pageSize 边界取值
- **维度**：§1 输入边界（分页参数）
- **来源**：AC-3 / IF-2
- **优先级**：中

#### TC-E-009 · pageNo 边界取值
- **维度**：§1 输入边界（分页参数）
- **来源**：AC-3 / IF-2
- **优先级**：中
- **GIVEN** 库中命中记录共 N 页

#### TC-E-010 · 空表导入
- **维度**：§1 输入边界（集合大小 = 0）
- **来源**：EX-2
- **优先级**：中

- **THEN** 返回成功、`code` 为 0、`imported` 为 0、`skipped` 为空

#### TC-E-011 · 工时精度边界
- **维度**：§1 输入边界（数值精度）
- **来源**：AC-4 / IF-3（`work_hours DECIMAL(5,2)`）

- **WHEN** 导入
- **THEN** `workHours` 精确保留两位小数并可入库
  ⚠️ **歧义点/待定衔接**：两位小数的**舍入模式**（四舍五入 / 截断 / HALF_UP）源未定义、`calcHours` 实现不在片段——见 澄-3，不据此写死具体舍入结果

#### TC-E-012 · 部门字段长度边界

- **来源**：IF-3（`dept VARCHAR(64)`）

- **GIVEN** 一行部门名恰为 64 字符，另一行为 65 字符
- **WHEN** 导入
- **THEN** 64 字符正常入库；65 字符 ⟨预期待定 · 源未定义超长部门名的处置（截断 / 报错 / 跳过） · 现状未查得（落库对超长值的处置不在片段，DB 列约束为 `VARCHAR(64)`） · 不得据此写死断言；见待澄清 澄-11⟩

## 3. 错误处理 (TC-ERR)

#### TC-ERR-001 · 上传非 .xlsx 文件
- **维度**：D5 错误响应契约
- **来源**：EX-1 / IF-1
- **优先级**：高
- **GIVEN** 一个非 .xlsx 文件（如 .csv / .xls / 伪装扩展名）
- **WHEN** 提交导入
- **THEN** 返回 `code` 为 4001、`message` 为「仅支持 .xlsx 格式」语义，且不产生任何入库记录
  （源 EX-1 + IF-1 已定义该应然结果——文档有定义、代码未在片段中查得校验点，按文档优先填，不标歧义）

#### TC-ERR-002 · 非法工号行整行跳过、不阻断其余
- **维度**：D5 错误响应 / 部分失败容错
- **来源**：BR-1 / AC-1
- **优先级**：高
- **GIVEN** 一个 .xlsx，含若干合规行与至少一行非法工号
- **WHEN** 导入
- **THEN** 非法工号行整行跳过、进入 `skipped` 并报告其行号；其余合规行正常入库，`imported` 为合规行数

#### TC-ERR-003 · 上班晚于下班的异常行跳过
- **维度**：D5 错误响应 / 部分失败容错
- **来源**：BR-3
- **优先级**：高
- **GIVEN** 一个 .xlsx，含一行上班时间明确晚于下班时间
- **WHEN** 导入
- **THEN** 该行整行跳过、进入 `skipped` 并报告，其余行不受影响

#### TC-ERR-004 · 跨月行跳过
- **维度**：D5 错误响应 / 部分失败容错
- **来源**：BR-4
- **优先级**：中
- **GIVEN** 一个 .xlsx，含一行打卡日期不在基准月份内
- **WHEN** 导入
- **THEN** 该行整行跳过、进入 `skipped` 并报告
  （「基准月份」来源不明——见 澄-4，本条断言「跨月即跳过」的规则本身）

#### TC-ERR-005 · 缺「打卡日期」必填列的行跳过、不阻断其余
- **维度**：D5 错误响应 / 部分失败容错

## 4. 状态迁移 (TC-ST)

> **说明**：本任务数据模型 `t_attendance` 无显式状态字段（无 `status` 列、DDL 无状态机），因此不存在传统意义的合法/非法状态迁移。唯一可视为「迁移」的是一条 (工号, 日期) 记录的**存在性**：不存在 → 首次导入后存在 → 再次导入同键时按 BR-2 应「覆盖为最新」。故本区仅两条，且其预期与 D1/BR-2 冲突同源。这是如实反映领域，不是漏抽。

#### TC-ST-001 · (工号, 日期) 首次导入 → 新增
- **维度**：D1 去重（存在性迁移）
- **来源**：BR-2 / IF-3（`uk_emp_date`）
- **优先级**：中
- **GIVEN** 库中不存在 (工号 A, 日期 D) 的考勤
- **WHEN** 导入含 (工号 A, 日期 D) 的一行
- **THEN** 该键从「不存在」迁移到「存在」，库中新增一条对应考勤记录

#### TC-ST-002 · (工号, 日期) 已存在 → 再次导入应覆盖为最新
- **维度**：D1 去重（存在性迁移）
- **来源**：BR-2 / IF-3（`uk_emp_date`）
- **优先级**：高
- **GIVEN** 库中已存在 (工号 A, 日期 D) 的一条考勤
- **WHEN** 再次导入同键、字段值不同的一行
- **THEN** ⟨预期未定 · 冲突未裁定 · 源：以最后一次为准、覆盖为最新值、仍只一条（BR-2） · 实现：`saveBatch(ok)` 无 upsert 覆盖机制、`uk_emp_date` 冲突默认抛异常（`AttendanceServiceImpl.java:33`），「已存在 → 覆盖」这一迁移在现状下不成立 · 不得选定任一侧写死断言；见待澄清 澄-1⟩
