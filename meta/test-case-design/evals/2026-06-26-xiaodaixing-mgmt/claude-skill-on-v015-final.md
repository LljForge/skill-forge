### 待测事实集（本次用例的唯一来源，`来源` 字段回指此处编号）

**验收标准**（requirements §4）

- **验收标准 #1**：新表 `tjd_base_xiaodaixing_project` 建好（增量 DDL 入 `docs/sql/increment/`），含 36 业务列 + 批次列 + 审计/租户/逻辑删除/版本列。
- **验收标准 #2**：biz_web dev 环境「业务办理」下出现「小贷星管理」菜单，点进去是查询页。
- **验收标准 #3**：上传 `小贷星项目清单（20260531）.xlsx` → 66 行全部入库（含脏值保真），同批 `数据日期=20260531`、同一 `batch_no`。
- **验收标准 #4**：列表分页可用，常用筛选项可查。
- **验收标准 #5**：产出 prod 菜单接入操作说明文档（父节点=业务办理、路由 `/slp/xiaodaixing`、编码等）。
- **验收标准 #6**：后端可离线编译通过，无新增 Maven 依赖。

**业务规则**（requirements §3 + design ② ADR / ③ 五坑）

- **业务规则 R1**：菜单挂载 —— dev 环境改前端 mock 即可见；prod 菜单权威在仓外 `qfc-base-auth`，本工作区只出接入说明、不配 prod 菜单。
- **业务规则 R2**：表结构 1:1 保真 —— 保留 xlsx 全部 36 列、**不清洗**；重复列名「核销金额」加后缀区分（`write_off_amount` / `write_off_amount_2`）；空表头列起占位名；`是否逾期` 等脏值列原样入 `varchar`；**零丢失**。
- **业务规则 R3**：导入 = **只追加不去重** + 批次概念 —— 每次导入只追加、**不按项目号去重**；以 Excel「数据日期」列（仅首行有值，导入时 fill-down 到整批）作业务批次键；另加 `batch_no`（每次导入一个 UUID）+ `import_time` 兜底。
- **业务规则 R4**：查询 = 分页 + 常用筛选（客户名 / 项目号 / 所属机构 / 是否逾期 / 项目评级 / 数据日期(批次)）。
- **业务规则 R5**：复用既有依赖与范式 —— 表落 `base-service`、表名前缀 `tjd_`、导入复用已有 EasyExcel 4.0.3（零新引依赖）；前端复用 QmTable/QmForm/QmUploadFile 范式。
- **业务规则 R6**（design ADR-4 + requirements §6 风险②）：`数据日期` 首行也为空时，`data_date` 用上传时间兜底。
- **业务规则 R7**（design ADR-5）：`create_user` 显式取 `AppContext.getUserId()`，不只靠框架自动填充。
- **业务规则 R8**（design ③ 坑3）：查询/分页**手写 `is_deleted=0`** 条件，不靠自动逻辑删除（本库无 `@TableLogic`）。
- **业务规则 R9**（design ADR-2）：Excel 按 `@ExcelProperty(index=0..35)` **按列序映射**、全 String 读入，Service 内再转类型（避免脏值解析报错）。
- **业务规则 R10**（design ③ 36 列映射表）：idx → 列名 → 类型对照；金额清晰列 `decimal(24,6)`，脏值/日期序列号/比例/核销金额列 `varchar` 保真。
- **业务规则 R11**（design ③ 坑4）：前端上传绕过 axios 拦截器，须显式携带 `getConfigHeaders()`。
- **业务规则 R12**（requirements §5 非目标）：不做数据清洗；不做单条增/改/删；不做导入去重/校验/纠错；不做导出；不在本工作区配 prod 菜单。

**异常场景**（requirements §2.2 + §6 风险）

- **异常场景 #1**：上传空文件 → 提示「文件不能为空」，不入库。
- **异常场景 #2**：上传非 Excel（非 .xls/.xlsx）→ 提示「格式不支持」，不入库。
- **异常场景 #3**：Excel 含脏数据（重复列名、空表头列、`是否逾期` 为日期拼接脏值）→ **原样保真入库、不报错、不丢行**。
- **异常场景 #4**（风险①）：月度重复导入同一份清单 → 产生重复行（R3 只追加的预期代价），用 `batch_no` 区分批次缓解。
- **异常场景 #5**（风险②）：某月 Excel「数据日期」首行也为空 → 该批次业务日期缺失，导入时需容错兜底。

**接口契约**（design ① 数据流 + ② 深度自陈）

- **接口契约 #1**：`POST /xiaodaixing/tjdBaseXiaodaixingProject/importExcel`（multipart，参数 `file`）→ `Result`。
- **接口契约 #2**：`POST /xiaodaixing/tjdBaseXiaodaixingProject/search`（`SearchVO` → 分页结果）。
  > 坐标校正：design ① 写作 `/page`，实现与前端调用均为 `/search`（`TjdBaseXiaodaixingProjectController.java:56`、`biz_web/src/modules/slp/api/xiaodaixing/index.js`）。端点改名不改变功能语义，按铁律 2「坐标」判据读到即填死，不计为冲突。
- **接口契约 #3**：前端路由 `/slp/xiaodaixing`。
- **接口契约 #4**：对外暴露面 = 3 个（导入 + 查询 + 前端路由），无其它端点。

## 1. 功能测试 (TC-F)

#### TC-F-001 · 导入样例月度清单，全部数据行入库
- **维度**：D5 成功信封
- **来源**：验收标准 #3 / 业务规则 R3
- **优先级**：高
- **GIVEN** 一份符合月度清单格式、含 66 条项目数据行的小贷星 xlsx 清单，其中不存在「数据日期 + 项目号」相同的重复行、也不存在客户与项目号同时为空的空壳行，且台账中尚无该批次数据
  > 场景限定的由来：在用户裁决确立的覆盖语义（澄-1、澄-2 已裁）下，「66 行全部入库」只在清单本身无同组合键重复行、无空壳行时成立；该限定使本条与 TC-E-001/002/005 不相矛盾。
- **WHEN** 运营人员通过导入端点提交该清单
- **THEN** 导入以成功结果返回，且成功信息中告知本次处理的条数
- **THEN** 台账中该批次可查得 66 条项目记录

## 2. 边界测试 (TC-E)

#### TC-E-001 · 重复导入同一份清单
- **维度**：D1 幂等/去重
- **来源**：异常场景 #4 / 业务规则 R3 / **用户裁决**（2026-07-20：导入语义以实现的「按数据日期 + 项目号覆盖」为准，推翻 R3「只追加不去重」）
- **优先级**：高
- **GIVEN** 台账中已存在由某份月度清单导入的一批记录
- **WHEN** 将完全相同的那份清单再次导入
- **THEN** 台账中该批次的记录条数不变，同一项目号不产生重复行
- **THEN** 这些记录的批次号与导入时间被刷新为第二次导入的值
- **THEN** 这些记录的版本号在原值基础上递增一次

#### TC-E-002 · 同一文件内出现相同数据日期 + 项目号的两行
- **维度**：D1 幂等/去重
- **来源**：业务规则 R3 / **用户裁决**（2026-07-20：同上，文件内同组合键去重以实现为准）
- **优先级**：高
- **GIVEN** 一份清单，其中两行的项目号相同（同属一个数据日期批次），且其余列取值不同
- **WHEN** 导入该清单
- **THEN** 该项目号最终只入库一条记录，内容取文件中靠后那一行
- **THEN** 其余项目号的记录保持文件中的原有先后顺序

#### TC-E-003 · 数据日期整列为空时的批次日期兜底
- **维度**：§1 输入边界
- **来源**：异常场景 #5 / 业务规则 R6
- **优先级**：高
- **GIVEN** 一份清单，其「数据日期」列所有行（含首行）均为空
- **WHEN** 导入该清单
- **THEN** 导入不失败，本批记录仍带有非空的数据日期，取值为本次上传当日的日期
  ⚠️ **歧义点 歧-1**：源（业务规则 R6 / 异常场景 #5）只说「用上传时间兜底」，未定义兜底值的粒度与格式；此预期取自当前实现——以导入时间格式化为 `yyyyMMdd` 的 8 位日期（`TjdBaseXiaodaixingProjectService.java:259-261`），**非产品契约**，它不会发现此处的缺陷：实现改了它才会红，实现错了它不会红。须人工审查确认。

#### TC-E-004 · 只有表头、无数据行的清单
- **维度**：§1 输入边界
- **来源**：业务规则 R3 / 接口契约 #1

- **GIVEN** 一份格式合法但仅含表头、无任何数据行的 xlsx
- **WHEN** 导入该文件
- **THEN** 导入以成功结果返回，告知处理条数为 0，台账记录数不变

#### TC-E-005 · 客户与项目号均为空的空壳行
- **维度**：§1 输入边界
- **来源**：异常场景 #3 / 业务规则 R2 / **用户裁决**（2026-07-20：空壳行处置以实现的「跳过不入库」为准，R2「不丢行」不覆盖此情形）
- **优先级**：高
- **GIVEN** 一份清单，其中夹有若干「客户」与「项目号」两列同时为空、其余列或有零星残留值的行
- **WHEN** 导入该清单
- **THEN** 这些空壳行不入库，其残留值一并被丢弃
- **THEN** 其余数据行正常入库，导入不因空壳行报错，成功信息中的处理条数不计入被跳过的行

#### TC-E-006 · 金额列为空白或不可解析文本
- **维度**：§1 输入边界
- **来源**：业务规则 R2 / R10 / 异常场景 #3
- **优先级**：高
- **GIVEN** 一份清单，其中某几行的合同金额、放款金额等数值列为空白，另几行为无法解析成数字的文本
- **WHEN** 导入该清单
- **THEN** 导入不因该值中断，整行仍然入库
- **THEN** 该金额字段最终为空值（不保留原始文本）
  ⚠️ **歧义点 歧-3**：源（业务规则 R2「零丢失」）与「金额列用 decimal」并存，未定义金额列出现非数字时应置空、原样保留还是拒绝该行；此预期取自当前实现——不可解析时置为空值并记警告日志（`TjdBaseXiaodaixingProjectService.java:315-330`），**非产品契约**，它不会发现此处的缺陷。须人工审查确认。**注意此现状与「零丢失」在语义上有张力**（原始文本未被保留）。

#### TC-E-007 · 金额列带千分位分隔符或空白字符
- **维度**：§1 输入边界
- **来源**：业务规则 R10 / 异常场景 #3
- **优先级**：中
- **GIVEN** 一份清单，其中金额列的值形如带半角/全角千分位逗号、含前后空格或不间断空格的数字
- **WHEN** 导入该清单
- **THEN** 该金额被识别为对应数值入库，而非被判为不可解析而置空

## 3. 错误处理 (TC-ERR)

#### TC-ERR-001 · 上传空文件
- **维度**：D5 错误响应契约
- **来源**：异常场景 #1
- **优先级**：高
- **GIVEN** 一个内容为空的文件
- **WHEN** 提交该文件到导入端点
- **THEN** 导入以失败结果返回，失败信息为「文件不能为空！」
- **THEN** 台账记录数不变，无任何数据入库

#### TC-ERR-002 · 前端拦截非 Excel 扩展名
- **维度**：无领域维度（前端）
- **来源**：异常场景 #2

- **WHEN** 试图通过页面的导入入口选择该文件上传
- **THEN** 页面拒绝该文件、给出格式不支持的提示，文件不被送达导入端点

#### TC-ERR-003 · 非 Excel 内容直达导入端点
- **维度**：D5 错误响应契约
- **来源**：异常场景 #2 / 接口契约 #1
- **优先级**：中
- **GIVEN** 一个非空但内容并非合法 Excel 工作簿的文件
- **WHEN** 绕过页面直接提交该文件到导入端点
- **THEN** 导入以失败结果返回，失败信息可辨识为导入失败
- **THEN** 台账记录数不变，无任何数据入库
  ⚠️ **歧义点 歧-10**：源（异常场景 #2）要求提示「格式不支持」，但未指明该校验落在前端还是后端；后端侧现状为——不校验扩展名与内容类型，仅在解析抛错后统一返回「导入失败：<异常消息>」（`TjdBaseXiaodaixingProjectController.java:38-50`），提示语与源描述的「格式不支持」不一致。此预期取自当前实现，**非产品契约**，它不会发现此处的缺陷。须人工审查确认后端是否需补扩展名校验与专用提示。

#### TC-ERR-004 · 导入中途写库失败时的事务边界
- **维度**：D3 事务边界/回滚补偿
- **来源**：验收标准 #3 / 业务规则 R3
- **优先级**：高
- **GIVEN** 一份含多条数据行的清单，其中靠后的某一行在写库阶段会触发失败
- **WHEN** 导入该清单
- **THEN** 本次导入整批回滚，台账中不残留该批次的任何部分记录
- **THEN** 调用方收到失败结果而非成功结果
  ⚠️ **歧义点 歧-11**：源未定义导入的事务边界（部分失败时整批回滚还是留下已写入部分）；此预期取自当前实现——导入方法声明了对任意异常回滚的事务（`TjdBaseXiaodaixingProjectService.java:142`），**非产品契约**，它不会发现此处的缺陷。须人工审查确认。
