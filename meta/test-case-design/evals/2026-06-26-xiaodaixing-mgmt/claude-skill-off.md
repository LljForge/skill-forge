# 测试用例 — 小贷星管理（xiaodaixing-mgmt）

> 依据：`requirements.md` + `design.md`（两份文档为唯一输入）。
> 被测范围：base-service 两个后端端点（`/importExcel` 导入、`/page` 分页查询）+ 新表 `tjd_base_xiaodaixing_project` + biz_web 薄查询页 + 菜单接入。
> 端点前缀：`/api-zjq/xiaodaixing/tjdBaseXiaodaixingProject`（后端 8093）。

## 0. 说明与约定

### 0.1 用例编号规则
- `IMP-xxx`：导入（importExcel）功能
- `MAP-xxx`：Excel→PO 映射/批次填充纯逻辑（对应 design ④ 测试缝隙 `toPOList`，可单测）
- `QRY-xxx`：分页查询/筛选（page）
- `DDL-xxx`：表结构 / 建表脚本
- `MENU-xxx`：菜单接入（dev mock / prod 说明）
- `FE-xxx`：前端查询页
- `NFR-xxx`：非功能（编译、依赖、性能、安全、并发）
- `E2E-xxx`：端到端验收（贴合需求 §4 验收标准）

### 0.2 优先级
- P0 = 核心主流程 / 验收硬指标，必过；P1 = 重要分支/异常；P2 = 边界/健壮性；P3 = 增强项。

### 0.3 关键测试数据准备
- **标准样例**：`小贷星项目清单（20260531）.xlsx`，66 行、36 列，首行「数据日期」=`20260531`（第 2 行起该列为空），含脏值列（`是否逾期`=日期拼接、`本金/利息逾期起始日`=Excel 日期序列号、`核销金额`×2 可能非数字、`准备金计提比例`=`0.6` 形态、`(空表头)`列样例值`信用贷款`）。
- 需自造的变体样例见各用例「前置/数据」。

## 1. 导入 importExcel（IMP）

### IMP-001 【P0】标准清单导入成功、66 行全部入库
- 前置：库空（或记录已知基线行数 N）。
- 步骤：调用 `/importExcel` 上传标准样例 `小贷星项目清单（20260531）.xlsx`。
- 预期：
  1. 返回 `Result` 成功（success=true）。
  2. `tjd_base_xiaodaixing_project` 新增行数恰为 66（`count = N + 66`）。
  3. 无异常、无丢行、无脏值导致的解析中断。

### IMP-002 【P0】数据日期 fill-down —— 整批同一 data_date
- 前置：标准样例（仅首行「数据日期」=20260531，其余行空）。
- 步骤：导入后查询本批全部行的 `data_date`。
- 预期：66 行 `data_date` 均为 `20260531`（首个非空值 fill-down 到整批），无一行为空。

### IMP-003 【P0】批次键 batch_no —— 同批同一 UUID
- 步骤：导入标准样例，查询本批 66 行的 `batch_no`。
- 预期：66 行 `batch_no` 完全一致，且为一个非空 UUID 形态字符串。

### IMP-004 【P0】import_time 填充且同批一致（或落在同一次导入的时间窗内）
- 步骤：导入后查询本批 `import_time`。
- 预期：每行 `import_time` 均非空、为本次导入时刻；同批时间一致或极小差内。

### IMP-005 【P1】create_user 显式填充（ADR-5）
- 前置：以已登录运营账号 U 的上下文发起导入（请求期 `AppContext.getUserId()` 可取）。
- 步骤：导入后查询本批任一行 `create_user`。
- 预期：`create_user = U 的 userId`（**非 NULL**）——验证显式 set 生效，未依赖框架自动填充（模板表历史行 create_user 全 NULL 是反例）。

### IMP-006 【P0】只追加不去重 —— 同一份清单重复导入产生重复行（R3）
- 前置：已完成 IMP-001（库中 66 行，batch_no=B1）。
- 步骤：**再次**上传同一份 `小贷星项目清单（20260531）.xlsx`。
- 预期：
  1. 导入成功，不按项目号去重、不报唯一冲突。
  2. 总行数变为 132（追加 66）。
  3. 新批 66 行 `batch_no=B2`（≠B1）；两批 `data_date` 同为 20260531，靠 batch_no 区分。

### IMP-007 【P1】多月份多批次并存、可按批次区分
- 步骤：先导 20260531 清单，再导一份首行数据日期=`20260630` 的清单。
- 预期：库中两批共存，`data_date` 分别为 20260531 / 20260630，各自 batch_no 不同，互不影响。

### IMP-008 【P1】上传空文件被拒、不入库
- 步骤：上传 0 字节文件 / 空 multipart。
- 预期：返回失败并提示「文件不能为空」类信息；库中行数不变。

### IMP-009 【P1】非 Excel 格式被拒（后缀非 .xls/.xlsx）
- 数据：`x.txt`、`x.csv`、`x.pdf`、`x.png`。
- 步骤：逐一上传。
- 预期：返回失败并提示「格式不支持」类信息；不入库。

### IMP-010 【P2】伪装后缀文件（改名为 .xlsx 的非 Excel 内容）
- 数据：将 txt 改名 `fake.xlsx` 上传。
- 预期：EasyExcel 解析失败被捕获，返回可读错误、不入库、不 500 裸抛栈。

### IMP-011 【P0】脏值列原样保真入库、不报错、不丢行（R2 / 关键异常）
- 前置：标准样例（含脏值）。
- 步骤：导入后逐列核对脏值行。
- 预期（原样字符串保真，不清洗、不抛异常）：
  1. `is_overdue`（是否逾期，日期拼接脏值）→ 原文入 varchar(32)。
  2. `principal_overdue_start` / `interest_overdue_start`（Excel 日期序列号）→ 原样字符串，不被转成日期。
  3. `write_off_amount` / `write_off_amount_2`（可能非数字）→ 原样字符串入 varchar(64)，不因非数字而丢行或报错。
  4. `reserve_ratio`（如 `0.6` 比例）→ 原样字符串入 varchar，不丢精度/不四舍五入。

### IMP-012 【P1】重复列名「核销金额」两列各就其位、互不覆盖（ADR-2）
- 前置：xlsx 第 16 列与第 30 列均名「核销金额」。
- 步骤：导入后取一行，核对 `write_off_amount`（idx16）与 `write_off_amount_2`（idx30）。
- 预期：两列分别取到 xlsx 第 16、30 列的原值，**未相互覆盖**（按 index 映射生效，未按列名解析导致覆盖/崩溃）。

### IMP-013 【P1】空表头列按占位名入库
- 前置：xlsx 第 34 列表头为空。
- 步骤：导入后取该列 `guarantee_way_detail`。
- 预期：值为 xlsx 第 34 列原值（样例`信用贷款`），列存在且可查，不因空表头崩溃。

### IMP-014 【P1】金额列 null 安全 / 空单元格
- 数据：造一行使 `contract_amount` 等 decimal 列为空单元格。
- 步骤：导入。
- 预期：该 decimal 列入库为 NULL（或按约定 0），**不抛 NumberFormatException**、不丢行，其余列正常。

### IMP-015 【P2】金额列含千分位/货币符号/前后空格
- 数据：`合同金额` 单元格为 `1,234.56`、`￥1000`、` 100 `。
- 步骤：导入。
- 预期：按 toPO 转换约定处理——若无法转 decimal 则 null 安全兜底且不丢行（明确观察实际落库值，作为回归基线记录）。

### IMP-016 【P2】首行「数据日期」为空 → 上传时间/兜底逻辑（需求风险②）
- 数据：造一份所有行「数据日期」列均为空的清单。
- 步骤：导入。
- 预期：`data_date` 用兜底（上传时间/文件名）填充、非空、整批一致；导入不失败。

### IMP-017 【P2】仅表头、无数据行的 Excel
- 步骤：上传只有表头的 xlsx。
- 预期：导入成功、入库 0 行、不报错、返回可辨识结果（如导入 0 条）。

### IMP-018 【P2】列数少于/多于 36 的 Excel
- 数据：分别造 30 列、40 列的 xlsx。
- 预期：
  - 少列：缺失的高 index 列入库为 null，不越界报错。
  - 多列：多出的列被忽略（仅取 index 0..35），不报错、不错位。

### IMP-019 【P2】行内单元格换行 / 特殊字符 / emoji / 超长文本
- 数据：某文本列含换行、`<script>`、超出 varchar 长度的超长串。
- 预期：常规特殊字符原样入库；超长串按 DB 列长处理（截断或报可读错误，记录实际行为，不静默丢整行）。

### IMP-020 【P1】.xls（旧格式）与 .xlsx 均可导入
- 数据：同内容分别存 .xls 和 .xlsx。
- 预期：两者均能解析入库、结果一致（需求 §2.2 允许 .xls/.xlsx）。

### IMP-021 【P2】导入过程异常时的原子性/一致性
- 数据：造中途某行触发不可恢复异常的清单（如超严重结构损坏）。
- 预期：明确约定并验证——要么整批回滚不留半批（saveBatch 事务），要么给出清晰结果；不留下无 batch_no 的孤儿脏数据。

### IMP-022 【P3】多租户 tenant_id 填充
- 前置：带租户上下文导入。
- 预期：本批行 `tenant_id` 按当前租户填充、非空且一致。

## 2. Excel→PO 映射与批次填充纯逻辑（MAP，可单测，对应 design ④ toPOList）

> design ④ 明确唯一测试缝隙 = `toPOList(List<Excel>, ctx)` 纯函数，不 mock。以下为该函数单测。

### MAP-001 【P0】fill-down：首行有 data_date + 后续行空 → 整批一致
- 输入：3 行 Excel DTO，第 1 行 data_date=20260531，第 2、3 行该列空。
- 断言：输出 3 个 PO 的 data_date 全为 20260531。

### MAP-002 【P0】36 列各就位、index 顺序不错位
- 输入：一行 index0..35 分别填可辨识标记值。
- 断言：36 个 PO 字段逐一等于对应 index 的输入值（重点核 idx16→write_off_amount、idx30→write_off_amount_2、idx34→guarantee_way_detail）。

### MAP-003 【P0】金额列 null 安全
- 输入：decimal 类列传 null / 空串 / 非数字。
- 断言：不抛异常，落 null（或约定默认值），其余字段不受影响。

### MAP-004 【P1】脏值列全 String 透传
- 输入：is_overdue=日期拼接串、principal_overdue_start=日期序列号串、reserve_ratio=`0.6`。
- 断言：对应 PO 字段值 == 输入原串，未做任何转换。

### MAP-005 【P1】batch_no / import_time / create_user 一致填充
- 输入：多行 + ctx(userId=U)。
- 断言：所有 PO 的 batch_no 同一 UUID、import_time 同一值、create_user=U。

### MAP-006 【P2】首行 data_date 也为空 → 兜底
- 输入：全部行 data_date 空 + ctx 带上传时间。
- 断言：所有 PO data_date = 兜底值、非空、一致。

### MAP-007 【P2】空列表输入
- 输入：空 List。
- 断言：返回空 List，不 NPE。

### MAP-008 【P2】fill-down 取「首个非空」而非「第一行」
- 输入：第 1 行 data_date 空、第 2 行=20260531、第 3 行空。
- 断言：整批 data_date=20260531（验证是「首个非空」语义）。

## 3. 分页查询 page（QRY）

### QRY-001 【P0】默认分页查询返回台账列表
- 前置：库中已导入 ≥1 批数据。
- 步骤：POST `/page`（默认页码/页大小）。
- 预期：返回分页结构（含 records/total/current/size 或等价字段），records 非空。

### QRY-002 【P0】分页正确性（页码、页大小、总数）
- 前置：库中 66 行单批。
- 步骤：pageSize=10 逐页翻。
- 预期：total=66、共 7 页、第 7 页 6 条；各页无重复无遗漏、并集=66。

### QRY-003 【P0】仅返回未逻辑删除数据（手写 is_deleted=0，坑3）
- 前置：将某批中若干行 `is_deleted` 置 1。
- 步骤：`/page` 查询。
- 预期：is_deleted=1 的行**不出现在结果**、total 相应扣减（验证查询手写 is_deleted=0 条件，不依赖自动逻辑删除）。

### QRY-004 【P1】按客户名筛选（R4）
- 步骤：`customer=某客户名`。
- 预期：仅返回 customer 匹配的行（模糊/精确按实现约定，记录并回归）。

### QRY-005 【P1】按项目号筛选
- 步骤：`project_no=某项目号`。
- 预期：仅返回该项目号的行（重复导入场景下会跨批次返回多行）。

### QRY-006 【P1】按所属机构筛选
- 步骤：`org_platform=某机构`。
- 预期：仅返回该机构行。

### QRY-007 【P1】按是否逾期筛选
- 步骤：`is_overdue=某值`。
- 预期：仅返回匹配行（注意该列为脏值 varchar，验证筛选对原样值的匹配行为）。

### QRY-008 【P1】按项目评级筛选
- 步骤：`project_rating=某评级`。
- 预期：仅返回该评级行。

### QRY-009 【P0】按数据日期（批次）筛选
- 前置：库中 20260531、20260630 两批。
- 步骤：`data_date=20260531`。
- 预期：仅返回 20260531 批次行，不含 20260630。

### QRY-010 【P1】多条件组合筛选（AND）
- 步骤：`org_platform=X` + `is_overdue=Y` + `data_date=20260531`。
- 预期：返回同时满足全部条件的交集行。

### QRY-011 【P2】筛选无匹配
- 步骤：传不存在的客户名。
- 预期：返回空 records、total=0，不报错。

### QRY-012 【P2】筛选条件全空 = 全量分页
- 步骤：不传任何筛选。
- 预期：等价默认全量分页，与 QRY-001 一致。

### QRY-013 【P2】分页边界参数
- 数据：page=0 / 负数 / 超大页码 / pageSize=0 / pageSize 超大。
- 预期：按框架默认容错——非法值取默认或返回空页，不 500、不全表无分页返回。

### QRY-014 【P2】筛选值含 SQL 特殊字符 / 注入尝试
- 数据：`customer=' OR '1'='1`、含 `%` `_` 通配符。
- 预期：参数化查询，不发生注入、不越权返回全量；通配符按转义约定处理。

### QRY-015 【P2】按批次 batch_no 区分/查询（若支持）
- 步骤：以 batch_no 作为查询/展示维度。
- 预期：能定位某一次导入批次的数据（R3 批次管理目的）。

### QRY-016 【P3】排序（若列表支持）
- 步骤：按 import_time / data_date 排序。
- 预期：结果按指定列有序；默认排序稳定（建议按 import_time 或 id）。

### QRY-017 【P2】跨批次同项目号返回多行（呼应只追加）
- 前置：同一清单导入两次。
- 步骤：按该项目号查询。
- 预期：返回 2 行（不同 batch_no），验证只追加不去重在查询侧的可见性。
