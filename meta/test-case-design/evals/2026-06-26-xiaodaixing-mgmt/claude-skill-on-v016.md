### 待测事实集（本次用例的唯一来源，`来源` 字段回指此处编号）

**验收标准**（requirements 「验收标准」节）

- **验收标准 #1**：新表 `tjd_base_xiaodaixing_project` 建好（增量 DDL 入 `docs/sql/increment/`），含 36 业务列 + 批次列 + 审计/租户/逻辑删除/版本列。
- **验收标准 #2**：biz_web dev 环境「业务办理」下出现「小贷星管理」菜单，点进去是查询页。
- **验收标准 #3**：上传 `小贷星项目清单（20260531）.xlsx` → 66 行全部入库（含脏值保真），同批 `数据日期=20260531`、同一 `batch_no`。
- **验收标准 #4**：列表分页可用，常用筛选项可查。
- **验收标准 #5**：产出 prod 菜单接入操作说明文档（父节点=业务办理、路由 `/slp/xiaodaixing`、编码等）。
- **验收标准 #6**：后端可离线编译通过，无新增 Maven 依赖。

**业务规则**（requirements 「业务规则」节 R1–R5 + design ADR 派生）

- **业务规则 R1**：dev 环境菜单来自前端 mock；prod 菜单权威在仓外 `qfc-base-auth`，本次只产接入说明、不在本工作区配 prod 菜单。
- **业务规则 R2**：表结构 = 尽量 1:1 保真，保留 xlsx 全部 36 列、**不清洗**；重复列名「核销金额」加后缀区分（`write_off_amount` / `write_off_amount_2`）、空表头列起占位名、`是否逾期` 等脏值列原样入 `varchar`、**零丢失**。
- **业务规则 R3**：导入 = 只**追加**不去重（不按项目号去重）；批次 = 以 Excel「数据日期」列（该列仅首行有值，导入时 fill-down 到整批所有行）作业务批次键，另加 `batch_no`（每次导入一个 UUID）+ `import_time` 兜底。
- **业务规则 R4**：查询 = 分页 + 常用筛选，筛选项含 客户名 / 项目号 / 所属机构 / 是否逾期 / 项目评级 / 数据日期(批次) 等。
- **业务规则 R5**：复用既有依赖与范式；导入复用已有 EasyExcel 4.0.3，零新引依赖。
- **业务规则 R6**（design ADR-2）：Excel 按 `@ExcelProperty(index=0..35)` **位置**映射（否决按列名映射，因存在重复列名与空表头列）；**全 String 读入**后在 Service 转类型，避免脏值解析报错。
- **业务规则 R7**（design ADR-4）：`数据日期` 在 ReadListener 里取**首个非空值** fill-down 到整批；**首行也空时** `data_date` 用**上传时间**兜底。
- **业务规则 R8**（design ADR-5）：导入时 `create_user` 显式取当前登录用户（`AppContext.getUserId()`），不只靠框架自动填充。
- **业务规则 R9**（design ③ 坑3）：查询/分页**手写 `is_deleted=0`** 条件，不靠自动逻辑删除。
- **业务规则 R10**（design ④ 测试缝隙）：「Excel 行 → PO」纯转换须类型转换 **null 安全**（金额列），并完成 `数据日期` fill-down + `batch_no`/`import_time`/`create_user` 填充。
- **业务规则 R11**（requirements 2.1 主流程第 2 步）：运营人员上传月度清单、系统解析入库后，**列表可见新批次数据**；一次上传 = 一个批次。

**异常场景**（requirements 「关键异常」+「依赖与风险」）

- **异常场景 #1**：上传空文件 → 提示「文件不能为空」，不入库。
- **异常场景 #2**：上传非 Excel（非 `.xls`/`.xlsx`）→ 提示「格式不支持」，不入库。
- **异常场景 #3**：Excel 含脏数据（重复列名、空表头列、`是否逾期` 为日期拼接脏值）→ **原样保真入库、不报错、不丢行**。
- **异常场景 #4**（风险①）：月度重复导入同一份清单会产生重复行——这是 R3「只追加」的**预期代价**，用 `batch_no` 区分批次缓解。
- **异常场景 #5**（风险②）：某月 Excel 首行「数据日期」也为空 → 该批次业务日期缺失，导入时需容错兜底（见 R7）。

**接口契约**（design ① 数据流 + 36 列映射表；坐标经 Step 5 校准）

- **接口契约 #1**：`POST /xiaodaixing/tjdBaseXiaodaixingProject/importExcel`（网关前缀 `/api-zjq`），`multipart/form-data`，文件参数名 `file`，返回统一 `Result` 信封（`代码 biz/base-service/base-service-app/src/main/java/com/itgfin/slp/base/controller/TjdBaseXiaodaixingProjectController.java:38-51`）。
- **接口契约 #2**：`POST /xiaodaixing/tjdBaseXiaodaixingProject/search`（design 中写作 `/page`，实现为 `/search`——**叫法差异，非语义冲突**，坐标以代码为准），请求体为 SearchVO，返回 `Result.success(QmPage)`，当页数据在 `records` 下（`代码 …/TjdBaseXiaodaixingProjectController.java:56-59`、`biz_web/src/modules/slp/pages/xiaodaixing/index.jsx:141`）。
- **接口契约 #3**：前端路由 `/slp/xiaodaixing`（挂在「业务办理」菜单下）。
- **接口契约 #4**：36 列 index → 列名 → 类型 映射表。`decimal(24,6)` 列 = idx 12/13/14/15/17/21/22/25/26/29；**保真 `varchar` 列** = idx 16(重复列①)/18(是否逾期脏值)/19/20(日期序列号)/24(比例)/28/30(重复列②)/31/32；idx 0 = `data_date`（批次键）；idx 10 = `project_no`；idx 34 = 空表头列 `guarantee_way_detail`（`代码 docs/sql/increment/20260626_01_ddl_xiaodaixing-project.sql:12-62`）。
- **接口契约 #5**：批次/审计列 = `batch_no varchar(64)`、`import_time datetime` + 审计列（`tenant_id`/`create_user`/`create_time`/`last_modify_user`/`last_modified`/`is_deleted tinyint default 0`/`version bigint`）+ `id varchar(64)` 主键；表上**无唯一约束**，仅 `project_no`/`data_date`/`batch_no` 三个普通索引（`代码 …20260626_01_ddl_xiaodaixing-project.sql:56-61`）。
- **接口契约 #6**：服务边界 `/api-zjq` → base-service（8093）。
- **接口契约 #7**（坐标，Step 5 补）：列表默认排序为 `import_time` 降序（`代码 biz/base-service/base-service-domain/src/main/java/com/itgfin/slp/base/service/TjdBaseXiaodaixingProjectService.java:112-114`）。
- **接口契约 #8**（坐标，Step 5 补）：筛选匹配方式——`customer`/`projectNo`/`orgPlatform` 为模糊匹配，`isOverdue`/`projectRating`/`dataDate` 为精确匹配（`代码 …/TjdBaseXiaodaixingProjectService.java:128-133`）。**该划分是否即为期望，见待澄清 #7。**

## 1. 功能测试 (TC-F)

#### TC-F-001 · 样例清单全量入库、零丢行
- **维度**：§1 输入边界（有效类代表值）
- **来源**：验收标准 #3 / 业务规则 R2
- **优先级**：高
- **GIVEN** 一份 66 个数据行、36 列的月度小贷星项目清单 xlsx（即样例 `小贷星项目清单（20260531）.xlsx`），且台账中尚无该批次数据
- **WHEN** 通过导入端点提交该文件
- **THEN** 导入以业务成功信封返回，并回报本次处理的记录条数
- **THEN** ⟨预期待定 · 源（验收标准 #3 / R2「零丢失」）要求 66 行全部入库，实现会跳过「客户与项目号均为空」的行（`TjdBaseXiaodaixingProjectService.java:264-266`）—— **两侧冲突未裁定**，入库行数以裁定结果为准；见待澄清 #2 — 不得据此写死断言⟩

#### TC-F-002 · 数据日期 fill-down 到整批
- **维度**：§1 输入边界
- **来源**：业务规则 R3 / R7 / 验收标准 #3
- **优先级**：高
- **GIVEN** 一份「数据日期」列仅首个数据行有值（值为 `20260531`）、其余行该列为空的清单
- **WHEN** 导入该文件
- **THEN** 本批次全部记录的 `data_date` 均为 `20260531`，不存在为空的记录

#### TC-F-003 · 批次标识全批一致且每次导入互不相同
- **维度**：§1 输入边界
- **来源**：业务规则 R3 / 验收标准 #3
- **优先级**：高
- **GIVEN** 台账中已存在一个历史导入批次
- **WHEN** 再导入一份清单
- **THEN** 本次写入的全部记录共享同一个 `batch_no`，且该值与历史批次的 `batch_no` 不同
- **THEN** 本次写入记录的 `import_time` 被填充为本次导入发生的时刻

#### TC-F-004 · 36 列按位置各就其位
- **维度**：§1 输入边界
- **来源**：业务规则 R6 / 接口契约 #4
- **优先级**：高
- **GIVEN** 一份 36 列、各列取值互不相同且可区分的清单（含两个同名「核销金额」列与一个空表头列）
- **WHEN** 导入该文件
- **THEN** 每一列的值落在与其列序（index 0..35）对应的目标字段上，无错位
- **THEN** 第 16 列与第 30 列这两个同名「核销金额」分别落到 `write_off_amount` 与 `write_off_amount_2`，互不覆盖
- **THEN** 第 34 列（空表头）落到 `guarantee_way_detail`，值不丢失

#### TC-F-005 · 金额列按数值类型入库
- **维度**：§1 输入边界
- **来源**：接口契约 #4 / 业务规则 R10
- **优先级**：中
- **GIVEN** 一份清单，其被定义为 `decimal(24,6)` 的金额列（合同金额/放款金额/贷款余额等）取合法数字值，含小数位
- **WHEN** 导入该文件
- **THEN** 这些列以数值形态入库，数值与原表格一致（含小数精度）
- **THEN** 带千分位分隔符（半角或全角逗号）、含空格的金额文本同样被识别为数字入库，不被判为脏值（`代码 …/TjdBaseXiaodaixingProjectService.java:319-323`）

#### TC-F-006 · 导入人与租户记为当前上下文
- **维度**：§1 输入边界
- **来源**：业务规则 R8 / 接口契约 #5
- **优先级**：中

- **WHEN** 由该用户发起一次导入
- **THEN** 本次写入记录的 `create_user` 为该用户的标识，不为空
- **THEN** 记录携带当前租户标识；当前上下文无租户时落为系统默认租户（`代码 …/TjdBaseXiaodaixingProjectService.java:215`）

#### TC-F-007 · 导入后新批次在列表中立即可见

- **GIVEN** 一次刚刚成功完成的导入
- **WHEN** 紧接着按该批次查询台账列表

#### TC-F-008 · 分页列表返回成功信封
- **维度**：D5 成功信封

#### TC-F-009 · 单一筛选条件命中

## 2. 边界测试 (TC-E)

#### TC-E-001 · 重复导入同一份清单
- **维度**：D1 幂等/去重（幂等半边）
- **来源**：业务规则 R3 / 异常场景 #4
- **优先级**：高
- **GIVEN** 一份已成功导入过一次的月度清单（其数据已在库）
- **WHEN** 原封不动地再导入同一份文件
- **THEN** 第二次导入同样以成功告终，不因重复而被拒绝
- **THEN** ⟨预期待定 · **源与实现相抵触**：源（R3「只追加不去重」/ 异常场景 #4「重复导入会产生重复行」）要求记录翻倍并以 `batch_no` 区分；实现按「数据日期 + 项目号」覆盖既有记录、总量不变（`TjdBaseXiaodaixingProjectService.java:162-179`、`TjdBaseXiaodaixingProjectMapper.xml:5-50`）。**冲突未裁定，不得默认任一侧为准**；见待澄清 #1 — 不得据此写死断言⟩

#### TC-E-004 · 金额列为空
- **维度**：§1 输入边界（空值）

- **THEN** 导入不因空值报错，这些行照常入库
- **THEN** 对应金额字段为空值，而非被填成 0，也不导致整行丢失

#### TC-E-005 · 脏值列原样保真
- **维度**：§1 输入边界
- **来源**：业务规则 R2 / R6 / 异常场景 #3 / 接口契约 #4

- **THEN** 导入过程不报错、不丢行
- **THEN** 上述各列以表格中的原始文本形态入库（仅首尾空白被裁去），未被解析、转换或规整

#### TC-E-006 · 只有表头、零数据行
- **维度**：§1 输入边界（集合大小下界）
- **来源**：验收标准 #3（导入行数语义）/ 异常场景 #1（源对「空内容」定义了拒绝语义，但只覆盖空文件）

           · 当前实现：以成功信封返回、处理 0 条、不写库（`TjdBaseXiaodaixingProjectService.java:247-250`）—— **未确认**

#### TC-E-007 · 列数与 36 不匹配
- **维度**：§1 输入边界（集合大小上下界）
