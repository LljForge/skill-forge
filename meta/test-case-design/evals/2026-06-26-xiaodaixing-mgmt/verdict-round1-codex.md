# 小贷星管理测试规范盲评结论

下文只以两份原始事实源为真值。为压缩引用，`R:n` 表示 `requirements.md` 第 n 行，`D:n` 表示 `design.md` 第 n 行；`甲:n`、`乙:n` 同理。

## ① 甲/乙三指标对照表

| 规范 | 可直接落测数 | 断言了未溯源契约数 | 待补缺口数 |
|---|---:|---:|---:|
| 甲 | **11** | **0** | **6** |
| 乙 | **51** | **23** | **0** |

计数口径：以“一条有编号的用例”为指标 1/2 的计数单位；只有整条用例的预期都能由事实源支撑、且足够确定，才计入“可直接落测”。指标 2 只计明确断言了源中没有的具体行为；含“或按约定”“若支持”“记录实际行为”但没有选定具体结果的条目，只判为不可直接落测，不硬算成虚构契约。指标 3 按去重后的、明确要求先补契约才能测的缺口条目计数，不按文中重复提及次数计。

### 指标 1 计数底账与例子

- **甲 11 条，全部可直接落测**：`TC-F-001`～`TC-F-005`（5）、`TC-E-001`～`TC-E-004`（4）、`TC-ERR-001`～`TC-ERR-002`（2）。例如：
  - `TC-F-001` 的 66 行、`data_date=20260531`、同一 `batch_no`，直接来自 `R:25,32` 与 `D:17-18,50`。
  - `TC-E-002` 的 36 列、重复列、空表头及脏值保真，直接来自 `R:24,32` 与 `D:48,72-111`。
  - `TC-ERR-001/002` 的空文件/非 Excel 被拒、语义提示且不入库，直接来自 `R:16-18`；甲没有擅自写 HTTP 状态码、业务错误码或错误体字段。
- **乙 51 条可直接落测**，编号底账如下：
  - IMP（12）：`IMP-002`～`IMP-009`（不含 `IMP-010`），`IMP-011`、`IMP-012`、`IMP-013`、`IMP-020`。
  - MAP（6）：`MAP-001`、`MAP-002`、`MAP-004`、`MAP-005`、`MAP-006`、`MAP-008`。
  - QRY（11）：`QRY-002`～`QRY-009`、`QRY-011`、`QRY-012`、`QRY-017`。
  - DDL（7）：`DDL-002`～`DDL-008`。
  - MENU（5）：`MENU-001`～`MENU-005`。
  - FE（6）：`FE-001`、`FE-002`、`FE-004`、`FE-006`、`FE-007`、`FE-010`。
  - NFR（1）：`NFR-001`。
  - E2E（3）：`E2E-001`～`E2E-003`。
  - 代表例：`DDL-004` 可逐列按 `D:49,113` 核验；`MENU-005` 可按 `R:23,34` 与 `D:43,52` 检查交付物；`NFR-001` 可按 `R:27,35` 直接跑离线编译并核 Maven 依赖。

乙的 82 条完整对账为：**51 条可直落 + 23 条含未溯源具体契约 + 8 条其他不可直落 = 82 条**。后 8 条是 `IMP-014`、`IMP-016`、`IMP-021`、`QRY-015`、`DDL-001`、`FE-008`、`NFR-003`、`NFR-008`：它们主要用二选一、条件式或不可量化措辞，不能直接写成判定唯一的测试，但也没有被强行计入指标 2。特别是 `IMP-016` 的“上传时间/文件名”虽不符合最终设计 `D:50` 选定的上传时间，但“文件名”确实曾出现在 `R:46` 的候选中，因此严格按“源里根本没有”定义，不计指标 2。

### 指标 3 计数底账与例子

甲的“待澄清”区列了 7 项，但按题目对“未定义、需先补齐才能测”的严格定义，只计 **6 个真实缺口**：

1. **分页参数/边界契约**（`甲:205`）：源只说分页（`R:26,33`；`D:21-22,54`），没有参数字段名、默认页大小、上限及越界行为。
2. **筛选匹配语义**（`甲:206`）：源只列筛选字段（`R:14,26`），没有精确/模糊、大小写及多条件 AND/OR。
3. **鉴权失败三态**（`甲:207`）：`D:13,51` 只有请求头和用户上下文线索，通篇没有端点权限、匿名/过期/无权限行为及状态码。
4. **导入事务边界**（`甲:208`）：`D:18` 只写 `saveBatch`，通篇没有整批回滚、部分成功、补偿或重试契约。
5. **错误/成功信封细节**（`甲:209`）：`R:17` 只有提示语义，`D:54` 只有 `Result`/分页类型，未定义 HTTP 状态码、业务码和字段结构。
6. **仅表头、无数据行 Excel 的行为**（`甲:210`）：`R:17` 的“空文件”不等于合法但零数据行的 Excel，设计通篇也没有定义该等价类。

`甲:211` 不计：`D:50` 已明确首行也空时用**上传时间**，它是文档措辞差异提醒，不是“必须先补齐才能测”的未定义契约。乙的指标 3 为 **0**：虽然 `IMP-021` 写“明确约定并验证”，以及多条使用“或按约定/若支持”，但乙没有把任何一项明确登记为“事实源未定义、先补契约再测”的缺口；它们仍混在预期结果里，不能算合格的缺口标注。

## ② 指标 2 的逐条实例

### 甲：0 条

未发现甲把源中沉默的错误码、分页默认值、鉴权、事务、并发、缓存等写成具体断言。甲对这些点要么不生成用例，要么显式放入待澄清。

### 乙：23 条

1. **`IMP-001`（`乙:30-36`）写死 `Result.success=true`。** `D:54` 只定义“file → Result”，`R`/`D` 通篇没有 `success` 字段名或布尔值契约；照写会把未知信封结构固化进测试。
2. **`IMP-010`（`乙:77-79`）断言伪装成 `.xlsx` 的损坏内容必须被捕获、返回可读错误、不入库且不裸 500。** `R:17` 只定义空文件和扩展名非 `.xls/.xlsx`，`D:14-18,48` 只描述 EasyExcel 正常读取链；两份源通篇没有损坏工作簿的响应、异常映射或事务结果。
3. **`IMP-015`（`乙:105-108`）断言千分位、货币符号、前后空格无法转 decimal 时以 null 兜底且不丢行。** `D:121,123` 只定义金额列“null 安全”，没有 `1,234.56`、`￥1000`、空格或非数字金额的解析/容错契约；`R:41` 反而明确本次不做导入校验/纠错。
4. **`IMP-017`（`乙:115-117`）断言仅表头 Excel 导入成功、入 0 行。** `R:17` 只覆盖 0 字节/空 multipart 意义上的空文件，`R`/`D` 通篇未定义“合法工作簿但无数据行”的成功或失败行为。
5. **`IMP-018`（`乙:119-123`）断言少于 36 列时补 null、多于 36 列时静默忽略。** `R:24,30` 与 `D:48,72-111` 定义的是标准 36 列和 index 0..35 映射，通篇没有畸形列数的宽容策略。
6. **`IMP-019`（`乙:125-127`）断言特殊字符原样入库，超长值只能截断或报可读错误且不能静默丢整行。** `D:72-113` 只给固定列长，源中没有超长输入的截断/拒绝选择、错误可读性或逐行保留保证。
7. **`IMP-022`（`乙:137-139`）断言 `tenant_id` 按当前租户自动填充且非空。** `D:113` 只要求存在 `tenant_id` 列；`D:17,51,121` 明确填充的是 `create_user`，两份源通篇没有 tenant_id 的取值来源、非空性或自动填充契约。
8. **`MAP-003`（`乙:155-157`）把“非数字金额”也规定为不抛异常并落 null/默认值。** `D:121,123` 的 null 安全只支撑 null/空值边界，未定义任意非数字串的输出或“不抛”保证。
9. **`MAP-007`（`乙:171-173`）断言空 List 必须返回空 List 且不 NPE。** `D:121` 只给出 `toPOList(List<Excel>, ctx)` 的测试缝隙和正常转换职责，源中没有空集合输入契约。
10. **`QRY-001`（`乙:183-186`）断言省略分页参数时存在默认分页并返回非空页。** `R:26,33` 与 `D:21-22,54` 只说“分页”，通篇没有参数可省略、默认页码/页大小或默认结果行为。
11. **`QRY-010`（`乙:223-225`）写死多筛选条件按 AND 取交集。** `R:14,26` 只列可筛字段，`D` 通篇没有多条件组合语义；AND 是典型的未溯源具体契约。
12. **`QRY-013`（`乙:235-237`）断言非法页码/页大小只能回落默认或返回空页，且不能 500、不能退化为无分页全表。** 两份源通篇没有分页合法区间、默认值、上限或非法参数处理。
13. **`QRY-014`（`乙:239-241`）断言查询一定参数化且 `%`/`_` 按某种转义约定处理。** `D:22` 只说明基类分页和 `is_deleted=0`，源中没有 SQL 特殊字符、通配符转义或注入输入的对外契约。
14. **`QRY-016`（`乙:247-249`）断言支持时可按 `import_time/data_date` 排序且默认排序稳定。** `R:14,26` 与 `D:21-22` 只定义筛选和分页，通篇没有排序字段、方向、稳定性或默认序；标题中的“若支持”不能为“默认排序稳定”提供来源。
15. **`DDL-009`（`乙:287-288`）断言 `version` 有初始值。** `D:49,113` 明确 `is_deleted DEFAULT 0`，但对 `version` 只定义 `bigint` 类型，通篇没有默认值/初始值。
16. **`FE-003`（`乙:324-326`）断言页面有重置动作，且会清空表单并恢复全量分页。** `R:7,12-14` 和 `D:21,35` 只定义查询页、筛选和分页，未定义重置控件及行为。
17. **`FE-005`（`乙:332-333`）断言导入成功必须 toast 并自动刷新列表。** `R:13` 只要求导入后列表可见新批数据，`D:35` 只指定复用组件；源中没有 toast 形态或成功回调自动刷新契约。该用例中的错误信息语义有源，但整条仍混入了未溯源成功行为。
18. **`FE-009`（`乙:345-346`）断言上传中必须有 loading、防重复点击并在完成后恢复。** 两份源通篇没有上传交互状态或重复点击策略。
19. **`NFR-002`（`乙:359-361`）断言 66 行“秒级”、1 万行必须分批 `saveBatch`、可接受时间完成且不 OOM。** `R:32` 只给 66 行验收样例，`D:18` 只写 `saveBatch`；源中没有性能阈值、万行容量目标、分批策略或内存指标。
20. **`NFR-004`（`乙:368-370`）断言未登录/无权限调用两个端点都必须被拦截。** `D:13` 的配置请求头和 `D:51` 的 `AppContext` 只是上下文线索；`R`/`D` 通篇没有两个端点的鉴权要求、权限模型或匿名失败行为。
21. **`NFR-005`（`乙:372-374`）断言两个并发导入请求必须批次独立、互不串批且数据不错乱。** `R:25`、`D:50` 只定义“每次导入一个 UUID”，通篇没有并发隔离、线程安全、原子性或并发上限契约。
22. **`NFR-006`（`乙:376-378`）断言超大文件超过配置/业务上限时要可读拒绝、不 OOM、不悬挂。** 两份源通篇没有文件大小/行数上限，也没有超限响应和资源保护契约。
23. **`NFR-007`（`乙:380-381`）断言上传时间兜底值必须与真实批次可区分或另有标识。** `D:50` 只决定“首行也空时用上传时间”，没有额外标志、格式区分或防混淆契约。

这些条目不是单纯“多测一点”：如果原样写成回归测试，会把实现偶然行为或作者偏好误升格为产品契约。

## ③ L1 赢家：甲

**执行者镜头选甲。** 甲给出 11/11 条可直接落测、0 条未溯源具体契约；需要追问的 6 个缺口被集中暴露，可以一次性找产品/设计拍板。乙虽然有 51 条可用内容，绝对数量远高于甲，但混在 82 条里还有 23 条会制造错测试、8 条不能形成唯一判定；执行者必须先重新做一次事实源审计，才知道哪些能写。

对执行者而言，指标 2 比指标 3 更痛：显式缺口造成的是可见的前置等待，能批量澄清；虚构断言会先进入代码，再造成假失败、错误放行或把实现偶然行为固化成契约，返工通常跨测试、实现和评审三处。甲覆盖较窄是成本，但“少而可信”比“多且夹雷”更适合作为直接实施规范。

## ④ L2 赢家：乙

**富草稿镜头选乙，但只能作为待清洗草稿。** 它有 82 条候选，覆盖导入、纯映射、查询、DDL、菜单、前端、工程约束和 E2E；其中 51 条确实有源、可直接转为测试，明显比甲的 11 条更利于 QA lead 做覆盖盘点。甲主动把 DDL、编译、交付文档等视为门而未展开成用例，作为“尽量全的起点草稿”信息密度不足。

代价必须明说：乙的“完整”中至少 23 条是未溯源契约、另有 8 条无法给出唯一 oracle。QA lead 应保留 51 条 grounded 用例，把 23 条改写为候选风险/待决策问题，把 8 条补成明确契约或删除；不能把 82 当成可直接下发的测试基线。

## ⑤ 一句话总评

**本质上，甲是稀疏但契约纪律强的执行规范，乙是覆盖宽却把大量沉默点写成断言或含糊预期的候选清单；实证结论是沉默点值得写全，但应写成显式缺口/待决策项，绝不能为了“完整”替事实源补造预期。**
