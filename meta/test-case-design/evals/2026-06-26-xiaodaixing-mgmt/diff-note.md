## 二、押错的预测 = 最强证据

挑本样本时押的是：**D3 事务欠文档 → 裸模型会漏 → 维度表点名翻盘**。结果押错：

- **claude-off（裸）l.135**：「要么整批回滚不留半批（saveBatch 事务）…不留孤儿脏数据」
- **codex-off（裸）E18**：「使 saveBatch 中途写失败 → 事务回滚全部本批行；无残留半批」

两个裸模型，在 **D3 根本没写进文档**的情况下，凭测试工程师本能照样独立抓到事务回滚。样本 #1 还能辩「他们从 design 读到的」；本样本 design 里没有，**维度表最招牌的「召回会漏的」卖点在最有利的条件下被证伪**。

D5 错误码同理：四格全部收敛到「码未定义 → 标出来」。**没法「召回」一个源文档里不存在的错误码**——所以维度表在欠文档维度上也无差异化。

## 三、必须诚实的修正：把「维度表」拆两层

两 skill-on 的待澄清清单（claude-on 6 条、**codex-on 13 条**）系统、可交付、PM 能直接补；裸格也标了缺口（codex-off l.35/36 列了分页缺省/错误码/事务原子性未定义）但**散在前言**。故：

- **维度表「内容」（D1–D6 具体维度）= 教已知**——模型自带，两任务两模型四次证明。
- **维度扫描「机制」（逼逐维记 已覆盖/N-A/待澄清）+ 待澄清必备出口 = 有边际价值**——把散落缺口逼成结构化、逐维、可对账的台账。

→ 修正「砍维度表」的粗判：**该砍的是内容厚度（教科书式 6 条详解 + 显然维度），该留的是「逐维强制交代 + 待澄清」的骨架。**

## 九、盲评判决 + v0.1.4 验证（2026-07-18）

§一~§八 都在量 skill 自己的机器（门B命中、待澄清数、维度扫描）——**全是 skill 拿自己的价值观自评**。本节补上缺的那个判决：**下游视角盲评「skill 产物 vs 裸跑写全」哪个更有用**。

### 9.1 判决轮（v0.1.3-postslim vs 裸跑）· 产物 `verdict-round1-{claude,codex}.md`

两裁判（Claude+Codex）盲态、去品牌标甲/乙、位置对调、对 `requirements/design` 核 grounding，三硬指标 + 两镜头：

| | Claude 裁判 | Codex 裁判 |
|---|---|---|
| 裸跑断言未溯源契约（造错测试） | 6硬+~9软 | **23** |
| skill 断言未溯源契约 | **0** | **0** |
| L1 执行者镜头 | **skill** | **skill** |
| L2 富草稿镜头 | 裸跑 | 裸跑 |

两裁判独立收敛同一处方：**「沉默点值得写全，但应写成显式缺口/待决策项，绝不替事实源补造预期」＝「skill 的纪律套在裸跑的广度上」。** 这直接暴露 v0.1.3 的缺陷：它把「场景有据·预期沉默」的用例整条弃进待澄清、丢了覆盖（L2 输）。→ 催生 v0.1.4「预期待定」第三条道（见 CHANGELOG v0.1.4）。

### 9.2 验证轮（v0.1.4 vs 裸跑）· 产物 `claude-skill-on-v014.md` + `verdict-v014-{claude,codex}.md`

改完同任务重跑 skill-on（v0.1.4）、再送同样两裁判：

### 9.3 对最初那问的实证回答

「skill 写的用例比裸跑真的更好吗」——**分用途，不是一句话的好**：
- **交接给执行者落测** → skill（尤 v0.1.4）更好、且编造安全性碾压（0~1 vs 2~20）。
- **要一份最广的过目草稿** → 裸跑更好（但含大量得删/改的编造 + 越界面）。

v0.1.4 把 skill 主场（交接规范）做强了，代价 0。**两处 runner 应用瑕疵**（toPOList 单测缝隙被门B误弃、1 条编造滑过）记在 CHANGELOG v0.1.4 残，非设计缺陷。
