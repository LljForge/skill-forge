### 三处修正

**F1 · 入口① 去 recon 结构耦合（核心）**

原 `SKILL.md` Step 1 入口表把 recon 产物的**目录路径 + `§2.2/§3/§4` 章节号 + design 分节**写进本体。两个毛病：① 章节号漂移——`§2.2 关键异常` 对 recon **模板**（单一「主流程」子节，见 `templates/requirements.md`）成立，但真实多场景任务把 §2 撑开后，关键异常漂到 §2.4，照号取会取错整段；② 更根本地，它把「recon 产物」这一种使用场景的文档 schema 焊进本体，违背 test-case-design 应保持独立自包含（#7）——recon 产物只是携带四类事实的一种来源，不是特权入口。

**改**：入口回归本体已有的源无关四类事实抽象；各来源（对话/PRD、接口文档、recon 产物）一律平权，**按语义定位、不绑任何来源的固定章节号/路径/分节**。原先复审列出的「章节号漂移」「漏 §2 场景正文」「design §测试缝隙 未表态」三条发现，本是「本体贴着 recon 结构写」的同一个病，一并消。接口契约只取被测契约本身、来源夹带的测试实现不带入，门 B 出口再兜。

**F2 · 悬案 2 结案（D1 一行两态）**

`v0.1.0` 悬案 2：D1 幂等/并发两半边判据不同，维度扫描表一行两态没说填哪个。真实任务撞上（幂等半→待澄清、并发半→N/A），实证印证候选解「取更严的一半」站得住 → 写进门 C：一行内两半边状态不一致时行状态取更严的一半，理由列分注两半边归宿。

**F3 · 混合跨栈任务补指引**

原文只二分「纯后端 / 纯前端 E2E」。真实任务是后端+前端混合，靠推导处理（后端半走维度表、前端半走纪律层）。补一句显式指引，免得下次重新推导。

## v0.1.1 — 重命名 + description 纠偏

**`test-case-authoring` → `test-case-design`**。「测试用例设计」是行业标准说法；`references/backend-api.md` §1 的等价类划分 + 边界值分析，在教科书里本就归在「用例设计方法」名下。

**description 两处失真，一并修正**：

1. 原写「**显式调用，不自动触发**」——照抄了 `codex-task` / `skill-tempering` 的模式，但没检查前提。那两个是重型委托 / 重型 meta 动作，误触发代价高；写测试用例不是这类，用户说「给这个接口写测试用例」它就该触发。已改为主动触发。
2. 原写「**第一版仅后端接口，前端 E2E 不在范围**」——把**进度**写进了**触发面**。description 该写职责边界（如 `codebase-exploration` 的「全局结构 vs 单模块精读」），那是永久的；「仅后端」下个版本就假了，且主动拒绝了既定的后续方向（E2E）。现 description 只描述职责；范围降为正文里的行为指令——遇 E2E 任务，纪律层照常适用，如实告知领域维度未覆盖，不拿后端 6 条硬套。

保留的职责边界：**产「测什么」的规范，不产可跑的测试代码**。这条不随版本变，也是本 skill 区别于测试代码生成器的地方。

**v0.1.0 的设计文档与实施计划仍在旧名下**，未改：`docs/superpowers/specs/2026-07-17-test-case-authoring-design.md`、`docs/superpowers/plans/2026-07-17-test-case-authoring.md`。它们是那天的历史快照，不是活规范——改名不该回头篡改「当时做了什么」。下方 v0.1.0 记的也是当时的账，同样保留原样。

**行为验证状态未变**：仍是零真实任务验证。改名不产生任何实证。

## v0.1.0 — 首版（孵化）

**定位**：根据一次开发任务或业务领域，产出「执行无关」的测试用例规范——只回答「测什么」，「怎么测」留给下游执行者（另一个 AI agent / skill）。设计文档：`docs/superpowers/specs/2026-07-17-test-case-authoring-design.md`。

### ⚠️ 成色声明

**纪律层与维度表均未在正向场景验证。**

- **纪律层**（三铁律 + 出口三门）：零真实任务验证，是设计推演的产物。
- **维度表 D1–D6**：拄自 `incubating/module-spec-baseline/references/behavior-dimensions.md`。该表有真实来历（生自 GitNexus 对照复盘 + 5 模块实测，`782a7ec` 的 "redo" 印证了回退重做），**但**：① 其 `evals/` 是空目录、复盘产物已在 `eb475fa` 清除，实测结论无法独立复核；② 它验证的是**逆向 as-built** 场景，本 skill 是**正向**——这 6 条在正向下够不够、准不准，**均未验证**。

**第一版不建 evals**，直接拿真实后端接口开发任务跑。理由：纪律层零验证时，先撞真实任务比先造 eval 架子有信息量；反面教材就在隔壁（`module-spec-baseline/evals/` 是从无提交的空目录）。

**跑第一次真实任务时要观察三件事**：
1. **三门有没有真拦住东西** —— 一次都没拦住 = 门是摆设，或任务太浅、样本无效。
2. **维度表有没有召回本来会漏的** —— 这是它存在的唯一理由。没召回到任何东西，说明 6 条拄来的维度在正向不适用。
3. **「待澄清」有没有真挡住编造** —— 若它空着但用例齐全，多半是在偷偷补来源（门 A 假过）。

### 已定位、待实证裁决的悬案（首次复盘一并处理）

以下**不是新发现的 bug，是已经诊断清楚、但缺一个真实数据点才好裁的题**。记在这里，免得复盘时被重新发现一遍。

**悬案 1 · D4 的 §3 引用票悬空**

`references/backend-api.md:72` 的 D4 前提句写「没依据不生成（**见 §3**）」，但 §3 清单（`:95`）只有限流/防刷、多租户隔离、版本兼容三项，**不含缓存**。门 C（`SKILL.md:196-197`）把 §3 治下维度闭集化并显式列名之后，这张引用票就悬空了：门 C 判 D4「非 §3 维度 → 事实集沉默 → 走待澄清（问用户）」，D4 自己的前提援引 §3 说「不生成、也不问」——**两条相反指令**。

成因：修 D1 时建立的范式是**先声明归属再援引**（D1 写的是「并发上限半边**算限流/防刷类，受 §3 管**」，且门 C 把它显式列进闭集）。D4 只有援引、没有归属声明，门 C 也没收它——范式没回头套用到 D4。D4 内部还有轻微张力：`:71` 常见漏法写「**没问**有没有缓存层」，这话本身在主张「该问」，与 `:72` 的「也不问」拧着。

**为什么留到复盘**：两种归宿都不产生幻觉契约（要么多问一句「有没有缓存层」，要么 N/A 掉一条维度），风险有界。而「事实集沉默时该不该问缓存」本质是**经验问题**——问缓存诱导用户虚构需求的风险远低于问限流，推演不如一个真实数据点。

**候选解**：(a) 照 D1 范式给 D4 补归属声明并列进门 C 形式 2 的闭集；(b) 删掉 `:72` 的「（见 §3）」，让 D4 沉默时正常走待澄清——与 `:71` 常见漏法的主张一致。**倾向 (b)，但由实证定。**

**悬案 2 · D1 一行两态，扫描表没说填哪个**

`backend-api.md:49` 的 D1 有两个半边判据（幂等半边默认点名 / 并发半边受 §3 管），但 `SKILL.md:228-233` 的扫描表是**一维度一行**。写接口 + 事实集对幂等和并发都沉默时，幂等半边→待澄清、并发半边→不生成不问，**D1 那一行填哪个状态？**

可推导（取更严的一半 = `⚠ 待澄清`，并发半边按 §3 本就不该留痕），但没明说。一句话可封：「D1 两半边判据不同时，行状态取更严的一半」。等复盘确认这个推导在真实任务里站得住再写死。

**悬案 3 · 设计文档 §6.1 / §6.4 仍是二态摘要**

`docs/superpowers/specs/2026-07-17-test-case-authoring-design.md:228`（门表 C 行写「已覆盖 **或** N/A + 具体理由」——`或` 断言了二元闭集，不准确）、`:265`（产物骨架）。

**不阻断的理由**：正下方 `:230` 就是完整正确的三形式规则（含「非 §3 维度沉默 → 不是 N/A，是待澄清」），读的人必然两句都读到，照它做不会做出被禁止的事；且**运行时文件是 SKILL.md，已是完整三态**，设计文档标着「设计完成、待实施」，agent 跑的时候不读它。**但仍应补**——设计文档是活得最久的规范件，成本两个单元格。

### 关键设计取舍

| 取舍 | 决定 | 为什么 |
|---|---|---|
| 先例 vs 基座 | `cexll/myclaude@test-cases` 作**先例参考**，不作基座 | 把它从「通用手工用例」硬掰成「Java 接口 + 执行无关契约」= 架构大改，正是 recon 批次 A/B/C 翻车后立规矩要避开的坑。只吸收其四类分区、覆盖矩阵、等价类划分三样 |
| 防泄漏手段 | **砍字段** > 加原则 | test-cases 证明：模板有 `Test Steps` 字段，四处原则宣示也拦不住，且其唯一 AAA 范例自带 `Click "Login" button` 反例 |
| 复用 msb | **零依赖 + 拄 6 条** | msb 仍孵化（无 evals、agent 对表的消费分批接入未完），抽公共层 = 焊死在移动靶；且两张表服务对象不同（逆向契约 vs 正向用例），共用会互相拖累演进 |
| 门的形态 | **内联三门**，不派 sub-agent | msb 教训是「门要全」而非「要加 agent」，且明确记有「门 2/3/4 折进 grounding 一趟、成本不涨」 |
| 门 B 分级 | **硬词表 + 软判断**两级 | msb 首版「grid + grounding 双门过了」，A/B/C 复盘照样发现软实现泄漏（异步线程内/删后插/签名派生）——这类不含禁用词，词表拦不住 |
| 覆盖矩阵 | **双向** | test-cases 只做正向（每条需求有用例）；反向（每条用例有来源）才是正向场景最需要的防编造门 |
| 事实集落盘 | **必须编号落盘**进产物「概览」，非可选存档 | 不落盘，用例 `来源` 字段回指的就是当场编的号、用完即弃——门 A 退化为 agent 自我声明；设计 §7 唯一的验证手段（真实任务人工复核）也查不了这个最核心的门 |
| 第一版范围 | **仅后端接口** | 后端与前端 E2E 仅共享纪律层，维度层几乎不重叠。纪律层零验证时不摊两个领域，翻车可归因 |
| 接 recon | **不接**，只读其产物 | 接进去破坏 recon「自包含、零外部 Skill 依赖、丢到任何环境都能独立跑」，且丢掉「直接给业务领域」入口 |
