## v0.1.5 — 代码作第二输入源 + 「⚠️ 歧义点」取代大部分「预期待定」

> ⚠️ **本版未经外部盲评验证。最后一个经盲评验证的版本是 v0.1.4。**
> 本版的证据只有两轮自跑（`evals/…/{claude,codex}-skill-on-v01{5,6}.md`）+ 三轮静态复查，
> 未做 A/B、未做下游交接复验。用它之前请知悉这一点。

### 起因

用户指出：文档是用来**圈定测试范围**的，不是唯一真值源；代码本来就可以读。此前 skill 把「不读代码」当成设计前提（源于铁律 2 的正向定位），是把一条自我设限当成了硬约束。

### 核心改动

- **铁律 2 · 源分两型**：文档 ground 范围与预期；代码（可选）ground 结构与坐标。判据是「**把这个值换成另一个，算不算功能变了**」——不算＝坐标（读到即填死），算＝答案（只能提供现状）。
- **流程 5→7 步**：新增 Step 5「空位查证」、Step 6「冲突上报」。
  - **Step 1–4 必须不看代码**，这是安全机制：顺序反了会照着实现写测试，连 bug 一起固化。
  - Step 5 四情形：文档有定义→按文档填；文档有定义但代码不同→**冲突**；文档沉默代码有→**按代码填 + 标 `⚠️ 歧义点`**；两边都没有→`⟨预期待定⟩`。
- **`⚠️ 歧义点` 取代大部分 `⟨预期待定⟩`**：源沉默时不再卡住等裁决，按现状填 + 标记，**审查权交给人但不打断人**；`⟨预期待定⟩` 降为罕见降级出口。标记必须写全三件事，尤其**「它不会发现此处的缺陷」**——这是本版用「发现能力」换「不打断」的代价，不写清楚标记就退化成没人看的空标签。
- **冲突（源与实现相抵触）单列为第三类发现**，排「待澄清」最前，两侧注出处，不许默认哪侧为准。
- 编号分两套：`歧-N` / `澄-N`。

### 演进过程（三版，前两版的教训值得留）

1. **v0.1.5 初版**加了 Step 6「批量裁决」，把所有决策空位摆给用户 → 用户反馈**「问题太多了」**。实测：Claude 13 条、Codex 24+ 条，用户只能整包搁置。
2. **加闸修正**（出口闸 + 禁止捆绑 + §3 不问）→ Claude 改善（预期待定 31→17、抓到 2 处冲突），**Codex 反而更差**（39 条、`来源:代码` 归零——它压根没查那一类）。诊断：Step 5 有指令无台账，「没查」与「查不到」在产物里长得一样。
3. **用户决断简化**：只有文档与代码不一致才问；其余一律按文档/代码填 + 标歧义点，人工审查、要改由用户主动发起。**这一刀砍掉的是产生问题的机制本身，不是又补一道闸**——前两版每加一条规则就得配一道闸、一份账，正因为「问人」这条路自带复杂度。

### 三轮静态复查（记此备查：为什么每轮都有发现）

三轮各用了不同镜头，因此各有收获：①旧机制删干净了吗（残留措辞）②新写的自己站得住吗（**含第一轮修改引入的回归**：门 C 对齐歧义点时反手诱导凭空造场景，破铁律 2）③没改的部分还成立吗（**门 B 被架空**——它一字未改，但用例正文里现在有了 provenance，硬拦正则会命中「现状：会插入一条新记录」这类合法描述，软拦会把 `<path>:<line>` 判成实现细节，**成批误删歧义点用例**，与当年 `toPOList` 同一失效模式）。

**结论**：SKILL.md 是不可编译、不可运行的程序，静态复查＝人脑模拟单条路径，第 N 轮仍会有发现，且严重度未见收敛（第三轮那条最严重）。修改本身也带同样出错率。**要提高置信度只能跑，不能再读。** 两次真跑找到的问题比三轮复查加起来更实质。

### 已知未解

- **门 B 观察面**（`toPOList` 单测缝隙被误弃）仍未修。本版把「在哪一层可观察」从铁律 2 的代码可 ground 清单里**删掉了**——收回一个没有闸支撑的承诺，消除矛盾，但问题本身留在 `BACKLOG.md` A1。
- `description` 未提「代码是可选第二输入」，而它现在影响 skill 需要什么运行环境。改 description 会动触发行为，留待定。
- 铁律 2「想测但没依据→回头问用户」与 Step 6「别的一律不问」措辞仍有张力（复查三 ②，判为次要未改）。

> **纪律不能留给模型自觉。** 即便是一个已被证明懂这条规则的模型，在「写出能跑的测试」这个执行压力下也不会应用它。

**根因**：`THEN ⟨预期待定 · 源未定义 X；见待澄清 #N⟩` **只描述状态、不下达指令**。陌生 agent 读完仍需自行决定怎么办，而「自行决定」在模型间方差极大。下游不受控（下游 skill 未定、agent 可能是 Codex 也可能是别的）→ 必须按最差那档设计，**闸焊在产物里**。

同时暴露一个无关的上游真缺口：事实集 IC6 列映射表没钉全（「数据日期」「合同金额」「放款金额」的 index 缺失），两个下游都被迫自行假设。Claude 标了 `假设 A3/A4` 并集中成常量，Codex 直接用了。

## v0.1.4 — 铁律 3 开「预期待定」第三条道（盲评 2 裁判证据驱动）

**据一场盲评对照实验给铁律 3 补一条出路。** 实验设计：拿 v0.1.3 瘦身版 skill-on 产物 vs 裸跑写全产物（同任务 xiaodaixing、同生成模型），去品牌盲标甲/乙、两裁判位置对调，Claude + Codex 各自对着 `requirements.md` + `design.md` 真值核 grounding，出三硬指标（可直接落测 / 断言未溯源契约 / 待补缺口）+ 两镜头（L1 执行者 / L2 富草稿）。产物 `evals/2026-06-26-xiaodaixing-mgmt/verdict-round1-{claude,codex}.md`。

### 实验结论（两裁判盲态 + 对调 + 对源核，逐字复现）

- **断言未溯源契约数**：裸跑 6 硬~9 软（Claude 判）/ 23（Codex 判）；**skill 版 0**。这是 skill 的真价值——裸写在源沉默处把预期编死成断言，落测即造「把实现偶然行为固化成产品契约」的错测试。
- **L1 执行者镜头 → skill 赢**（两裁判一致）：虚构断言的返工成本（跨测试+实现+评审）比显式缺口的前置提问更痛。
- **L2 富草稿镜头 → 裸跑赢**（两裁判一致）：裸跑广度（82 条 vs skill 11 条）大体是真覆盖、非编造，边缘面枚举有价值；skill 把这些整条弃进待澄清、丢了覆盖。
- **两裁判独立收敛的处方一致**：「沉默点值得写全，但应写成显式缺口 / 待决策项，绝不替事实源补造预期」＝「skill 的纪律套在裸跑的广度上」。

### 改法：拆「场景沉默」与「预期沉默」，给后者开第三条道

根因：铁律 2/3 把一条用例当**整体**判——THEN 无源即整条弃进待澄清。但坑清单里那些「造错测试」的用例**场景有据**（筛选字段存在、36 列映射有定义），只是**预期**（oracle）源里没定。此前 skill 把「场景沉默」和「预期沉默」用同一把弃案闸处理了。

- **铁律 3 补一段**：分清场景（GIVEN/WHEN）与预期（THEN）。场景无据→不生成（尤其 §3）；**场景有据、只预期沉默→写案不弃案**，THEN 写「预期待定」（点明源未定义 + 列候选 + 回指待澄清，绝不选定具体值），缺口照进待澄清区。
- **用例格式**：`THEN` 可待定，加 `TC-QRY-010` 范例；`来源` 对待定用例回指的是**场景**依据。
- **门 A**：THEN 预期待定 ≠ 无源，其 `来源` 回指场景输入面依据、回指得上即放行，别误删。
- **门 C**：加状态 `◐ 场景已覆盖·预期待澄清 · <TC-ID>`，同占「已覆盖」与「待澄清」两栏语义，别塌成纯待澄清。

### 为什么不破门 / 不重新放进编造

门 B 完全不动（THEN待定 仍是 THEN 槽、无执行细节）；**§3 仍硬压制**（限流/多租户/版本/并发上限的场景本身无据 → 走「场景无据→不生成」，不开待定道）；scalpel 只切「场景有据·预期沉默」窄带。拿裁判坑清单验算：QRY-010(AND/OR)、IMP-018(≠36列)、IMP-017(仅表头0行)、D5(错误码)、IMP-022(tenant_id) 转为待定案（拿回广度、0 编造）；NFR-002(性能)、NFR-005(并发隔离)、NFR-006(文件上限) 仍压制。自检信号仍在（待定案让待澄清区保持非空）。

改完拿 xiaodaixing 重跑 skill-on（v0.1.4）、再送同样两裁判盲评 v0.1.4 vs 裸跑。产物 `evals/.../claude-skill-on-v014.md` + `verdict-v014-{claude,codex}.md`。

### 已定位、待实证裁决的悬案（首次复盘一并处理）

以下**不是新发现的 bug，是已经诊断清楚、但缺一个真实数据点才好裁的题**。记在这里，免得复盘时被重新发现一遍。

**悬案 1 · D4 的 §3 引用票悬空**

`references/backend-api.md:72` 的 D4 前提句写「没依据不生成（**见 §3**）」，但 §3 清单（`:95`）只有限流/防刷、多租户隔离、版本兼容三项，**不含缓存**。门 C（`SKILL.md:196-197`）把 §3 治下维度闭集化并显式列名之后，这张引用票就悬空了：门 C 判 D4「非 §3 维度 → 事实集沉默 → 走待澄清（问用户）」，D4 自己的前提援引 §3 说「不生成、也不问」——**两条相反指令**。

成因：修 D1 时建立的范式是**先声明归属再援引**（D1 写的是「并发上限半边**算限流/防刷类，受 §3 管**」，且门 C 把它显式列进闭集）。D4 只有援引、没有归属声明，门 C 也没收它——范式没回头套用到 D4。D4 内部还有轻微张力：`:71` 常见漏法写「**没问**有没有缓存层」，这话本身在主张「该问」，与 `:72` 的「也不问」拧着。

**为什么留到复盘**：两种归宿都不产生幻觉契约（要么多问一句「有没有缓存层」，要么 N/A 掉一条维度），风险有界。而「事实集沉默时该不该问缓存」本质是**经验问题**——问缓存诱导用户虚构需求的风险远低于问限流，推演不如一个真实数据点。

**候选解**：(a) 照 D1 范式给 D4 补归属声明并列进门 C 形式 2 的闭集；(b) 删掉 `:72` 的「（见 §3）」，让 D4 沉默时正常走待澄清——与 `:71` 常见漏法的主张一致。**倾向 (b)，但由实证定。**

**悬案 2 · D1 一行两态，扫描表没说填哪个**

`backend-api.md:49` 的 D1 有两个半边判据（幂等半边默认点名 / 并发半边受 §3 管），但 `SKILL.md:228-233` 的扫描表是**一维度一行**。写接口 + 事实集对幂等和并发都沉默时，幂等半边→待澄清、并发半边→不生成不问，**D1 那一行填哪个状态？**

可推导（取更严的一半 = `⚠ 待澄清`，并发半边按 §3 本就不该留痕），但没明说。一句话可封：「D1 两半边判据不同时，行状态取更严的一半」。等复盘确认这个推导在真实任务里站得住再写死。

**悬案 3 · 设计文档 §6.1 / §6.4 仍是二态摘要**

`docs/superpowers/specs/2026-07-17-test-case-authoring-design.md:228`（门表 C 行写「已覆盖 **或** N/A + 具体理由」——`或` 断言了二元闭集，不准确）、`:265`（产物骨架）。

**不阻断的理由**：正下方 `:230` 就是完整正确的三形式规则（含「非 §3 维度沉默 → 不是 N/A，是待澄清」），读的人必然两句都读到，照它做不会做出被禁止的事；且**运行时文件是 SKILL.md，已是完整三态**，设计文档标着「设计完成、待实施」，agent 跑的时候不读它。**但仍应补**——设计文档是活得最久的规范件，成本两个单元格。

### 关键设计取舍

| 取舍 | 决定 | 为什么 |
|---|---|---|
| 先例 vs 基座 | `cexll/myclaude@test-cases` 作**先例参考**，不作基座 | 把它从「通用手工用例」硬掰成「Java 接口 + 执行无关契约」= 架构大改，正是 recon 批次 A/B/C 翻车后立规矩要避开的坑。只吸收其四类分区、覆盖矩阵、等价类划分三样 |
| 防泄漏手段 | **砍字段** > 加原则 | test-cases 证明：模板有 `Test Steps` 字段，四处原则宣示也拦不住，且其唯一 AAA 范例自带 `Click "Login" button` 反例 |
