## 1. D4 多栈仓跨栈边(前端→后端 HTTP)+ §6「单一权威源」对多栈的诚实性(理论对标:theory-foundation §3 D4)

**问题**:依赖矩阵只采各栈内部 import 边;多栈仓(Java+TS,用户项目几乎全是)前端→后端 HTTP 跨栈边不画,但 §6 自称「单一权威源」——对多栈名不副实。
**候选修法**:
- ~~**轻量(措辞诚实化、零 dogfood)**~~ **已于 v1.0 落地**:§6「单一权威源」加限定「仅栈内 import/装配边的权威源;跨栈运行时边(前端→后端 HTTP)属暂缓的 ②④、不在矩阵内」(line 170 + 产物 schema 表)。
- **重量(开 ②④ 能力、必 dogfood)**:retroactive linking——前端出站 HTTP 槽 + 后端 endpoint 槽,合并阶段按 path 延迟匹配,标「跨栈/HTTP 候选边、低置信」。撞慢变闸(path 快变),须抽象到慢变形状粒度。**仍开放**。
**待触发条件 / 搁置原因**:用户倾向「多栈专项单独找项目实跑验证后再针对性优化」。轻量已做;重量与 ②④ 解禁一并设计。

## 3. D5-B 单域超大模块 vs 单次精读容量(双闸正交)(理论对标:theory-foundation §3 D5-B)

**问题**:单域超大模块(如 300+ java)按业务内聚不拆该保一个,却超下游单次精读的上下文容量;曾有硬阈值被删(怕误伤单域大模块)。
**候选修法**:内聚闸(不拆)+ 容量闸(独立)双闸正交——容量闸只触发「标注超容量、建议分子域/拓扑分批 brief」,**绝不触发改边界**;阈值用 token 预算非文件数(ArchAgent adaptive grouping)。**坑**:极易退化成文件数硬阈值、复活 v0.3.0 删掉的项目魔数。
**待触发条件 / 搁置原因**:更罕见 + 改起来烫手(撞硬阈值血泪史)。同样**盯 slp**——出现某单域大到单次精读爆容量再做。属契约段,须自带 dogfood。

## 4. ②④ 入口与执行流 / API 表面能力槽(理论对标:theory-foundation §3 ②④)

**问题**:SKILL.md 留而未建的两个能力槽。
**候选修法**:② 入口借 call-path 有界 k-hop 追踪、只记慢变骨架链;④ API 表面优先复用契约文件(OpenAPI/proto)否则源码路由拼接、先认全类目再枚举(呼应 §5.5.1);④ 与 D4 后端 endpoint 槽**复用同一信号源**(一举两得)。**坑**:执行流易变,只记抗重构骨架、不画逐行调用栈(守慢变闸)。
**待触发条件 / 搁置原因**:真·能力扩张,与 D4 重量版同源,独立立项;留槽不建是 v0.6 起的有意设计(②③④ 仍有顾虑)。

## 5. 精确计数硬闸对「指令/枚举量词」假阳性(ergonomics,非语义缺陷)

**问题**:v0.12 机械 grep 闸(`[0-9]+ *(个|处|次…)`)会命中「按需选 1-2 个代表」(阅读指令量)、「1处理中」(状态机枚举码)这类**非提交漂移**的良性量词——slp/flycat 两个 executor **独立都撞到**。当前靠逐命中判定绕开,与「期望字面 0 命中」框定有张力。
**候选修法**:自检条允许「逐命中豁免标注」(命中但判定为非漂移量词时记一句豁免理由),而非强制全文字面 0;或闸正则排除已知良性形态。**坑**:别开成语义豁免后门(v0.12 正为堵此)——豁免须留痕可复核。
**待触发条件 / 搁置原因**:纯 ergonomics、低优;下次顺手或假阳性变多时做。

## 7. 微服务自暴露 Feign 的配方种子提示(理论对标:理论充分,属配方结晶)

**问题**:`@FeignClient(name=本服务名)` 是本服务对外暴露的 SPI(供别的微服务调)、**非调外部系统的 adapter**——slp executor 靠现场读 name 指向自身才识破,§5.3 配方层无种子提示,弱模型易误标 Adapter/外部依赖。
**候选修法**:§5.3 Java/Spring Cloud 行加种子「`@FeignClient` 的 name/value 若 = 本服务 `spring.application.name` → 本服务自暴露 SPI、非外部 adapter」。**坑**:跨服务调用(name=别的服务)仍是真 adapter,别一刀切。
**待触发条件 / 搁置原因**:微服务栈(Spring Cloud)样本更多时结晶;轻量、可随手做。
