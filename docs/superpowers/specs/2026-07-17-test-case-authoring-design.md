## 1. 背景与动机

### 1.1 要解决的问题

根据一次开发任务、或一个用户给出的功能业务领域，产出**测试用例规范**。除功能测试外，还要覆盖各种边界场景。

关键约束（需求方原话）：

> 写测试用例不用关注后续怎么测，只需关注要测的内容。具体怎么测交给后面的自动化测试执行者来定。

即产物是**「测什么」的规范**，与「怎么测」解耦。下游执行者是**另一个 AI agent / skill**，由它决定用什么框架、怎么构造请求、怎么断言。

需求方技术栈：Java 后端接口测试 + Vue / React / 微信小程序 E2E。

### 1.2 为什么不用现成 skill

调研了开放 skill 生态（`npx skills find`，多轮关键词）。结论：**没有匹配的现成 skill**。

| 候选 | 装机 | 否决理由 |
|---|---|---|
| `cexll/myclaude@test-cases` | 1K | 最接近，但见 1.3 —— 作为**先例参考**，不作基座 |
| `softaworks/agent-toolkit@qa-test-planner` | 4K | 大而全 QA 工具箱（Figma 校验/缺陷报告/回归套件），边界场景无专门方法论 |
| `github/awesome-copilot@pytest-coverage` | 11.6K | 仅 25 行，行覆盖率驱动 —— 与边界覆盖**相反**（边界 bug 恰恰藏在已覆盖的行里） |
| `trailofbits/skills@property-based-testing` | 4.3K | 质量过硬，但「不要用于」清单首条即排除集成/E2E 与带副作用代码 —— 正是本项目主场景 |
| `cognitedata/builder-skills@test-coverage` | 1.9K | 硬绑自家 Flows app 的 80% 门禁 + vitest 配置 |
| `owl-listener/designer-skills@test-scenario` | 861 | 可用性测试（UX 研究），与功能边界无关 |
| BDD/Gherkin 系 | ≤110 | 方向对（Given-When-Then 天然执行无关），但全是个人仓半成品 |

**旁证**：本仓 `docs/集成测试skill-调研.md` 2.2 独立发现「`integration` 一词被系统性误用」——19K 装机的 api-test-suite-builder 假设依赖已就绪，10K 的 test-master 用 `DELETE FROM users` 删表。与本轮发现同一规律：**测试类 skill 的装机量与质量基本不相关**。

### 1.3 先例参考：`cexll/myclaude@test-cases` 的可取与致命伤

**可取（本设计已吸收）**：
- 四类分区 TC-F / TC-E / TC-ERR / TC-ST
- 需求 × 用例覆盖矩阵
- 等价类划分 + 边界值分析（有效区间 18-100 → 测 17/18/100/101）

**致命伤（本设计针对性修复）**：

1. **原则四处宣示、零处校验，且自带反例。** 它在 SKILL.md 核心原则 #1、testing-principles.md 开篇、Good Test Cases、Common Pitfalls #1 四处写了「测行为不测实现」，但 Step 5 校验清单只查覆盖率（需求/场景/风险），**没有一条查执行细节泄漏**。更糟的是它 AAA 模式的唯一范例自带反例：

   ```
   3. Click "Login" button (Act)
   ```

   **范例的示范力远大于原则的宣示力。**

2. **根因是字段设计。** 它的模板有 `Test Steps` 字段——一个叫「步骤」的槽位天然只能填操作序列。字段决定内容，原则拦不住自己的字段。

3. **边界清单是通用语言级**（null/空串/最大长度/特殊字符/并发），不认领域。接口测试真正翻车的幂等重试、鉴权过期、事务回滚、分页越界，它一条都不提醒。

### 1.4 与 `module-spec-baseline` 的关系：零依赖 + 拄条目

本仓 `incubating/module-spec-baseline` 有高度相关的资产：行为优先纪律、`references/behavior-dimensions.md` 维度登记表、openspec Scenario 格式（WHEN/THEN）。

**核实过的成色**（重要，别误引）：
- 维度表**有真实来历**：v1.1.0 生自 GitNexus 对照复盘（发现「非 HTTP 入口与横切契约整类漏」），v1.2.0 经 5 模块实测打磨、并**砍掉** B4 出站签名。提交历史佐证：`53508e4`（B0 地基）→ `782a7ec`（B1 redo，"redo" 印证了回退重做）。
- **但**：`evals/` 是空目录（从无提交）；CHANGELOG 引用的 `docs/module-spec-baseline-optimization/` 已在 `eb475fa 过程产物清理` 删除。**实测结论无法独立复核，CHANGELOG 是仅存记录。**
- **且**：它验证的是「逆向 as-built 后端模块」场景。对正向 + 前端 E2E **零覆盖**。

**两条硬性不兼容，决定必须新建而非复用它**：
1. **方向反了**：它是逆向 as-built，反幻觉底线要求「每条必须追溯真实代码事实」，明文「specs 描述现状，不是愿望」。本 skill 是正向——代码可能还不存在。这是地基冲突。
2. **前端整块在范围外**：它明文「不做前端 UI 细节建模」。

**结论：零依赖 + 拄条目。** 新 skill 自建维度表，把 6 条后端维度当「有来历的输入」抄入并标注出处与成色；不调用、不 import、不抽公共层。理由：module-spec-baseline 仍在孵化（无 evals、agent 对表的消费分批接入未完），抽公共层等于焊死在移动靶上；且两张表服务对象不同（逆向契约 vs 正向用例），共用会互相拖累演进。它烂尾或大改，都伤不到本 skill。

## 2. 定位与边界

**一句话**：根据一次开发任务或一个业务领域，产出**执行无关的测试用例规范**——只回答「测什么」，把「怎么测」留给下游执行者。

| 项 | 取向 |
|---|---|
| 产物 | 测试用例规范 markdown（下游是 AI agent，markdown 足够；重点是接口稳定） |
| 抽象层次 | **执行无关**：可写被测契约，禁写测试实现（判据见 4.2） |
| 第一版范围 | **仅后端接口** |
| 触发 | 显式调用，不自动触发 |
| 外部依赖 | **无**（不依赖 recon-driven-dev、不依赖 module-spec-baseline、不需要 openspec CLI） |

**明确不做**：写测试代码、选框架、mock 策略、覆盖率、前端 UI（第一版）、bug-hunting。

**为什么第一版只做后端接口**：后端与前端 E2E 是两个领域——后端从 API 契约推导（边界是幂等/事务/鉴权/错误码，有 6 条已验证维度可拄），前端从用户旅程推导（边界是授权拒绝/网络切换/后台恢复/分包加载失败，**维度表零基础、需从零建并从零验证**）。两者仅共享纪律层，维度层几乎不重叠。纪律层本身尚未在任何真实任务上验证过——同时上两个领域，等于拿一套未验证的纪律去摊两张未验证的表，翻车时分不清是纪律错还是维度缺。

### 3.2 输入：双入口 → 归一成「待测事实集」

| 入口 | 来源 | 取什么 |
|---|---|---|
| ① recon 产物 | `docs/recon-driven-dev/<日期>-<change>/` | `requirements.md` §2.2 关键异常、§3 业务规则、§4 验收标准；`design.md` 的接口契约 |
| ② 直接给 | 对话 / PRD / 接口文档 | 同上四类事实，缺的当场问 |

**对 recon-driven-dev 零依赖**：走入口 ① 只是读它落好的 markdown 文件，不调用它、不要求它跑过。recon 的「自包含、零外部 Skill 依赖、丢到任何环境都能独立跑」不受影响。

**待测事实集**（归一化后的中间产物，不落盘）四类：验收标准、业务规则、异常场景、接口契约。**这是流程唯一的输入面**——不管从哪个入口来，后续步骤只认它。

**落点冲突已化解**：recon 立有硬原则「不引入跨会话持久件、不发持久编号」，但那约束的是 recon 自己的产物。本 skill 是独立 skill，产物落自己的目录，不受该原则约束。

### 3.3 流程

```
1. 输入归一化 → 待测事实集
2. 抽场景（按四类分区）
3. 逐条对维度表点名扫（不是"想想边界"，是拿清单点名）
4. 产用例
5. 出口自检一趟，过三门
```

## 4. 核心机制

### 4.1 砍掉「测试步骤」字段（结构性防御）

**整个设计最关键的一刀。** test-cases 必然写出 `Click "Login" button` 的根因是它模板里有 `Test Steps` 槽位。

本 skill 的用例**没有步骤字段**，改用 GIVEN / WHEN / THEN：

```markdown
#### TC-E-003 · 重复提交取消请求
- **维度**：D1 幂等/去重
- **来源**：业务规则 §3.2 / 验收标准 #4
- **优先级**：高
- **GIVEN** 一个处于开启状态的订单
- **WHEN** 对它连续发起两次相同的取消请求
- **THEN** 订单最终处于已取消状态，且只产生一条取消记录
- **THEN** 第二次请求以幂等方式返回成功，不报冲突错误
```

GIVEN = **状态事实**，WHEN = **意图**，THEN = **可观察结果**。三个槽位都没地方塞「点哪个按钮」「用 RestAssured 怎么断言」。

**这是结构性防御，不是纪律宣示**：不靠模型自律，靠没有那个槽位。

### 4.2 执行无关的界线：执行者替换测试

界线必须划准，否则会抽象到没法用。对后端接口测试：

- **可写（被测契约本身）**：接口语义、HTTP 状态码、业务错误码、响应字段、状态名
- **禁写（测试实现）**：用什么框架、怎么构造请求、怎么造数据、mock 谁、怎么断言

**判据 —— 执行者替换测试**：「把执行者换成人手测 / RestAssured / Postman / Playwright，这条用例还成立吗？」成立 → 执行无关；不成立 → 泄漏。

> ⚠️ 这与 module-spec-baseline 的黑盒判别测试**防的不是一个东西**：它防 spec 里混进类名/SQL/内部结构；本 skill 防用例里混进执行方式。**判据不能照搬。**

### 4.3 「来源」字段是防幻觉锚

正向场景独有的硬问题。module-spec-baseline 敢写「每条必须追溯真实代码事实」，因为它逆向已有代码，**有 grounding 锚**。本 skill 正向，代码可能还不存在——**没有锚就会编造用例**。

**解法：待测事实集就是锚。** 每条用例的 `来源` 字段强制回指某条验收标准 / 业务规则 / 异常场景 / 接口契约。**无源即非法**：想测但事实集里没依据，不是偷偷补用例，是回头问用户、补进事实集再生成（泄压阀见 6.4「待澄清」）。

比 test-cases 硬一档：它是「链接到需求（可选做到）」，本 skill 是「无源即非法（出口门校验）」。

### 4.4 维度表与四类分区是**正交**的两个轴

不讲清楚，实现时必打架。

- **四类分区**（TC-F 功能 / TC-E 边界 / TC-ERR 错误 / TC-ST 状态迁移）= **归档轴**，用例写完往哪放。抄自 test-cases。
- **维度表** = **召回轴**，保证不漏。它**不是分类法，是点名清单**：拿着表逐条问「这条维度在本次任务里有没有对应用例」。

一条维度会散落到多个类：D1 幂等的「重复提交」落 TC-E，「并发超限被拒」落 TC-ERR。**别试图让维度对齐分类**。

## 5. 维度表（`references/backend-api.md`）

分两层。

### §1 输入契约边界法（方法，非清单）

等价类划分 + 边界值分析。抄自 test-cases 唯一的真材实料：有效区间 18-100 → 测 17 / 18 / 100 / 101。这层管「单个字段怎么取值」。

### §2 领域维度清单（点名扫）

第一版 6 条，全部拄自 module-spec-baseline：

| 维度 | 测什么 | 出处 |
|---|---|---|
| D1 幂等/去重/并发 | 重复提交的后果、并发上限与拒绝策略 | msb B7 |
| D2 鉴权前置 | 无凭据/过期返回什么、哪些路径豁免 | msb B5 |
| D3 事务边界/回滚补偿 | 部分失败是整批回滚还是留脏数据 | msb B1 |
| D4 缓存读一致性 | 写后立即读，是否有读旧值窗口 | msb B6 |
| D5 错误响应契约 + 成功信封 | 错误码/状态码映射、分页缺省 | msb C1 |
| D6 异步最终一致 | 「接口返回成功 ≠ 下游已就绪」 | msb A2 |

**每条必须标注成色**：`[来源：module-spec-baseline behavior-dimensions · 后端逆向场景实测 · 未在正向场景验证]`。把真实成色写在脸上，别让后来的人（或三个月后的自己）误以为这是正向验证过的。

### §3 不默认登记（元规则，抄自 msb 并升级）

> 限流/防刷、多租户数据隔离、版本/向后兼容 —— 很多项目根本没有，默认写进会臆测出不存在的契约。

**对正向场景风险更高**：msb 至少能去代码里探测有没有，本 skill 连代码都没有，更容易凭空造出「应该限流」的用例。**升级为硬规则：这类维度只有在事实集里有明确依据时才生成用例，否则连问都不问。**

## 6. 出口三门与产物

### 6.1 出口自检一趟，过三门

不分三趟——依据 msb 实测教训「门 2/3/4 折进 grounding 一趟、成本不涨」。

| 门 | 查什么 | 判据 |
|---|---|---|
| **A · 无源门** | 每条用例的「来源」能回指事实集某条 | 回指不上 → 删除或回头补事实集，不许留 |
| **B · 执行泄漏门** | 用例里没混进执行方式 | 两级，见下 |
| **C · 维度覆盖门** | 维度表逐条有交代 | 每条标「已覆盖（列 TC-ID）」或「N/A + 具体理由」；**不许空、理由不许写"无"** |

### 6.2 门 B 分两级——词表拦不住软泄漏

依据 msb 一手教训：首版「grid + grounding 双门过了」，A/B/C 复盘照样发现**软实现泄漏**（异步线程内 / 删后插 / 签名派生）。这类不含任何禁用词，硬词表一个字都拦不住。

**硬拦（机械可查，grep 级）**：框架/工具名（RestAssured、Postman、JUnit、Mockito、Playwright）、UI 操作动词（点击、输入、滚动、跳转）、mock/stub 指令、断言方式（assertEquals、校验 JSON path）。

**软拦（逐条判断）**：执行者替换测试。最易漏的是**状态事实 vs 构造方式**：

- ✅ `GIVEN 一个处于开启状态的订单`（状态事实，执行者自行决定怎么造）
- ❌ `GIVEN 向 t_order 表插入一条 status=OPEN 的记录`（构造方式，已替执行者做决定）

两句描述同一前置，后者是软泄漏的典型形状。

### 6.3 覆盖矩阵做双向

test-cases 只做正向。双向才闭环：

- **正向**：每条事实 ≥ 1 条用例 → 防漏测
- **反向**：每条用例 ≥ 1 条来源 → 防编造（门 A 的产物形态）

反向是 test-cases 没有的，而它恰恰是正向场景（无代码可锚）最需要的。

### 6.4 产物结构

```markdown
# 测试用例规范 — <任务名>

## 概览          输入来源 / 事实集摘要
## 1. 功能测试 (TC-F)
## 2. 边界测试 (TC-E)
## 3. 错误处理 (TC-ERR)
## 4. 状态迁移 (TC-ST)
## 覆盖矩阵      事实 × 用例，双向
## 维度扫描表    D1-D6 逐条：已覆盖 / N/A + 理由
## 待澄清        事实集缺口
```

**「待澄清」是必备出口，不是附录。** 正向场景一定撞上事实集不全（需求没写清超时怎么办、错误码没定）。规则：**不许猜，落这里回头问用户**。这是门 A 的泄压阀——没有它，门 A 只会逼模型偷偷编一条来源。

## 7. 验证方式

**第一版不建 evals**，直接拿一次真实的后端接口开发任务跑。

理由：纪律层零验证，先撞真实任务比先造 eval 架子有信息量。反面教材就在隔壁——`module-spec-baseline/evals/` 是个从无提交的空目录。

**要观察三件事**：

1. **三门有没有真拦住东西** —— 一次都没拦住 = 门是摆设，或任务太浅、样本无效。
2. **维度表有没有召回本来会漏的** —— 这是它存在的唯一理由。没召回到任何东西，说明 6 条拄来的维度在正向场景不适用。
3. **「待澄清」有没有真挡住编造** —— 若它空着但用例齐全，多半是模型在偷偷补来源（门 A 假过）。

**CHANGELOG v0.1.0 明写**：「纪律层与维度表均未在正向场景验证」。照 module-spec-baseline 标注成色的做法执行。

## 8. 关键设计取舍（记录 why）

| 取舍 | 决定 | 为什么 |
|---|---|---|
| 基座 vs 先例 | test-cases 作**先例参考**，不作基座 | 把它从「通用手工用例」硬掰成「Java 接口 + 执行无关契约」= 架构大改，正是 recon 批次 A/B/C 翻车后立规矩要避开的坑 |
| 复用 msb | **零依赖 + 拄 6 条** | msb 仍孵化（无 evals、分批接入未完），抽公共层 = 焊死在移动靶；两张表服务对象不同，共用互相拖累 |
| 防泄漏手段 | **砍字段** > 加原则 | test-cases 证明：有 `Test Steps` 字段，四条原则也拦不住 |
