# recon-driven-dev 实战驱动打磨机制 · 设计稿

> 日期:2026-06-23 · 状态:设计待实施
> 目标:为 recon-driven-dev 建一套**与具体开发任务无关**的验证机制——拿任意真实开发任务跑这个 skill,在过程中系统捕获 skill 自身的缺陷信号,经分诊→修复→验证→沉淀,让它越用越稳,并能回答「整个 skill 是否实测到位」。

## 0. 两条贯穿铁律

1. **机制完全从 recon-driven-dev 自身长出**。验证维度、每个构件的存在理由,都基于它自己的[全局护栏](../../../skills/recon-driven-dev/SKILL.md)与[维护宪法 MAINTAINING.md](../../../skills/recon-driven-dev/MAINTAINING.md) 推导,不照搬任何其他 skill 的方法论。
2. **绝不耦合其他 skill**。recon-driven-dev 自 v0.4.0 起已与其他 skill 完全解耦;本机制的文档、构件、措辞均不得引用、对齐、参照任何外部 skill,不走回头路。

> 特别地:其他薄编排型 skill 用「薄度」当审美/验证维度——本 skill **不是**薄编排件,而是**自包含五阶段流水线**(各阶段能力内联拥有、reference 有实打实细则)。它的命是「自包含 + 分层 + 单一权威」,不是「薄」。对应的防膨胀概念是它自己宪法里的**「单路径承载软上限」**。

## 1. 成功标准:地板 + 天花板

「整个 skill 实测到位」= 两条同时满足:

- **地板(覆盖账本)**:skill 的每一条可观测路径单元,都至少被一种覆盖手段碰过、且判定「指令够用」(不带 `⚠`)。
- **天花板(缺陷收敛)**:被碰过的路径,连续 N 轮实跑(建议 N=2)不再冒出新的 skill 级改进点。

地板保证「每条路径都碰过」,天花板保证「碰过的地方够稳」。

## 2. 总览拓扑

```
┌──────────────────── 实测会话(对象层 · 保持纯净)────────────────────┐
│  拿任意真实开发任务,用 recon-driven-dev 跑全程                       │
│                                                                      │
│  ①─⏸─②─⏸─(③?)─⏸─④─⏸─⑤─⏸     每个 ⏸:主 agent 除呈现产物,        │
│   └───────────┬───────────┘       附一行「skill 体验自评」           │
│               │                    (顺 / 卡在哪 / 指令含糊 / 降级没走通)│
│           跑完整任务                        ↓ 落 监督笔记             │
│               └─ 派复盘 agent:读 监督笔记+实走路径+产物,对抗补刀   │
│                  抓主 agent 自评漏掉的 skill 缺陷                    │
│                                                                      │
│  产出A = 监督笔记(厚·业务现场)   产出B = 改进点清单(薄·交接物)    │
└────────────────────────────┬─────────────────────────────────────┘
                             │  交接物 = 改进点清单 + 覆盖账本增量
                             ▼
┌──────────────────── 打磨会话(元层 · 专注)──────────────────────┐
│  读改进点清单 →                                                    │
│   ① 三分诊:skill 缺陷? / 任务特例? / 模型自身失误?              │
│        └─ 仅「skill 缺陷」入流程(防特例·失误污染 skill)          │
│   ② 处置门槛:局部低风险 → 当场改 ; 涉护栏·动骨架·单样本 → 搁置   │
│        ├─ 改:改 skill + 验证(消费者映射 + 对抗 rubric)→ CHANGELOG│
│        └─ 搁置:进 BACKLOG(problem / 候选修法 / 待触发条件)       │
│   ③ 回填覆盖账本:该格 触发手段 + 「指令够不够用」判定            │
└──────────────────────────────────────────────────────────────────┘
```

**两个内置防污染闸**:
1. **会话分离**——实测会话只跑+记笔记,改 skill 在另起的打磨会话做。元层动作不污染实测纯净度;主 agent 不在被改对象的会话里改它。
2. **三分诊**——任务特例 / 模型失误不被误当 skill 缺陷而乱改 skill。

## 3. 资产分层:常驻薄协议 + 即弃厚现场

| 资产 | 体量/寿命 | 落点 | 运行时读? |
|---|---|---|---|
| 实测打磨**协议**(怎么监督/复盘/分诊/验证) | 稳定 · ~1 屏 | `skills/recon-driven-dev/MAINTAINING.md` 新增一节 | 否 |
| **BACKLOG**(搁置候选改造项) | 缓增 · 维护者备忘 | 新建 `skills/recon-driven-dev/BACKLOG.md` | 否 |
| **覆盖账本**(路径单元 × 触发状态) | 一份表 · 随实跑更新 | 新建 `skills/recon-driven-dev/EVAL-COVERAGE.md` | 否 |
| **监督笔记**(每跑一份 · 含业务现场) | 量大 · 易脏 | `docs/recon-driven-dev-eval/<date>-<task>/supervision.md` | 否 |
| **改进点清单**(交接物) | 每跑一份 · 用完即弃 | `docs/recon-driven-dev-eval/<date>-<task>/improvements.md` | 否 |

设计依据(全部本 skill 自身):
- 协议/BACKLOG/账本进 skill 包但**运行时不读**——符合本 skill「运行脊柱 vs 维护资产分层」:SKILL.md 只持运行脊柱,这三者是维护资产,不进脊柱、不增加任何执行路径的 reference 载入量(不触碰「单路径承载软上限」)。
- 监督笔记/改进点清单**不进 skill 包**——它们量大、含具体业务现场,放 skill 的开发仓 `docs/` 下,归档即弃。这与全局护栏「不引入跨会话持久件」**不冲突**:该护栏约束的是**业务开发产物**(directed-report 等随归档清);监督笔记是**改 skill 用的元层维护资产**,本就该持久,归属维护侧。
- 监督笔记/改进点清单不进被开发的业务项目——保持本 skill 与被开发项目的产物边界(业务产物随归档清),元层资产不混入业务项目。

## 4. 覆盖账本设计

### 4.1 路径单元定义

skill 在某条执行路径上**给出独立指令、且可能独立失效**的最小观测点。判据:*走到这里时 skill 有没有专门的话要说?这话照做会不会单独出问题?* 是 → 一个单元。

### 4.2 覆盖手段四态(让「细」可实施的关键)

不假装每格都被真实压过,而是用最合适、最低成本的手段覆盖每一格,诚实标注强度:

- `✓真实实跑` — 真实任务自然走过。
- `▲人造场景` — 靠选项切换触发(选子 agent 执行 / 选丢弃收口等),动作真实、成本极低。
- `◇桌面走查` — 读那段 skill 指令 + 静态核它自洽可执行,**不实跑**。用于无法真实触发的格(环境依赖降级、需人为埋错的失真类)。真实任务偶然撞上时可顺手升 `✓`。
- `·未触发` — 还没碰过。

**地板达成** = 每格 `≥◇` 且「指令够用」不带 `⚠`。

## 5. 实测会话:监督协议

### 5.1 现场层:每 ⏸ 主 agent 自评

主 agent 在 skill 既有的每个 `⏸` 暂停点,除呈现阶段产物外,**附一行「skill 体验自评」**写进监督笔记:这一步顺 / 卡在哪 / 哪句指令含糊 / 降级路径有没有走通,带现场证据(哪句 skill 指令导致什么)。零新增停点——嵌入既有 ⏸。

监督笔记结构:
```
# 实测监督笔记 · <date> · <task-slug>
> 实跑 recon-driven-dev 全程的现场记录。元层维护资产,归档即弃。

## 走过的路径单元
（列账本格号）

## 逐 ⏸ 体验自评
- <阶段><⏸>  顺/卡/含糊/降级 + 现场证据  [→ IP-NN]
```

### 5.2 复盘层:整任务跑完派复盘 agent

每个完整任务跑完,派一个**上下文隔离的复盘 sub-agent**,输入「监督笔记 + 实走路径单元 + 全部产物(directed-report/requirements/design/review/tasks/代码)」。

- 职责:专猎主 agent **照 skill 做了、却被指令误导而不自知**的盲区(自评的天然盲点)。
- 边界:只评 **skill 指令本身**在这次实跑暴露的问题,**不评业务代码质量**(那是 skill ⑤ 自己的职责)。
- 产出:补充改进点,合并进清单、标 `▲复盘补`。

> 派隔离 agent 给对抗视角,是 recon-driven-dev 本就有的派发范式(①③⑤ 全用它);复盘 agent 只是同一派发/降级范式在「评 skill 自身」上的再次运用——无隔离能力时同样降级主会话自跑。

### 5.3 交接物:改进点清单

```
# 改进点清单 · <date> · <task-slug>   (交接物 → 打磨会话)
IP-NN [▲复盘补]? | 来源 <阶段/⏸/路径单元号> | 现象:<skill 指令怎么让 AI 卡/错/含糊> | 初判 <skill缺陷?/任务特例?/模型失误?> | 影响面 <每次X/某分支> | 严重 <高/中/低>
```

「初判」只是实测会话的快速归类;最终定诊在打磨会话。

## 6. 打磨会话:分诊 → 处置 → 验证 → 沉淀

### 6.1 三分诊(防污染 skill 的核心闸)

对每条改进点定诊,可派隔离对抗 agent 复核(防主观):

- **skill 缺陷** — skill 指令本身的问题(含糊 / 缺动作 / 降级没说清 / 停点冗余 / 判据漂移 / 跨阶段衔接断)。**仅此类入修复流程。**
- **任务特例** — 本项目/环境特殊导致,非 skill 通用问题。→ 可回写被开发项目的文档,**不改 skill**。
- **模型自身失误** — skill 指令没错,是 AI 没照做 / 能力波动。→ 至多提示 skill 某处措辞可加固,本质不算 skill 缺陷。

依据本 skill 维护宪法对「通用 vs 特定」的取向:复用资产只放通用约束,项目/环境特有的事实不污染 skill。

### 6.2 处置门槛

确认为 skill 缺陷后:

- **当场改** — 局部 · 低风险 · 单样本足以确认(某句指令含糊、某 ⏸ 模板缺一句):改 SKILL.md/reference + 验证 + 立 CHANGELOG。
- **进 BACKLOG 搁置** — 涉护栏权衡 / 动骨架 / 单样本不足以定方向 / 疑似需多实跑样本佐证:记 `problem 事实 / 候选修法 A/B/C / 搁置原因或待触发条件`。

BACKLOG 结构:
```
## N. <标题>(来源:<date> 实跑·<阶段>发现)
**问题事实**:…
**候选修法**(讨论于 <date>,未定):A. … B. … C. …
**搁置原因 / 待触发条件**:…
```

### 6.3 改后验证 rubric(从 recon-driven-dev 自身护栏 + 宪法推导)

**A. 结构契约没破(改对了没)**
- **自包含性** — 没引入外部 skill 调用 / 脚本 / 跨会话持久件依赖。【唯一硬约束】
- **单一权威源** — 没制造第二处判据复述;若动了某判据的家,按名引用它的地方(模板维度名、产物落点 / 命名空间)同步改、没漏。
- **分层归位** — 运行约束落脊柱、治理规则落 MAINTAINING、判据落 reference,没放错层。
- **派发-降级对称** — 改了派发侧契约,主会话降级侧同步;反之亦然。
- **阶段衔接·不回流** — 线性没破、把关没堆下游、跨阶段信号产出义务(② 给评审决策点的信号、② 给 ⑤ 的测试缝隙约定)没断。

**B. 改动本身够格(改好了没)**
- **可执行性** — 改的指令是 AI 真能照做的动作工单,不是陈述体 / 职位说明书,不依赖它没有的命令或信号。
- **单路径承载** — 改动没让某条执行路径(内联 / 子 agent)单次 run 载入的 reference 量超软上限。【防膨胀】

**验证怎么跑**:① 受影响阶段的**消费者映射**——改了谁 → 谁按名/按序消费它 → 只重验那条链,不读该产出的环节不必重跑;② 派一个**隔离对抗 agent** 逐条过上面这份 rubric。

**诚实定性**:改完如实区分「能力提升」还是「工程收益」(可维护 / 单一权威 / 抗漂移多属后者),CHANGELOG 照实写,不 oversell。

### 6.4 回填账本 + 沉淀

- 回填覆盖账本:本次走过的格 → 更新触发态 + 「指令够用」判定 + 关联改进点号 + 最近实跑日期。
- 已采纳改进沉 CHANGELOG(what + why);搁置项沉 BACKLOG。

## 7. 收敛判定

- **地板达成**:覆盖账本每格 `≥◇` 且不带 `⚠`。
- **天花板达成**:被碰过的路径,连续 2 轮真实实跑不再新增 skill 缺陷类改进点(BACKLOG 与 CHANGELOG 在这 2 轮无 skill 缺陷新增)。
- 两者皆达 → 宣布「当前版本实测到位」,机制转入低频维护(skill 后续大改后,受影响路径单元的覆盖态重置为 `·未触发`,按本机制重测)。

## 8. 落地清单(交付物)

**新建 / 改动 skill 包内(常驻资产)**:
1. `skills/recon-driven-dev/MAINTAINING.md` — 新增「实测打磨协议」一节(监督协议 + 三分诊 + 处置门槛 + 改后验证 rubric + 收敛判定的权威),并在 README 维护资产说明里挂指针。
2. `skills/recon-driven-dev/BACKLOG.md` — 新建,带文件头说明(运行时不读 · 元层备忘)。
3. `skills/recon-driven-dev/EVAL-COVERAGE.md` — 新建覆盖账本,初版填入 §4.3 全集、触发态全 `·未触发`。
4. `skills/recon-driven-dev/README.md` — 目录结构补 BACKLOG.md / EVAL-COVERAGE.md 两行。

**仓内维护侧(即弃现场,按次产生,不进 skill 包)**:
5. `docs/recon-driven-dev-eval/<date>-<task>/{supervision.md, improvements.md}` — 每次实跑产生,模板见 §5。

## 9. 自洽性说明

- 与全局护栏「不引入跨会话持久件」的关系:见 §3,该护栏约束业务产物,本机制资产是元层维护资产,归属不同、不冲突。
