# 复盘：Markdown TOC 实跑在阶段③六轮未通过后终止

## 结论先行

本次没有进入实现阶段。阶段①和②顺利完成，阶段③连续运行 6 个上下文隔离的 fresh reviewer：前 5 轮均为“有条件通过”，第 6 轮升级为“不通过”。六轮共提出 14 条阻断/须改和 2 条可改；每轮开始前，上一轮须改均已回到②修订并由用户重新批准。失败形态不是“同一缺陷修六次仍失败”，而是**每个 fresh 全量 reviewer 在不同轮次重新发现此前未报告的潜伏边界**，使评审面持续外扩，最终没有收敛到④。

最关键的观测是：第 5 轮才报告空 slug 行为，第 6 轮才报告绝对缩进问题；但本次只逐轮归档 review，没有冻结 v1～v5 的 requirements/design 输入与修订 diff，因此**不能从现存产物机械证明它们何时首次引入**。可以确认的是 fresh reviewer 的发现集合明显逐轮变化；“每轮完整重判”提供了新视角，却没有旧项核销、修订邻接扫描和上限后范围冻结契约。

## 六轮为什么没有收敛

### 1. 不是旧发现没核销，而是发现面逐轮漂移

每轮 reviewer 都是 `fork_context=false` 的 fresh agent，只读取当前 `directed-report.md`、`requirements.md`、`design.md`，没有读取前轮 review。这样避免了旧结论锚定，但代价是复评没有旧项核销表，也无法区分旧项残留、修订回归和全新发现。六轮的发现集合明显变化；由于没有保存每轮输入版本，不能进一步精确量化“潜伏缺陷”和“修订派生”的占比。

### 2. “完整性凭常识全判”没有收敛边界

`review-agent.md` 要求完整性/一致性/可实施性凭常识全判，探针只是地板、非天花板。对这个小 CLI，fresh reviewer 可以持续把“合理但此前未写死”的字符、Markdown、安装、文档和测试细节提升为须改。过滤门只滤不改档的噪声；只要 reviewer 认为会影响一个验收项，就可继续扩面。

### 3. 两轮上限能止损，但“继续修”后的契约不完整

R3 后主会话正确触发两轮上限并暂停。用户为实跑选择继续修后，skill 只给“继续修/降范围/终止”，没有规定继续修后应：冻结需求范围、只复核旧发现、扫描邻接范围，还是再次完整全量评审。主会话沿用 fresh 完整评审，R4–R6 因此每次重新打开全域发现面。

### 4. 主设计也确有真实缺口

不能把六轮全归咎于 reviewer。R1 的 slug 碰撞、R2 的标签转义/围栏语法、R6 的跳级缩进都是真实可交付风险；初版②自评只扫占位符/矛盾/歧义/范围蔓延，不负责这些深度问题。问题在于深度评审没有高召回地一次性聚合主要缺口，而是串行释放，导致评审成本远超一个小工具的实现成本。

### 5. 评审迭代本身反向扩大了设计

为了让每条发现可机械裁决，设计从一个小型 TOC 工具扩到 Unicode category、全局 slug 探测、Markdown 标签转义、精确围栏语法、干净 venv、README stdout diff、VSCode 六向量证据等。多数单点合理，但叠加后出现 sprawl；阶段③从“拦截坏方案”变成“持续完善规格百科”。

## 对 skill 的初步判断

- **高置信 skill 缺陷**：多轮只归档 reviewer 输出，不冻结该轮 requirements/design 输入或修订 diff；fresh “复评”也不消费前轮 review，无法机械核销旧项。
- **高置信 skill 缺陷**：两轮上限后的“继续修”缺少范围冻结和复评口径；正确修法不能简单变成“只核旧项”，而应是旧项核销 + 修订差异/邻接面的有界扫描 + 超范围新建议另行裁决。
- **skill 缺陷 × 模型波动 × 任务放大**：review-agent 的开放式常识全判没有发现分类与比例尺；但要求首轮一次发现全部边界同样不现实，不能把本次 Markdown 细节沉积进常驻 prompt。
- **skill 时态缺口 × 模型解释偏差**：R4 的 C9 建议不是纯模型失误；①允许写未定时点的“当前状态”，③又要求把后续真值差异上报为订正。
- **任务选型是放大器，不是单因根因**：它天然可拆 3 个 task 并适合批次 C，但同时跨 Unicode、Markdown、打包、README 与真实预览器，边界密度高。未触发⑤是实测编排目标未达，不能反推 runtime skill 应降低③门槛。

## 对批次 C 的影响

本次不能声称批次 C 五条通过。设计阶段已经为两种 verification-mode、3 个任务和子 agent 执行准备了契约，但没有任何实现、失败测试、task-agent、task-reviewer、manual evidence 或整支 code-reviewer 实际运行。终止态只能证明 preflight 的 START_SHA 与 untracked sentinel 保留机制仍在，不能替代⑤的真实回归。

## 建议带入打磨会话的问题

1. 两轮上限后若用户选“继续修”，是否强制先冻结 scope，并把下一轮改为“旧发现核销 + 修订触及面邻接扫描”，禁止再次无界完整评审？
2. ③ 是否需要把一轮中的发现按“直接违反已批准要求 / 新增边界完善”分层，后者默认可改或进入实现期验证，而非持续阻断设计？
3. 是否在第三轮仍新增与修复无关的须改时，自动建议“降范围/接受风险/终止”，避免小任务规格化 sprawl？
4. 如何提高首个 review-agent 对既有 design 的稳定召回，而不是靠多 fresh reviewer 串行补洞？
