# 隔离复盘结论

## 总体判断

主 agent 在阶段③的主路径上基本照 skill 行事：每轮收到非“通过”结论后回②修订、让用户重新批准、再派上下文隔离 reviewer；R3 后也按两轮上限暂停，并在用户明确选择继续后才延长。因此，六轮未收敛不能归为“主 agent 擅自多审”或“旧项未修”。

本次暴露的是三类因素叠加，不能单因化：

1. **高置信 skill 缺陷**：③的“复评”实际仍是无历史的全量开放式重判；逐轮留存只保存 reviewer 输出，不保存该轮输入版本或修订差异；两轮上限后的“继续修”又没有下一状态、范围冻结、复评口径和新上限。skill 因而既不能机械核销旧项，也不能证明后续新项究竟是早已潜伏、修订诱发还是 reviewer 漂移。
2. **skill 缺口 × 模型波动**：`review-agent.md` 要求“凭常识全判”、探针只作地板且非天花板，却没有把“已批准要求的直接违反”“为封闭规格而新增的边界”“本轮修订引入的回归”“可延期风险”分开。fresh reviewer 的召回波动本身是模型属性；skill 缺陷在于没有为这种可预期波动提供有界的恢复协议，而不是没能保证首轮一次发现所有问题。
3. **任务特例放大**：该任务虽代码量小，却同时跨 Unicode 分类、Markdown 围栏/列表渲染、slug、打包安装、README 契约和真实预览器证据，开放边界密度高，容易触发“凭常识全判”的扩面。它不是唯一根因，但不能像主复盘那样完全排除任务选型的放大作用。

R4 对 C9 阶段号的“事实被推翻”也不宜纯判模型失误：报告写的是未带时点的“当前 phase 为①”，而 skill 同时要求 reviewer 对被后续证据推翻的事实报订正；生命周期自然推进与事实证伪之间缺少契约区分，reviewer 是被两条指令共同诱导。另一方面，本次从未进入④⑤，不能把任何尚未发生的⑤问题或“未覆盖批次 C 下游路径”反焊成③的产品缺陷。

## 主复盘中应纠正/收紧的归因

1. 将“R5 空 slug、R6 绝对缩进都从早期 design 就存在”收紧为**待证假设**。`references/requirements-design.md:107` 只要求留 review，不留 v1～v5 输入快照；当前证据能证明它们最迟在对应轮次被发现及 v6 中的状态，不能完整证明首次引入轮次。尤其 R6 的“README 必写空 slug 规则”明显是 R5 产品策略确定后的派生闭环（`review-r5.md:40-43` → `review-r6.md:14`），不属于初版潜伏项。
2. 不要把六轮新增统一叫“fresh 召回漂移”。至少拆成：早先未召回的直接冲突、为封闭开放规格而新增的决策、上一轮修订产生的派生闭环/潜在回归、模型或时态解释偏差。没有 per-round 输入版本时，无法进一步精确量化四类占比。
3. 将 R4 C9 从“模型失误/过归因”改成“**skill 时态缺口诱发、模型作了不理想解释**”。`directed-report.md:13` 的“当前”没有报告时点，而 `review-agent.md:38` 确实要求发现后续真值不一致时上报；这不是纯粹没照指令做。
4. IP-04 的“继续修后缺契约”可从疑似提升为高置信 skill 缺陷，但修法不应直接冻结为“只核既有问题”。那会制造相反的漏检；应保留对本轮修订差异及紧邻影响面的有界回归扫描，同时把超出冻结范围的新建议单独交用户裁决。
5. IP-03 应拆根因，不宜把“六轮发现不全”整体要求 skill 用更多探针修到首轮高召回。`MAINTAINING.md:6-15,111-115` 本就要求防膨胀和单路径承载；把六轮所有具体边界塞回 reviewer prompt 会沉积成任务百科。应修的是跨轮状态、发现分类和停止协议，而不是把本次 Markdown/Unicode/打包案例常驻化。
6. IP-05 中“未触发⑤、没完成批次 C 覆盖”是**实测编排目标未达**，不是运行时 skill 应降低③门槛的证据。R6 已给出“不通过”，按 `requirements-design.md:99-101` 阻止进入④是契约内正确行为；不得为了覆盖⑤而把真实③结论绕过去。任务选型可定为放大器/批次安排不佳，但不是单独的 skill 修复理由。
7. 本次没有进入⑤（`run-state.md:6-7,20-21`；`retrospective-staging.md:51-53`）。任何关于 task-agent、TDD、manual-evidence、task-reviewer 或整支 code-reviewer 的缺陷都仍是未发生、未验证，不能从③的不收敛外推。
