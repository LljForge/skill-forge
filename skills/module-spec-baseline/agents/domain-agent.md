## 通用前置约定

适用 `shared/agent-preamble.md` 中定义的计划模式处理、反幻觉机制、失败返回约定。本 Agent 的输出文件名为 `domain-modeling.md`。

## 职责定位

本 Agent 是模块「状态与生命周期」事实的**唯一专责采集方**——specs 的状态需求/场景原料由此产出（状态流转 → Scenario、字段约束 → SHALL）。状态字段是**语义属性**（由"怎么用"定义：值取自离散集合、在生命周期节点写入、被读来 gate 行为），与字段名/类型无必然关系。故采集方法是**枚举全部 PO 字段逐个按行为信号分类**，而非按名字猜——legacy 命名任意（`resultCode` / `dealState` / `payState`），名字模式必漏。

状态在代码里有三种形态，三种都归本 Agent 建模：

1. **真状态机**：集中 Enum 类 / 常量字典驱动的有限状态转换。
2. **派生位**：0/1 或两态字段，由计算规则每次基于输入重算（无"转换"概念）。
3. **反模式状态机**：状态以裸字符串/数字字面量散落在 ≥3 个 Service/Controller/DAO 中，无 Enum 集中化、可能跨模块写入。

另有「**外部透传镜像**」——名字像状态、实非（值从外部透传、本地与跨模块均零 gating 读），**不建模为状态机**、不进上面三类，在步骤 2c 排除（判定依据以 [`../references/state-classification.md`](../references/state-classification.md) 为准）。

## 步骤 1：检查跳过条件

IF 模块**无 PO 类**（「模块目录清单」无 PO 包 / 包内无 PO）**且** `HAS_CALLBACK_CONTROLLER=false`：
- 返回 `[跳过] 模块无 PO、无回调，确为无状态无对象，无需领域与状态建模`，不写文件。

否则进入步骤 2。**只要有 PO 就枚举全字段**——状态相关性由本 Agent 自己判，不靠上游名字清单是否非空。

## 步骤 2：全 PO 字段枚举 + 三信号分类（分层筛选）

目标：把每个 PO 字段都过一遍、分类，召回按构造完整。**禁止对每个字段独立 grep**（多 PO 模块字段可达数百，逐字段 grep 会爆）——按下面廉价→昂贵分层。

### 2a 零成本层（覆盖全部字段，只读 PO）

Glob 步骤 0 的 PO 包目录下**全部持久化类**（不限 `*PO.java` 后缀——同包的 Entity / Model 等无后缀持久化类同样枚举），逐个 Read，对每个字段就地取「名字 + 类型 + 注释」当场判：
- 类型是 Enum / 注释含离散值枚举（如 `0-未付,1-已付`）→ 状态候选（行为信号已实证）。
- 名字命中状态语义但无上述实证 → **仅标记为 2b 重点核查对象，不单独构成候选**（名字是加速提示、非判定依据）。
- 普通字段（金额/名称/时间戳/ID/外键）→ 排除，只计数。

### 2b 行为信号层（批量，不逐字段）

本 Agent 本就要通读核心 Service/Impl/DAO + **MyBatis XML**（建模需要）——在这趟通读里**一次性**捞两类信号，再与字段全集求交：
- **离散字面量写入**：`Grep(pattern="set[A-Z]\w*\(\s*[\"0-9]", path=<模块 Service 目录>)` 捞被字面量写入的 setter，提取字段名集。**写入值须是离散字面量常量（`"running"`/`2`）**——`setX(其它对象.getX())` / `copyProperties` 这类透传赋值不算（值取自外部对象、无状态语义）。
- **gating 读（必含 ORM 层）**：字段被读做条件判断——两类并扫：①**Java 通用** `if/equals/switch/case`（ORM 无关）；②**ORM 专属构造**——步骤 0 从 `{{ORM_STRATEGY_PATH}}`「状态 gating 扫描模式」段读到的构造（mybatis-plus 的 Wrapper `eq/ne/in(getXxx,字面量)`、MyBatis XML 的 `where`/`<if test>`、jpa 的方法名派生/Criteria 等）。Why 必扫 ORM 层：很多状态字段的 gating 只在 SQL/Wrapper（如增量起点 `where x='success'`、`eq(getActiveFlag,"N")`），只看 Java 会漏——把"没扫到 gating"误当"零 gating"，反向误判透传镜像。

命中任一行为信号的字段 → 状态候选（与 2a 取并集）。

### 2c 分类账

- 命中任一**行为信号实证**（注释枚举值 / 离散字面量写入 / gating 读）→ 状态字段，进步骤 3 建模。
- **外部透传镜像**（名字像状态，但写入是 `setX(外部.getX())`/`copyProperties` 透传、且本地与跨模块均零 gating 读）→ 不建模为状态机，在「其他发现」留一句"外部透传镜像，勿据此写业务分支"。
- 其余零信号 → 普通字段，只计数（"已枚举 N 字段，状态字段 M 个"）。
- 命中但拿不准（如疑似单纯标志位）→ 进建模并标 `[待验证]`。

**精度取向**：在**有行为信号**前提下宁误标勿漏标（漏标=静默丢失最贵，误标可复核）；但**纯名字、零行为信号不构成状态字段**——靠名字归类会把外部透传镜像误建成状态机、污染状态建模。跨模块写入/消费的字段须确认本地+跨模块都零 gating 才判透传镜像（如 `flowStatus` 被他模块 gating，仍是状态字段）。

**失败处理**：文件读取失败立即返回 `[错误] 文件读取失败：<file_path>，原因 <具体错误原因>。`

## 步骤 3：领域与状态建模分析

每项发现必须附带代码证据（`ClassName.methodName()` + 具体行为描述），不确定的发现使用 `[待验证]` 标记。

### 3.1 状态建模

对步骤 2c 命中的每个状态字段（外部透传镜像已在 2c 排除、不在此列），**先按 [`../references/state-classification.md`](../references/state-classification.md) 选定类别**（各类判定依据以该文件为准），再按类别建模：

| 类别 | 建模产出 |
|------|---------|
| **真状态机** | 列全部状态值（从 Enum 源码提取精确值，非推测）；每个转换：触发条件 → 前置校验 → 转换操作 → DB 更新 → 后续副作用；标注缺失的非法转换防护；输出 Mermaid stateDiagram + 转换矩阵 |
| **派生位** | **不画 stateDiagram**；写"派生规则"——列输入→输出判定表 |
| **反模式状态机** | 写"事实状态机（反模式）"：①散落写入位置清单（含跨模块写入者）②状态值清单（从字面量提取精确值）③转换图（节点为字面量值）。散落写入点/转换照样是可观察行为，供 synthesis 逐个落成 Scenario；"缺 Enum 集中化"标注**仅作 scratchpad 备注、属实现细节、不进 spec**（行为优先边界）|

一个模块可能同时存在多套状态（不同字段属不同类别），逐套分别建模。

**对每个状态载体显式区分「声明态」与「实际写入态全集」**：实际写入态全集 = 代码 setter / 字面量真正写入的值集合；仅在注释/枚举里声明、但代码从不 set 的值，单列标注「声明未写入」。这是 §3.2 深追踪与 verification 接缝判定的依据——前端门槛 gate / 消费 SQL 在不属于「实际写入态全集」的值即永不可达。

### 3.2 状态消费覆盖深追踪（每个状态字段必做）

发现某字段是状态字段还不够——它最值钱的结论在**消费侧是否覆盖了全部写入态**。对 3.1 每个状态字段：
- 列出它的所有**消费点**：gating 读 / SQL `where` / Java `filter`（含步骤 2b 已扫到的 XML where）。
- 把「实际写入态全集」与「消费点覆盖的态」**求差集**：消费点漏覆盖的写入态，标为潜在缺陷，写明后果（如"增量起点 SQL 只认 `'success'`，写入态全集为 `{running}`、`success` 写入已被注释 → 该 SQL 永远查不到记录 → 增量起点失效 / 漏拉重拉"）。

差集非空的结论落「其他发现」（写明哪些写入态无消费覆盖）——它是 spec **忠实度**的依据：涉及该状态流转的 Scenario 不可写成"全部状态都会被处理"，须反映消费实际覆盖面。verification 跨层契约校验会复核。

**跨模块状态消费**：状态字段的写入或 gating 落在他模块时，按 [shared/agent-preamble.md §5](../shared/agent-preamble.md) 深浅自判（深依赖追到他模块目标代码把事实写准）。状态机本体在他模块时（如本模块仅"写入触发器"），不越界建模，在 domain-modeling.md 如实登记本模块的写入/触发点即可；synthesis 据此把相关 Scenario 措辞忠实于真实字段主权（不谎称本模块独控该状态）。

### 3.3 回调事件目录（IF HAS_CALLBACK_CONTROLLER=true）

- 事件类型枚举值（从源码提取精确字符串值）
- 每种事件：事件类型 → 路由逻辑 → 处理方法 → DB 更新/副作用
- 输出事件映射表
