# 状态字段分类（domain-agent 建模权威）

`domain-agent` 对每个候选状态字段**先选定类别再建模**；`spec-synthesis-agent` 据类别决定怎么把状态落成 Scenario。判定标准在此（单一权威，两个 agent 共用）。

把"裸字符串散落"误当健康状态机会掩盖问题、污染后续场景；故分类是质量门，不是可选项。

## 四类判定

| 形态 | 判定依据 | 对 spec 的意义 |
|------|---------|---------------|
| **真状态机** | **准入硬门：能指出集中 Enum 类 / 常量字典（≥3 个状态值集中定义在 1 个文件）**；状态由有限转换函数推进。指不出集中定义 → 禁归真状态机，按散落程度改判反模式/派生位 | 每个状态转换 = 一条 Scenario（WHEN 处于态 X 且触发 Y / THEN 转到态 Z）；非法转换防护 = 拒绝类 Scenario |
| **派生位（非状态机）** | 字段只有 0/1 或两态，由计算规则每次基于输入重算（无"转换"概念） | 派生规则 = 约束/计算类 Requirement（WHEN 输入满足 X / THEN 该位为真）；**不建状态转换 Scenario** |
| **反模式状态机（裸串散落）** | 状态以字符串/数字字面量散落在 ≥3 个 Service/Controller/DAO，**无 Enum 集中化**；可能跨模块写入。**前提：先排除外部透传镜像**——仅当本模块与跨模块均零 gating 读且值透传自外部时才归镜像；被他模块 gating 的（即便本模块零 gating）仍是反模式状态字段 | 散落写入点/转换照样是可观察行为，**每个写入点/转换落成 Scenario**；状态值集就是 Scenario 的态。`缺 Enum 集中化` 是**实现细节、不进 spec**（它是 module-brief 的陷阱小节素材，不是行为契约） |
| **外部透传镜像（形似状态、实非）** | 字段名像状态，但写入是 `setX(外部对象.getX())`/`copyProperties` 透传、且**本地与跨模块均零 gating 读**（状态机本体在他模块/外部） | **不建模为状态机、不出状态 Scenario**；如确有可观察影响，作普通字段约束处理 |

## 边界情形

- **复合形态**：派生位/标志同时满足反模式判据（裸串散落 ≥3 处、无 Enum）时——如全项目通用启停位 `is_used`——主归其形态类（派生位/标志），不在两类间硬选。
- **字典 FK 型别字段**：值域开放的字典外键（如证件类型 `cardType`），即便有字面量 gating（`==魔数`/`字典名.equals` 做型别分支）也**既非状态机、也非派生位**——gating 是型别判别非生命周期转换。其 gating 若构成可观察约束（如"身份证类型走特定校验"），落约束类 Scenario，但不建状态机。

## 典型误判信号（命中任一即禁归"真状态机"）

- 转换图节点是字符串字面量（`"SEND_OK"`/`"MAKING"`），而非 Enum 类型名 + 值。
- 找不到对应的 Enum 类 / 常量字典（≥3 值集中定义）。
- 状态值以 `setStatus("XXX")` 形式直接出现，而非 `setStatus(Enum.X)`。

> 注意：这些误判信号里的 `setStatus("XXX")`、Enum 类等是 domain-agent **判类别用的代码取证**，属 scratchpad 范畴；最终 spec 里只留状态的**可观察行为**，不留这些实现痕迹（见 [openspec-spec-format.md](openspec-spec-format.md) §3 行为优先边界）。
