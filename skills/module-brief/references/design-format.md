# design.md 格式（技术白盒）

`survey-agent` 写 `design.md` 的格式权威。这份是**给接手/维护本模块的开发者看**的技术地图——允许类/方法/路径/SQL 等代码标识。刻意轻量：建立认知够用即可，不追求穷尽。

## 骨架（5 节）

```markdown
# {{MODULE_CN_NAME}} — 设计说明

> 本文档记录 {{MODULE_CN_NAME}} 的技术结构。由 module-brief 自动生成、供人快速建立认知（非穷尽规范）。

## 1. 分层结构概览
（分层方式、组件划分、关键目录；可含极简 Mermaid 组件图）

## 2. 关键类 / 接口入口
（核心类/接口清单 + 一句话职责；带标识如 `OrderService` / `OrderController#submit`。挑关键的，不罗列全部）

## 3. 核心调用链（1–2 条）
（最主干的业务流程 Controller→Service→DAO 走一遍；可含极简 Mermaid 时序图。有远程调用时点出降级/重试等坑）

## 4. 数据模型概览
（核心实体 + 关系，可含极简 ER 图；**不穷举字段**，只点关键字段与约束。状态字段：点名全部，挑最核心的一个画 stateDiagram，其余一句话）

## 5. 陷阱与护栏
（改这块会被咬的雷、改 A 必连带改 B 的协同约束。只留最狠的几条；无则写 N/A）
```

## 刻意不做（相对深度分析）

- 不穷举 PO 字段（只点关键字段）。
- 不画全部状态机（只点名 + 挑核心画一个）。
- 不做跨模块依赖的合并/去重重机制（提一句"依赖谁"即可）。
- 不要求符号锚点机械可验、不做完备性对账门——**给人看，锚点放宽**：用 `OrderService#cancel` 或口语化"见取消逻辑"都行，重点是让人找得到，不是机器可 grep。

## 第 5 节「陷阱与护栏」写法（最值钱、也最容易注水）

只收 §1-4 与 requirements 导不出来的**耐久雷**和**协同约束**。两种写法：

- **协同护栏**：`改 <A> 时 → MUST 连带 <B> → 否则 <Z 错算>`（带指向，如"见 XxxService 取消逻辑"）。
- **陷阱**：`<标题>：<现象 + 原因 + 改时注意>`。

**入选测试（防注水总闸，每条都过；不过 = 删，没有"保留 + 标注它偏通用"的折中）**：
1. **模块替换测试**：把模块名换成另一个模块，这条还成立吗？还成立 = 通用规范（事务/日志/异常/命名/空值、`@MapperScan` 注册新 DAO 这类），**直接删**——它的家在项目级 CLAUDE.md，不在这。**边界性通用项一律删**：不要留着再加一句"这条偏通用规范"——那还是注水，读者要的是本模块独有的雷，不是夹带的通用常识。
2. **能否写成"改 X 会踩 P"**？写不出 = 是普通业务陈述/事实，删。

❌ 通用（删） vs ✅ 模块特有（留）对照：

| ❌ 删（换个模块也成立） | ✅ 留（本模块特有） |
|---|---|
| "异常要捕获并记日志" | "取消订单必须同步反写库存占用表，否则库存虚高" |
| "状态字段建议用枚举" | "增量同步起点 SQL 只认 `success`，但代码已不再写该值 → 永远查空" |

只留最狠的几条（不设硬上限，但宁缺毋滥）；禁示例代码块；扫一眼就能用的极简体量。陷阱来源 = survey 读模块时顺手撞见的 top 几条，不另起验证 agent、不做穷尽排查。

## 自检（写完过一遍）

- 5 节齐全（缺的显式 N/A，不省略）。
- 第 5 节每条过"模块替换测试"+"能写成改 X 踩 P"——不过的删。
- 无示例代码块塞进陷阱小节。
- 核对门已过——`design.md` 每个代码标识已 grep 核验，查无的已标 `[待确认]`。
