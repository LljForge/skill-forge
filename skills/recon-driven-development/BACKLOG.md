## 1. 基线漂移无规定动作(来源:2026-06-10 实跑·实施期发现 1)

**问题事实**:④ 实施期"同步基线"步骤(ff 拉取 7 个提交)恰好改到 ② 设计锚点文件(ThreeVerifyClient / UserBusinessServiceImpl),靠执行者自觉 diff 复核设计前提后才继续;lite 与 superpowers 均无此规定动作。若漂移真触及设计前提("设计当时以为的代码现状"被新提交改变,如校验返回方式变更导致"零新增提示代码"前提塌掉),现行流程没有"停下问用户还是回 ②"的路由规则。

**候选修法**(讨论于 2026-06-10,未定):
- A. ④.1 委托 writing-plans 时注入计划约束:凡含"同步基线"步骤,后面必须跟一步"diff 检查设计锚点文件;改到设计依赖的行为就停,在 ⏸ 让用户决定回 ② 还是继续"。(与既有"落点/命名"注入约束同型,不算重排 superpowers 链)
- B. 仅 ④ 正文加提醒、不强制进 tasks.md。
- C. 不改,接受为已知取舍。

**搁置原因**:用户决定暂且搁置,留待后续(可能等更多实跑样本)再定。

## 2. 构建命令与实际环境脱节,发现未闭环(来源:2026-06-10 实跑·实施期发现 2/3)

**问题事实**:④ 构建验证时,项目文档(GMZB-NJZL CLAUDE.md / build-commands.md)记的 `mvn clean install ...` 在该机器不可复现——实际可用组合是「Maven 3.6.3 二进制 + settingsnew.xml + 局域网 Nexus 192.168.8.6:8081」(PATH 上的 3.9.9 因 HTTP blocker 与来源追踪必败)。烧 4 轮排查 + 一次用户介入。AI 摸出答案后只记在监督笔记、未回写项目文档,闭环未合。

**候选修法**(讨论于 2026-06-10,未定):
- A. 两层都做:skill ④ 收尾加一句「实施期发现项目文档与实际环境脱节,收尾时提议回写项目文档」;同时把可用构建命令回写 GMZB-NJZL build-commands.md。
- B. 只回写项目文档,skill 不动(「提议回写」属模型常识,写入=教已知;但本次恰恰没闭环)。

**搁置原因**:用户决定暂且搁置。注意:构建命令未回写前,该项目下次构建验证会原样重烧排查成本。

## 3. 产物提交「留待 ④ 收尾」的真空期与覆盖面(来源:v1.4.7 对抗评审)

**问题事实**:v1.4.7 把 ② 产物提交推迟到 ④ 收尾统一定,对抗评审指出该规则只闭合 happy path,存在四个缝:
- **实现链卷走**:同仓布局(代码与 docs/code-dev 同仓)下,④.2 实现链 commit 自管,一个 `git add -A` 就可能把未提交的流程产物卷进实现 commit,统一决策点被抢先架空。候选:④ 委托时声明「docs/code-dev/** 不入实现链 commit」。
- **中途终止无人接**:③ 不通过后放弃、或在任一 ⏸ 长期停摆时,④ 收尾永不发生,提交决定悬空。候选:「轨中途终止时,产物提交在终止点一并定」。
- **保留分支场景**:finishing 选「保留分支自行处理」时,代码尚未集成,「按仓惯例」未把分支去向纳入决定输入。候选:提交决定顺延到分支集成时再定。
- **①③④.1 产物覆盖面**:规则措辞挂在 ② 局部,directed-report.md / review.md / tasks.md 三个产出时点无显式指引(④ 收尾括注笼统兜住但读者到 ④.3 才见到)。候选:升为「核心流程」节一处全局声明。

**搁置原因**:超出 v1.4.7 已批的措辞范围,「实现链卷走」一条还触及「别在 ④ 重排/干预 superpowers 链」护栏,需单独权衡;实跑样本(子仓布局)未踩中任何一缝。⏸ 收尾前确认是现行兜底。
