# recon-driven-development（侦察驱动开发 · 轻量四阶段工作流）

开发任务的**轻量四阶段流程**：定向分析 → 需求设计 → 评审 → 落地。开头做一份只针对本次改动的"摸底"，喂给后面的需求、设计、评审、实现。

> **本 Skill 不自动触发，由用户显式调用。** 完整规则见 [SKILL.md](SKILL.md)，本文件只做总览与导航。

---

## 这是什么

一条**即时开发轨**：不先摸清楚就拍方案，十有八九漏掉"原来别处也在用这张表"这类坑，等写代码才发现要返工。lite 用一份用完即弃的定向摸底兜住这类遗漏。

四阶段，每阶段结束**暂停**等用户确认。产物落本次改动目录 `docs/recon-dev/<YYYY-MM-DD>-<change-name>/`，收尾时连目录一起归档进 `docs/recon-dev/_archived/`。

**前置**：superpowers 插件是唯一硬前置（②④ 委托它）——会话里没装就不开轨、引导安装。

```
① 定向分析   一句话粗需求 → 钻落点 + 消费面(全集) → directed-report.md
             ⏸ 过报告质量门（① 自守，防空转的真牙）
② 需求+设计  据①，委托 brainstorming → requirements.md + design.md
             ⏸
③ 评审       派发评审 sub-agent → 综合质量门（三维度 + 定向报告验收）→ review.md
             ⏸
④ 落地       拆 tasks.md ⏸ → 全委托 superpowers 跑实现链（含分支收口）→ 文档归档 ⏸
```

**定位是"薄编排件"**：薄的是流程（不教怎么 grep / 追链、不复述子技能本职），不是结论；料（落点现状 / 实答 / 锚点）要厚。`①→②③` 线性不回流，各阶段在自己出口守住产物质量。

## Skill 设计哲学

> 全文：[SKILL-DESIGN-PHILOSOPHY.md](SKILL-DESIGN-PHILOSOPHY.md)（通用设计哲学的单一权威源，适用于任何 Skill；其他项目可手动引用该路径）。

**核心命题**：一个 Skill 的价值不在于"教会模型怎么做"，而在于**装模型自己导不出来的东西**。给强模型补盲区（不会主动扫的陷阱）和加约束（下游依赖的契约），别手把手教它已知流程——教已知既费上下文又会过时。

**七条原则（摘要）**：

1. **内容 = 契约 + 陷阱，不教已知** —— 判别只一句："这行模型自己能导出来吗？能 → 删。"该留：输出 schema 契约、召回底线、栈/域特有陷阱。
2. **改动 → 消费者映射** —— 每个产出都有读者；新增内容必须有消费者，验证只跑受影响的消费者，改名前先查谁按名引用。
3. **区分骨架问题与内容问题** —— 多数是内容问题；正交、可线性扩展的骨架别动，重写它是负 ROI。
4. **去重要根治** —— 超集关系（A ⊃ B）用一份文件 + 围栏标注"A 特有"段，一份文件不会和自己漂移。
5. **单一权威源 + 分层职责** —— 每个事实只有一个家，跨层重复 = 漂移源。
6. **证据先行 + 诚实区分"能力 vs 工程收益"** —— 大改前用真实数据验证；强模型时代多数"改进"是工程收益（可维护/鲁棒/显式），别 oversell。
7. **通用 vs 特定的分界** —— 复用型 Skill 只放通用知识，项目特有事实归 CLAUDE.md，正文只写当下规则、版本注解归 CHANGELOG。

**适用边界**：弱模型兼容时显式指令该留；契约/schema 即便"导得出"也留（作用是统一形态，不是教学）；召回底线防偶发遗漏，是地板不是教材。

## 如何迭代这个 Skill

> 全文：[MAINTAINING.md](MAINTAINING.md)。改本 Skill **自己**（SKILL.md / 模板 / 流程）时读它；改业务代码不读。

- **走 `/skill-creator`**，动手前先读 `SKILL-DESIGN-PHILOSOPHY.md` 对齐，每处改动拿它的判别尺自检。
- **大改先确认、小改直接做**：结构性大改（动骨架 / 流程 / 产物形态）先呈现方案、能用真实数据就验证再动手；措辞/补漏类小改可直接做。
- **改前必备份**：把当前文件整套拷进同级目录 `~/.claude/skills/recon-driven-development-workspace/skill-snapshot-pre-v<新版本号>/`。
- **验证三条**：① 改动→消费者映射，只重验受影响阶段（如改 ③ 维度名必须同步改 `references/templates/review.md`，否则静默失配）；② 落地前多 agent 对抗评审（哲学薄度 / 一致性 / 盲区 / 可执行性四维度）；③ 诚实定性"能力 vs 工程收益"。
- **守住"薄"**：别给定向分析加规则表、别复述子技能本职、别读写长期文档或发持久编号。
- **版本号（semver）**：MAJOR = 稳定里程碑 / 不兼容大改；MINOR = 行为改动 / 新能力 / 产物形态变化；PATCH = 纯修订 / 可执行性补漏；纯文档整理类不改行为的可**不发版**（备份照做）。
