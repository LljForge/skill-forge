## 改名 + 迁库（2026-06-15 · 非行为改动、不动版本号）

原名 `code-dev-workflow-lite` → `recon-driven-development`。新名直接点出本轨灵魂——① 定向分析（现状侦察 / 摸底）先行、驱动其后需求/设计/评审/实现全程；并借 superpowers 家族的 `*-driven-development` 命名（对齐其 `test-driven-development` / `subagent-driven-development`），以同款"姓氏"标明本 Skill 系骑在 superpowers 上的一脉。源从 `~/.claude/skills/` 迁入 `~/Projects/AI/skill-forge/skills/`、并入 skill-forge 的 git 仓统一管理（旧仓 `.git` 未带过来）。**流程 / 产物形态 / 规则一字未改**，故按本仓「迁移类非行为改动可不发版」惯例不动版本号。

## v1.5.1 — ②④ 派发处接缝收紧：override 显眼化 + 闸对齐子技能原生停点 + ④ 点名链重构（PATCH · 可执行性）

源自 GMZB-slp「借据关联决议」实跑监督（EXECUTION-REPORT §4.1/§5.3）：lite↔superpowers 接缝处,子技能的「默认终点」需执行者主动压制、但压制指针不够显眼——这是「全委托」模式的「可实施性税」。逐轮讨论收敛出一条统一取向:**lite 只保留 superpowers 导不出的 override / 护栏;可点名子技能的链 / 停点当「挂点」,但不复述它的本职、也不在它的原生停点旁另竖一道门**。

- **② brainstorming 派发处（两件）**:(a) 把「委托边界」run-on 重排为**只压住它这 3 个默认终点**清单（产物 / 谈完 / 提交）,并**删掉「多轮讨论 / 提方案 / 逐段过设计全要」半句**——那是复述 brainstorming 本职（违哲学 #1「不教已知」）;顺带修了「两条边界 vs 三件事」计数瑕疵。(b) 把「复审门」从「产物」条括号拎到 ② 的 ⏸ 行、砍掉枚举,收成「即 brainstorming 谈完让你过目那道停点就是 ② 这道 ⏸、别另叠」——防 lite 的「每阶段暂停模板」与 brainstorming 自带门重复成**双停**。产物形态 + 终点改向两条 ③ 刚需不动。
- **④ 落地节重构为「点名链 + 各节点挂注意事项」**:原 ④ 为守护栏而**回避点名**链,满是「之后…那个停点…收口时」的间接指代、读着乱（用户连追数轮才逼出症结）。改为**显式点名 superpowers 完整链**——`writing-plans 拆计划 → 停下问选执行方式 → using-git-worktrees 起隔离工作区 → executing/subagent 实现(TDD+多轮评审它自管) → finishing 收口`——再把 lite 注意事项挂到对应节点:writing-plans（落点 tasks.md + **⏸ 审计划:就在「选执行方式」停点、别答菜单跳审**）/ worktree（原「独立子仓提醒」移此挂点:对子仓操作）/ 实现（它自管、lite 不插手;唯一接——superpowers 偏"能绕就绕、绕不过才卡停"、不会主动喊你回头重判设计,故 design 被实现证伪时 lite 补一停 surface 给用户,即「事实订正」④ 落点;到 ④ 该落点的"留痕"半边已无下游消费者、降为捎带,价值在"暂停"半边）/ finishing（分支去向是你的决定）/ 归档（链外、lite 专属 + ⏸ 收尾确认）。
- **链经 superpowers 源码核实补正**:落地前聚焦复评据 `using-git-worktrees`/`writing-plans`/`executing-plans`/`subagent-driven-development` 源码,纠了草稿两处错——「实现」被写薄成纯写码（实含 TDD + 每任务两段评审 + 最终全量评审）、worktree 节点错排到 writing-plans 之前（实为「at execution time」、在选执行方式之后）。另:④ 早稿曾拟「节奏 → 拆完别立刻续跑实现」压制,经核 `Execution Handoff`:134-144 证伪（writing-plans 拆完不自动续跑、而是抛菜单停下问人）、撤回——真缝（执行者答菜单跳审）改用「审计划闸对齐到那个菜单停点」解,不加门。
- **配套放宽一条护栏**:末尾「❌ 别在 ④ 重排实现链」原一刀切禁「点名链」、与新结构冲突;放宽为「可点名链节点当挂点、但别逐步重教每步怎么做」——禁的本意（别复制 superpowers 内容、别替它驱动）不变。
- **守单一权威 + 薄**:②④ 的 override / 挂点都不复制到第二处、不另设「接缝总表」（守哲学 #5）;每个挂点逐条验过、只放 superpowers 导不出的料,无复述本职。BACKLOG #4（writing-plans 菜单跳审）、#5（brainstorming 两道门）经本版解掉,回原 3 条。
- **诚实定性**:工程收益（②④ 派发约束从「埋在散文 / 间接指代」升为「override 逐条可见、闸对齐原生停点、链显式点名」、降低漏压与跳闸概率）,非能力跃升。**非铁闭**——仍靠执行者读到并照做。落地经四视角对抗评审（②）+ 两轮 ④ 聚焦复评 + 用户逐句追问层层逼出最终结构。

## v1.5.0 — ① 数据态诚实契约 + 封存报告事实订正（MINOR · 补盲区 + 规则例外）

源自 GMZB-slp「决议编辑后展示与版本管理」实跑监督（EXECUTION-REPORT）。该次 ① 纯静态读码把软删根因判错——静态推断「某字段值不匹配」,实为该字段被写空致软删条件永不命中;**根因一度判错、仅因用户事后查库才纠**(③ 又发现订正后仍有「NULL vs 空串」口径差,静态连订正完都没钉死)。取证辨真伪后只动两处真问题;P3(writing-plans 的 TDD 摩擦:属 superpowers 域 + 护栏禁重排 + 实跑证「过度悲观」)、P4(Task0,洞察已被 P1 吸收)、P5(流程偏重,定位合理)均判不动。

- **P1 · ① 软/硬事实分等（横切契约,补盲区）**:① 此前边界「只摆现有代码的事实」把「`eq(A,B)` 这行代码(硬事实、有锚点)」与「A 字段运行时实际取值(纯静态推不准)」当同种事实写,下游分不清哪条是猜的。补**横切四样的标注约定**:软推断标 `[静态推断·未核验]`(简称 `[未核验]`);触发用**句式触发器**(结论形如「X 字段实际取值为…」「这条分支实际走没走到」即标)、**不写 bug 分类清单**(守「别给定向分析加识别-动作规则表」)。边界放开到含运行时/数据态。
- **P1 · 机会核验折进质量门（不新增独立流程阶段·折进既有质量门、无数据手段即跳过）**:质量门原只「抽查锚点」验硬事实,补两条——①「锚点真实」加**反向探针**堵漏标(结论暗含运行时假设却当硬事实写的,补标);②「支撑坑的软推断核验」:scope 锚到**支撑样③坑(尤其反直觉陷阱/协同护栏两味)**的 `[未核验]` 条(避开「动摇根因」这种 ① 时点还算不出的下游概念),主会话用**现成只读零副作用**手段定点实测、升 `[实测]` 或推翻,够不着坦诚留标记。实测归主会话(钻取优选 code-explorer 工具面纯读码、查不了库);没数据手段的项目停标记态、契约仍兑现(可移植)。
- **P1 · 「谁用谁验」接棒纪律**:标记是接力棒——② 据 `[未核验]` 做改动决策前先补验,够不着则在 design.md 显式记「此决策建于未核验推断 X」,堵「① 没验、② 也忘验」真空(正是 slp 失败形态)。
- **P2 · 封存报告「事实订正」例外（规则开窄口）**:原「不回写已封存 directed-report」对「② 探到 ① 没写的新面」(前向)是对的,但对「下游证据推翻 ① 写下的事实」一刀切会留错误导下游(本次靠 user override 打订正块才免)。开**窄口**:推翻 ① 事实 → 原文条目下追加订正块(范式固定)、原文标 `[已订正]`(防「保留原文不标已废 = 两条矛盾并存」)。**分工**:评审 sub-agent 只把推翻真实记进 review.md、绝不碰 directed-report;核验与落订正由主会话做。**范围回流仍禁**;落点 stage-agnostic,并钉死「④ 实现链途中证伪」处置(主会话暂停链落订正、不算重排;动摇设计带订正回 ③ 重判该点——诚实承认这条 ④→③ 回边)。
- **review-agent.md + review.md 模板补 ③ 承接面**(终检揪出的结构洞):③ 把判据全外包给 review-agent.md,但它写于本版前、对 `[未核验]`/证伪无感知——「谁用谁验」接力棒与「事实订正」在 ③ 都断头(SKILL 正文宣称②③④通用、③ 这端却没人接)。补最小感知(仍守 v1.4.4 只读 +「核验/订正归主会话」):一致性维度加一格——发现/判档依赖 ① 的 `[未核验]` 主张就先定点源码核、核不动则在 review.md 标「依赖未核验推断 X、建议主会话补验」;源码核出 ① 某条事实为假就记发现「① 事实被推翻 + 证据」、**只记 review.md、绝不改 directed-report**;模板「定向报告验收」加「① 事实被推翻」槽供主会话落订正。(曾起意「review-agent.md 不动」,被终检证伪——那条没实现「reviewer 记录、主会话核验」架构,reviewer 得先知道有这回事才记得了。)

## v1.4.7 — 首次实跑回馈：① 覆盖地板 + 落盘纪律、② 提交时机（PATCH · 可执行性）

源自 GMZB-NJZL「实名三要素后移」实跑监督（supervision-2026-06-10）发现的三个真实摩擦，各补最小措辞；另两项高影响发现（基线漂移路由、构建文档回写）经讨论搁置进 BACKLOG.md（本版新立，维护者备忘、运行时不读）。

- **① 验收单补「覆盖地板」+ 质量门补核对项**：实跑中落点现状表已列 getAuthUrl、验收单却没问它的消费方，①② 双漏、靠 ③ 兜底。补两侧：写侧地板（逐载体至少一问覆盖其调用/消费方；一问可打包、但实答按载体分项，防伞形问句混答漏单个载体；不问须显式标理由，防默掉）+ 门侧逐载体核对（**指针式**引用覆盖地板、不复述口径，防双写漂移；缺口定点打回补摸、不整报告重钻）。地板形态要求随派发包下发——清单是钻取 agent 现摸出来的，执行者收不到规则就只能靠门返工。

- **防误删备忘**：报告现有两层地板，分工不同勿合并——覆盖地板 = 问句层结构覆盖（每个载体有人问），① 质量门当场核；召回地板（四样④）= 检索行为层（顺着消费面追到底），③ 验。「一问可打包」半句属工程保险（防机械一载体一问推高钻取成本），非能力项。
