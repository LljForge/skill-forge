## 1. 设计决策（本质与取舍）—— 最该留的部分

### 1.1 ⑦ 是"判断步"，无法做成脚本硬门
"这页平不平衡、清不清楚"是设计判断，无法用退出码表达。所以 ⑦ 拆**两层**：
- **机械层**：每页是否真渲染、非空白 —— 可脚本化。
- **判断层**：每页是否平衡/层次清/不溢出/不半空 —— 只能人看，只能"协议硬"。

### 1.2 机械层委托"任意渲染能力"，不点名工具
镜像 deckcraft 自身哲学（抽取委托 document-skills，不自带解析器）：把"HTML→图"委托给本机已有的任意能力（Playwright / 系统浏览器 headless / 截图类 skill / 浏览器 MCP），**skill 只定能力 + 产出契约，不绑工具**。
取舍依据：要"看真像素"必须有绘制引擎，**纯 Node 拿不到像素**；最贴近"零依赖/即拷即用"的是 shell 调系统已装浏览器，而非 npm 装 Playwright。故不写死任何一个。

### 1.3 "硬"分两处落地，各归其位
- **判断层 = 协议硬**：逐页看 + 逐页记结论 + **未对每页取得合格渲染并亲眼看过，不得交付**；无渲染能力时转"必须拿用户截图"。
- **⑥ 非空检查 = 脚本硬门**：纯 Node、零依赖、真退出码，是机械层里能真正自动化的那部分。

### 1.4 非空检查绑"稳定 HTML 元素名"，不绑 deckcraft 类名
判定"含图形元素"按 `img|svg|figure|canvas`（HTML 标准元素名，几乎不改），**不按 `.dc-chart`/`.dc-graph`**（deckcraft 类名，会改）。
理由：类名改名时，绑类名的检查会"扫不到→以为没内容要查→静默绿灯放过"（最坏的假阴性）。绑元素名则不烂。
**防烂靠 fixture 哨兵测试**：用"带稳定 tag 但不带 dc-* 类名"的 figure/svg/canvas 断言非空——若日后把判定误绑到类名，这些用例会变红。

### 1.5 数据契约在"构造端"守，不回扫产物
`chart-render.mjs` 的 `num()` 已在构造时把脏值归一为 0（`Number.isFinite` 兜底，**绝不 NaN**）。故 ⑥ **不**去回扫 `.dc-chart` 验图表数据——冗余且脆。原则：**契约在数据进来的地方守，不靠回扫 HTML 重验。**

### 1.6 早校准（④）+ 末尾逐页（⑦）
只在末尾看 = 盲写 N 页、末尾才发现系统性毛病（每页都堆成左窄列 / 某主题层次普遍读不清）= 一次性返工 N 页。
→ ④ 写完头 1–2 页先渲染看一眼锁版式（**用 `--out` 落到 `./.deckcraft-tmp/` 临时路径，不污染 `docs/decks/`**），⑦ 末尾仍逐页全检。

### 1.7 不把 ⑥ 写成"规则表"
正文写**不变量**（"没有空页"），把"当前类名叫什么"指向 `components.md`/渲染脚本这个活的事实源；校验清单**指向单一事实源**（SKILL ⑥ / `validate-deck.mjs`），不在多处复述以免漂移。

## 5. 待定 / 未决（尚未落地）

### 5.1 修 ⑦"改 hash"事实 bug 【待定·确认必做】
⑦ 机械层"逐页"条写"（改 hash / 翻页）"，但**本模板靠 `.slide.active` + `goto(n)` + 键盘翻页，无 hashchange，hash 切不动页**（`template.html` 闭包 `let i` / `goto` / `keydown`）。照字面做会翻不动、得自己逆向。→ 改为正确机制。

### 5.2 机械层补"几何客观信号" 【待定·建议做】
实跑暴露：**满分辨率截图在 Read 时被缩放，制造"半空/压缩"假象**，一度差点误把第 7 页当溢出去"修"。真正能定论的是引擎几何（`scrollHeight==clientHeight`、overflow=0、同构页对比），不是肉眼看缩略图。
→ 机械层契约补：**出图前把各页归一到统一宽度再 Read**；并把 `overflow = scrollHeight − clientHeight` 作为**客观信号**纳入截前断言。属工具无关的契约增强，与 1.2 一致。

### 5.3 可选 `render.mjs` 骨架 【待定·真正的取舍】
逐页驱动 `.active`、等绘制、归一宽度、断言非空+overflow、逐页出图。一次解决 5.1+5.2 的操作性（省去每次逆向 template 切页机制），但**重新引入一个 Playwright 专用脚本**，与"不点名工具/零脚本"取向有张力。
→ 可框成"用 Playwright 就用这个骨架，用别的能力照契约自己实现"的**可选项**。等拍板。

### 5.4 提速方案（保"全覆盖"又砍串行读图）【提案】
- **拼图（contact sheet）**：21 张缩略拼成一张大图，**一次 Read** 看全貌找可疑页，再只对可疑页满分辨率单看 → ~21 次读图压成 1+几次。
- 配合 5.2 的 overflow 客观信号兜底 → 满分辨率只读被标记的几页。

### 5.5 未结案：新版（21 页）不如旧版（25 页）【未结案 · 勿当已解决】
用户反馈"新的产物反而还不如旧版"，排查被打断（本记录即在此处中断）。
- **已有客观差**：旧 deck 25 段、新 deck 21 段（少 4 页）。
- **候选假设（均未验证）**：① 标准档过度压缩、丢了实质（违反"实质下限"）；② subagent 的 2 处"修复"（P9 ul→.steps、P12 压单行）反而劣化；③ 自动生成质量本就不及那份经人工 curate 的旧样例（旧 deck 是 git 历史里"以干净重跑构建作为样例"、被挑过的）；④ 二者生成过程/详略口径本就不同，非同条件可比。
