# deckcraft 摄取管线参考

本文件指导运行时模型把用户材料（md / txt / docx / pdf）摄取为一套自包含 HTML 幻灯片。

> 文中 `<SKILL>` 指代本 skill 所在目录的绝对路径（如 `~/.claude/skills/deckcraft`），执行命令时替换为真实路径；输入材料 / 输出 deck 仍按当前项目相对。详见 `SKILL.md` 顶部「路径约定」。
读取与图表落地的细节见 `references/extraction.md`；版式组件与确切片段见 `references/components.md`；主题令牌见 `references/themes.md`。

整条管线分五步：**读取 → 内容策略 → 版式映射 → 灌进插槽 → 输出落盘**。
本文件覆盖内容策略、版式映射与输出落盘三步；读取（含图表 H 决策树）在 `extraction.md`。

---

## 一、内容策略：归纳 / 精炼

> **实质下限（贯穿所有档与材料类型）**：精炼 ≠ 只留标题。**把一整章压成一句标题是"丢内容"，不是"精炼"。** 每页必须保留足以支撑其主张的实质；宁可多拆一页，也不要把论点饿成口号。
> 实质的单位是**保留的独立论点数量**，与每条的长短解耦：一个来源章节/段落有几条独立论点，就应有几条对应的要点/卡片活下来。**调档调的是每条展开到哪一层，不是丢掉多少条论点**——精简档每条留关键词级线索、标准档一句说完整、详尽档完整展开（为什么 / 边界 / 例子），但**三档活下来的论点条数相同**。
> **区分两个层级（关键）**：**独立论点**（distinct claim，如"签约要走这个流程"）受地板保护、三档都得活下来；而**同一论点内部的并列子项 / 步骤**（那一串操作步）**不是**独立论点，它们合多粗、铺多细由档位的"粒度"决定（见下「详略三档」的粒度列）——精简可把 7 步并成 3 句粗线索，详尽则一步一条铺开。地板管的是论点别被饿成口号，**不是**禁止精简合并同一论点的内部步骤。

所有材料（md / txt / docx / pdf）都是连续文稿，统一**归纳与精炼**：从段落里提炼主张，砍掉铺陈，把长句压成短要点。这正是 `tests/fixtures/sample-doc.md`（一篇 Markdown 业务回顾）变成 `tests/fixtures/sample-doc.slides.html`（4 张幻灯片）所做的事——摘要段被提炼成一张标题页 + 一张 KPI 页，风险段被压成两张对照卡，下季目标段成为收束页。

归纳允许重写措辞，但**不臆造数据**（细则见下「结构约束」节，不重复）。

## 二、结构约束（所有材料都遵守）

- **每页一个主张**：一张幻灯片只讲一件事。多个论点拆成多页，不要堆叠。
- **要点 ≤6 条**：一页内的列表 / 卡片不超过 6 条；超了就拆页或合并同类项。
- **标题短促**：`h1` / `h2` / `.ctitle` 用名词短语或一句断言，不写整段。参考 `sample-doc.slides.html` 的 `营收同比增长 18%`、`要盯住的两点`。
- **不臆造数据**：缺的数字不补、不估、不 round；原文没有的结论 / 百分比一律不加。读不到的数值按 `extraction.md` 的「诚实标注」降级。
- **填满版面，别只占左半**：每张是一块 16:9 宽屏画布，你是设计师——目标是信息清晰、版面平衡、填满画布而不拥挤。横向思考宽屏：并排、分栏、图文左右，别默认把内容堆成左对齐的一窄列、把右半页空着。手段有双栏列表（`ul.cols-2`）、双栏 / 多栏卡片（`.cols`）、给列表或正文配一个右侧视觉元素（图表 / 金句 / KPI 卡 / 整图）等——见第五节调色板，按内容自己挑、自己拿捏密度。**页数是结果不是目标**：一页装满，自然更少页（交付前还会在 SKILL ⑦ 看一眼成品再调，见反馈回路）。

### 详略三档（由用户在开始前选定，默认「标准」）

三档调的是**两面：每条主张展开到哪一层（纵向·深度）+ 源材料切成几个单元（横向·粒度）**，都不是丢掉多少条论点。给的是性格，不是配额——具体几条、几页由你按内容拿捏。

| 档 | 本质：自给度 · 阅读模式 | 每条展开到（纵向·深度） | 单元如何切（横向·粒度） | 篇幅倾向（涌现） |
|----|------|----------|----------|----------|
| **精简 / 演讲骨架** | 靠**在场讲者**补全，页面是提词；读者本就懂 | 关键词级**线索**——唤起讲者要说的话，铺陈全留给口头 | **少而粗**：合并相邻单薄节、编号步归并成更少粗要点、收粗顶层单元 | 自然偏少 |
| **标准（默认）** | 页面**自己讲清大意**，不在场也抓得住主线 | **一句**说完整——含它为何成立的最硬支撑；丢例子细节 / 过渡 | **按论点自然拆**：一个论点一个单元，不强合也不细铺 | 中等，随论点数长 |
| **详尽 / 讲义** | 页面**独立成文**，无讲者也能重建完整论证 | 完整展开——主张 + 为什么 + 边界 / 坑 + 关键例子 | **多而细**：充分铺开，子点 / 步骤可各自成条 | 自然偏多 |

**脊梁**：三档保留的**独立论点条数相同**（实质地板不变）——档位变的是「每条展开多深（纵向）」+「单元切多粗（横向）」这两面，两面都从"自给度"长出来，不是两个能各自失步的旋钮；篇幅是它俩的涌现结果。**装饰形态**（用 `p.body` 还是 `ul`）由内容意图决定、**不在这张表里**（见第五节调色板）；但**结构粒度**（切多粗）是档位的事，**就在这张表里**。

**自检**：若某来源章节/段落本有好几条独立要点、你却只落了一句标题，多半压过头了——回到上面的"每条主张展开到"补回实质。这一自检在 SKILL ⑦「设计自检」的「档位核对」里再验一次。

## 三、多来源合并

- **默认合成一套 deck**：多个输入文件默认产出**一份** HTML，按主题逻辑（而非文件边界）重新组织章节。
- **目录通配按同一规则**：传入目录或 glob（如 `materials/*.md`）时，把命中的每个文件当作一个来源，仍合成一套 deck。
- **不同来源不混揉**：不同文件 / 章节的内容不可拌进同一个论断或同一组列表，除非它们本就在讲同一件事且你明确判断可合并。
- **护栏 —— 先确认范围**：当**来源文件 > 8 个**，或**合计页数 > 40 页**时，**先停下来向用户确认范围和篇幅**（要全部？要哪几份？目标多少页？），不要默默吞掉一大批材料生成超长 deck。

## 四、篇幅

- **内容优先，不设页数上限**：页数由内容决定，**不要为凑数注水，也不要为压数砍掉实质**。每个关键论点该有一页就给一页（遵守「每页一个主张 + 要点 ≤6 + 实质下限」）。源材料越大、章节越多，deck 自然越长——这是对的，不要硬压。
- **页数是涌现结果，不是档位独立设的旋钮**：它由「每条展开多深 × 论点数（实质地板不变）」自然落定——精简偏少、标准中等、详尽偏多（判法见上「详略三档」表"篇幅倾向（涌现）"列，此处不重述）。
- **用户指定优先**：用户给了明确页数（如「压到 12 页」）或详略（「只要摘要」「展开讲」）时，以用户为准——这是唯一的硬约束来源；用户没给就内容优先、不自设上限。
- 注意：**幕封面 / 标题页 / 致谢页都计入 `sections`**（`validate-deck` 把它们和内容页一起数），但它们不占用任何"页数预算"——因为没有预算。

## 五、版式调色板

下面是你的**版式调色板**——`references/components.md` 里的真实组件，各擅长表达不同意图。**组件名、确切片段、数据签名是系统事实（照用别改）；用哪个、怎么搭配、怎么填满画布，你作为设计师自己判断**，别把"擅长表达"读成"必须这么用"。

**形态归内容意图，与档位无关**（这里说的是**装饰形态**——用哪个组件；**不含结构粒度**——单元切多粗那是档位的事，见第二节三档表的粒度列）：用哪个组件由**内容意图 + 你的设计判断**决定，不由档位决定。档位只改变「哪些意图会升起」的分布——所以 `p.body` 在精简档里**自然**少见（"展开背景 / 为什么"的意图很少升起），这是**涌现结果，不是禁令**：某页真有独句 thesis 或必须的一句背景，照用。

| 内容意图 | 用什么 | 擅长表达 | 确切组件 / 片段 |
|----------|--------|----------|------------------|
| 对比 | 两栏对照 | 新旧、优劣、双侧并列 | `.cols` 容器套两个 `.card`；正向一侧加 `.card.new`（绿底+绿左边框）。见 components.md「要点卡 `.cols` / `.card`」 |
| 指标 | KPI 卡 | 结果汇报、数字摘要 | `.cols` 套 `.card`，`h3` 直接写数字 + `p` 写说明，如 `<div class="card"><h3>1,240</h3><p>新客户 · 华东领跑</p></div>`。见 components.md「数据 / KPI」 |
| 要点列表 | `ul`/`ol`（可二级嵌套、可双栏） | 一组有实质的并列要点、机制拆解、清单 | `<ul><li><span class="lead-in">小标题</span>：说明</li></ul>`，**一条主张下还要拆子点**时可 `li > ul` 嵌套；想分栏铺满宽度就加 `class="cols-2"`。见 components.md「要点列表」 |
| 正文 | `p.body` | 需要一两句讲清背景 / 原因时 | `<p class="body">…</p>`，多段用多个；保持结构化不堆大段。见 components.md「正文段落」 |
| 数据图 | 渲染图表 | 有 categories / 数值序列要可视化 | `chart-render.mjs` 的 `renderChart(spec)`，`spec = { type:'bar'\|'line'\|'pie', title?, categories:[..], series:[{name?, values:[..]}] }`，产出 `<figure class="dc-chart">` 占位，运行时 ECharts 渲染（支持多系列、随皮肤 setTheme 换肤）。**`values` 必须是纯数字**。数据契约见 extraction.md |
| 流程 | 流程图（dc-graph） | 步骤、流程、关系、架构 | `graph-render.mjs` 的 `graphToSvgBlock(graph)`，`graph = { nodes:[{id,label,type?}], edges:[{from,to,label?}] }`，产出 `<figure class="dc-graph">` 静态 SVG。见 components.md「流程图 `figure.dc-graph`」 |
| 金句 | 引用 | 单句观点、过渡页核心句 | `<blockquote class="quote">…</blockquote>`。见 components.md「引用 `blockquote.quote`」 |
| 整图 | 整图特写 | 一张图本身就是主体（截图、示意图） | 相对路径 `<img src="img/...">`（build 时内联），**外面必须包 `<div class="slide-body">`**——定高弹性区让竖图按本页剩余空间自适应缩放、不顶破页面（缺了 validate 直接报错）。见 components.md「内嵌图片」 |
| 分章 | 幕封面 | 章节分隔、驱动左上角幕导航 | `<section class="slide cover">`，含 `.cnum` 幕号 + `.kicker` 幕标签 + `.ctitle` 幕标题。见 components.md「幕封面 `.slide.cover`」 |

辅助标记（贯穿所有版式）：

- 章节开场 / 全篇开场用标题页（`.kicker` + `h1` + `.lead`），强调词包 `<span class="accent">`。
- 主线内容 `data-tier="main"`，可跳过的深度补充 `data-tier="deep"`。
- 需要逐元素入场用 `data-reveal style="--i:N"`；需要按键逐步揭示用 `.frag`。
  细节见 components.md「渐进揭示与分层」。

## 六、输出命名与落盘

幻灯片 HTML 片段写好后，用 `build-deck.mjs` 灌进固定模板 `template.html`，再落盘。

### 灌装命令

```bash
node <SKILL>/build-deck.mjs \
  --template <SKILL>/template.html \
  --slides ./.deckcraft-tmp/slides.html \
  --title "2026 Q2 业务回顾" \
  --images ./.deckcraft-tmp \
  --dir docs/decks
```

`build-deck.mjs` 内部：

- `replaceSlots(template, {title, slides})` —— 把 `<!-- DECKCRAFT_TITLE -->` 和 `<!-- DECKCRAFT_SLIDES -->` 两个插槽替换为标题（转义）与幻灯片片段。
- `inlineImages(html, baseDir)` —— 把相对路径 `<img src="img/...">` 按 `--images` 基目录（即 `./.deckcraft-tmp`，slide 的 `img/xxx.png` 拼接成 `./.deckcraft-tmp/img/xxx.png`）读取并 base64 内联；`http:` / `https:` / `data:` 外链不动；文件不存在则原样保留。**注意**：`--images` 给工作目录本身、不要给到 `img/`，否则会拼成 `img/img/...`。
- `buildDeck({templatePath, slides, title, imageBaseDir})` —— 串起上面两步，返回完整 HTML 字符串。
- `resolveOutputPath(dir, title)` —— 由标题推导文件名并落地（见下）。

也可用 `--out <文件>` 直接指定完整输出路径，绕过 `resolveOutputPath`。

### 落盘目录

- **默认 `docs/decks/`**：所有产出默认写到工作根目录下的 `docs/decks/`。该目录可能不存在，且 `build-deck.mjs` **不会自建目录**——`--dir` 前先 `mkdir -p docs/decks`。
- **用户指定优先**：用户给了明确输出路径就用用户的。
- 临时产物（`slides.html` / 抽取的图片）始终在 `./.deckcraft-tmp/`，不与正式产出混放。

### 标题优先级

`resolveOutputPath` 用标题生成文件名，标题按优先级取：

1. **用户指定**的标题（最高）。
2. 文档**大标题**（如 Markdown 的 `# 2026 Q2 业务回顾`）。
3. **文件名**（兜底）。

### 文件名安全化与同名追加序号

`resolveOutputPath` 会把标题里的 `/ \ : * ? " < > |` 替换为 `_`、空白压成 `-`、截到 80 字、削去首尾的 `-_.`；为空则退回 `deck`。
若目标 `<safe>.html` 已存在，自动追加序号 `<safe>-2.html`、`<safe>-3.html`……，**不覆盖**已有文件。

### 灌装后必须校验

落盘后跑 `validate-deck.mjs` 验收：

```bash
node <SKILL>/validate-deck.mjs <deck.html>
```

它返回 `{ok, errors, sections}`，`ok` 时退出码 0。校验项：两个插槽必须已被替换、6 套主题（clean / fresh / elegant / terminal / business / passion）的 `data-theme='<key>'` 变量块必须齐全、至少有一张 `.slide`、不得残留未内联的本地 `<img>`。有 `errors` 就回去修，别交付。
