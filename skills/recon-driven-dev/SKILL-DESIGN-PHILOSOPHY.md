# Skill 设计哲学

> 通用 Skill 设计哲学（**单一权威源**）。在 module-depth-analysis v1.10 重构中提炼，适用于任何 Skill；MDA 作为贯穿示例（斜体标注）。
> 其他项目需要时，直接引用本文件路径即可（手动引用，不自动加载）。本文描述设计取向；具体版本演进见各 skill 自己的 CHANGELOG。

## 核心命题

**一个 Skill 的价值不在于"教会模型怎么做"，而在于装模型自己导不出来的东西。模型越强，这条越成立。**

给强模型补盲区（它不会主动去扫的陷阱）和加约束（下游依赖的契约），而不是手把手教它已知的流程。教一个聪明模型它已会的东西，既费上下文又会过时——文档僵在那，模型的知识反而更鲜。

## 七条原则

### 1. 内容 = 契约 + 陷阱，不教已知
判别测试只一句：**"这行，模型自己能导出来吗？能 → 删。"**
该留三类：① **输出 schema 契约**（下游依赖的稳定形态）；② **召回底线**（地板非天花板："至少覆盖，遇变体照追"，别让固定清单把模型锚死）；③ **栈/域特有陷阱**（模型不会主动去扫的盲区，以"分析指令"形式写，不是名词解释）。
该删：框架识别特征、通用流程步骤——模型比文档懂得更全更新。
*MDA：删策略文件的"识别特征"（归检测器），补"`LambdaQueryWrapper.eq` 也是状态 gating"这类陷阱。*

### 2. 改动 → 消费者映射（系统视角）
Skill 是生产者-消费者系统，每个产出都有读者。设计与验证都靠这条：
- **新增内容必须有消费者**，否则空转——陷阱写进文件却没 agent 读它 = 白写。
- **验证只跑受影响的消费者**，不读该产出的环节不必重跑。
- **改名 / 改结构前，先查谁按名引用它。**
*MDA：gating 陷阱要接 domain 才生效；verification/synthesis 不读策略，故改策略矩阵不必重跑它们；"关键词表/分析模式"标题被按名引用，改名即破坏。*

### 3. 区分骨架问题与内容问题
正交、成本是加法（非乘法）、可线性扩展的结构是好骨架，重写它是负 ROI。动手前先判：是骨架坏了，还是内容坏了？多数是内容。
*MDA：技术栈维度矩阵（web×orm×rpc，加一个 ORM 只 +1 文件）骨架不动，只重写内容。*

### 4. 去重要根治：超集关系用一份文件
复制粘贴是漂移的原罪。A ⊃ B 时，核心写一次 + 围栏标注"A 特有"段。一份文件不会和自己漂移。
*MDA：MyBatis-Plus ⊃ MyBatis，两份字节级相同的文件合并为一份 + 围栏「Plus 附加」段。*

### 5. 单一权威源 + 分层职责
每个事实只有一个家。跨层 / 跨文件重复同一事实 = 漂移源。
*MDA：识别归检测器、gating 模式随栈归策略文件、schema 契约归策略。本文档自身也遵循此条——权威放 docs，skill 内只留指针。*

### 6. 证据先行 + 诚实区分"能力 vs 工程收益"
- **大改前用真实数据验证**（改完关键产出仍幸存 → 瘦身安全）。
- **强模型时代的诚实**：很多"改进"在已验证场景测不出能力提升——价值是工程性（可维护 / 鲁棒 / 显式），不是能力跃升。别 oversell，如实写 CHANGELOG。
- **显式入 spec = 买保险**：即便当前强模型不需要，写明仍为更弱模型 / 未来漂移 / 模型运气兜底。
- **随机性差异要多跑 + 一致性矩阵**，单跑测不出（如召回率从 2/3 提到 3/3）。
*MDA：tms/paybill 实测高价值发现幸存证明瘦身零回归；isWrite 召回新旧都 3/3，诚实记为"工程收益非能力跃升"。*

### 7. 通用 vs 特定的分界
复用型 Skill 只放**通用**知识；项目 / 环境特有的事实归项目配置（CLAUDE.md 等），不污染可复用资产。正文只描述当下规则，版本注解归 CHANGELOG。
*MDA：栈通用的 Wrapper gating 入策略；项目特有的"双 XML 根目录"归 CLAUDE.md。*

## 改 Skill 时的自检清单

- [ ] 这段内容模型自己导得出来吗？导得出就删。
- [ ] 改的是骨架还是内容？动骨架先三思。
- [ ] 同一事实是否在多处重复？合并到单一权威源。
- [ ] 改了标题 / 结构？谁按名引用过它？
- [ ] 有真实数据验证吗？它证明的是能力提升还是工程收益？诚实写。
- [ ] 放进复用资产的，是通用知识还是项目特有？

## 适用边界（别教条）

- "删模型已知"针对**强模型**。若 Skill 需兼容弱模型，显式指令是保险，该留。
- **契约 / schema 即使模型"导得出"也要留**——它的作用是**统一**多次产出的形态（让下游可依赖），不是教学。
- 召回底线即使模型"大概知道"也要留——它防的是偶发遗漏，是地板不是教材。
