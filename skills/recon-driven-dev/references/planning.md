# ④ 写计划 细则（基线计划六条 + 分层计划 + 执行方式决策）

> ④ 阶段的细则,由 SKILL.md ④ 指针引入。产物 `tasks.md`。无论 ⑤ 怎么执行,基线六条都写;选了子 agent 执行,再把计划升级到自包含。

## 基线计划六条（两条执行分支都写）

### 1. 曳光弹垂直切片

每个任务是穿透各层的**薄端到端切片**、各自可演示 / 可验,**不按技术层横切**(别"先建所有 model、再建所有 service")。切的依据是文件责任,见第 4 条。

*极小改动退化*:改动小到一个原子 commit 时,曳光弹退化为单切片——右尺寸(第 2 条)自然吸收此情形,不强行套"垂直切片 + 各自可演示"。

### 2. 任务右尺寸

任务 = **值得一个新评审者过一道闸的最小单元**——只在"评审者可能通过 A 却驳回 B"处切。把 setup / 配置 / 脚手架 / 文档折进它服务的那个任务,别因技术步骤单列。

### 3. prefactor 先行

**先让改动变容易,再做容易的改动**。该重构铺路的,排在功能任务前面。

### 4. 文件结构图（单点声明:按职责切、不按层）

列任务前先映射"**建 / 改哪些文件 + 各自职责**":改在一起的放一起,一个文件一个清晰责任。切任务依这条责任边界、不按技术层——这是第 1 条切片的根据。

### 5. 每任务验收标准 + 依赖顺序

每任务落**可观察的验收 checkbox**(把"可测"写成"做成什么算过"),blocker 先排。

**无占位符 + 可验收 checkbox 是两条执行分支都要的计划卫生底线**——不是子 agent 才升级;内联分支若允许占位符计划,会削弱 ④ 的 `⏸` 审计划这道最便宜拦截点。

### 6. ④ 自评（设计 ↔ 任务机械对账）

出 `tasks.md` 前对账:**设计每条 → 有任务接**;任务间引用的类型 / 签名 / 文件名一致。

**限机械对账、不复跑 ③ 的深度判据**(方案好坏是 ③ / 整支评审的活,搬进来 = 第二权威源)。

## 分层计划（随执行方式升级）

基线六条**总写**。按计划末尾的执行方式决定是否再升级:

- **选内联执行**:基线六条即够。
- **选每任务一个新子 agent**:把计划升级到**自包含**——每任务补 `Interfaces`(Consumes / Produces 的**精确签名**:函数名、参数与返回类型)+ **任务级完整文件路径**,喂给零上下文的 per-task 子 agent。(无占位符已是基线底线、见第 5 条;此处升级专指精确签名 + 完整路径。)

## ⏸ 执行方式决策（前移到 ④ 末）

计划主体写完后停一道,选 ⑤ 怎么执行:**内联**(主上下文逐任务跑) vs **每任务一个新子 agent**。前移到这里,是为了按选择把计划写对(子 agent 要自包含计划)。这个选择同时决定 ⑤ 走哪条执行分支。
