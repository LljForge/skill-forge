# ② 需求 + 设计 细则（对话契约 + 两份产物 + design.md 必覆盖清单 + 评审决策点判据）

> ② 阶段的细则,由 SKILL.md ② 指针引入。本文持四块单一权威:**对话契约 / `design.md` 必覆盖清单 / ② 自评 / 评审决策点判据**——脊柱与其余 reference 只指针、不复述。

## 职责与接棒纪律

**职责**:据 ① 的现状定"要什么"和"改哪些 / 怎么改",带着定向报告这个上下文,在主对话里**内联谈细需求与设计**(不依赖任何外部技能),**谈完一次产出两份**。

**边界**:① 是参考起点、**不是权威上限**——需求一细化就可能够到 ① 没覆盖的代码,**该自行补探就探**;自己探到的、① **没写的新面**记进 `design.md`(前向),**不回写已封存的 `directed-report.md`**(唯一例外:证伪 ① 的事实主张时走「事实订正」,见 SKILL.md)。代码既有的坑在 ① 里、这儿不重列。

**接棒纪律**:据 ① 的 `[未核验]` 条目做改动决策前,**先据手段补验**(升 `[实测]` 或推翻);够不着则在 `design.md` **显式记**「此决策建于未核验推断 X」——别让 ① 留标记放行的软推断在这儿被无声当真。

## 对话契约(怎么谈)

- **一次一个问题**澄清(优先多选),聚焦目的 / 约束 / 成功标准——这层喂 `requirements.md`。
- 定方案前**提 2-3 个不同方案**带取舍,领头给推荐 + 理由,别一头扎进第一个解法。
- **分节呈现设计**,每节确认再往下,别一次甩完只问一遍。

## 四条硬规则(强模型为"帮上忙"最爱跳过,必须明写)

- **HARD GATE**:呈现需求 + 设计并经**用户**(非模型自判)批准前,**不写任何实现 / 脚手架 / 代码**。
- **"太简单不需要设计" override**:该门适用于**每个**任务——单函数工具、配置改动、todo 列表也一样,不许以"这个很简单"自行豁免。
- **范围分解触发器**:若请求横跨多个独立子系统,**先分解**(列独立块、关系、build 顺序),别花澄清问题去细化一个该拆的项目;分解后逐个子项目走本轨。
- **别猜、先问(don't guess)**:设计对话中撞到**实质不确定 / 分叉**(选它会改需求 / 方向、爆炸半径或接口形态)——**先问用户,不自行拍板当既成事实**。有合理默认可自决,但须按 ADR 记进 `design.md` 必覆盖清单②(决策 + 被否方案 + 理由),让用户在 `⏸` 看得见你替他拍了什么。两条出口(**问** / **ADR 记**)必走其一,绝不静默拍板。(承 superpowers `executing-plans`「Ask for clarification rather than guessing」+ `brainstorming` Q&A;逃生阀承本文件必覆盖清单② 与接棒纪律。)

**反合理化红旗**(强模型最爱把模糊点自己拍掉,一律是红旗、非豁免):

| 冒出的念头 | 现实 |
|---|---|
| "这个显然就该 X" | 是实质分叉就 surface,"显然"是你的默认、不是用户的 |
| "用户八成想要 Y" | 猜想 ≠ 确认;无明显默认就问 |
| "先按常见做法做,回头再改" | 静默拍板;要么问、要么 ADR 记 |
| "问了显得没主见" | 漏问才是缺陷;带推荐地问才是专业 |

## 两份产物

### `requirements.md` —— 业务视角

要做成什么、约束、验收标准。形态见 [`templates/requirements.md`](templates/requirements.md)。

### `design.md` —— 本 Skill 自产设计内容(无外部技能代劳)

在 ② 对话里谈出**设计本体**(架构 / 数据流 / 关键决策 + 取舍 / 风险),按复杂度伸缩(简单的几句、有讲究的多写),扎根现有代码模式,只改服务本次目标的代码、不顺手大重构。两个透镜帮你谈深:

- **深模块 / 窄接口**(用「**窄接口**」指接口暴露面——**深度即杠杆、非行数比**):好接口窄、藏住实现复杂度,暴露面越小越好。
- **ADR 精神**:关键决策连「**被否方案 + 为什么**」一起记;随本次改动归档、**不引跨会话持久层**。

设计写成什么形态由对话生成、按复杂度伸缩,**不套死章节骨架、也不"完全无骨架"**——用下面的必覆盖清单约束(规定"每条都要覆盖",不规定"必须有哪几节")。

#### `design.md` 必覆盖清单(检查锚点 · 单一权威)

这份清单**会被写进每份 `design.md` 当检查锚点**——②自评、④自评、③定向报告验收、⑤整支评审的 Spec 轴都拿它**逐条对**。四要素必须覆盖,形态按复杂度伸缩:

- **① 架构 / 数据流** —— 本次改动的结构与数据走向。
- **② 关键决策 + 被否方案 + 理由** —— 含一句**深度 / 接口自陈**:逐项列出「本方案接口暴露面 = N(枚举哪 N 个)、为何算深模块」(可检查,不是泛泛一句"接口很干净")。
- **③ ① 锚定层** —— 圈定要改的契约子集 + 自行补探到 ① 没列的新发现(③ 定向报告验收拿这层跟 ① 的实答逐条对;它也是本次改动的**爆炸半径**所在)。
- **④ 测试缝隙** —— 在哪测、mock 哪里、**缝隙越少越好(理想 = 1)**;列出本次的测试缝隙数。**定缝隙形态前先客观核**目标模块的测试惯例与设施够不够 skill 默认 mock 假设(**模块既有测试抽纯函数还是 service-mock**、有无 mock 框架 / runner)——键于**可验证事实**、非主观「难测」(后者仍是 ⑤ `implementation.md:59` 红旗块的务实跳过);不达假设时直接把缝隙约定为**降级形态**(模块惯例零 mock / 无 mock 框架 → 核心判定抽纯函数测、编排留集成期;无 runner → 最轻可执行 / 文档化手动验证)并在 design 显式记,⑤ 照此执行**不算偏离**。这一条在 ② 的**设计分节确认流里谈定**(是分节确认的一节、**不另起独立 `⏸`**),并焊到 ⑤ 的 TDD——⑤ 在这里约定的测试缝隙上写失败测试。

## ② 自评(仅轻量卫生扫描)

出两份前扫一遍**就地修**:占位符(TBD / 待定 / 无理由的"本次不涉及")、内部矛盾、真歧义(能两种读法的挑一种写死)、范围蔓延。**只扫这四类、不复跑 ③ 的判据**(深度评审是 ③ / 整支评审的活,这儿重复 = 第二权威源)。

> **② 自评不覆盖「定向报告验收」**——查 design 有没有静默违反 ① 的坑 / 护栏,是 ③ 的活。这正是下面"跳过 ③ 的代价"的由来。

## 评审决策点判据(③ 是否进行 · 信号清单 + 代价提示 · 单一权威)

② 出口的评审决策点据本判据给「**建议评审** / **可安全跳过**」+ 理由,用户拍板。脊柱只点钩子,完整判据在此。所需信号由 ② 的产物/自评产出(承载处见下「② 端信号产出义务」)。

**强烈建议评审的信号:**
- ① 侧:协同护栏 / 反直觉陷阱 / 残留 `[未核验]`。
- ② 侧:触多个测试缝隙 / 多个 ADR 决策 / 爆炸半径大(① 锚定层牵动面广)。

**可安全跳过:** 改动小 / 单个测试缝隙 / 无非显性坑。

**两组信号冲突时**(skip 侧与「强烈建议」侧同时命中):以 ① 侧坑 / 护栏 / 残留 `[未核验]` 为准、**建议评审**——skip 侧信号只反映 ② 自评可见面,① 坑恰是 ② 自评盲区(理由见下「代价提示」,不另列)。

**代价提示(必须讲给用户):** 跳过 ③ = 丢掉「**定向报告验收**」——查 design 有没有**静默违反 ① 的坑 / 护栏 / 契约**;而 ② 自评**不覆盖**这一项(只扫四类卫生问题)。故"① 坑多"被设为强烈建议评审的信号,是有意的安全网。

### ② 端信号产出义务(链向评审决策点)

② 的产物/自评须让上面判据要读的信号**可读取**,各信号的承载处:

| 信号 | 承载处 |

| ① 协同护栏 / 反直觉陷阱 / `[未核验]` | 承自 `directed-report.md`,在 ① 锚定层(必覆盖清单③)与接棒纪律里带出 |
