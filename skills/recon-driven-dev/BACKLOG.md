## 1. ⑤ TDD / clean-baseline 三硬闸预设「测试设施存在」,无「环境无 runner」的 sanctioned 降级路径(来源:2026-06-24 applet-appid-feature-gating 实跑 · ⑤a/⑤b 发现 · 原 IP-02)

**问题事实**:`implementation.md:22` clean-baseline「先跑现有测试确认绿底,再写新码」、`:24` TDD 铁律「先写失败测试→亲眼看它失败→转绿」、`:59` 反合理化红旗块「太简单不用测=红旗」三处均隐含「环境有测试框架 / runner」前提。目标环境微信原生小程序无构建 / 无框架 / 无 runner,全文无「环境客观无任何 test runner 时如何落 clean-baseline / TDD」的明文降级,AI 卡在「必须写失败测试却无处运行」;与 `SKILL.md:8`「丢到任何环境都能独立跑」自包含承诺正面张力。现场已自收敛出 workaround:featuresOf 抽纯模块 + 裸 node 断言脚本红→绿,wx 依赖项走手动验证——说明指令未真死锁正确行为,只是没明示该降级。
**候选修法**(讨论于 2026-06-24,未定;均须守:触发条件做成「客观无 runner」可验证事实、**非**主观「这个难测」,否则重开 `:59` 红旗块严防的「务实跳过」豁免口子):
- A. 在 `implementation.md:24` TDD 铁律或 `:59` 红旗块尾补一句客观触发降级——「环境客观无任何 test runner(无框架 / 无构建 / 无可跑断言的 node 等)时,在 ② 约定的测试缝隙上以最轻可执行手段验红绿(如裸 node 断言纯逻辑);框架强耦合且确无 runner 的部分走文档化手动验证清单、在 design.md 显式记。触发是『客观无 runner』、不是『难测』——后者仍是红旗」。
- B. 挂在 `:22` clean-baseline 下,对称处理「无现有测试故无绿底可跑」的空白。
- C. 前移到 `requirements-design.md:47` 测试缝隙清单,要求 ② 先确认环境测试设施可用性、无 runner 时预先约定降级手段,⑤ 只执行。
**搁置原因 / 待触发条件**:涉护栏权衡(降级措辞贴着 `:59` 红旗块的「务实跳过」红线,触发条件「客观事实 vs 主观判断」的边界设计需斟酌)+ 单样本不足以定修法落点(挂 clean-baseline / TDD / 还是前移 ②)。待第二个「无测试栈环境」实跑样本佐证此为高频真空白后定稿。借鉴溯源已核:TDD 红绿源自 superpowers,`CHANGELOG` v0.3.0「明确砍掉」清单不含「无测试环境降级」,故非有意砍、不算胀回。

## 2. ② 接口 / 落点契约在 ②→④→⑤ 流转中与实现失同步(两面:设计侧落点未谈定 + 实施侧正当偏离无留痕)(来源:2026-06-24 实跑 · ②d/④a/⑤c 发现 · 合原 IP-03 + IP-04)

**问题事实**:
- (设计侧 · 原 IP-03)`requirements-design.md:47` 必覆盖清单④「测试缝隙」只规定「在哪测 / mock 哪 / 缝隙数」,不约束缝隙逻辑的文件落点 / 形态。落点的单一权威其实在 `planning.md:21-23` §4「文件结构图(单点声明:建 / 改哪些文件 + 各自职责)」——清单④ 与 §4 是分属 ②设计期 / ④计划期的两个权威源,二者间缺一句「测试缝隙的可测性若倒逼文件边界,在设计期就让 §4 接住」的衔接钩。致 featuresOf 在 design(:64 `featuresOf(appId)`、`hasFeature` 内部调它)+ tasks(T1 钉 util.js)钉死落点,⑤ 才被迫抽成独立 `utils/features.js`、签名变 `featuresOf(appId,configs)`(规避 config↔util 循环依赖 + 可 node 测),design / tasks 留下与实现不符的死信息。
- (实施侧 · 原 IP-04)⑤ 期为正当工程理由(规避循环依赖)偏离 ②已批准的接口签名 / 落点后,无 sanctioned 留痕路径:`planning.md:33` ④自评「签名一致」是写计划期快照、够不到实施期漂移;`SKILL.md:120-130` 事实订正只覆盖「推翻①事实」、不覆盖「②接口的正当偏离」;`implementation.md:61` false-①-premise pause 只覆盖「design 建在错前提」。design.md 未封存(`SKILL.md:66` 允许前向回写),但无一句点名 ⑤ 期偏离接口该走此回写。
**候选修法**(讨论于 2026-06-24,未定):
- 设计侧:在 `planning.md` §4(落点的家)加一句衔接钩——「§4 划文件职责时,若 ②清单④ 约定的测试缝隙要求(如可独立单测)与某文件的既有框架 / 循环依赖冲突,在此把缝隙逻辑划成独立职责文件、别拖到 ⑤ 才抽」。**明确否决**在清单④ 加「落点子要素」——会与 §4 制造第二权威源,且因下游(②自评 / ④自评 / ③定向报告验收 / ⑤Spec 轴)全按「design.md 自带检查锚点」泛引用「逐条对」,新子要素会被凭空卷入四处对账。
- 实施侧:在 `implementation.md:25-26`(per-task 评审 / continuous-execution 旁)加一句指针式——「⑤ 期为正当工程理由偏离 ②已批准的接口签名 / 落点(非 design 出错)→ 回写未封存的 design.md(同 `SKILL.md:66` 前向机制)+ 一句偏离理由,使 per-task spec 符合性评审对齐更新后的 design;区别于事实订正(推翻①事实)与 false-①-premise pause(design 建在错前提)」。**明确否决**扩 `SKILL.md` 事实订正覆盖②——会把「推翻事实」与「正当偏离」两种语义混进同一机制、触碰跨全程总账。子 agent 分支须同步(`implementation.md:25` 末已声明该闸由派发子 agent 承担)。
**搁置原因 / 待触发条件**:两面均为「跨阶段衔接灰区」、主体含模型失误成分(§4 已有落点权威、design.md 已可回写,主 agent 未用足);单样本不足以定「衔接钩的家在设计侧还是计划侧」「两面是否同一落点合修」。待第二个「测试缝隙可测性倒逼文件边界 / ⑤ 正当偏离②接口」样本佐证此洞频次后,合并定稿。消费者映射已核:修 §4 由 §4「按职责切」既有机制自然接住、修 implementation 由 ⑤ 两分支按序消费,均不触发额外具名引用同步。

## 3. 复盘协议(元层)输入「全部产物」未交代归档后路径,与 ⑤k 归档时序跨文档无衔接(来源:2026-06-24 实跑 · 元层复盘 / ⑤k 发现 · 原 IP-08)

**问题事实**:`MAINTAINING.md:48` 复盘层规定复盘 agent 输入「监督笔记 + 实走路径单元 + 全部产物」,对「全部产物」在哪取零交代;而 `SKILL.md:106` / `implementation.md:40` 的 ⑤k 归档 ⏸ 把对象层产物从 `docs/recon-driven-dev/<date>-<task>/` 整体移进 `_archived/<date>-<task>/`(监督资产则在另一棵树 `docs/recon-driven-dev-eval/<date>-<task>/`、标「归档即弃」、不被搬)。两份文档不交叉引用,本次主 agent 靠问用户才定复盘与归档先后。注:先后其实从设计可推定——`MAINTAINING.md:48` 明写复盘在「整任务跑完」做、⑤k 是对象层末步,故复盘必晚于归档、产物此时已在 `_archived/`;本次复盘 agent 实际也拿到了正确的 `_archived/` 路径,缺陷未致故障。
**候选修法**(讨论于 2026-06-24,未定):在 `MAINTAINING.md:48`「输入『…全部产物』」后补一句指针式衔接——「复盘在整个对象层运行(含 ⑤k 归档)跑完后做,故『全部产物』此时已在 `_archived/`(按 implementation.md 归档处取);监督笔记 / 改进点仍在未归档的 `recon-driven-dev-eval/`」。守:纯元层(不随运行载入、对五阶段零影响)、单一权威(`_archived/` 规则只指针引归档处的家、不复述路径规则)、防膨胀(一句话)。
**搁置原因 / 待触发条件**:先后可从设计推定 + 本次未致故障(复盘 agent 拿到了正确路径),单样本不足以在「补一句加固 vs 判作可推定不动」间定向;防过度修——为「字面没说但可演绎、且未致错」的衔接补丁动权威源有「文档完备性洁癖」之嫌。待第二个「复盘 agent 真传错归档前旧路径致故障」样本即升级当场改;若再不复现则直接关。
