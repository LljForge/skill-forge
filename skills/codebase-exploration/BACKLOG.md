## 1. D4 多栈仓跨栈边(前端→后端 HTTP)+ §6「单一权威源」对多栈的诚实性(理论对标:theory-foundation §3 D4)

**问题**:依赖矩阵只采各栈内部 import 边;多栈仓(Java+TS,用户项目几乎全是)前端→后端 HTTP 跨栈边不画,但 §6 自称「单一权威源」——对多栈名不副实。
**候选修法**:
- **轻量(措辞诚实化、零 dogfood)**:给「单一权威源」加限定「各栈内部的权威;跨栈运行时边属暂缓的 ②④」,§6 改为「栈内 import 权威 + 跨栈派生候选」两层。与 cbx targets 既有口径对齐。
- **重量(开 ②④ 能力、必 dogfood)**:retroactive linking——前端出站 HTTP 槽 + 后端 endpoint 槽,合并阶段按 path 延迟匹配,标「跨栈/HTTP 候选边、低置信」。撞慢变闸(path 快变),须抽象到慢变形状粒度。
**待触发条件 / 搁置原因**:用户倾向「多栈专项单独找项目实跑验证后再针对性优化」。轻量措辞可随任意一轮顺手做;重量与 ②④ 解禁一并设计。

## 2. D5-A 超大仓卡片折叠(~70 子包破「5 分钟读懂」)(理论对标:theory-foundation §3 D5-A)

**问题**:超大仓产出几十张 8 行卡片 + 上百行矩阵,破灵魂「新人 5 分钟读懂」,schema 无折叠 / 取舍指引。
**候选修法**:Shneiderman overview→detail + 重要度排序——加分组概览层(伞形包/层聚 5–9 区),高扇出 hub 详写卡片、外围折叠一行 + 按需 /module-brief;矩阵升级折叠式分区矩阵(反向边永远冒泡顶层)。**坑**:折叠是展示层、不倒灌改边界;外围留地标不删条目。
**待触发条件 / 搁置原因**:触发场景在超大仓。**盯 slp(1666 java)**——下次在 slp 上跑时若卡片数破「5 分钟读懂」即出列。有真实样本再设计,不凭空。

## 3. D5-B 单域超大模块 vs /module-brief 容量(双闸正交)(理论对标:theory-foundation §3 D5-B)

**问题**:单域超大模块(如 300+ java)按业务内聚不拆该保一个,却超下游 /module-brief 上下文容量;曾有硬阈值被删(怕误伤单域大模块)。
**候选修法**:内聚闸(不拆)+ 容量闸(独立)双闸正交——容量闸只触发「标注超容量、建议分子域/拓扑分批 brief」,**绝不触发改边界**;阈值用 token 预算非文件数(ArchAgent adaptive grouping)。**坑**:极易退化成文件数硬阈值、复活 v0.3.0 删掉的项目魔数。
**待触发条件 / 搁置原因**:更罕见 + 改起来烫手(撞硬阈值血泪史)。同样**盯 slp**——出现某单域大到 /module-brief 爆容量再做。属契约段,须自带 dogfood。

## 4. ②④ 入口与执行流 / API 表面能力槽(理论对标:theory-foundation §3 ②④)

**问题**:SKILL.md 留而未建的两个能力槽。
**候选修法**:② 入口借 call-path 有界 k-hop 追踪、只记慢变骨架链;④ API 表面优先复用契约文件(OpenAPI/proto)否则源码路由拼接、先认全类目再枚举(呼应 §5.5.1);④ 与 D4 后端 endpoint 槽**复用同一信号源**(一举两得)。**坑**:执行流易变,只记抗重构骨架、不画逐行调用栈(守慢变闸)。
**待触发条件 / 搁置原因**:真·能力扩张,与 D4 重量版同源,独立立项;留槽不建是 v0.6 起的有意设计(②③④ 仍有顾虑)。
