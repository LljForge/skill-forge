# 代码探索 codebase-exploration

帮**第一次接手陌生项目**的新人快速建立结构认知：有哪些模块、各自干什么、模块间怎么调、哪里有坑、从哪读起。

## 灵魂：产物是项目结构地图（活文档），不是给分析师看的报告

**定位**：本 Skill 产出的是**项目结构地图——一份活文档**。它记录慢变的架构结构，可作人接手的项目资产，也可作 AI 编码时的架构指引。地图只写抗重构的慢变结构——这既是产物不易过时的根，也是后文「能力准入闸」的依据（易变细节已被剔除，不进产物）。慢变纪律天然抬高权威度：读者看到的内容，在下次大重构前基本成立。

读者是**还不懂业务、没有可信标准答案**的新人。三条底线：

1. **读者优先**——讲业务，不讲分析过程。读者要的是"这模块叫什么、干啥、入口在哪、改要注意啥"，不是"被注入 31 次 / `grep -c` = 40"。度量数字是**分析者下结论的工具**，结论可保留，过程留分析者脑里。
2. **边界不甩判断**——模块该合该分由 Skill 用度量定默认，产物不要求读者拍板**边界**。但「**要不要对某模块花一次 `/module-brief` 深析**」这类地图看不透的取舍（薄底座有没有藏雷、值不值），老实说明情况交读者判更诚实（见第 3 条）。
3. **校准收窄**——置信标记（高/中/低）**只留在真正模棱两可的边界**，不在产物头部贴一刀切的「草稿·非权威」自贬。慢变结构判断拿得准的就写实，拿不准的地方才标低置信并导向 `/module-brief <模块>` 核实；**不脑补、不武断**。

**与邻居 Skill 的分工**（互不依赖）：

| Skill | 性质 | 何时用 |
|------|------|--------|
| **codebase-exploration**（本） | 广、快、跨栈、活地图 | 刚接手，先建立全局认知 |
| module-brief | 选定**一个**模块、读进上下文产出需求+设计认知文档 | 探索后要深入读懂某个模块 |
| module-spec-baseline | 把模块现状逆向成 openspec 规范基线（长期、过 validate） | 要为某模块引入 openspec 治理 |

## 慢变纪律闸

本 Skill 产物只写抗重构的慢变结构。这张表既是**产物过滤器**、也是**能力准入闸**——进 Skill、进产物都必须过这一关：

| 进（抗重构的慢变结构） | 出（易变细节，交 `/module-brief`） |
|---|---|
| 模块边界与职责、分层 | 行号、精确文件/类计数 |
| 依赖**形状与方向** | 穷举 PO/类清单 |
| 入口约定、横切约定 | 随提交而变的度量 |
| 核心实体↔模块归属 | 具体实现细节 |
| 真陷阱（同表多映射 / hub / 触发器隐式耦合） | — |

**impact（影响面/改动半径）**因随每次提交而变，落「出」栏，故排除在地图之外——留 `/module-brief` 或临时查询处理。这条排除就是闸门的示范：不是内容不重要，而是它不满足「慢变」准入条件，强写进地图反而让地图快速过时。

**度量数字处理**：沿用「度量数字的处理」节规则——翻成读者结论、全文 ≤ ~8 处；超了说明在写报告，不在写地图。

## 信号配方层

**本 Skill 不携带 16 份硬编码栈配方。** 携带的是：通用信号槽模型 + 通用名模式 + 框架约定速查表（派生种子，非穷举）+ 派生协议 + Java 首份结晶参考。配方只回答一件事：「这个栈里，每个信号槽该 grep 什么」（机械层）。机械 vs 智能的铁律见 §5.5 护栏。

### §5.1 通用信号槽表

每个信号槽对应一类可 grep 的机械锚点，覆盖所有能力的采集底线。Java 参考填法是种子示例——换栈时按「跨栈含义」列派生该栈写法，不替换通用槽。

| 信号槽 | 服务能力 | Java 参考填法 | 跨栈含义 |
|---|---|---|---|
| 模块单元 | ① | package 目录 | 该栈"候选模块边界"是什么（TS 特性目录 / Go package / Python package） |
| 导入依赖边 | ① | `import <root>.` | 跨文件/模块依赖怎么表达 |
| 依赖装配（强度） | ① | `@Autowired`/`@Resource`/构造注入 | 依赖怎么接线 → 定 Injected vs Import-only |
| 入口点 | ①⑤ | Controller / `@Scheduled` / main | 路由/CLI/定时/消费者/main |
| 数据访问 | ① | MyBatis / `@TableName` | ORM/repo → 表/实体映射（辅助锚定模块） |
| 自声明清单 | ① | Maven `<modules>` / `@ComponentScan` | 构建/模块声明（起骨架） |
| 横切件 | ⑤ | `@Aspect` / HandlerInterceptor | AOP/middleware/拦截器/装饰器/filter/guard |
| 约定锚点 | ⑤ | base 类 / 全局异常处理器 / SecurityConfig | 认证/错误/日志/配置/事务/缓存 在哪定义 |

### §5.2 通用名模式（直接可 grep）

以下模式是真·语言无关的公共约定事实，已被多栈验证。它们帮助区分「入口点」与「工具方法」——入口点优先识别，工具方法降权、不当入口。每个栈在通用模式基础上**叠加**该栈特有入口名（如 Django 的 `views.py`、Express 的 `router.`），**不替换**通用模式。

**入口点通用名模式：**

- `^(main|init|bootstrap|start|run|setup|configure)$`（忽略大小写）
- `^handle[A-Z]`、`^on[A-Z]`、`Handler$`、`Controller$`
- `^(process|execute|perform|dispatch|trigger|fire|emit)[A-Z]`

**噪音/工具名模式（降权，非入口）：**

- `^(get|set|is|has|can|should|will|did)[A-Z]`、`^_`
- `^(format|parse|validate|convert|transform)`（i）、`^(log|debug|error|warn|info)$`（i）
- `^(to|from)[A-Z]`、`^(encode|decode)`（i）、`^(serialize|deserialize)`（i）
- `Helper$`、`Util$`、`Utils$`、`^utils?$`、`^helpers?$`

### §5.3 框架约定速查表（派生种子，非穷举）

从公共框架约定蒸馏，覆盖主流生态；agent 探测到栈后查表起步、现场补。**作派生起点，不是完整配方**——新栈缺行照槽模型 + 通用名模式现场派生；某栈跑顺了**结晶**进本表成下一份参考。

| 栈 | 入口/路由约定 | 依赖装配 | 数据访问 | 横切件 |
|---|---|---|---|---|
| JS/TS | Next.js `pages/*`·`app/**/page.*`·`app/**/route.ts`(api)·`layout.*`；Expo `*+api.*`/screens；Express/MVC `routes/`·`controllers/`·`handlers/` | NestJS `@Injectable`/构造注入；否则手工 wiring | Prisma/TypeORM/Drizzle；`schema.prisma` | NestJS guard/interceptor/middleware；Express middleware |
| Python | Django `views.py`·`urls.py`；FastAPI/Flask `@app.get`/`APIRouter`；入口 `main.py`/`manage.py` | 多显式构造；FastAPI `Depends` | SQLAlchemy/Django ORM | 装饰器 / middleware |
| Go | `main.go`；handlers/routes/controllers 目录；gin/echo/fiber 路由 | 显式构造/wire | gorm/sqlx | middleware 链 |
| Java/Spring（**Java 结晶参考**） | `@RestController`/`@Controller`·`@Service`·`@Scheduled` | `@Autowired`/`@Resource`/构造注入 | MyBatis/`@TableName`/JPA | `@Aspect`/HandlerInterceptor；全局异常处理器；SecurityConfig |

### §5.4 栈探测协议

按 manifest 文件识别栈，再取对应配方起步：

- `pom.xml` / `build.gradle` → **Java**，查 Java/Spring 行
- `package.json` + `tsconfig.json` → **TS/JS**，查 JS/TS 行
- `pyproject.toml` / `requirements.txt` → **Python**，查 Python 行
- `go.mod` → **Go**，查 Go 行
- 其余栈：照 §5.1 槽模型 + §5.2 通用名模式现场派生，补进速查表结晶

**多栈仓库**：各栈独立跑各自配方，再合并；合并时模块单元按各自栈边界，跨栈依赖边按文件归属栈判强度。

### §5.5 护栏

**机械/智能分工铁律**：配方只定「grep 什么」（机械，1.A 用 Bash grep 已足够）；归并/强度/反向异味/约定识别是**当前 agent 的智能任务**（需项目特有锚 PO/语境/拓扑判断），**不要做成脚本**（脚本化需硬编码项目锚表、违通用定位）。**禁把"派生配方"当借口把智能归并也机械化。** 未来某子动作反复机械重复才考虑建辅助脚本，现在不预设。

**许可证护栏**：不抄 GitNexus 源码、不依赖它；借的是「框架怎么摆文件」这类公共约定事实，用自己的话重表述。

### §5.5.1 完整性闸（「枚举/无」类断言总则）

「枚举所有 X」「未发现 X」「无 X」跑在临时心算的子集上必假阴性。铁律：**下这类断言前，候选集须是从 1.A `import` 全集（+扁平分层同包补扫）机械派生的完整集合，逐项核后才可断言**，不凭印象取子集。

- **候选集机械生成、粒度=「边/表面」非「文件」**：即 1.A 已采的全部跨模块边，或某类对外表面/锚点的栈等价物全集；规模由模块数/表面数界定、非文件数——大仓按此粒度仍可控，**不退化为逐文件穷举**。逐项判向/判有无仍是 agent 智能任务、不脚本化。
- **判「无」按类反证**：机制类「无 X」（无全局异常处理器/无认证/无某横切）→ grep 该机制全部栈等价锚点；文件类「无 X」（仓内无某类文件）→ `find`/`grep -rl`。**任一命中即不得写「无」。**
- **「枚举某类表面」先认全类目**：先识别有哪几**类**对外表面（鉴权/路由/消费者…），再各类内枚举——免得漏掉一整类还自称已枚举（如只认主鉴权、漏旁路 token 守卫）。表面属开放集：给代表 + 「按栈现场补」，不强求穷尽冷门协议。
- **完整性只约束「验证」、不扩产物**：要求**分析者核得全**、**不要求产物列得全**。产物照旧——⑤ 写代表锚点 + 不穷举使用点，第六节按既有「代表非穷举」契约列跨子模块边（每边取代表锚点、不加调用点行），≤~8 度量、8 行卡片、慢变。**查得全 ≠ 写得全。**
- **召回上限**：候选集召回受 1.A 上限约束——SPI/反射/XML 等无 import/无同包痕迹的装配边可能漏在集外，属**已知缺口**，不假装「已穷尽」。

## 能力①：模块测绘（跨栈）

### 核心认知：包 ≠ 模块

顶层包结构会骗人，**不能 `ls 包` 当模块**。模块的本质是高内聚低耦合——用**代码连接关系**定边界，类名只用于事后起名（名字是最弱信号，异名/跨切面会让它失效）。

### 关键架构约束：模块粒度 = `/module-brief` 单次可消化

地图的**唯一下游消费者**是 `/module-brief <模块>`——会把模块全部代码读进上下文做深度分析。若把伞形包当成单个模块，跑 `/module-brief` 直接爆上下文。所以：

> **每个候选业务模块按业务内聚性判**（凭类名读一遍看是不是单一域），跨多个不相关业务域的伞形包必须继续拆。**不设硬文件数阈值**——大 ≠ 多域，单域大模块（如 tms 88 java）该一个就一个。文件数仅作展示证据，不据阈值判过宽。

伞形包（如典型遗留 600+ java 主数据包）必须拆到每子模块单一业务内聚；拆够后每个子模块对应一条 `/module-brief <子模块>` 调用。

### 测绘两步

1. **依赖矩阵采集**（先做）——**严格分 4 个 sub-step 顺序执行，不要合并成"一站式大脚本"**

   Why 显式分步：步骤 1 含 1 个机械子任务（量大手累）+ 3 个智能子任务（要现场判）。混在一起做容易写 300+ 行 Python 一锅端，反而把智能任务也机械化、撞上"扁平分层归子模块需项目知识"的循环依赖。分步走：机械的让 Bash 做、智能的留给当前 agent 现场判。

   - **1.A 采集依赖边（机械、可借助 Bash）**：按§5 信号配方层，跑「导入依赖边」「依赖装配」「数据访问」三个信号槽的 grep，采集跨文件/模块的依赖原始信号。**非 Java 栈**：换该栈对应配方（见§5.3 速查表 + §5.4 栈探测协议），原理同。

     **Java 栈补充**：用一行 `grep -rE "^import <本项目根包>\." <后端 src 根> --include="*.java"` 拿 import 全集。**扁平分层项目还需追加同包补扫**：扁平分层下多子模块共享一包（如 `com.x.masterdata.service`），同包类互调不产生 import 语句、grep import 必漏；需在 flat-package 内对被注入/引用的类名做 grep 补扫源码（如 `grep -rE "\b<TargetClass>\b" <flat-package>`）。**只做机械扫描**——无需现场写 Python 脚本枚举/聚合（按本节最后「关于辅助脚本」段：Bash 短组合已足够，写脚本封装属于把智能任务一并塞进去的反模式）。

   - **1.B 归子模块（智能、当前 agent 现场判）**：拿 1.A 的输出，对每条依赖边判源/目标各属哪个 kebab 子模块。以 Java 为例，其它栈同理按各自配方派生。
     - 包即模块（每子模块独立子包，如 `com.x.order.*`）：按包路径直接归。
     - 扁平分层（多子模块共享 `service/dao/po` 一包）：按**类名前缀** + 第二节卡片「锚 实体/表」交叉判定（如 `import com.x.masterdata.po.AppCenter` 按 AppCenter 前缀归到含此实体的 `mdm-app-cfg`）。
     - 与本子模块取差集 = 本模块的跨子模块依赖。

   - **1.C 强度分档（智能）**：对每条边判一档（以 Java 为例，其它栈同理按各自装配方式判）：
     - `Injected`：注入边 `@Autowired` / `@Resource` / 构造注入
     - `Import-only`：仅 import 无注入
     - `Adapter`：目标在 `api.*` / `kd.*` 等基础设施包
     **强度档区分力依赖「装配是否注解显式」**：当本仓装配**非注解显式**（手工 wiring / Lombok `@RequiredArgsConstructor` 构造注入无字段注解 / 扁平分层同包 import 即用 / 无 DI 框架的前端栈）→ 三档会坍缩为**近全 `Import-only`**。**这是真实结论、非缺陷**——产物显式声明「本仓装配非注解显式、强度档区分力弱」即可，不硬造区分；此时「目标是否基础设施/外部」仍能有效分出 `Adapter`。
     **护栏：坍缩须先验明、不殃及注解显式边**——坍缩声明只适用于装配**确非**注解显式的仓（先验明本仓装配方式再决定）。仓里**确有** `@Autowired`/`@Resource`/显式构造注入的边，**每条按其真实装配分档，不得因坍缩心态把真注入边降为 `Import-only`**。
     adapter/util/共享 enums 列入但标 `Adapter` 或弱依赖；本子模块内 import 不列。

   - **1.D 反向异味标注（智能）**：底层模块 import 上层模块（如 mdm-employee import ccm.budget）= 拓扑反向，标 `(反向异味)`。判向先用"基础设施 < 主数据底层 < 业务编排 < 顶层"项目层级直觉粗排，再回头标。**双向耦合不一定算反向异味**：若一方是横切/审计/日志服务出站调用，视作正常（如 customer-enterprise 调 MdmLogSyncApiService 记日志属正向，不标异味）。**「未发现反向边」须对 1.A 完整候选集（见 §5.5.1）逐边核后才可断言、非乐观默认**：候选集含**「域→域（平级）」边**、不止「下层→编排层」——平级跨域 `@Autowired`/`@Resource` 注入正是反向边高发处；逐边判向核完才能写「无反向边」（漏标真双向环会误导接手者）。

   此 4 步产出依赖矩阵全集（第六节产物），是后续拓扑序与卡片「依赖摘要」的**单一权威源**。

   **关于辅助脚本（防误用）**：当前 SKILL 不携带 Python 脚本——1.A 用 Bash grep 已足够，1.B-D 是当前 agent 的智能任务（写脚本归类需项目特有锚 PO 表硬编码、违通用 Skill 定位）。若未来某子步骤反复出现可机械化的子动作（多次跑测都重写），再为该具体动作建辅助脚本；现在不预设。

2. **业务语义提取**（人话、后做）

   每个子模块抽 1 句"干什么" + 1-3 条关键线索，凑模块卡片。来源按强度：**DDL 表/字段 `COMMENT`（最强）** > `@ApiModel`/`@Api(tags)`/`@ApiOperation` > 类/方法 JavaDoc > API 路径业务名。**线索带来源标注**（`[DDL]` / `[path]` 等）；脚手架占位注解（`tags="xx控制器"`）剔除。非 Java 栈用各自等价注解/注释约定，原理同。

### 信号源（召回底线）

**跨栈总则：换该栈的 ORM 映射 / DI 注入 / 路由表，原理同**——下列信号槽在所有栈中都存在，只是表达形式不同，按§5 配方层取各栈对应 grep。

- **起骨架**：项目自声明清单——`@MapperScan` / `@ComponentScan` / `@EnableFeignClients` / Maven `<modules>` 列举的包（Java）；TS workspaces / `go.mod` / Python package `__init__.py`（其它栈）。先判构建单元是按业务拆还是按技术层拆，后者要下沉到包/目录。
- **纵向绑定**：ORM 映射（Java: `@TableName` / Mapper XML `namespace`/`resultType`；其它栈见§5.3）、DI 注入边。
- **跨切面 = 共享度**：被多少不同文件引用；高共享（审计/同步/配置/加解密切面）拎进**全局基础设施**，不当业务模块。
- **三视角交叉**：代码关系 / 路由树（API 路径 / CLI 路由 / 消费者注册）/ 数据（DDL/schema）——三方互证则高置信，分歧则标低置信。

### 栈陷阱（模型不会主动扫，必须主动查）

> **跨栈同理**：以下条目以 Java 为典型例；换该栈 ORM/DI/路由原理同，明显 Java 专属的陷阱已标注，对应栈的同类陷阱由§5 配方/agent 现场识别。

- **伞形包**：类跨多个不相关业务簇 → 按簇拆，拆到每子模块业务内聚。（Java 典型例，跨栈同理：任何按技术层平铺的顶层目录都可能是伞形）
- **扁平分层**：`service/dao/po` 平铺多模块，边界在**类名前缀**不在目录；依赖矩阵采集要按前缀+锚 PO 交叉判定。（Java 典型例，跨栈同理：换该栈 ORM/DI 组织方式，原理同）
- **适配器集合**：某包/目录 `controller=0` 且子包清一色外部系统名 → 对外部系统客户端集合，**归基础设施、非业务模块**。
- **`Base*` 等假前缀**：无区分度的通用前缀，聚类失效 → 下沉看内聚不看名。
- **审计/日志伪装成模块**：某 Service 族 0 调用方 + 表名逐一镜像业务表 → 横切审计层，非业务模块，哪怕有 Controller/路由暴露查询。**写入路径必须核实别脑补**：grep DDL 的 `CREATE TRIGGER`（Java 典型例：MySQL 可能用 `create definer = X@'%' trigger ...` 语法，关键字不连写）、AOP 切面（Java）/ middleware（其它栈）、MyBatis 拦截器（Java 典型例，对应栈同类陷阱由配方/agent 识别）各一遍；只有名字痕迹无定义 → 标"写入路径存疑"。
- **扁平 CRUD 编排外移**：注入图极稀疏时，业务编排散在 Controller/Handler（一个方法调多个 Service）→ 看流程必须读路由入口，识别这种横跨多团的"编排团"（不落 PO/entity）。
- **同表多 PO**：多个 PO/Model 映射同一张表，且命名可能误导（如 `XxxBankAccountPO` 实际映射 `company_account` 表）→ 数据层陷阱，易改错对象。（Java/MyBatis 典型例；对应栈同类陷阱由配方/agent 识别）
- **依赖方向望名生义**：命名/表名前缀**不定**依赖方向。典型反例：`tms_bfe_flow_history` 表名带 `tms_` 但实属 bfe 模块、方向是 tms 调 bfe；`fin`/`bfe` 体量小看着像"附属"，实际是 tms 上层依赖的底层服务。**依赖方向必须来自依赖矩阵（机械事实），不能靠名字 / 前缀 / 体量推**。判错让"底→顶"反向，新人就不是从地基往上盖。

## 能力⑤：约定与横切机制

**目标**：让人/AI 在改代码前，知道"请求先过谁"和"加东西要守什么规矩"——不是列出所有用到的地方，而是给出每条约定的机制、定义点、怎么遵守这三件事。

### 测绘什么（慢变）

**① 横切件**——"请求先过谁"

识别 AOP/middleware/拦截器/装饰器/guard/filter 等横切件，关注**顺序与作用域**（哪些路径被覆盖、哪些例外放行）。信号槽：§5.1「横切件」槽（`@Aspect` / HandlerInterceptor / NestJS guard/interceptor / Express middleware / 装饰器 / middleware 链）；速查表 §5.3「横切件」列。

**② 项目约定**——"加东西要守什么规矩"

测绘六类慢变约定，每条给「在哪定义（代表锚点） + 怎么遵守（一句）」：

| 约定类 | 典型锚点（Java 参考） | 跨栈同理 |
|---|---|---|
| 认证 | SecurityConfig + JwtAuthFilter | NestJS AuthGuard / Django middleware |
| 错误处理 | `@ControllerAdvice` 全局异常处理器 | Express error-handler / FastAPI exception_handler |
| 日志 | AOP `LogAspect` / Logback 配置 | middleware 日志 / structlog |
| 配置 | `application.yml` / `@Value` 绑定类 | `.env` + config loader |
| 事务 | `@Transactional` + 传播策略定义 | SQLAlchemy session / Django transaction.atomic |
| 缓存 | `@Cacheable` / Redis config Bean | cache decorator / django-cacheops |

> 表内「典型锚点」是**示意参考**；写进产物前必须对本项目 **grep 核实**实际定义点（见下方「校准」），不得照抄本表。

### 信号

- **横切件槽**（§5.1 + §5.3 横切件列）：`@Aspect` / HandlerInterceptor；NestJS guard/interceptor/middleware；Express middleware；Python 装饰器/middleware；Go middleware 链——按 §5.4 栈探测协议取对应行。
- **约定锚点槽**（§5.1 + §5.3 横切件列）：base 类 / 全局异常处理器 / SecurityConfig；速查表 §5.3「横切件」列的锚点是种子，现场补充。
- **共享度辅助**：被多文件引用 = 候选横切件（§能力① 信号源「跨切面 = 共享度」法，引用即触达，不穷举）。

### 慢变纪律

**只记「机制 + 定义点（代表锚点） + 怎么遵守（一句）」，不穷举使用点。**

**可执行约定规则范例（逐字）**：

> `认证 — 所有 Controller 经 JwtAuthFilter（security/JwtAuthFilter）；加新接口默认受保护，放行须进 SecurityConfig 白名单。`

该范例格式即全部六类约定的输出标准：一条一行、机制+锚点+操作规则、读者扫一眼知道"我加接口该怎么做"。

### 校准（延续灵魂）

- **约定确认不了标存疑**，绝不脑补——写进产物前必须有锚点文件支撑。
- **判「无」与判「有」对称都要锚点 — 按 §5.5.1 完整性闸逐项核**：下「无 X」前正向反证——机制类（无全局异常处理器/无认证/无某横切）grep 该机制全部栈等价锚点（全局 error：Java `@ControllerAdvice`/`@ExceptionHandler`、Express 4-arg `app.use((err,req,res,next))`、FastAPI `exception_handler`）；文件类（仓内无某类文件）`find`/`grep -rl` 反证。**枚举对外表面（如鉴权）先认全类目再各类枚举**，免漏旁路 token 守卫这类第二表面。**命中却写「无」= 事实错误**；确空才可下「无」。
- **写入路径必须 grep 核实**（trigger / AOP / 拦截器，呼应 v0.1.1 审计陷阱）：横切件名字有了但找不到切面定义 → 标「写入路径存疑」；勿因命名推断就当真。
- 非 Java 栈：换该栈等价锚点 grep（如 NestJS `useGlobalGuards` / Express `app.use` / Django `MIDDLEWARE`），原理同。

### 边界

⑤ **不当业务模块**——横切件与约定独立成节，与 ① 的业务卡片完全分开；不得因"某 Service 有日志逻辑"就将其归入横切件（业务 Service 夹带日志不等于它是横切件）。

**文件头**：`generated_by` / `generated_at` frontmatter + 头部声明「**项目结构地图（活文档）**」（可疑边界已标低置信，核实见 `/module-brief`）。

**正文七节**：

| 节 | 内容 |
|---|---|
| 一 技术栈与分层 | 栈 / 构建运行 / 分层，一行带过 |
| 二 模块总览 + 模块清单 | ① 能力：跨栈模块单元 + 读者卡片（边界/职责/入口/依赖摘要/置信） |

| 五 上手顺序 | 拓扑序从第六节派生、按层分组每层一表；末列 ✅跑/⚖️你定/⬜不跑 + 理由 + `➡️ /module-brief` 命令 + 顶部三态图例 |

| 七 新鲜度与重生 | 见下方第七节处理段 |

**模块卡片模板**（每模块严格按此结构，不超 8 行）：

```
### `<模块 kebab>` <名>（<伞形包名> 子模块）　置信：<高/中/低>
- **干什么**：一句业务人话（20–50 字）
- **入口**：主要入口 2–3 个（读流程从这进）
- **锚 实体/表**：核心 2–3 个，带 [来源] 一句
- **依赖摘要**：1 行业务侧重（全集见第六节）
- ⚠️ **要注意**（若有）：具体陷阱一行一条
```

**伞形包子模块标题必须带 `(<伞形包名> 子模块)` 归属标识**；独立单一模块（如 `tms` / `bfe`）标题不带括号归属。

**省略**：不要塞内聚度量、被注入次数、完整 scope 实体/PO 列表、grep 命令、判型依据展开——这些是**分析者的工具**，不是**给读者的内容**。

**第六节「模块依赖矩阵」schema**：

```
| 源模块 | 目标 | 强度 | 锚点 |
|---|---|---|---|
| mdm-customer-enterprise | mdm-app-cfg | Import-only | AppCenterService |
| mdm-customer-enterprise | mdm-dictionary | Import-only | DictionaryService |
| mdm-employee | ccm-budget | Import-only (反向异味) | <锚点类> |
| ccm-paybill | mdm-employee | Injected | BaseEmployeeService |
| mdm-customer-enterprise | api/edp/channel | Adapter | EdpCallService |
```

**完整度契约**：跨子模块依赖**必须全列**（含 Import-only 弱依赖）；adapter 列入并标 `Adapter`；本子模块内 import 不列；反向依赖标 `(反向异味)`，方向以"基础设施 < 主数据底层 < 业务编排 < 顶层"层级直觉判，可疑标低置信；**锚点取代表、非穷举**。

**「改动前必读」节处理**：陷阱一句一条（同表多映射命名误导 / 触发器隐式耦合 / 垃圾表（如 `x_*` 与 `_bak` 前后缀表） / 编排在 Service 还是 Controller 散落）。高危耦合 hub 一句话点名（如「`XxxAppCfgService` 是全项目配置中心，改它牵连面广」），不堆数字。

**「上手顺序」节处理**：拓扑序从第六节依赖矩阵机械派生，**按层分组（基础设施 → 主数据底层 → 主数据外围/编排 → 业务上层 → 顶层），每层一张表**——表格让"要不要跑 `/module-brief`"一列对齐、读者一眼扫到。分组标题保留层名（"上手顺序"的顺序感在分层，不在表内）。

每层表**三列**：`模块 | 干什么 | 要不要跑 /module-brief`。**末列把「判定 + 理由 + 命令」并在一格**：以 ✅/⚖️/⬜ 起头（emoji 在最左、竖向自然成一条可扫的纹），后接理由/读法，**可跑的命令用 ➡️ 标出**（⬜ 不跑无命令）。**§5 开头给一行三态图例**（✅ 跑 / ⚖️ 你定 / ⬜ 不跑 各一句）。

末列判定拿地图够得着的两个信号——**扇出**（第六节，机械可靠）+ **逻辑看着厚不厚**（卡片侧弱启发，满屏泛型 CRUD 就当薄）——取「或」判；"内部藏没藏雷"要深读代码甚至连库 DESC、**超出地图能力，不作判据**。**拿得准才下结论、拿不准就交读者**，三态：

- **✅ 跑 — 〈为什么值得〉 ➡️ `/module-brief <kebab>`** —— 逻辑域 / 高扇出 hub。理由点出值得（状态机 / 被 N 家依赖）。**反直觉护栏：hub 逻辑可能最薄、却最该析——依赖它的人最多、地图又看不进它内部有没有雷；别因"看着薄"判它不跑。**
- **⚖️ 你定 — 〈说明情况〉；省事跳过让消费方就地读 / 保险 ➡️ `/module-brief <kebab>`** —— 公共码表 / 薄底座这类**薄逻辑 + 扇出不低 / 拿不准**的。说明情况（薄、但被 N 家依赖、地图看不透内部）+ 摆两条路。**地图不替读者预先指定它该折进谁**（替消费方判折向超出地图能力）。低扇出又明显是纯码表的，直接写"疑不用跑、消费方就地读即可"。
- **⬜ 不跑 — 〈读法〉** —— 全局基础设施 / 横切件 / 适配器集合 / 框架配置。给"读一个代表"的扫法：
  - 适配器集合 → "按需选 1-2 个代表读"（指出代表如 `api/oa/SyncApiService`）
  - AOP 切面 → "读 `aspect/<file>` 一个反例，其它同模式"
  - 触发器 → "看 DDL `TRI_<某表>_INSERT` 一个反例，其它同模式"
  - 配置中心 → "读 Service + 表注释一头一尾即可"
  - WSDL/IDL stub → "禁手改、不要读，仅升级时整体重生成"

**守住边界**：以上"⚖️ 交读者判"只用于"要不要花一次 `/module-brief`"这类地图看不透的取舍；**模块该合该分（边界）仍由地图用度量定、不甩给读者**（见「灵魂」第 2 条）。

反向异味边**表后单列「反向异味告警」一段**（不塞进表）：列出第六节标 `(反向异味)` 的边，提示跑上层 `/module-brief` 时一并核实方向。

### 度量数字的处理（关键）

度量数字（被注入 N、共享度 N）是**分析者下结论的工具**，**不是产物内容**。处理原则：

- **分析时**：大胆 grep、大胆数，用度量定边界、识别 hub、判跨切面。
- **写产物时**：把数字翻译成读者能用的结论。"被注入 31 次" → 「全项目配置中心，改它牵连面广」。
- **少量例外**：有些数字本身是"陷阱警示"，可保留（如「DDL 有 40 个触发器，改主数据表字段必须同步改触发器」）。
- **精确计数 = 0（硬闸）**：正文**禁出现「N 处 / N 次 / N 个文件」精确计数**（"被注入 31 次""AppContext 388 处""51 处文件"）——既违慢变（随提交而变）、又最易失准。一律翻成读者结论，或退标**范围/数量级**（"约几十处""上百文件"）。`~8 处`上限是给**数量级/陷阱数字**的预算，**不含**精确计数（不是"精确计数也可来 8 个"）。
- **必留数字写前 grep 核准**：极少数确需写出的数量级数字，**写入前 grep 实测一遍**再标、宁标范围不随手估——大仓尤其禁"凭印象给个数"（实跑见 388 对源码失准、"约 N"系统性低估近 2×）。
- **数量上限**：全文度量数字出现不超 ~8 处。超了说明在写报告，不在写地图。
