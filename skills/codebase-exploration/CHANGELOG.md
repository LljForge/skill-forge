## v0.11.0 — 2026-06-24（D6 路径接地闸 + D7 逻辑↔物理映射输出：补 reflexion map-resolution 缺位）

### 背景

理论底座（[theory-foundation.md](theory-foundation.md)）的核心发现：**D6/D7 是同一理论缺位的两面**——本 skill 缺了 Reflexion Model 的「map 条目必须 resolve 到真实 source 实体」约束。D6 写锚点路径不回查 = 写了一条无法 resolve 的 map 条目；D7 造 kebab 逻辑名却丢掉它到源码的映射 = 给了一个无法 resolve 的高层名。六个研究维度独立收敛到同一修法。本版按理论范式落地。

### 实际改动

| 缺口 | 文件·节 | 改动 |
|---|---|---|
| **D6** | SKILL.md ⑤校准 | 加**路径接地闸**：⑤ 锚点路径落笔前对**整串路径** grep/find 反查、**类名∧路径双命中**才写实（grep 到类名只证机制在、不证路径对——同名类可能在别的包/重构后移包）；只命中类名而路径核不准 → 降级只写类名或标「定位存疑」。**「机制存在」与「路径属实」是两道独立闸**。措辞用「锚点定位串」避开 line 247「写入路径」（审计写入链路）撞名 |
| **D7** | SKILL.md 产物 schema（卡片模板 + 省略条 + 上手顺序） | 卡片加**条件 `scope` 字段**：扁平分层/造名 kebab（包路径里不字面存在）输出 1.B **已算出**的「kebab → 类名前缀/包 glob」（reflexion-style map、代表非穷举、前缀 `^` 锚定防过匹配）；下游据 scope grep 而非 grep 不到的 kebab。**包即模块（kebab≈目录）免此行——不画蛇添足**。省略条澄清（省穷举清单、代表 scope 选择器要给）；上手顺序加命令定位说明 |
| — | SKILL.md 自检 ×2 | 卡片条加 D7（扁平分层有 scope、包即模块免）；⑤条加 D6（路径接地闸已过） |

### 理论支撑

theory-foundation.md §2（D6/D7 同源）+ §3：D6 借 reflexion map-resolution + code-intelligence（Kythe/SCIP 符号必派生可核、grep 类名 ≠ 路径对）；D7 借 reflexion map 文件（逻辑名 + 源码选择器）、4+1 logical vs development view、RMTool 前缀选择器 `^` 锚定防过匹配。

### 取证

双目标 dogfood（executor×2 → 对抗式 assessor）：
- **master-data**（Java 扁平分层伞形包）：压 D6/D7 **真实失败面 + 修复生效**——锚点路径是否整串反查、`mdm-*` kebab 是否带 scope 输出。
- **edoc**（原 GMZB-NJZL，Java+TS）：补 **TS 侧路径接地表达** + 验改动**不回归**；首跑当变异度基线。
- 对照组 GitNexus 已按用户要求弃用,准确度校准改源码抽样核 + 首跑基线。

**判决:ship-with-followup,D6/D7 两个 PASS、零自身回归。** 产物 `runs/{masterdata,edoc}-2026-06-24-v0.11-*`。

- **D6 路径接地真落地(非装饰)**：回源 grep 两目标全部锚点、整串 class∧path 双命中无漏网,**实战各救一次错判**——master-data `JWTRealm` 在 `com/itgfin/shiro/` 而非 config(只 grep 类名会误挂)、edoc `OpenApiInterceptor` 虚构名 → 降级写真实 `WebLogInterceptor`、`EdocExceptionHandler` 物理落反直觉的 `edoc-common`。两道独立闸生效(`@EnableCaching` 在但 `@Cacheable` 不在 → 写「实走手工 RedisTemplate」)。TS 侧相对路径接地成立、无 Java 残留。
- **D7 scope 两形态都对**：master-data 扁平分层 7 个 `mdm-*` scope 回源全 grep 命中、`^Base*` 家族 `^`锚定+业务词互斥**无过匹配跨吞**；edoc 13 域 + master-data 的 ccm/tms/bfe/fin **正确免 scope(独立子包/`@MapperScan` 佐证、非漏写)**——验证「包即模块不画蛇添足」。
- **不回归 PARTIAL(唯一扣分、非 D6/D7 引入)**：卡片 4–7 行未超 8、scope 增信息非增噪、1.B-D 全心算无脚本、慢变守住；**唯精确计数硬闸(v0.7.0 D1)有违例**——`130 个 Java 文件`(实测 148 失准)/`168 @Autowired`(实测 169)/`48 个并列类`。assessor 定性「既有纪律执行回归、非 D6/D7 新机制问题、不阻断发布」。

**本版折入的 dogfood 措辞增强(三条、纯增量,同 v0.9 先例)**：D6 加「反查用绝对路径防 cwd 漂移」「非 Java 栈路径=文件/import 路径」(edoc executor 实测踩 cwd 漂移 + TS 侧 import 别名);D7 加「同词根多子模块需验前缀互斥(`^Base*` 家族)」(master-data 实测靠业务词互斥、补显式提醒)。

**后续候选(v0.12)**：精确计数硬闸执行回归——dogfood 在本版产物上抓到 3 处违例(130 已失准、正中硬闸预言),说明 v0.7.0 硬闸措辞强度不足以拦「出完整图时散文/陷阱注里随手写精确数」。属既有纪律强化、与 D6/D7 正交,单独立项(自检「慢变闸生效」条强调收敛后逐处复核正文精确计数、翻数量级)。

## v0.10.0 — 2026-06-24（§5.5 点状白名单澄清：红线划在「机械 vs 智能」、为 1.A 机械层开有围栏的小口）

### 背景

v0.9.0 dogfood（GitNexus/TS）浮出一处 §5.5 措辞软肋：「1.A 用 Bash grep 已足够 / 不要做成脚本」的字面一刀切，与 TS/Python **相对 import 路径 resolve** 的实操有张力——纯 Bash 不顺手、易错，executor 用 ~20 行机械 python 做 resolve+计数却得自标「措辞 vs 实操张力点」。

executor 自评开的药方是「把红线整体放松到『脚本不得承载智能判断』」——v0.9.0 对抗式契约 lens 用 CHANGELOG v0.3.3 实证**否决**了它：该措辞与 v0.3.3 已退化的「临时脚本可以」同构（加 1 行宽松契约 → subagent 把 1.C 写单段 Python、8min→17.5min）。正确收口是**点状白名单澄清而非放松红线**：真问题（TS import-resolve 摩擦）用有围栏的小口解决，不给「智能任务脚本化」任何语义滑坡。本版即落地该收口。

### 实际改动

**红线重新表述：划在「机械 vs 智能」、不划在「脚本 vs 非脚本」**——1.A 机械层（grep 采集 + 路径 resolve + 计数聚合）可用任意工具含短 inline 脚本；唯一红线是脚本止于 1.A、不得触及 1.B-D 的归类/强度/反向判断。

| 文件·节 | 改动 |
|---|---|
| SKILL.md §5.5 机械/智能铁律（line 115） | 「1.A 用 Bash grep 已足够」补「纯 Bash 不顺手时短 inline 脚本亦可，红线见 1.A 末段」；末句「建辅助脚本」→「**预置**辅助脚本（1.A 临时 inline 不在此列）」，区分「持久预置」与「临时 inline」。**「智能任务不要做成脚本」核心一字未动** |

### 设计决策记录

依据 [docs/skill-design-philosophy.md](../../../docs/skill-design-philosophy.md) 七条原则：

- **原则 1（不教已知）**：砍聚类算法/强度计算/归属判定——强模型本就会做。留契约（依赖矩阵 schema、完整度契约）+ 陷阱（伞形/扁平/Base*/审计伪装/依赖方向望名生义）。
- **原则 2（改动→消费者映射）**：核心消费者 = MDA v1.23 boundary `dep_doc` 列。codebase-map 第七节依赖矩阵直接喂 MDA boundary "本模块 → kebab" 判定的输入源。次级消费者 = 新人读地图理解模块边界。
- **原则 3（骨架 vs 内容）**：6 节产物 + 8 行卡片骨架不动，正交可扩；新加第七节是骨架横向加节（线性扩展，原则 3 边界范围内），非重写骨架。
- **原则 4/5（去重 + 单一权威源）**：拓扑序从依赖矩阵派生而非独立写——消除"卡片依赖行 vs 拓扑序"漂移源。卡片「依赖摘要」明确为业务侧重描述，完整度归第七节单一权威。
- **原则 6（证据先行 + 诚实）**：4 模块抽样实测（mdm-customer-enterprise/mdm-employee/ccm-paybill/mdm-app-cfg）证实漏列形态各异（弱依赖/反向异味/横向），非个例。诚实定性：本次是**契约补全 + 教学瘦身的工程收益**，不是能力跃升（强模型 grep import 一次也能出依赖图）；价值在统一形态、防"卡片依赖按业务感觉裁"的静默缺陷、承接 MDA v1.23 消费者通路。
- **原则 7（通用 vs 特定）**：所有改动用通用形态描述（"跨子模块 import"、"按业务内聚性判"），不绑 `com.itgfin` 等项目标识。硬阈值去掉，保持多项目复用。

**砍掉的备选**：
- **依赖矩阵带数值列（被注入次数 / import 次数）**：违哲学度量数字处理（"翻译成读者能用的结论、不入产物"），且会复活"分析过程出现在产物里"的反模式。砍。
- **强度档加更多分类（"事件订阅"、"消息总线"）**：扩展过早，3 档（Injected/Import-only/Adapter）已覆盖本仓库观察到的所有形态；后续项目遇到新形态再补。砍。

### 已知妥协

- **扁平分层下子模块归属判定靠"前缀 + 锚 PO 交叉"启发式**：强模型够（看 `import com.x.masterdata.po.AppCenter` 能判到 `mdm-app-cfg`），弱模型可能漂移。本期识别但不强行机械化——硬规则要求"全锚 PO 反查表"会过度膨胀，且本仓库仅一处扁平分层（`com.itgfin.masterdata`），多项目复用价值未必抵成本。
- **反向异味标注先有鸡先有蛋**：判反向需要先有拓扑序，但拓扑序又从依赖矩阵派生。当前解：先用"基础设施 < 主数据底层 < 业务编排 < 顶层"项目层级直觉粗排，再回头标。可能漏判（如同层模块互调难判方向），属启发式可接受。
- **依赖矩阵全集会变长**：本仓库 16 个业务模块预计依赖矩阵 60-100 行（卡片本身 ~16×8=128 行），整体产物会比 v0.2.4 长 ~30%。换来的是依赖图完整度从 60% → 95%+ 与拓扑序自动准确，权衡合理。

## v0.2.0 — 2026-05-26(读者优先排版 + 伞形包粒度收敛硬约束)

### 背景

iter-3 落地试跑(在本项目跑全项目梳理)暴露两个用户反馈的问题:

1. **可读性差**:产物文档把分析者的工作笔记(度量数字 `被注入 31`、`@Autowired 计 152 次`、`grep -c=40`、自检对照表 等)塞给读者新人——读者要的是"模块叫什么/干什么/入口在哪/改有啥坑",不是"分析者怎么下结论的"。
2. **粒度爆炸预防**:masterdata(607 java)在 iter-3 落地只拆 4 簇(其中"基础主数据"一簇含 300+ java/50+ Service),用户跑 `/module-depth-analysis 基础主数据` 必爆 MDA 上下文——**地图模块粒度是受下游消费(MDA 单次可吞下)约束的硬指标,不是粒度偏好**。

### 实际改动

| 项 | 文件 | 改动 |
|----|------|------|
| 新增「关键架构约束」节 | SKILL.md | 明确"模块粒度 = /module-depth-analysis 单次可消化",硬上限 ≤ ~100 java / ≤ ~20 顶层 Service;伞形包必须拆够 |
| 测绘四步增"粒度收敛" | SKILL.md | 第 2 步专门测每个候选团 java/Service 数,超就继续拆 |
| 产物 schema 从七节改六节 | SKILL.md | 删除「边界决策记录」「自检对照」入产物;新增「五 改动前必读」专列陷阱;模块清单改为 8 行卡片模板(干什么/入口/锚 PO/依赖/⚠️) |
| 「度量数字的处理」专节 | SKILL.md | 度量是分析工具不是产物内容,翻译成结论;全文上限 ≤ ~8 处 |

| references/example.md 重写 | example.md | 改虚构电商示例为读者卡片风格,展示伞形包拆够粒度(payment 拆 3 子、masterdata 拆 5 子,各 ≤ ~50 java) |

### 取证(iter-3 with_skill 全项目梳理 vs iter-2 SKILL 真实落地的 `docs/codebase-map.md` 作 old_skill baseline)

| 指标 | old_skill | with_skill (v0.2.0) | Δ |
|---|---|---|---|
| Pass Rate(7 条新 assertion) | 29%(2/7) | **100%(7/7)** | **+71%** |
| Time | 666s | **479s** | **-28%**(更快) |
| Tokens | 111k | **88.8k** | **-20%**(更省) |

assertion 设计直接对应两个反馈:masterdata 拆 ≥ 6 子 / 子模块 ≤ ~100 java / 度量 ≤ 8 处 / 不含自检表 / 卡片读起来流畅 / 探索声明 + MDA / 上手顺序。

**意外收获**:更省更快不是 trade-off,是**双赢**——精简产物 ≠ 工作量减少,而是 agent 不再耗 token 堆度量过程,产出直接到结论,效率反而上升。

**关键质性差异**:
- **masterdata 拆分**:old_skill 4 簇(基础主数据/应用与权限/审计/金蝶 EAS) → with_skill 11 子(employee/company-org/bank-finins/finance-dict/app-config/echat/sync-orchestration/action-log/quota/mdm-log/kingdee-k3),每个 ≤ 13 顶层 Service,**符合 MDA 单次可消化**。
- **可读性**:old_skill 模块条目混杂 24 张 PO 列表/grep 命令/`find ... =0`/`@Autowired 计 152 次`,新人需要的"干什么/入口/陷阱"被淹没;with_skill 严格 8 行卡片,信息密度高、可读。
- **自检对照**:old_skill 在产物里留 8 条 `[x]` 自检 checklist,泄漏分析者内部活;with_skill 在内部完成,产物干净。

### 设计决策记录

- **粒度上限为何 ≤ ~100 java / ≤ ~20 Service**:经验阈值,对应 MDA 在 Sonnet/Opus 上单次能舒适消化的代码量;不是绝对线,但超了就要警觉。
- **为何六节而非七节**:原"边界决策记录"节是分析者下结论的中间产物(团内/团外强度数字表),读者用不上;归类成果直接体现在模块清单(卡片"依赖"行)+ 第五节陷阱中。
- **「度量数字 ≤ 8 处」上限**:全文是地图不是报告——超 8 处的度量数字必然遮蔽业务信息;少量保留是给陷阱警示用(如"DDL 40 个触发器,改字段必须同步改触发器")。
- **未做**:不强制规定每个伞形包"必须拆几个"——拆够粒度即可(用 java/Service 数硬指标判);拆几个由业务簇自然决定。

## v0.1.1 — 2026-05-26(影子表写入路径校准强化)

### 背景

iter-1 实测两个 with-skill agent 对"影子表写入路径"给了相反结论——masterdata 用例标"DDL 无 `CREATE TRIGGER` → 写入路径存疑"(校准对,但事实判错),full-project 用例武断"由 `TRI_BASE_*` 触发器写"(没核实)。问题:同一 SKILL 在两个 run 给不一致结论,且都没真正核实。SKILL 原文只描述"多由触发器/AOP/拦截器旁路写",未指示核实 → agent 行为不一致且都可能错。

### 实际改动

| 项 | 文件 | 改动 |
|----|------|------|
| 「审计/日志伪装成模块」陷阱条目 | SKILL.md | 加"写入路径必须核实、别脑补"指令:grep DDL 的 `CREATE TRIGGER`、AOP 切面、拦截器各一遍;只看到 `TRI_*` 等名字痕迹却无 `CREATE TRIGGER` 定义 → 标"写入路径存疑、需深入核实",**绝不武断断定** |

### 取证(iter-2 重跑 with-skill × 2,baseline 沿用 iter-1)

- 两个 agent **都 grep 实证 → 都找到 40 个 trigger 定义**,且自发用更宽 grep 模式发现:**DDL 用 MySQL `create definer = mdm@... trigger ...` 语法,不是关键字 `CREATE TRIGGER` 连写**——**纠正了 iter-1(包括我们手工 grep 也漏掉)的事实错误**:9 张 `mdm_log_base_*` 中 8 张由 trigger 写,1 张(`employee_add`)由 Java save。
- pass rate 100% 不变(已饱和);time 256s → 601s(**+2.4x**);tokens 77k → 101k(+32%);tool calls 26/29 → 46/69(翻倍)——agent 多做了核实工作。
- 跨 run 方差消除:两个 with 现都准确归因,不再"一个校准存疑、一个武断断定"。

### 设计决策记录

- **诚实定性:工程/鲁棒收益,非能力跃升**。pass rate 已饱和 100%,iter-2 测不出 pass rate 提升;真收获是消除跨 run 方差 + 纠正事实错误(40 trigger 真相)。代价 time 2.4x。对"一次性探索"场景可接受。
- **保留的理由**:校准是本 Skill 灵魂(产物给新人看,武断结论会误导);grep 核实是栈陷阱(模型常省事不核实),值得显式入 spec(对齐设计哲学原则 1+6)。
- **未做**:补 `grep CREATE TRIGGER` 的具体正则(MySQL `definer` 语法变体)——指令上写"grep + 名字痕迹核实"已让 agent 自发探索更宽匹配,模型导得出的不教(原则 1)。

## v0.1.0 — 2026-05-26(首版)

### 背景

为"第一次接手陌生项目、既不懂业务也没有可信标准答案"的新人设计的代码探索工具。区别于 `module-depth-analysis`(权威文档生成):**本 Skill 产出是启发式探索草稿、非权威、可能错;校准优先于整齐**。

设计思路在 masterdata 伞形包(本仓库最难拆的 607 java/扁平分层包)上做了三轮方法对照——命名前缀法 → 关系聚类法 → 关系+度量+校准——坐实"关系聚类 + 共享度抓跨切面 + 三视角交叉 + 客观度量定边界 + 代码中文提语义 + 混乱预警"是穿透"包 ≠ 模块"陷阱的方法。

### 核心设计(灵魂)

- **非权威/校准优先**:产物头部声明"探索草稿·非权威";置信标记 + 校准存疑明示;可疑处导向 `/module-depth-analysis <模块>` 核实。
- **边界用度量定,绝不甩给用户**:新人不懂业务、不能当裁判;"该合该分"用团内/团外连接强度数字给默认+置信分,禁止"需人裁决"。
- **业务语义从代码中文提取**:DDL 表注释最强 > `@Api(tags)` > 路径,每条标来源,禁臆测。
- **栈特有陷阱固化**(7 条):包≠模块、伞形包、扁平分层、适配器集合、`Base` 假前缀、审计层伪装成模块、扁平 CRUD 编排外移 Controller、同表多 PO 命名误导。
- **可扩展能力清单骨架**:当前 1 能力(模块测绘),将来加新探索能力是加节、不动骨架(原则 3 加法非乘法)。
- **零耦合**:不依赖也不写 `module-depth-analysis` 的产物;Skill 互不感知。

### 实测取证(在本仓库 master-data-be)

| 测试用例 | with-skill | baseline(无 Skill) | Δ |
|---|---|---|---|
| **masterdata 伞形包**(最难) | **100%**(7/7) | 43%(3/7) | **+57%** — baseline 命名法把 MdmLog 当模块、脑补"触发器写"、无置信/无度量 |
| **全项目顶层结构** | **100%**(6/6) | 67%(4/6) | +33% — baseline 顶层结构靠强先验做对了不少,但 masterdata 未拆/无性质判型 |
| **综合** | **100% ± 0%** | **55% ± 17%** | **+45%** |

代价:with 比 baseline 慢约 2x、tokens +50%——对"一次性探索"可接受。

### 设计决策记录

- **价值在"难处"最大,不假装全面碾压**:伞形包内部拆分 / 陷阱穿透 / 校准——baseline 系统性失效;顶层结构(模型有强先验)差距小。诚实定性。
- **范例 `references/example.md` 用虚构示例**(电商订单服务,Spring Boot + JPA):格式锚点,内容脱敏不绑任何真实项目——避免在该项目跑测时污染对照。
- **与 `module-depth-analysis` 完全独立**:不当其门禁、不被其依赖;模块测绘可能错,绝不作为权威分析的输入。
- **non-discriminating assertion 暴露**:"无甩锅词"(baseline 也 0)、"非权威声明"(baseline 也自发加)——下轮若加 assertion 应换更尖锐的(如"边界决策附量化数字")。
