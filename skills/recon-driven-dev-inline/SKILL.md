# recon-driven-dev-inline（侦察驱动开发 · 自包含五阶段流水线工作流）

一条**即时开发轨**:开头做一份只针对本次改动的"摸底",喂给后面的需求、设计、(评审)、写计划、实施。**自包含、零外部 Skill 依赖**——各阶段能力都内联在本 Skill 内,丢到任何环境都能独立跑。

各阶段细则下沉 `references/`,本文件只持**跨阶段的脊柱**:流程图、阶段衔接与所有 `⏸` 暂停、起步约定、全局护栏,以及每阶段「一句话职责 + 指向 reference 的指针」。**指针只点名该 reference 里非默认的具体动作,细则不在脊柱复述。**

## 核心流程

五阶段,每阶段结束**暂停**等用户确认。产物都落本次改动目录 `docs/recon-driven-dev-inline/<YYYY-MM-DD>-<change-name>/`,收尾时连目录一起归档。③ 评审**可选**,由 ② 出口的决策点拍板。

```
① 定向分析   一句话粗需求 → 钻落点 + 消费面(全集) → directed-report.md
             ⏸ 主会话实测门(① 自守、防空转的真牙)
② 需求+设计  据①(参考非上限,可自行补探),内联对话澄清需求与设计 → requirements.md + design.md
             ⏸
       └─ ⏸ 评审决策点:据 ①② 信号给「建议评审 / 可跳过」+ 理由,用户拍板 ─┐
③ 评审(可选) 派评审 sub-agent → 三维度 + 定向报告验收 → review.md
             ⏸
④ 写计划     拆 tasks.md(曳光弹切片 + 右尺寸 + 验收/依赖序)
             ⏸ 审计划   →   ⏸ 执行方式(内联 / 子 agent)
⑤ 实施       两分支 gates(clean-baseline / ISOLATE / TDD / per-task 评审)+ 收尾前整支评审
             ⏸ FINISH 收口菜单   →   ⏸ 归档
```

流向:`① → ② → (③?) → ④ → ⑤`,**线性不回流**(唯一例外:事实订正,见末尾全局护栏)。

**为什么先摸代码再谈需求**:不先摸清楚就拍方案,十有八九漏掉"原来别处也在用这张表"这类坑,等写代码才发现要返工。

## 派发与降级(① / ③ / ⑤ 整支评审 共用)

凡需上下文隔离的 sub-agent 能力(① 侦查、③ 评审、⑤ 收尾前整支评审、⑤ 子 agent 实现),走同一范式:

- **有隔离能力**(本会话有 `Task` / 通用 agent):**派**——`Read` 对应的自带 prompt 体(`recon-agent.md` / `review-agent.md` / `code-reviewer.md`),以其正文作 `Task` 的 prompt、把本次的路径与上下文追加在末尾,用通用 `Task` 当搬运工。**不赖任何特定内置 agent**(如 `Explore`)——它们的 prompt/模型不归本 Skill 控、质量会飘。
- **无隔离能力**:**降级**主会话自跑同一套契约/判据——没有新视角与隔离收益,须把契约当外部尺逐条核、把检索噪声尽量挡在叙事外。

契约/判据**与跑在哪一级无关**,派发只是执行方式。各 prompt 体是本 Skill 自带的、不调任何外部 skill。

## 起步:建本次产物目录

AI 直接为本次改动定个 kebab-case 短名(无需问用户),冠以当天日期,创建 `docs/recon-driven-dev-inline/<YYYY-MM-DD>-<change-name>/`;**该目录已存在(活跃或归档)就换名 / 加后缀,别覆盖前一次的产物**。

## 每阶段暂停模板

```
阶段 X 完成,产出 <file>。请审阅。
✅ 满意 → 进入下一阶段   ✏️ 需调整 → 告诉我哪里改
```

## ① 定向分析 → `directed-report.md`

**职责**:动需求/方案前,把这次需求主题落进的那片现有代码摸清楚——有啥、牵动谁、埋了什么雷——按主题摸**全集**、全带锚点,喂给 ②。只摆现状事实、不出方案。它是给 ② 的**现状参考、不是权威上限**。

细则(派发路由 / agent 降级 / 主会话实测门 / ① 阶段封存边界)见 [`references/directed-analysis.md`](references/directed-analysis.md)。

**⏸ 这道质量门是「主会话实测门」、不是泛质量门**——收到报告后,主会话亲自**扫本会话可用的只读手段(含 MCP),对支撑坑的 `[未核验]` 软推断当场取真实值比对**,而非只复核"四样齐全"就放行。具体动作住 reference;**别让半成品报告进 ②**。

## ② 需求 + 设计 → `requirements.md` + `design.md`

**职责**:据 ① 的现状定"要什么"和"改哪些 / 怎么改",内联对话谈完一次产出两份。① 是参考起点、不是上限,需求够到 ① 没覆盖的代码就**自行补探**,新发现记进 `design.md`(前向),不回写已封存的 directed-report。

细则(对话契约 / 三硬规则 / 两份产物 / **`design.md` 必覆盖清单** / **评审决策点判据** / ② 自评)见 [`references/requirements-design.md`](references/requirements-design.md)。

**HARD GATE**:呈现需求 + 设计并经**用户**(非模型自判)批准前,**不写任何实现 / 脚手架 / 代码**——完整规则住 reference,脊柱只亮牌。

**② 端信号产出义务**:② 的产物/自评须让下面这些信号**对评审决策点可读取**——① 暴露的协同护栏 / 反直觉陷阱 / `[未核验]`,与 ② 的**测试缝隙数 / ADR 决策数 / 爆炸半径**。否则决策点"据 ①② 信号给建议"无据可依。

**⏸** 两份产出后暂停让用户过目;别在这道停点之外再叠一道。

## ⏸ 评审决策点(③ 是否进行)

② 出口后,skill 据上面「② 端信号产出义务」列的 ①② 信号,给「**建议评审** / **可安全跳过**」+ 理由,**用户拍板**。**判据(完整信号清单 + 代价提示)单一权威在 [`references/requirements-design.md`](references/requirements-design.md)**;脊柱只点一个钩子——**跳过 ③ 的代价 = 丢掉「定向报告验收」**(查 design 有没有静默违反 ① 的坑/护栏),而 ② 自评**不覆盖**这项,故"① 坑多"是强烈建议评审的信号。

## ③ 评审(可选) → `review.md`

**职责**:判方案好坏。③ 不在主上下文内联自评——**派一个上下文隔离的评审 sub-agent**,新视角去自评偏见,评审的中间推理不回灌主对话。

**派发**:把 ①② 的三份产物(`directed-report.md` + `requirements.md` + `design.md`)+ 一句改动概述交给评审 sub-agent,按上文「派发与降级」走(`Read` `review-agent.md` 作 prompt)。它产 `review.md` + 三档结论(通过 / 有条件通过 / 不通过)+ 修订清单,回主 agent 一段简短结论。判据全住 [`references/review-agent.md`](references/review-agent.md),脊柱不复述;后两档不自动分流,在下面这道 `⏸` 由用户据修订清单定是否回写 ② 改设计。

**⏸**。

## ④ 写计划 → `tasks.md`

**职责**:把 design 拆成小而独立、各自可演示/可验的任务,排好依赖序。

细则(基线计划六条 / 分层计划 / 执行方式决策)见 [`references/planning.md`](references/planning.md)。

- **⏸ 审计划**——写码前最便宜的拦截点,审过再往下。
- **⏸ 执行方式**:计划末尾选**内联** vs **每任务一个新子 agent**。这个选择决定计划是否升级到自包含形态(精确 `Interfaces` + 任务级完整路径),也决定 ⑤ 走哪条执行分支。

## ⑤ 实施 → 代码 + 归档

**职责**:按 ④ 的计划在隔离工作区内落地、收口、归档。

细则(两条执行分支 gates / 非显性陷阱块 / 实施前冲突预检 / 收尾前整支评审 / 文件交接 / 进度账本)见 [`references/implementation.md`](references/implementation.md)。脊柱点名几个非默认硬闸:

- **两分支共有硬闸**:clean-baseline、ISOLATE(优先原生 worktree 工具)、**在 ② 预先约定的测试缝隙上写失败测试**(TDD)、per-task 两阶段评审(spec 符合 + 代码质量)。
- **实施前**:先做「计划 ↔ 全局约束 ↔ 评审 rubric」的冲突预检——**一次批量问**用户、**扫净则静默继续**。
- **收尾前**:派隔离子 agent 跑 [`code-reviewer.md`](references/code-reviewer.md) 做整支代码评审(Standards + Spec 两轴),无隔离能力降级主会话自跑——**派/降级只看隔离能力,与 ④ 选的执行分支正交**。
- **⏸ FINISH** 收口菜单(本地合并 / push+PR / 保留 / 丢弃) → **⏸ 归档**(本次目录移进 `_archived/`)。

## 全局护栏

- ✅ **报告必须落「发现」(落点现状 / 实答 / 锚点)——这是料、不是套路**。薄的是流程(别教怎么 grep / 追链),不是结论;空有问句没答案才是真薄。
- ✅ **`① → ② → (③?) → ④ → ⑤` 线性、不回流**——各阶段在出口守住自己产物的质量,不把把关堆到下游;③ 不打回 ① 补摸底,② 的盲区 ② 自己探。
- 🔁 **「薄」是审美、不是否决权**——吸收按"对新 Skill 好不好"逐条判,经评估保留的才留。防膨胀不靠口号:落地后**实测每条执行路径(内联 / 子 agent)单次 run 实际载入的 reference 量,设一个"单路径承载"软上限**;超了就回头评增量价值偏低的吸收项能否再降级。
- ❌ **别给定向分析加判断规则表 / 检索套路**("碰到契约就 grep 调用方"这类识别-动作对照表)——识别 AI 会,动作溶进验收单 + 坑的 why 即可,别把"薄摸底"胀成重型分析机器。
- ❌ **别让半成品报告进 ②**——① 的主会话实测门必须真卡,不是拖到 ③ 才发现。
- 🔁 **`design.md` 不套章节骨架、也不"完全无骨架"**——改用**必覆盖清单**(四要素检查锚点,形态按复杂度伸缩)。清单的要素与判据单一权威在 `requirements-design.md`,脊柱与其余只指针、**不在此列举要素**。
- ❌ **别把判据复述进 SKILL.md / 模板**(单一权威源)——`review-agent.md`(设计三维度)、`implementation.md`(per-task 两阶段)、`code-reviewer.md`(整支两轴)、`requirements-design.md`(评审决策点判据 + `design.md` 必覆盖清单)各有唯一家,脊柱 / 模板**只路由不复述**(复述 = 第二权威源、会漂移)。
- ❌ **别把 provenance / 借鉴出处写进 agent 本体**(`recon-agent.md` / `review-agent.md` / `code-reviewer.md`)——可执行 prompt 只放角色 / 契约 / 判据,出处归 CHANGELOG。
- ❌ **别让评审 sub-agent 替 ① 重做系统钻取**——它只对正在判的主张做定点源码核验;全量摸底是 ① 的活,回流是禁的。
- 🔁 **④⑤ 的实现步骤本 Skill 内联拥有**——别委托外部技能、别引脚本,也别在既定 gates / 停点之外加冗余评审;本次吸收全部是内联重写,未引入任何外部 skill 调用。
- 🔁 **不引入跨会话持久件、不发持久编号**——读了长期文档反而引入漂移与冲突,故否决项目级持久 `CONTEXT.md` / ADR 库。**进度账本是本次产物目录内的 per-change scratch、随归档清,不属此列。**

### 事实订正(线性不回流的唯一例外 · 跨全程总账)

directed-report 一过 ① 质量门进入 ②,即视为**封存**。此后任何环节用证据**推翻它写下的一条事实主张**(区别于「① 没写的新面」——那走 `design.md` 前向、不算订正),就在**原文条目正下方追加订正块、原文标 `[已订正]`**:

```
⚠️订正:原文已被推翻 → 真相:… | 证据:file:line 或查询结果
```

**落点(重映射到五阶段)**:②③ 阶段证伪在对应 `⏸` 落、动摇设计走 ③ 三档;**④ 写计划 / ⑤ 实施途中**推翻——主会话落订正块(诚实留痕、不算重排链),**动摇设计就就地暂停、surface 给用户定方向**(续做 / 回 ②③ 改),本 Skill 不自动重判。

**① 封存前** directed-report 还是草稿、**直接改不计订正**(此即 ① 阶段的封存边界,`directed-analysis.md` 用指针引、不另持一份)。核验与落订正**由主会话(主 agent)做**;评审 sub-agent 只把推翻真实记进 `review.md`、**绝不碰 directed-report**。
