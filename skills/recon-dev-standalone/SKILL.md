# recon-dev-standalone（侦察驱动开发 · 轻量自包含四阶段工作流）

一条**即时开发轨**:开头做一份只针对本次改动的"摸底",喂给后面的需求、设计、评审、实现。**自包含、零外部 Skill 依赖**——②④ 的能力内联在本 Skill 内,丢到任何环境都能独立跑。

## 核心流程

四阶段,每阶段结束**暂停**等用户确认。产物都落本次改动目录 `docs/recon-dev-standalone/<YYYY-MM-DD>-<change-name>/`,收尾时连目录一起归档。

```
① 定向分析   一句话粗需求 → 钻落点 + 消费面(全集) → directed-report.md
             ⏸ 过报告质量门(① 自守,防空转的真牙)
② 需求+设计  据①(参考非上限,可自行补探),内联对话澄清需求与设计 → requirements.md + design.md
             ⏸
③ 评审       派发评审 sub-agent → 综合质量门(三维度 + 定向报告验收)→ review.md
             ⏸
④ 落地       拆 tasks.md ⏸ → 内置实现链(隔离工作区 + TDD + 实现期评审 + 收口)→ 文档归档 ⏸
```

**为什么先摸代码再谈需求**:不先摸清楚就拍方案,十有八九漏掉"原来别处也在用这张表"这类坑,等写代码才发现要返工。

**自包含**:本 Skill **不依赖任何外部插件、无硬前置**。①③④ 用到的 sub-agent 派发都是 harness 内置能力(`Task`/`Explore`/原生 worktree 工具),且一律配主上下文降级——环境没有就降级、不阻断开轨。

**起步**:AI 直接为本次改动定个 kebab-case 短名(无需问用户),冠以当天日期,创建 `docs/recon-dev-standalone/<YYYY-MM-DD>-<change-name>/`;**该目录已存在(活跃或归档)就换名 / 加后缀,别覆盖前一次的产物**。

**每阶段暂停模板**:
```
阶段 X 完成,产出 <file>。请审阅。
✅ 满意 → 进入下一阶段   ✏️ 需调整 → 告诉我哪里改
```

## ① 定向分析 → `directed-report.md`（灵魂 · 现状侦察）

**职责**:动需求/方案前,把这次需求主题落进的那片现有代码摸清楚——有啥、牵动谁、埋了什么雷——按主题摸**全集**、全带锚点,喂给 ②。**边界**:只摆现状事实(不细化需求、不定改动范围、不出方案、不判好坏)——**含码上读不到的运行时/数据态**,按下文「软/硬分等」标记与核验。它是给 ② 的**现状参考、不是权威上限**。

输入是用户一句话粗需求,产出一份摸底报告（directed-report.md）——它是 ②③ 共用的上游参考。

### 钻取外包（内置派发,三级降级）

翻代码的体力活(实际去搜"谁调用了这个 Service"、追调用链)**优先派给钻取 sub-agent**,别在主上下文亲自 grep——避免主对话被一堆中间检索结果污染。降级阶梯:

- **有内置 `Explore` / 通用 sub-agent(`Task`)→ 派给它**(零依赖、最可移植;把检索噪声挡在主上下文外)。
- **都没有 → 主 agent 自己 grep / 追链**(零依赖地板;尽量别把中间检索灌进主叙事)。

报告契约与质量门**与跑在哪一级无关**——派发只是执行方式,输出形态(下文四样)不变。

### 怎么喂给钻取 agent

**不写检索步骤表**。把"粗需求 + 下面四样里的验收单问句 + 几条坑的方向"整包丢给钻取 agent——**验收单的问句本身就是检索意图**:答不全就继续翻,答全了才算摸够。具体怎么搜、搜哪些文件,由 agent 自己定。派发 prompt 里声明:**"四样"的形态要求(含覆盖地板、含软/硬事实分等标记)随包附上、按其形态返回;最终回复即报告正文、将直接落盘,不得含过程性叙述**;交回仍夹带的,落盘前剪掉再存——剪裁线即四样:之外剪、之内不动。

### 报告必须现写的四样

定向分析不是固定清单,是**每次改动现写**的四样:前两样是"摸到了什么"(发现,报告的肉),后两样是"按什么标准摸够"(章程)。四样全是关于**现状**的事实/推断(或地板),**不预判要改什么**。

**① 落点现状(这次需求落在哪些现有契约上)** —— 一句话说清需求大意,再列它**落在的现有契约载体**(Service 接口 / DTO 字段 / 枚举值 / 端点 / 表字段之类),每个带 file:line 锚点。只标"现有代码长这样、牵动谁",**不预判要改哪些**(改动范围是 ② 的事)。它是发现的源头,后面三样都挂在这片现状上。

**② 验收单 · 逐载体实答** —— 围绕 ① 这些现有契约提开放问句(检索意图、驱动摸得更广,不写"去 grep 谁调用它"、AI 自己会);但报告**按载体逐项落实答**——① 落点现状清单里的**每个契约载体**,都紧跟一行"谁在调用 / 消费它"的实答(调用方 / 消费方清单,带锚点);答不出标"未发现"、不许空着,确实不必问的显式标"不问 + 一句理由"、不许默掉。**这层"逐载体都有实答"即覆盖地板**——缺哪个载体一眼可见,① 质量门当场核;一问可打包多个同类载体、但**实答按载体分项**(防伞形问句混答漏单个载体)。**真正的肉是答案**,下游能依赖的就是这份实答。地板非天花板,清单外该问的照问;按需求主题摸**全集**(宁可多摸),②(设计)再从中收窄。

**③ 几条"容易栽的坑 + 为什么 + 锚点"** —— 每条是 AI 凭直觉扫不出、不递就容易踩的非显性点,说清"为什么会咬人",带 file:line 锚点(没锚点的坑是空中楼阁、没法验)。模型自己想得到的常识不算坑、别写。两味:
- **反直觉陷阱**:东西不在你以为的位置、或不叫你以为的名字——真起作用的逻辑藏在第一眼不会看的地方(如状态判断藏在 SQL/Wrapper 里、字段不叫 status 叫 dealState),光按直觉找就漏判、误判。
- **协同护栏**:改 A 必须连带改 B,否则 C 错算——改动型分析里这味最值钱。

**④ 一句召回地板** —— "至少要覆盖到调用方和消费方,遇到没列到的变体也照样追——这是地板、不是天花板。"

**【软/硬事实分等 · 横切四样】** 报告每条主张分两等——**硬事实**:码上读得到、带 file:line 锚点(质量门抽查锚点验它);**软推断**:结论依赖一个码上读不到的运行时取值/行为(纯静态推不准)。**凡软推断必须标 `[静态推断·未核验]`(下文简称 `[未核验]`)**——结论形如「X 字段(实际/运行时)取什么值」「这条分支实际走没走到」「数据实际状态是…」这类**靠运行时真实情况、非代码字面**的,一律标;**写下一条像事实的结论时反问一次:它成立是否依赖代码字面读不到的运行时取值/路径?是→标**(连看着像硬事实的隐式推断也兜住)。一条主张里只要有一环依赖读不到的取值/路径,整条按软推断标(宁可标)。**标记是「谁用谁验」的接力棒**:谁据某条 `[未核验]` 做决定,谁在那步据手段补验;没人据它决策的,留标记即可、不必验。给下游(②③)一把尺:哪些可依赖、哪些得当心。

### ⏸ 审报告(① 的质量门——防空转的真牙)

定向分析很薄,半成品报告喂进 ② 就让后面全建在沙上。**质量门长在这儿、当场守**(不是拖到 ③):出报告前自检 + 审报告时逐项核对——

- **四样齐全**?
- **锚点真实**?(抽查几个跳过去看、对得上当前代码;**且锚点支撑的结论别暗含码上读不到的运行时取值/路径假设——暗含却当硬事实写的,补标 `[未核验]` 按软推断处理**)
- **支撑坑的软推断核验了吗**?(挑**支撑样③坑、尤其反直觉陷阱 / 协同护栏两味**的 `[未核验]` 条——核验目标就是那条里那个读不到的取值/走向:**先扫本会话可用工具面(含 MCP)**,命中**现成只读、零副作用**的手段(不为此搭环境 / 写新测试)就把真实值取出来比对,一致→升 `[实测]`、不一致→主会话就地改正(报告未封存、直接修);够不着就坦诚留 `[未核验]` 放行。报告自标「可能 / 非目标 / 无法确认」的边缘条留标记即可。实测是**主会话(主 agent)这道门**的活、不指望只读码的钻取 agent。)
- **样②逐载体都有实答、踩住覆盖地板**?(对着 ① 落点清单**看有没有缺行**——答不出标"未发现"、确实不必问的标"不问 + 理由";缺行或理由站不住的,定点打回补摸即可、不必整报告重钻)
- **坑是真·非显性料**、不是放哪都对的通用废话?

不过门 → 打回钻取 agent 重摸,**绝不让半成品报告进 ②**。**过门即封存**:报告一过这道门进入 ②,directed-report 视为封存,此后任何推翻走护栏「事实订正」(过门前是草稿、就地直接改)。

## ② 需求 + 设计 → `requirements.md` + `design.md`（做决定）

**职责**:据 ① 的现状定"要什么"和"改哪些 / 怎么改"。**边界**:① 是参考起点、**不是权威上限**——需求一细化就可能够到 ① 没覆盖的代码,**该自行补探就探**;自己探到的、① **没写的新面**记进 `design.md`(前向),**不回写已封存的 directed-report**(唯一例外:证伪 ① 的事实主张时就地「事实订正」,见护栏)。**接棒纪律**见下。代码既有的坑在 ① 里、这儿不重列。

带着定向报告这个上下文,**在主对话里内联谈细需求与设计**(不依赖任何外部技能),**谈完一次产出两份**。

**对话契约(怎么谈)**:
- **一次一个问题**澄清(优先多选),聚焦目的 / 约束 / 成功标准——这层喂 `requirements.md`。
- 定方案前**提 2-3 个不同方案**带取舍,领头给推荐 + 理由,别一头扎进第一个解法。
- **分节呈现设计**,每节确认再往下,别一次甩完只问一遍。

**三条硬规则(强模型为"帮上忙"最爱跳过,必须明写)**:
- **HARD GATE**:呈现需求 + 设计并经**用户**(非模型自判)批准前,**不写任何实现 / 脚手架 / 代码**。
- **"太简单不需要设计" override**:该门适用于**每个**任务——单函数工具、配置改动、todo 列表也一样,不许以"这个很简单"自行豁免。
- **范围分解触发器**:若请求横跨多个独立子系统,**先分解**(列独立块、关系、build 顺序),别花澄清问题去细化一个该拆的项目;分解后逐个子项目走本轨。

**两份产物**:
- `requirements.md` —— 业务视角:要做成什么、约束、验收标准。形态见 `~/.claude/skills/recon-dev-standalone/references/templates/requirements.md`。
- `design.md` —— **本 Skill 自己产出设计内容**(没有外部技能代劳):
  - **设计本体**:架构 / 数据流 / 关键决策 + 取舍 / 风险——在此对话谈出,按复杂度伸缩(简单的几句、有讲究的多写),扎根现有代码模式,只改服务本次目标的代码、不顺手大重构。
  - **① 锚定层**:圈定要改的契约子集 + 自行补探到 ① 没列的新发现(③ 的"定向报告验收"要拿这层跟 ① 的实答逐条对)。
  - **不设僵化分节骨架**——按上面两块的内容要求写,别套死"必须有哪几节"的模板。

**② 接棒纪律**:据 ① 的 `[未核验]` 条目做改动决策前,**先据手段补验**(升 `[实测]` 或推翻);够不着则在 `design.md` **显式记**「此决策建于未核验推断 X」——别让 ① 留标记放行的软推断在这儿被无声当真。

**② 自评(仅轻量卫生扫描)**:出两份前扫一遍占位符(TBD / 待定 / 无理由的"本次不涉及")、内部矛盾、真歧义(能两种读法的挑一种写死)、范围蔓延,就地修。**只扫这四类、不复跑 ③ 的判据**(深度评审是 ③ 的活,这儿重复 = 第二权威源)。

**⏸**:两份产出后暂停让用户过目;别在这道停点之外再叠一道。

## ③ 评审 → `review.md`（裁判 · 派发评审 sub-agent）

**职责**:判方案好坏。③ 不在主上下文内联自评——**派发一个自包含、上下文隔离的评审 sub-agent** 来判:它没参与 ② 造设计,新视角去自评偏见,评审的中间推理也不回灌主对话。

**派发**:把 ①② 的三份产物(`directed-report.md` + `requirements.md` + `design.md`)+ 一句改动概述,交给评审 sub-agent——**Read `references/review-agent.md`,以其正文作 Task 的 prompt、把这些路径与概述追加在末尾**,判据全在那(本节不复述)。它产 `review.md` + 三档结论 + 修订清单,回主 agent 一段简短结论。

**agent 选择**(软依赖,同 ①):本会话有隔离 sub-agent 能力(`Task` / 通用 agent)就派,工具面给 `Read`/`Grep`/`Glob` + `Write`(使用约束见 review-agent.md 入口、不在此复述);没有就降级主 agent 自己跑 review-agent.md 那套判据——**降级态主 agent 已带 ② 上下文、没有新视角与隔离的收益,须把判据当外部尺逐条核、显式自查确认偏误**。缺三份中任一:派发态回「RETRY:缺 X」、降级态就地停判,都补齐再派/再判,别拿半成品评。

**三档结论去向**:**通过**(进 ④)/ **有条件通过** / **不通过**;后两档不自动分流——在下面这道 ⏸ 由用户据修订清单定是否回写 ② 改设计(守「不回流 ①」)。

⏸。

## ④ 落地（内置实现链 · gates + 非显性陷阱块）

③ 通过的 design 在本 Skill 内**内联落地**(不委托任何外部技能)。按下列 gates 推进,每个 gate 写的是**要达成什么 / 何时停**——具体怎么做(TDD 红绿步骤、git / worktree 命令)模型自己会,不在此教。

**Gates(依序)**:

1. **拆 `tasks.md`**(落本次目录),把 design 拆成小而独立、各自可测、各自一个 commit 的任务。**⏸ 用户审计划**——写码前最便宜的拦截点,审过再往下。
2. **clean-baseline gate**:进工作区后**先跑一次现有测试确认绿底,再写新码**;红底 → **停下问**用户(继续 / 先查)——否则分不清新坏旧坏。
3. **ISOLATE gate**:别直接在 main / master 上动手。顺序——**先检测是否已隔离 → 优先用原生 worktree 工具**(如 `EnterWorktree`)**→ 没有才 `git worktree` 兜底**;普通 checkout 里建 worktree 前先征得用户同意(除非已声明偏好)。
4. **TDD 铁律**:每个特性 / 修复**先写失败测试 → 亲眼看它按预期失败 → 写最小代码转绿(输出干净)→ 重构保持绿**。修 bug 先用失败测试复现。
5. **per-task 两阶段评审(按序)**:每个任务实现 + commit 后——**① 先 spec 符合性评审**(不缺、**不过度造**额外没要的东西)**② 再代码质量评审**;有发现 → 改 → **复评**,任一阶段未清不进下一任务。**这道评审区别于 ② 自评、也区别于 ③**(③ 评设计,这里评每个已实现任务对设计的符合度)。
6. **continuous execution**:开跑后一路做到底,**别在任务间反复问"要继续吗"**;只在 blocker(缺依赖 / 反复验证失败 / 指令不清 / 计划有洞)时**停下**——计划错**上报用户**、别静默绕过。
7. **FINISH 闭合菜单**:全部任务做完后——**先复验全测试绿 → 检测工作区 / 分支态 → 给固定闭合菜单**(本地合并 / push+PR / 保留原样 / 丢弃;若 detached-HEAD / 外部托管则**去掉"本地合并"**)**→ 精确执行所选一项**。**绝不在收尾抛开放式"接下来干嘛"**。
8. **归档**(链外):分支收口后把本次目录移进 `docs/recon-dev-standalone/_archived/`(目录名已带日期、直接移;产物文档是否 commit 按所在仓惯例一并定)。**⏸ 收尾前确认**。

**非显性陷阱(模型不会主动想到,保留)**:
- **worktree 检测假阳性**:`GIT_DIR != GIT_COMMON` 在 **submodule** 里也成立 → 下结论前用 `git rev-parse --show-superproject-working-tree` 排除(返回路径 = 在 submodule、按普通仓处理)。
- **项目内 `.worktrees/`** 须**先 gitignore 并提交**,再在其下建 worktree(否则 worktree 内容被提交进仓);用 `git check-ignore` 验。harness / 全局托管的目录不必查。
- **收口顺序**:要集成则**先合并 → 在合并结果上验测试 → 再移除 worktree → 最后删分支**(先删分支会因 worktree 仍引用而失败)。
- **清理 provenance 门**:**只清你自己建的** worktree;push+PR 与"保留原样"两档**必须留着** worktree(用户要据 PR 反馈迭代);只在合并 / 丢弃两档清;`git worktree remove` 前先 `cd` 回主仓根、再 `git worktree prune`。
- **丢弃需打字确认**(如用户键入 `discard`)才执行任何破坏性删除,别凭一句随口的"行"就强删分支 / 移 worktree。
- **TDD 反合理化红旗**:「待会儿测 / 已手测过 / 太简单不用测 / 删码可惜 / TDD 教条我务实」一律是**红旗、非豁免**;若生产代码先于失败测试写了,**删掉重来**(别"留作参考"或"改改用")。测**真实行为**而非 mock 调用次数;一个测试一个行为;难测 = 设计味道(解耦 / 注入依赖)、非跳过理由。
- **独立子仓 caveat**:代码在被 `.gitignore` 排除的独立子仓里时,worktree / 提交 / 收口都对**子仓**做(对容器仓做盖不到子仓里的代码)。
- **false-① -premise pause**:实现中若发现 design 建在 ① 的错误前提上——**暂停、surface 给用户**(续做 / 回 ②③),按护栏「事实订正」留痕,别静默绕过、也别自行重判。

**执行方式**:默认主上下文内联跑。若改用 sub-agent 逐任务实现,**须附交接护栏**:每个 sub-agent 给**全任务文本 + 场景上下文**(别让它读计划文件或继承会话史);同一工作区**不并行**跑多个实现 agent;卡住的 agent **别静默原样重派**——换更多上下文 / 更强模型 / 更小任务。

## 守住"薄"（护栏）

- ✅ **报告必须落"发现"(落点现状 / 实答 / 锚点)——这是料、不是套路**。薄的是流程(别教怎么 grep / 追链),不是结论;空有问句没答案才是真薄。
- ✅ **①→②③ 线性、不回流**——各自在出口守住自己产物的质量(① 守报告、② 守需求/设计、③ 综合判),不把把关堆到下游;③ 不打回 ① 补摸底,② 的盲区 ② 自己探。
- ✅ **事实订正(线性不回流的唯一例外)**——directed-report 封存后,任何环节用证据**推翻它写下的一条事实主张**(区别于「① 没写的新面」——那走 design.md 前向),就在**原文条目正下方追加订正块、原文标 `[已订正]`**(范式:`⚠️订正:原文已被推翻 → 真相:… | 证据:file:line 或查询结果`)。**评审 sub-agent 只把推翻经一致性/真值裁决真实记进 review.md、绝不碰 directed-report;核验与落订正由主会话(主 agent)做**。修的是事实记录、非范围回流。**落点**:②③ 阶段证伪在对应 ⏸ 落、动摇设计走现有 ③ 三档;**④ 实现链途中**推翻——主会话落订正块(诚实留痕、不算重排链),**动摇设计就就地暂停、surface 给用户定方向**(续做 / 回 ②③ 改),lite 不自动重判。
- ❌ **别给定向分析加判断规则表 / 检索套路**("碰到契约就 grep 调用方"这类识别-动作对照表)——识别 AI 会,动作溶进验收单 + 坑的 why 即可。手痒加规则,定向分析会从"薄摸底"胀成一套重型分析机器,违背本轨的轻量定位。
- ❌ **别让半成品报告进 ②**——① 的质量门必须真卡(见 ①「⏸ 审报告」),不是拖到 ③ 才发现。
- ❌ **别给 `design.md` 套死内容骨架**——设计内容(架构 / 数据流 / 决策 / 风险)由 ② 对话谈出(本 Skill 自产、无外部技能代劳),另补 ① 锚定层(要改的契约子集 + 补探新发现);但"谈出"不等于"套模板",规定 design.md "必须有哪几节"会把它压成填表——形态按复杂度伸缩,内容由对话生成。
- ❌ **别把判据复述进 `review.md` 模板或 SKILL.md ③**——判据(三维度 + 定向报告验收 + 逐条发现形态 + 三档映射)归 `references/review-agent.md`(单一权威);SKILL.md ③ 只派发/路由、模板只填空。复述 = 第二权威源、会漂移(同 design.md 骨架的坑)。
- ❌ **别把 provenance / 派发理由写进 `references/review-agent.md`**——借鉴出处、为什么隔离好,家是 CHANGELOG;可执行 prompt 只放角色 / 契约 / 判据。
- ❌ **别让评审 sub-agent 替 ① 重做系统钻取**——它只对正在判的主张做定点源码核验;全量摸底是 ① 的活,回流是禁的。
- ❌ **④ 的实现步骤本 Skill 内联拥有**(拆计划→审→隔离工作区→TDD→两阶段评审→收口→归档)——**别把它再委托给外部技能,也别在既定 gates / 停点之外加冗余评审**;实现中推翻 ① 事实走「事实订正」、动摇设计就暂停 surface,不自动重判。④ 写的是 gates(达成什么 / 何时停)与非显性陷阱,**别退化成 TDD / git 的机制教程**(那些模型自己会)。
- ❌ **不读 / 不写任何长期分析文档,不发持久编号**——读了长期文档反而引入漂移与冲突。
